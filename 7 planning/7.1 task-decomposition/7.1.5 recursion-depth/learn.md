# 7.1.5 Recursion Depth — 递归深度与终止条件

## 简单介绍

递归深度（Recursion Depth）是指任务分解过程中从顶层目标到最底层原子步骤的层数。决定"分解到多细才算够"是一个关键的工程权衡——太浅则子任务仍然复杂难以执行，太深则规划开销指数级增长且管理复杂度飙升。

## 基本原理

任务分解是一个递归过程：顶层目标 → 子任务 → 子子任务 → ... → 原子步骤。每递归一次，任务粒度减小的收益递减，而管理成本递增：

- **层数太少**：子任务过于复杂，Agent 在执行时可能仍需要再次规划，实际上相当于把规划负担推迟到了执行阶段
- **层数太多**：分解树过于庞大，部分叶子步骤可能过于琐碎（如"点击按钮1"→"点击按钮2"），增加了不必要的规划开销和 token 消耗

## 背景

在经典 AI 规划中，层次任务网络（HTN）通过领域专家定义的方法（methods）来控制分解深度。每个方法指定了如何将一个任务分解为子任务，以及何时停止分解。在 LLM Agent 中，不再有领域专家定义的方法，因此如何自动确定分解深度成为一个开放问题。

## 之前针对这个问题的做法与结果

1. **固定深度分解**：设定全局最大深度（通常 3 层），超过深度后强制视为原子任务。结果：简单可预测，但对不同复杂度的任务缺乏适应性。
2. **"可一步完成"判断**：让 LLM 判断当前任务是否"足够简单到可以一步完成"。结果：理论上灵活，但 LLM 的判断不稳定，有时高估有时低估任务的复杂性。
3. **自适应深度**：根据任务的 token 数、涉及的实体数、工具调用次数预估合适的深度。结果：比固定深度好，但预估算法的准确度有限。

## 核心矛盾

**完备性 vs 效率**：越深的分解越能保证每个子任务都是可执行的（完备性高），但分解本身消耗的 token 和时间呈指数增长。需要在任务完成率和规划成本之间找到平衡。

## 当前主流优化方向

1. **基于执行反馈的深度控制**：先做浅层分解，如果某个子任务执行失败（或 Agent 在执行时需要额外规划），说明分解不够细，再对该子任务进行递归细化。这是最优的工程方案——"需要多深就分多深"。
2. **任务复杂度预评估**：在执行分解前，先让 LLM 评估任务的复杂度，给出推荐深度。复杂度评估考量因素包括：涉及的工具数量、信息的异构程度、步骤间依赖的密度。
3. **深度与工具能力匹配**：分解深度应与可用工具的能力粒度对齐——如果工具有"搜索+提取+分析"集成的能力，就不需要分解到搜索→提取→分析三个独立步骤。
4. **预算约束的深度决策**：在总 token 预算或总时间预算约束下，计算最优分解深度（深度越浅消耗越少但失败风险更高，深度越深消耗越多但成功概率更高）。

## 实现的最大挑战

```python
class AdaptiveDecomposer:
    def __init__(self, llm, max_depth=5, min_depth=2):
        self.llm = llm
        self.max_depth = max_depth
        self.min_depth = min_depth
    
    async def decompose_with_depth(self, goal: str, context: dict) -> list[TaskNode]:
        # Phase 1: 预评估任务复杂度
        complexity = await self._assess_complexity(goal)
        
        # Phase 2: 确定目标深度
        target_depth = self._compute_target_depth(complexity)
        
        # Phase 3: 带深度约束的分解
        return await self._decompose(goal, context, depth=0, target_depth=target_depth)
    
    def _compute_target_depth(self, complexity: dict) -> int:
        """基于复杂度评估计算目标深度"""
        base = self.min_depth
        
        # 工具数量越多，需要越深
        if complexity["tool_count"] > 5:
            base += 1
        # 信息源越异构，需要越深
        if complexity["info_heterogeneity"] > 0.7:
            base += 1
        # 依赖密度越高，需要越深
        if complexity["dependency_density"] > 0.5:
            base += 1
        
        return min(base, self.max_depth)
    
    async def _assess_complexity(self, goal: str) -> dict:
        prompt = f"""
        评估以下任务的复杂度，返回 0-1 范围的分数:
        {{
            "tool_count": float,         // 完成此任务可能需要调用几种不同工具
            "info_heterogeneity": float, // 信息来源的多样性
            "dependency_density": float, // 子步骤间依赖的紧密程度
            "domain_rarity": float,      // 领域的罕见程度（越罕见越需要深入分解）
            "estimated_steps": int       // 粗略估计的总步骤数
        }}
        
        任务: {goal}
        """
        ...
```

最大挑战在于**深度-成功率的量化关系未知**：当前无法精确知道多一层分解能提升多少成功率，更多依赖经验规则。不同任务类型的最优深度差异极大——"写一封邮件"可能只需 1 层，"做一个市场调研"可能需要 3-4 层。

## 能力边界和结果边界

- **能力**：可以动态调整分解深度，适应 80% 以上的常见 Agent 任务
- **边界**：对于超复杂任务（30+ 步骤），分解树的广度比深度更难管理，深度适当但广度可能爆炸
- **结果**：自适应深度相比固定 3 层，能减少 20-30% 的不必要分解，同时保持相同的任务成功率

## 核心优势

自适应深度控制将有限的上下文窗口和 token 预算分配到"最需要细化"的部分——简单任务的步骤少、分解浅，复杂任务的步骤多、分解深。这本质上是一种**注意力分配机制**。

## 工程优化方向

- 建立"深度经验表"：记录不同任务类型的最优深度，随着使用经验积累逐步优化
- 补偿机制：如果总 token 预算不足，优先保证前几层分解完整，最后一层用 ReAct 模式"边想边做"代替预分解
- 可视化深度分布：调试时展示每棵子树的实际深度，帮助识别"过度分解"和"分解不足"的模式

## 适合场景

- 任务复杂度波动大的通用 Agent
- 需要在 token 预算和完成率之间做权衡的生产系统
- 作为分层规划器中"何时停止分解"的决策模块
