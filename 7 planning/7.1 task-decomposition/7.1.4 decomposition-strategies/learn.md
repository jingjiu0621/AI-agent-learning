# 7.1.4 Decomposition Strategies — 分解策略（基于功能 / 数据流 / 依赖）

## 简单介绍

任务分解没有"一招鲜"的通用方法——不同性质的任务需要不同的分解策略。选择正确的分解策略是 Agent 规划有效性的关键决定因素。本节覆盖三种主流的分解范式：基于功能的分解、基于数据流的分解和基于依赖的分解。

## 基本原理

三种分解策略对应看待任务的三种不同视角：

| 策略 | 视角 | 分解依据 | 典型产出 |
|------|------|---------|---------|
| 功能分解 | "需要做什么" | 子能力 / 子职责 | 模块化子任务 |
| 数据流分解 | "数据如何变换" | 数据的输入-处理-输出链路 | 流水线步骤 |
| 依赖分解 | "哪些先做哪些后做" | 前置条件 / 资源约束 | 拓扑序 |

## 背景

传统软件工程中，功能分解（Function Decomposition）是结构化编程时代的产物——将系统按功能划分为模块。数据流分解（Dataflow Decomposition）源于 ETL 和流处理系统。依赖分解（Dependency Decomposition）则来自项目管理（PERT 图、关键路径法）。Agent 的任务分解必须综合这三种视角，因为 Agent 面临的任务往往同时涉及功能模块、数据流转和时序约束。

## 之前针对这个问题的做法与结果

1. **纯功能分解**：按"做什么"划分（搜索、分析、生成）。结果：直观易懂，但忽略了数据依赖可能导致步骤顺序错误。
2. **纯数据流分解**：按"数据如何流经系统"划分。结果：适合 ETL 类任务，但面对非确定性任务（如"头脑风暴创意方案"）时难以定义明确的数据变换。
3. **混合分解**：先用功能分解确定子任务集合，再用数据流/依赖分析确定执行顺序。结果：当前最实用方案，在两阶段过程中可能增加一次 LLM 调用。

## 核心矛盾

**通用性 vs 最优性**：功能分解通用但可能低效（错过了并行机会），数据流分解高效但需要更精确的领域知识，依赖分解正确但可能过度约束。没有一种策略在所有场景下最优。

## 当前主流优化方向

1. **策略选择器**：先用 LLM 判断任务类型（组合型/流程型/探索型），然后根据类型选择最优分解策略。
2. **多策略融合**：对同一任务同时生成功能分解和数据流分解，然后合并/去重，取各自优势。
3. **策略切换**：分解过程中根据已发现的子任务性质，动态切换分解策略（如顶层用功能分解，叶子层用数据流分解）。
4. **元分解**：让 LLM 先分析任务特征，生成分解策略方案，然后再执行分解——即"先规划怎么分解，再分解"。

## 实现的最大挑战

```python
class DecompositionStrategySelector:
    def select_strategy(self, task_description: str) -> str:
        """根据任务特征选择最优分解策略"""
        features = self._analyze_task(task_description)
        
        if features["has_clear_data_flow"]:
            # 数据从源头到目标的变换路径清晰
            return "dataflow"
        elif features["has_independent_sub_goals"]:
            # 任务可拆分为多个独立的能力模块
            return "functional"
        elif features["has_strict_prerequisites"]:
            # 步骤之间有严格的先后约束
            return "dependency"
        else:
            # 不确定时，功能分解是最安全的基线
            return "functional"
    
    def _analyze_task(self, task: str) -> dict:
        prompt = f"""
        分析以下任务的特性，返回 JSON:
        {{
            "has_clear_data_flow": bool,    // 是否存在"输入→处理→输出"的明确链路
            "has_independent_sub_goals": bool, // 是否能拆分为多个独立的能力模块
            "has_strict_prerequisites": bool,  // 是否存在严格的先后约束
            "task_type": "compositional" | "procedural" | "exploratory",
            "recommended_strategy": "functional" | "dataflow" | "dependency" | "hybrid"
        }}
        
        任务: {task}
        """

```

最大挑战是**任务特征分析的准确性**：LLM 可能错误分类任务类型，导致选择不合适的分解策略。缓解方案是"试错+回退"——如果一种策略生成的子步骤在执行时频繁失败，自动切换到另一种策略重新分解。

## 能力边界和结果边界

- **能力**：能覆盖 80% 以上的常见 Agent 任务类型，显著提升分解质量
- **边界**：高度混合型的任务（既需要功能划分又有复杂数据流）需要人工设计分解方案
- **结果**：正确选择策略可让任务成功率提升 20-40%，执行时间减少 30-50%

## 核心优势

掌握多种分解策略并懂得选择，是高级 Agent 规划系统与简单提示词工程之间的分水岭。单一策略的 Agent 在面对"陌生任务类型"时很容易产生不合理分解，而多策略 Agent 有更大的鲁棒性。

## 工程优化方向

- 对高频任务类型自动记录"最优策略经验"，形成策略选择缓存
- 引入"分解质量评分"反馈回路，让 Agent 自主学习哪些策略适合自己的任务领域
- 实现"策略热切换"：执行过程中发现分解不合理，立即切换策略重新分解未完成部分

## 适合场景

- 任务类型多样、无法预定义的通用 Agent
- 需要处理组合型任务（多模块协作）和流程型任务（多步骤流水线）混合的场景
- 作为高层规划器的核心组件，为下层具体策略选择提供决策依据
