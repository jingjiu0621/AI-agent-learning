# 3.3.6 regression-test — Prompt 回归测试套件

## 简单介绍

Prompt 回归测试是确保 Prompt 优化不会"治好一个症状、引发两个新病"的关键机制。当你在改进 Prompt 时，修改可能会破坏之前已经正确处理的场景——回归测试套件捕捉这类退化。它是 Prompt 优化工程的"安全网"，让你的优化过程可以放心地大胆尝试。

## 基本原理

回归测试的核心是一个**经过标注的测试集** + **自动化评估管线**：

```python
class RegressionTestSuite:
    def __init__(self):
        self.test_cases = []  # (input, expected_output, evaluator)

    def add_case(self, input_text, expected, evaluator="exact_match"):
        self.test_cases.append((input_text, expected, evaluator))

    def run(self, prompt) -> TestReport:
        results = []
        for input_text, expected, evaluator in self.test_cases:
            output = llm_call(prompt, input_text)
            passed = self._evaluate(output, expected, evaluator)
            results.append({
                "input": input_text,
                "expected": expected,
                "actual": output,
                "passed": passed
            })
        report = TestReport(results)
        report.print_summary()
        return report
```

每次 Prompt 改动后运行回归套件，对比改动前后的通过率，若通过率下降则说明引入了回归问题。

## 背景

回归测试在传统软件开发中是标准实践——没人会在不改单元测试的情况下提交代码。但在 Prompt 工程中，回归测试的普及程度远低于传统开发。很多团队直到遭遇了惨痛教训（上线后才发现之前能正确处理的场景被破坏了）才开始重视。

## 之前针对这个问题的做法与结果

在回归测试普及前，Prompt 优化通常是"在少数测试样例上人工验证"。结果是：
- 优化针对样例集过拟合，对其他输入表现不佳
- 之前能正确处理的边缘案例在新版本中悄然失效
- 团队无法快速回滚到可靠的已知版本，因为不知道上次"全部通过"是哪个版本

引入回归测试后，团队能在大规模优化前快速筛选出"至少没有退化"的候选版本。

## 核心矛盾

回归测试的核心矛盾是**测试覆盖率与维护成本的平衡**。覆盖率高意味着更安全，但：
- 每个测试案例需要人工标注"正确输出"，成本高
- 高频运行回归测试消耗大量 API 调用费用
- Prompt 行为合理变化时需要更新测试标注（例如输出格式微调），维护工作量大
- Agent 系统的输出具有非确定性，同一输入可能产生多个"可接受"的输出

## 当前主流优化方向

- **自动生成测试用例**：用 LLM 从真实用户日志中自动提取测试案例并生成期望输出
- **模糊匹配评估**：用 LLM-as-judge 替代精确匹配，允许同义但不同表述的输出通过测试
- **分级回归测试**：全量测试集（慢而全）+ 烟雾测试集（快而精），日常开发只跑烟雾测试
- **差异对比报告**：回归测试不仅报告"通过/失败"，还报告输出内容的语义差异

## 实现的最大挑战

- **Oracle 问题**：Agent 任务的"正确输出"往往不止一种。定义一个覆盖所有可接受输出的评估标准本身就是一个难题
- **输出方差**：同一 Prompt 对同一输入连续运行两次可能产生不同输出，导致测试结果不稳定（flaky tests）
- **测试漂移**：应用的业务逻辑或需求变化时，原有的回归测试可能需要更新，但人往往忘记更新
- **工具调用测试**：Agent 的回归测试不仅需要验证最终输出，还需验证工具调用的顺序和参数——测试难度远高于纯文本任务

## 能力边界与结果边界

- 回归测试能捕捉"可定义"的退化，但不能捕捉"感觉不对劲"的退化（如语气变化、风格漂移）
- 测试覆盖率和实际效果保障之间不存在线性关系——覆盖 80% 的场景可能需要 80% 的成本
- 回归测试是必要不充分条件——通过了回归测试不代表 Prompt 在真实场景中表现良好
- 对于高度开放的生成任务（如创意写作），回归测试的价值有限

## 与其他技术的区别

- vs A/B 测试：A/B 测试比较两个 Prompt 版本的"优越性"，回归测试确保新版本"不更差"
- vs 单元测试：传统单元测试验证确定性的输入输出关系，Prompt 回归测试处理非确定性输出
- vs 评估集：评估集衡量 Prompt 在某个维度上的绝对表现，回归测试衡量版本间的相对退化

## 核心优势

- **防止倒退**：优化时可以大胆尝试，因为知道安全网会捕捉破坏性改动
- **快速反馈**：几秒钟到几分钟内就能知道改动是否破坏了已有功能
- **团队协作基础**：不同成员可以并行优化不同部分，相互信任基础建立在回归测试上
- **文档化行为**：回归测试集本身就是 Prompt 行为的可执行文档

## 工程优化方向

- 将回归测试集成到 CI/CD 流水线中，每次 Prompt 变更自动触发
- 建立测试案例贡献机制：生产环境中发现的失败案例自动加入回归测试集
- 使用语义相似度评估替代精确匹配，减少 flaky test
- 对测试案例进行重要性分级：核心流程（必须通过）、重要功能（尽量通过）、边缘场景（参考信息）
- 定期审查回归测试集：移除过时案例，更新变化的需求标注

## 适合场景的判断标准

- Prompt 被用于生产环境，且错误有实际成本
- 团队规模大于 1 人，多人维护同一组 Prompt
- Prompt 的变更频率较高（每周至少一次）
- 有稳定的评估标准和自动化评估能力
