# 3.3.3 ab-testing — A/B 测试：指标设计、对照组、统计分析

## 简单介绍

A/B 测试是科学验证 Prompt 改动的核心方法。当你要判断"新版 Prompt 是否比旧版更好"时，A/B 测试提供了从直觉驱动转向数据驱动的决策框架。在 AI Agent 场景下，A/B 测试不仅关注准确率，还需考虑延迟、成本、格式稳定性等多维指标。

## 基本原理

A/B 测试的核心是一个受控对比实验：
1. 将输入样本随机分配到控制组（A：旧 Prompt）和实验组（B：新 Prompt）
2. 在相同的评估条件下运行两组
3. 收集预定义指标
4. 用统计方法判断两组差异是否显著

```python
import random
import numpy as np
from scipy import stats

def ab_test(samples, prompt_a, prompt_b, eval_fn, alpha=0.05):
    random.shuffle(samples)
    split = len(samples) // 2
    group_a = [eval_fn(s, prompt_a) for s in samples[:split]]
    group_b = [eval_fn(s, prompt_b) for s in samples[split:]]

    t_stat, p_value = stats.ttest_ind(group_a, group_b)
    significant = p_value < alpha
    effect_size = np.mean(group_b) - np.mean(group_a)

    return {
        "significant": significant,
        "p_value": p_value,
        "effect_size": effect_size,
        "mean_a": np.mean(group_a),
        "mean_b": np.mean(group_b)
    }
```

## 背景

A/B 测试在互联网产品领域已经有数十年应用历史。但在 Prompt 工程中，传统的 A/B 测试框架直接套用会面临独特挑战——LLM 输出具有高方差（同一 Prompt 多次运行结果不同）、高成本（每次评估调用 API）和高维度（需要评估的方面很多）。

## 之前针对这个问题的做法与结果

早期的 Prompt 优化很少做严格的对比测试。典型做法是"跑几次感觉一下"或"拿几个例子比较输出"。结果是：大量真实退化的改动被误认为"改进"上线，因为单次运行的随机波动被误判为真实效果。引入 A/B 测试后，团队发现大约 30-50% 的"优化"实际上在统计上并不显著。

## 核心矛盾

A/B 测试的核心矛盾是**统计置信度与成本约束的对立**。要达到足够的统计功效（检测出真实差异），需要足够的样本量。但对于 LLM 应用，每个样本的评估成本很高（API 费用 + 延迟），样本量往往受限于预算。需要用合理的实验设计在有限样本下获取最大信息量。

## 当前主流优化方向

- **配对测试**：同一输入用新旧 Prompt 各跑一次，消除输入方差
- **序贯测试**：持续监测结果，提前停止已能做出判断的实验，节省成本
- **多臂老虎机**：动态分配更多流量给表现更好的 Prompt，减少实验成本
- **贝叶斯 A/B 测试**：提供"B 比 A 好的概率"而非二元的显著/不显著

## 实现的最大挑战

- **指标噪声**：LLM 输出的高方差意味着需要更多样本来达到统计显著
- **指标多样性**：Agent 系统的成功是多维的（准确率 + 成本 + 延迟 + 安全性），单一指标无法反映全貌
- **评估自动化瓶颈**：自动评估（LLM-as-judge）有偏差，人工评估无法规模化
- **顺序效应**：先跑 A 再跑 B 可能引入时间相关的混淆变量（模型更新、负载变化等）

## 能力边界与结果边界

- A/B 测试告诉你"哪个更好"，但不告诉你"为什么更好"
- 能检测出 5% 以上的效果差异（前提是有足够的样本量）
- 不能替代探索性分析——只回答验证性问题，不回答发现性问题
- 单个 A/B 测试的结果不能直接泛化到其他场景

## 与其他技术的区别

- vs 消融实验：消融实验旨在理解每个组件的独立贡献，A/B 测试旨在做二选一的决策
- vs 多臂老虎机：老虎机在实验和利用之间平衡，A/B 测试只做纯实验
- vs 交叉验证：交叉验证用于评估模型的泛化能力，A/B 测试用于比较两个 Prompt 的差异

## 核心优势

- 提供科学的决策依据，减少"拍脑袋"优化
- 量化改动效果，帮助团队确定投入产出比
- 建立归因机制，知道每次上线改动的实际影响
- 积累组织记忆：A/B 测试结果存档后可作为后续优化的参考基线

## 工程优化方向

- 建立 Prompt 实验平台：统一管理对照组、实验组、指标和结果
- 自动化 A/B 测试流程：代码触发实验、自动收集指标、生成报告
- 构建多维指标看板：一次实验同时监控准确率、成本、延迟和异常率
- 设计"亏损保护"机制：当实验组表现显著差于控制组时自动回退

## 适合场景的判断标准

- 每次 Prompt 改动都需要上线验证时
- 优化方向不明确、需要数据辅助决策时
- 团队内部对"这个改动是否有效"存在分歧时
- 需要向利益相关者量化证明优化效果时
