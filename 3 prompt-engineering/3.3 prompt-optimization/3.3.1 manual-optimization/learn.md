# 3.3.1 manual-optimization — 人工优化方法论与最佳实践

## 简单介绍

人工优化是 Prompt 优化最基础也最直接的方法。它依赖工程师的直觉、经验和系统化的调试流程来逐轮改进 Prompt 质量。尽管自动化工具日益成熟，人工优化依然是理解 Prompt 行为、发现优化方向的必要技能。

## 基本原理

人工优化的本质是"假设-实验-验证"循环：
1. 观察当前 Prompt 在测试样本上的输出
2. 形成关于"为什么表现不好"的假设
3. 修改 Prompt 中特定的措辞、结构或指令
4. 在测试集上验证效果
5. 决定保留、回退或继续调整

## 背景

在 LLM 应用早期（GPT-3 时代），Prompt 优化完全依赖人工。人们发现同样的任务，不同措辞可能导致天差地别的结果。早期实践者积累了大量"Prompt 技巧"——少样本示例顺序、角色设定语气、指令措辞的具体程度等——这些技巧构成了人工优化的知识基础。

## 之前针对这个问题的做法与结果

最初的做法是"碰运气式优化"：改了 Prompt 后凭感觉判断好坏，没有系统化的评估。结果就是优化不可复制、效果不稳定。后来逐步演进出结构化的方法：
- **错误分析驱动**：先分析失败案例，再针对性修改
- **消融实验**：逐一移除或修改 Prompt 中的组件，衡量各自贡献
- **分步验证**：将复杂 Prompt 拆解为子模块独立测试

## 核心矛盾

人工优化的根本矛盾在于"直觉的不可靠性 vs. 直觉的必要性"。人类的语言直觉常常产生误导——"感觉更清晰"的 Prompt 实际效果可能更差。但另一方面，自动算法缺乏对任务语义的理解，无法进行概念层次的创新，只有人能提出"换个角度描述任务"这类根本性改进。

## 当前主流优化方向

- **结构性调试**：按照 Prompt 组件（角色、指令、示例、输出格式、约束）逐层排查
- **对抗性测试**：主动构造"可能让 Prompt 失败"的边界案例
- **分而治之策略**：将复杂 Prompt 拆解后再针对子问题优化

## 实现的最大挑战

- **评估疲劳**：人工判断输出质量需要消耗大量注意力，评估者容易疲劳且标准漂移
- **因果混淆**：多个改动同时引入时，无法确定哪个改动导致了效果变化
- **上下文遗忘**：修改几轮后忘记了最初版本的行为表现，导致改进方向偏离

## 能力边界与结果边界

- **上限高但速度慢**：顶级工程师的 Prompt 优化质量很高，但耗时巨大
- **依赖个人经验**：优化效果与工程师对 LLM 行为模式的理解深度正相关
- **不可规模化**：一个人只能同时维护和优化有限的 Prompt

## 与其他技术的区别

人工优化是**所有自动优化方法的上限参考**。自动优化算法搜索到的 Prompt 最终需要人工判断是否合理。自动优化擅长参数级精调，人工优化擅长结构级创新。

## 核心优势

- 能够发现根本性的设计缺陷（自动优化通常只在已有框架内调参）
- 积累的调试经验可以转化为更具泛化性的工程知识
- 不需要标注数据或计算资源，适合快速原型验证

## 工程优化方向

- 建立 Prompt 调试日志模板，记录每次改动的假设、变更和观察结果
- 构建边界案例库，避免反复陷入同样的失败模式
- 与 A/B 测试和回归测试结合，用数据验证直觉，而不是用直觉替代数据

## 适合场景的判断标准

- 项目早期、评估体系尚未建立时
- Prompt 整体结构不合理、需要重新设计时
- 只有少量示例且无法自动标注时
- 作为自动优化的方向指引（先用人工找到方向，再用自动精调参数）
