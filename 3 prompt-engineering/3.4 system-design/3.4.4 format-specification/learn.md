# 格式规范：输出结构与格式要求

## 简单介绍

格式规范（Format Specification）是 System Prompt 中定义 Agent 输出应遵循的结构和格式的模式。对 AI Agent 开发而言，格式规范远比"输出美观"重要——它是 Agent 与下游系统（代码解析器、工具调用框架、前端界面）之间的接口契约。一个精心设计的格式规范确保 Agent 的输出可以被可靠地解析、验证和使用，是整个 Agent 系统稳定运行的基石。常见的格式规范包括 JSON 结构、XML 标签、Markdown 模板、分隔符约束和枚举值限定等。

## 基本原理

格式规范的工作原理基于 LLM 的**模式匹配和延续能力**：

1. **模板跟随**：当 System Prompt 给出了一个输出模板（如 JSON Schema），LLM 会倾向按该模板的结构化路径生成内容。提供模板相当于在生成空间中划定了"合法区域"。
2. **前缀约束**：在 Prompt 中预先给出输出的开头几行，模型会沿着该格式继续生成。利用"自回归生成"的特性，开头的格式设定会强制后续内容遵循相同模式。
3. **判别式引导**：明确指定可接受的输出格式列表（"只输出以下格式之一：text、json、xml"），限制模型的输出空间。
4. **反例排除**：通过反面示例（"不要输出这种格式"）间接定义格式边界，帮助模型理解"什么不是有效输出"。

格式规范需要在**严格性**和**鲁棒性**之间做平衡：格式要求越严格，解析越可靠，但 Agent 在边界情况下的输出失败率也越高。

## 背景

早期 LLM 应用（2022-2023 年初）主要面向对话场景，输出格式以自然语言为主，格式要求仅体现在"用中文回答"、"分点列出"这类简单约束上。

随着 Function Calling / Tool Use 协议的普及（2023 年中），Agent 的输出需要被下游代码直接解析。JSON 格式成为事实标准。开发者发现 LLM 的 JSON 输出经常出现格式错误（缺少括号、多余的逗号、非标准值），催生了"强迫 JSON 输出"的各种工程技巧。随后，OpenAI 推出 JSON mode（强制输出有效 JSON），Anthropic 推出 Tool Use 的原生格式支持，但自定义格式规范的需求仍然广泛存在——因为 JSON 模式只解决了格式合法性，没有解决内容结构性。

## 之前针对这个问题的做法与结果

### 做法 1：纯自然语言描述格式
"请用 JSON 格式输出，包含名称和年龄字段"。
- **结果**：可靠性差。模型理解的 JSON 和开发者期望的 JSON 常有差异。字段名不一致、嵌套层级错误频发。解析失败率达 20%~40%。

### 做法 2：提供 JSON Schema
在 Prompt 中嵌入完整的 JSON Schema 或 TypeScript 类型定义。
- **结果**：显著提升格式一致性，但仍偶有格式破坏。长 Schema 会大量消耗 token，且 Schema 越复杂模型越难遵循。

### 做法 3：使用开源结构化生成框架
Guidance、Outlines、LMQL 等通过约束解码（对 Token 生成施加合法性限制）的方式强制结构输出。
- **结果**：格式遵循率接近 100%，但增加了系统复杂度和延迟。且只适用于代码集成场景，对 API 调用场景（只能控制 Prompt 文本）不适用。

## 核心矛盾

**格式的严格性与 Agent 的自然语言灵活性之间的矛盾**：严格输出格式（如 JSON）容易被下游系统消费，但限制了 Agent 表达复杂信息的能力。宽松格式（如 Markdown 自由文本）保留了表达力，但增加了下游解析的难度和出错率。纯结构化输出（如只允许输出 JSON）在某些场景下完全可行，但在需要混合文本解释 + 结构化数据的场景中成为一个棘手问题——Agent 需要在"讲人话"和"讲机器话"之间切换。

## 当前主流优化方向

1. **多模态格式输出**：设计分层的输出结构，例如"自然语言解释"作为 Markdown 文本、"结构化数据"嵌入在 JSON code block 中，两者共存于同一输出：

```markdown
## 分析结果
根据用户数据，推荐以下配置：

```json
{
  "recommendation": "upgrade",
  "confidence": 0.87,
  "options": ["plan_a", "plan_b"]
}
```
```

2. **格式验证与重试机制**：Agent 输出后先通过格式校验器（如 JSON parser、Schema validator），校验失败则提示模型"格式错误，请重新输出"的自修复流程
3. **动态格式选择**：根据任务类型自动选择合适的输出格式模板，而非全系统使用统一格式
4. **格式容忍解析**：构建容错的输出解析器，能从"格式不对但意图明确"的输出中提取有效信息
5. **原生结构化支持的利用**：优先使用模型 API 内置的结构化输出功能（如 JSON mode、Tool Use 定义），这些在模型层进行了格式优化，可靠性远高于通过 System Prompt 文本约束

## 实现的最大挑战

1. **嵌套结构的不稳定性**：层级越多（嵌套 JSON 超过 3 层），LLM 的格式错误率指数级上升。深层嵌套的闭括号匹配是常见问题。
2. **输出长度限制下的格式完整性**：当输出接近 Token 上限时，模型可能截断在 JSON 的中间位置，导致格式不完整。
3. **中文字段与格式兼容性**：中文字段名在 JSON 中需要引号包裹，某些解析器对 Unicode 处理不完善。中文内容使格式长度预测更困难。
4. **格式约束与内容质量之间的平衡**：过度关注格式可能导致 Agent 在内容质量上"偷懒"——它可能生成了完美的 JSON，但里面的值是空洞的。
5. **多格式切换的混淆**：当 Agent 需要输出多种格式（分析用 Markdown，数据用 JSON，操作指令用 XML）时，模型可能在格式之间混淆。

## 能力边界与结果边界

**能做的**：
- 将 Agent 输出的机器可解析率从 60%~70% 提升到 95% 以上（配合格式验证重试可达 99%+）
- 使下游系统无需编写复杂 NLP 解析器，直接用标准解析器消费 Agent 输出
- 为 Agent 输出提供明确的"通过 / 不通过"判定标准，便于自动化测试

**不能做的**：
- 不能保证 100% 的格式正确率——LLM 是概率模型，总存在格式异常的可能
- 不能替代内容质量的保障——格式正确不意味着内容正确
- 不能保证跨模型版本的一致性——同一格式规范在不同模型版本上的遵循程度可能不同
- 不能完全消除"格式正确但语义错误"的情况——字段类型对但值不对的 JSON 不会被 Schema 校验捕获

## 与其他技术的区别

| 对比 | 格式规范 | 约束列举 | 指令层级 |
|------|---------|---------|---------|
| 关注点 | 输出结构 | 行为边界 | 指令优先级 |
| 主要问题 | 机器可解析性 | 安全合规 | 冲突裁决 |
| 失败表现 | 解析异常 | 违规输出 | 行为混乱 |
| 补救手段 | 格式验证+重试 | 内容过滤 | 层级刷新 |

## 核心优势

1. **系统集成必备**——Agent 不只是一个对话界面，当它需要被系统"消费"时，格式规范是必要基础
2. **可测试性强**——格式合规性可以自动化验证，适合集成到 CI/CD 流水线
3. **降低下游复杂度**——标准化输出意味着下游系统不需要为每个 Agent 定制解析逻辑
4. **渐进式增强**——可以从简单的格式描述开始，逐步演进到 Schema 验证 + 自修复的完整体系

## 工程优化方向

1. **使用示例 + Schema 双重约束**：

```python
system_prompt = """
输出格式要求：
请始终使用以下 JSON 结构输出：
{
  "action": "<动作类型>",
  "parameters": { "<参数名>": "<参数值>" },
  "reasoning": "<简要推理过程>"
}

有效输出示例：
{"action": "search", "parameters": {"query": "天气"}, "reasoning": "用户询问天气信息"}

无效输出示例（不要这样）：
action: search
query: 天气
"""
```

2. **输出后校验流水线**：Agent 输出 → 格式校验器 → 通过则放行 → 失败则重试（最多 3 次）→ 超过重试次数则返回降级响应
3. **格式模板管理**：为不同任务维护独立的格式模板文件，通过动态 Prompt 组装按需注入，避免 System Prompt 过度膨胀
4. **容错解析器**：编写能处理常见格式错误（如多余的逗号、引号缺失、字段名不一致）的解析器，降低对模型完美格式输出的依赖
5. **格式版本控制**：输出格式随系统版本演进，需要在 System Prompt 中标记格式版本号，以便在不同版本间做兼容性处理

## 适合场景的判断标准

格式规范最适合以下场景：
- **Agent 输出需要驱动下游系统**：API 调用、数据库操作、UI 渲染等
- **多 Agent 之间通信**：Agent A 的输出是 Agent B 的输入，需要严格的接口契约
- **自动化测试需求**：Agent 的输出需要被自动化测试工具验证
- **日志和审计**：结构化输出便于存储、检索和分析

不适合的场景：
- **纯聊天机器人**：只有人类读者，格式约束得不偿失
- **探索性对话**：Agent 需要最大灵活性来适应用户不可预测的对话方向
- **快速原型验证**：早期阶段不需要关注格式规范，快速验证价值假设更重要
