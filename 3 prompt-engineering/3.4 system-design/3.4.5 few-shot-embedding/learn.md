# 少样本示例嵌入策略

## 简单介绍

少样本示例嵌入（Few-Shot Embedding）是指在 System Prompt 中嵌入若干输入-输出示例（demonstrations），以引导 LLM Agent 按照预期模式处理任务的技术。与 Zero-Shot（无示例）不同，Few-Shot 通过提供具体案例隐式传递了任务的"处理方式"，包括推理路径、输出格式、边缘情况处理策略等。在 AI Agent 开发中，精心选择的示例往往比冗长的指令描述更有效——因为"做给你看"比"告诉你该怎么做"更符合 LLM 的类比学习方式。

## 基本原理

少样本示例的工作原理建立在 LLM 的**上下文学习**（In-Context Learning, ICL）机制之上：

1. **模式提取**：模型从提供的示例中自动提取输入-输出映射模式，将其作为隐含的"规则"。即使不明确写出规则，模型也能通过示例推导出应该怎么做。
2. **分布偏移**：示例改变了模型输出的条件概率分布——高质量示例将概率质量从"不希望的输出"向"期望的输出"方向移动。
3. **注意力引导**：示例在前缀中的位置影响注意力分布，靠近输出的示例（即示例排列顺序）对生成的影响更大。
4. **格式锚定**：示例是输出格式的"活模板"，模型会模仿示例中的格式特征（标点、分段、字段顺序）。

示例在 System Prompt 中的位置选择是一个关键工程决策——放在 System Prompt 末尾（紧贴用户问题）通常效果最佳，因为与当前输入的"距离"最近。

## 背景

Few-Shot Learning 源自 GPT-3 论文（Brown et al., 2020）的核心发现——大语言模型仅通过观察少数示例就能执行新任务，无需参数更新。这一发现颠覆了"每个任务都需要微调"的传统 NLP 范式。

在 AI Agent 开发中，Few-Shot 的应用从最初简单的分类/问答任务，扩展到复杂的多步推理、工具调用链和对话管理。一个典型的生产级 Agent 的 System Prompt 中，示例部分可能占据 30%~60% 的 token 预算——这是一笔巨大的开销，必须有策略地管理。

## 之前针对这个问题的做法与结果

### 做法 1：大量示例
认为示例越多越好，一次性放入 20~50 个示例覆盖所有可能性。
- **结果**：token 消耗巨大，超出上下文窗口限制。更重要的是，实验表明示例超过某个临界点（通常 5~8 个）后边际效益急剧下降。大量示例还容易引入不一致的噪音示例，反而降低模型表现。

### 做法 2：随机选择示例
从已有数据中随机挑选示例嵌入 Prompt。
- **结果**：效果不稳定。选到"好"示例时表现好，选到"差"或"不相关"示例时表现差。示例的质量和相关性比数量更重要。

### 做法 3：静态固定示例
一次选定一组示例后长期使用，从不更新。
- **结果**：示例逐渐"过时"——随着系统演进，任务的边界和用户需求发生变化，固定的示例不再反映当前的最佳实践。

## 核心矛盾

**示例的代表性与多样性的矛盾**：示例越具有代表性（精准反映最典型的场景），模型在常见情况下的表现越好，但对边缘/非典型场景的泛化能力越弱。示例越多样化（覆盖更广的场景），模型在边缘场景的表现越好，但每个场景的示例数量有限，可能每个都"学不透"。本质上是"深 vs 广"的权衡——有限的 token 预算下，追求深度（少而精）还是广度（多而杂）。

## 当前主流优化方向

1. **动态示例检索**（Dynamic Few-Shot）：不固定示例，而是每次根据用户输入从示例库中检索最相似的 3~5 个示例。使用 Embedding 相似度匹配，确保每个请求获得最相关的示例。
2. **示例质量评分体系**：建立示例效果评估标准（任务成功率、格式遵循率、用户满意度），淘汰低质量示例，持续迭代高质量示例。
3. **分层示例策略**：将示例分为"通用示例"（放在 System Prompt 中，覆盖大多数场景）和"场景示例"（动态注入，应对特定场景），实现示例的混合管理。
4. **示例增强**（Example Augmentation）：通过自动化方式生成示例变体（使用 LLM 或规则），扩展示例覆盖范围而不增加人工标注成本。
5. **Chain-of-Thought 示例**：在示例中不仅展示最终输出，还展示推理过程，引导模型的"思考方式"而非仅关注"输出样式"。

## 实现的最大挑战

1. **Token 预算争夺**：示例是 token 消耗大户。在上下文窗口有限的情况下，示例与指令、工具描述之间需要精打细算地分配 token 预算。
2. **示例偏差放大**：如果示例中存在微妙的偏差（如偏向某些用户群体、忽略某些场景），模型会放大这些偏差至更广的分布。
3. **示例的顺序敏感性**：LLM 对示例的顺序高度敏感——同样的示例换一种排列效果可能完全不同。找到最优排列需要大量实验。
4. **跨场景迁移困难**：在一个任务上精心设计的示例组合，迁移到类似但不完全相同的任务时效果骤降。
5. **示例维护成本**：高质量示例需要人工精心编写和审核，随着系统功能迭代，示例需要持续更新，维护成本不低。

## 能力边界与结果边界

**能做的**：
- 显著提升 Agent 在具体任务模式上的表现（相比 Zero-Shot 通常提升 15%~40% 的任务成功率）
- 隐式传递复杂的行为规则——通过"模仿"而非"理解规则"来学习，这对某些模型的遵循效果更好
- 提供输出格式的"活模板"——比格式规范更直观、更少歧义
- 覆盖指令难以描述的边缘情况——"遇到这种情况就像示例中那样处理"比用文字描述场景更高效

**不能做的**：
- 不能教会模型全新能力——示例只能"激活"模型已有的能力，不能"创造"新能力
- 不能保证零错误——示例不能消除模型的概率性错误，尤其是在非示例覆盖的场景中
- 不能替代清晰的指令——如果示例意图不明确（"为什么这个例子要这么做？"），模型可能学到错误的模式
- 不能解决需要逻辑推理而非模式匹配的问题——有些任务需要真正的推理，仅靠示例不足以正确解决

## 与其他技术的区别

| 对比 | 少样本示例 | 格式规范 | 角色扮演 |
|------|-----------|---------|---------|
| 传递方式 | 模仿学习 | 结构定义 | 身份暗示 |
| 占 Token | 多（每个示例 50~500 token） | 中（Schema ~100~300 token） | 少（10~50 token） |
| 灵活性 | 中（更换示例即可调整） | 低（格式变更影响面大） | 高（角色词直接调整） |
| 最佳位置 | System Prompt 末尾 | System Prompt 中部 | System Prompt 开头 |
| 主要依赖 | 模型的类比能力 | 模型的格式遵循力 | 模型的角色理解力 |

## 核心优势

1. **直观有效**——"做给你看"比"说给你听"对 LLM 的效果通常更好
2. **隐式传递复杂约束**——示例可以展示指令难以精确描述的复杂行为模式
3. **动态可调**——更换示例比修改指令结构更灵活
4. **可组合性强**——可以从不同的示例库中检索组合，针对不同场景定制

## 工程优化方向

1. **示例库 + 检索系统**：

```python
class FewShotManager:
    def __init__(self, example_store):
        self.store = example_store  # Embedding + 示例对

    def retrieve(self, user_input, top_k=3):
        query_embed = embed(user_input)
        return self.store.similarity_search(query_embed, k=top_k)

    def build_prompt(self, system_instructions, user_input):
        examples = self.retrieve(user_input)
        example_section = "\n\n".join([
            f"## 示例 {i+1}\n输入: {e.input}\n输出: {e.output}"
            for i, e in enumerate(examples)
        ])
        return f"{system_instructions}\n\n{example_section}\n\n当前输入:\n{user_input}"
```

2. **示例 A/B 测试框架**：对候选示例组合进行在线 A/B 测试，用度量指标（任务完成率、用户满意度）决定示例的保留、替换或淘汰
3. **示例压缩技术**：研究如何用最短的文本传递最有效的示例信息，包括去除示例中不影响模型理解的冗余部分
4. **多轮对话示例**：为多轮交互场景设计对话式示例（多组 user-assistant 轮次），而不仅仅是单轮输入输出对
5. **示例元数据标记**：为每个示例标记适用场景、难度级别、依赖条件，支持更精准的示例检索和组合

## 适合场景的判断标准

少样本示例嵌入最适合以下场景：
- **输出模式复杂但可示例化**：多步推理、格式化报告、特定风格的响应
- **任务边界清晰但规则不明确**：某些分类任务"看到就知道"，但难以用规则描述
- **需要控制 Agent 的处理风格**：如何分析问题、如何组织回答、如何呈现推理过程
- **处理模式相对固定的高频率任务**：同一类任务反复出现，值得投入示例优化

不适合的场景：
- **每轮任务完全不同**：无法建立有效的示例库，示例选择几乎随机
- **Token 预算极其紧张**：上下文窗口无法容纳足够的有效示例
- **需要严格规则遵循的任务**：示例容易产生歧义，不如规则指令明确
- **示例本身就存在偏见风险**：当示例的选择和呈现可能引入系统偏见时
