# Graph of Thoughts (GoT) — 思维图提示

## 简单介绍

Graph of Thoughts（思维图）是推理提示技术的集大成者——它将推理过程建模为**有向图**而非树。这意味着推理节点不仅可以分叉（一个思路产生多个子思路），还可以**合并**（多个思路汇聚成一个更强的结论）、**回环**（重新审视之前的节点）、以及**跳跃**（跨层连接不同推理阶段）。

如果说 CoT 是一条直线，ToT 是一棵树，那么 GoT 就是一张网——它允许完全自由的推理拓扑结构。这种灵活性使 GoT 能够表达最复杂的推理模式，但同时也意味着最高的实现复杂度和算力成本。

## 基本原理

GoT 的核心创新在于用图结构替代树结构，支持以下图操作：

1. **推理节点（Thought Vertex）**：表示一个推理状态或中间结论
2. **推理边（Thought Edge）**：表示推理之间的依赖和转换关系
3. **图操作（Graph Operations）**：
   - **分叉（Fork）**：从一个节点产生多个新节点（类似 ToT 的分支）
   - **合并（Merge）**：多个节点汇聚成一个新节点（综合多条推理路径）
   - **回环（Loop）**：节点指向祖先，允许回溯和修正
   - **精炼（Refine）**：基于反馈对节点内容进行迭代改进

```python
class GraphOfThoughts:
    """思维图推理引擎"""

    def __init__(self):
        self.vertices = []   # 推理节点
        self.edges = []      # 推理边 (src, dst, type)

    def add_thought(self, content: str, predecessors: list[int] = None):
        """添加新的推理节点"""
        vid = len(self.vertices)
        self.vertices.append({"content": content, "depth": 0})
        if predecessors:
            for pred in predecessors:
                self.edges.append((pred, vid, "derived"))
        return vid

    def merge_thoughts(self, source_ids: list[int], instruction: str):
        """合并多条推理路径为一个综合结论"""
        sources = [self.vertices[v]["content"] for v in source_ids]

        prompt = f"""以下是对同一问题的多条推理路径，请综合它们：
        {chr(10).join(f'路径{i+1}: {s}' for i, s in enumerate(sources))}

        综合要求：{instruction}
        综合结论："""

        merged = llm.generate(prompt)
        return self.add_thought(merged, predecessors=source_ids)

    def loop_back(self, current_id: int, target_id: int, feedback: str):
        """回环：基于当前节点反馈重新审视之前的节点"""
        target = self.vertices[target_id]
        prompt = f"""原始推理：{target['content']}
        后续反馈：{feedback}
        请基于反馈重新审视上述推理，输出修正后的版本："""
        revised = llm.generate(prompt)
        # 回环产生新的修正节点，连接到原始节点和反馈节点
        new_id = self.add_thought(revised, predecessors=[target_id, current_id])
        self.edges.append((current_id, new_id, "feedback"))
        return new_id
```

## 背景

GoT 由威斯康星大学麦迪逊分校等机构在 2023 年的论文 "Graph of Thoughts: Solving Elaborate Problems with Large Language Models" 中提出。随着 ToT 的成功，研究者很快意识到树结构的局限性：树不允许**跨分支的信息共享**。但在许多实际推理场景中，不同推理路径中的洞察需要被合并才能产生最佳结论。

例如，在文档分析场景中：一条路径分析文档的财务面，另一条分析技术面，第三条分析市场面。这三条路径的结论需要在某个节点合并才能形成全面的投资建议。ToT 的树结构无法表达这种合并操作，而 GoT 通过图自然地支持了。

## 之前针对这个问题的做法与结果

| 做法 | 描述 | 效果与局限 |
|------|------|-----------|
| Tree of Thoughts | 树状推理，支持分叉和回溯 | 分支之间不能信息共享，在需要整合多视角的任务上表现不佳。对文档分析等"多输入→单输出"任务，ToT 缺乏合并机制 |
| 链式 ReAct | 推理 + 行动线性交替 | 工具调用后的一次性反馈，没有"多源信息汇聚"的机制 |
| 多 Agent 辩论（Multi-Agent Debate） | 多个 Agent 各自推理后进行辩论 | 虽然实现了多视角的整合，但 Agent 之间的通信成本高，且缺乏结构化的融合机制 |

## 核心矛盾

| 矛盾维度 | 树结构 (ToT) | 图结构 (GoT) |
|---------|-------------|-------------|
| 拓扑表达能力 | 有限（树 = 无环连通图） | 通用（支持任意图） |
| 分支间通信 | 不允许 | 允许（节点合并） |
| 实现复杂度 | 中等 | 高 |
| LLM 调用次数 | 多 | 非常多 |
| 推理控制难度 | 中等 | 高（需要避免无限循环） |
| 结果可解释性 | 好（树结构清晰） | 较复杂（图结构不易追踪） |

## 当前主流优化方向

1. **有界图推理**——限制图的深度、宽度和合并次数，将 GoT 的通用图表达能力约束在工程可控范围内。例如：只允许在特定层级合并（如最后 2 层）
2. **图结构自动发现**——让 LLM 自行决定何时分叉、何时合并、何时回环，而非由代码硬编码。LLM 在推理过程中自然生成图结构
3. **GoT + 向量数据库**——将推理节点向量化存储，通过相似度检索自动发现可合并的节点，实现大规模推理图中的"结构自组织"
4. **可微分图推理**——引入注意力机制或 GNN 来学习"哪些节点对最终结论更重要"，用于优化搜索策略

## 实现的最大挑战

1. **避免无限回环**——图结构允许回环，但如果没有有效的终止机制，推理可能陷入"不断修正但永远不结束"的死循环。需要设计严格的终止条件（最大迭代次数、收敛检测）
2. **图的规模控制**——一个未加限制的 GoT 图大小随推理步骤指数级增长。需要有效的剪枝策略来决定"保留哪些节点、合并哪些节点、丢弃哪些节点"
3. **排序与评估的复杂性**——在图中，节点之间可能存在复杂的依赖关系，如何确定节点的执行顺序（拓扑排序）、何时触发合并、在何种条件下终止某条分支，这些决策的复杂度远高于 ToT

## 能力边界与结果边界

**能做什么：**
- 需要多视角综合分析的任务（文档分析、论文审稿、投资决策）
- 复杂规划任务（需要将多个子计划的结论合并）
- 需要迭代改进的创意任务（写作、设计、代码生成）
- 存在多源信息需要融合的场景

**不能做什么：**
- 结构化程度低的任务——图表达能力的优势在简单任务上无法发挥
- 任何 ToT 不能做的任务（GoT 不会让 LLM 本身变聪明）
- 对延迟极其敏感的场景

**产生的结果：**
- 完整的推理图（可导出为 Graphviz 或 JSON 格式进行可视化）
- 融合了多视角的推理结论（综合而非单一）
- 可回溯的推理依赖链（知道每个结论来自哪些先验推理）

## 与其他技术的区别

| 对比技术 | 核心区别 |
|---------|---------|
| Tree of Thoughts | ToT 不允许分支合并；GoT 允许任意拓扑（合并、回环、跳跃） |
| Chain-of-Thought | CoT 是线性推理；GoT 是完全自由的推理网络 |
| Self-Consistency | SC 对多条独立路径投票；GoT 在推理过程中就有结构化的路径交互 |
| Multi-Agent Debate | 辩论是横向的 Agent 间交互；GoT 是纵向的推理结构图 |

## 核心优势

1. **最自由的推理拓扑**——能够表达 CoT（单链）、ToT（树）以及任何复杂推理结构，统一了所有推理模式
2. **多源信息融合**——唯一原生支持"从多种推理方向汇聚到一个结论"的技术
3. **迭代精炼**——图结构天然支持"先粗糙再精细"的推理模式（通过回环和精炼操作）
4. **推理过程的全局视角**——完整记录节点间的依赖关系，便于分析"哪些推理步骤对最终答案贡献最大"

## 工程优化方向

1. **渐进式图构建**——不要一次性构建完整的推理图，而是从简单的 CoT 开始，当检测到需要多视角分析时才扩展为图。这样在简单任务上不增加额外开销
2. **可定制的图模板**——为常见任务类型预定义图模板（如"分析报告模板"= 3 条分析分支 → 1 个合并节点 → 结论），避免每次从零构建
3. **可视化调试工具**——开发图结构的可视化工具（用 D3.js、Graphviz 等），便于开发人员理解 LLM 的推理过程，发现推理链中的断裂或不一致

## 适合场景的判断标准

**应该使用 GoT 的场景：**
- 任务显然需要多视角分析（评估报告、竞争分析、文档摘要）
- 推理过程中有明显的信息汇聚点
- 任务复杂到 ToT 的单向树结构不够用
- 有足够且稳定的 API 预算（GoT 的 token 消耗通常是 ToT 的 2-3 倍）

**不应使用 GoT 的场景：**
- 单一路径推理就能完成的任务
- 实时交互系统（GoT 的延迟通常以分钟计）
- 需要确定性输出的任务（图结构的随机性更高）
- 团队能力不足以实现和维护 GoT 系统
