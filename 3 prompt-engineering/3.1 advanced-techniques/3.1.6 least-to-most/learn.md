# Least-to-Most Prompting — 从简到繁提示

## 简单介绍

Least-to-Most Prompting（从简到繁提示）是一种受人类教学法启发的推理技术。它的核心理念非常直观：**复杂问题不是一次性解决的，而是先分解成简单子问题，按顺序逐一解决，每一步的答案都作为下一步的上下文。** 这就像教小孩解数学应用题——先教加法，再教乘法，最后教综合运算。

对于 Agent 系统来说，Least-to-Most 提供了一种极其自然的任务分解和执行框架。Agent 面对一个复杂指令时，不急于直接回答，而是先问自己："完成这个任务需要先解决哪些更简单的问题？"

## 基本原理

Least-to-Most Prompting 分为两个阶段：

**阶段一：问题分解（Decomposition）**
让 LLM 将原始复杂问题分解为一系列从简到繁的子问题。每个子问题应该：
- 比原始问题更简单
- 是原始问题的必要组成部分
- 可以从之前的子问题答案中推导

**阶段二：顺序求解（Sequential Solving）**
按照分解后的顺序依次回答每个子问题，每个子问题的回答都可以参考之前子问题的答案。

```python
def least_to_most(complex_question: str):
    """
    从简到繁提示：先分解，再逐步求解
    """
    # Phase 1: 问题分解
    decomposition_prompt = f"""你是一个问题分解专家。请将以下复杂问题分解为一系列从简单到复杂的子问题。
    要求：
    - 子问题应该是回答原始问题的必要前置步骤
    - 每个子问题比前一个稍微复杂
    - 最终的子问题直接回答原始问题
    - 用编号列表输出

    复杂问题：{complex_question}
    分解："""

    decomposition = llm.generate(decomposition_prompt)
    sub_questions = parse_numbered_list(decomposition)

    # Phase 2: 顺序求解，逐步积累上下文
    context = []
    for i, sub_q in enumerate(sub_questions):
        if i == 0:
            # 第一个子问题，无前置上下文
            solve_prompt = f"Q: {sub_q}\nA:"
        else:
            # 后续子问题，带上所有之前的问答
            qa_history = "\n".join(
                [f"Q{q+1}: {sq}\nA{q+1}: {sa}"
                 for q, (sq, sa) in enumerate(zip(sub_questions[:i], context))]
            )
            solve_prompt = f"""以下是为回答最终问题而逐步推导的问答记录：
            {qa_history}

            请基于以上信息回答下一个子问题：
            Q{i+1}: {sub_q}
            A{i+1}:"""

        answer = llm.generate(solve_prompt)
        context.append(answer)

    # 最终答案已在最后一个子问题的回答中
    return context[-1]


# 实际运行示例：
# 原始问题："一个篮球和一副球拍共 220 元。球拍比篮球贵 200 元。篮球多少钱？"
# 分解结果：
#   1. 篮球和球拍的总价是多少？（已知：220 元）
#   2. 球拍比篮球贵多少钱？（已知：200 元）
#   3. 如果总价是 220 元、差价为 200 元，篮球的价格是多少？
```

## 背景

Least-to-Most Prompting 由 Google Brain 在 2022 年的论文 "Least-to-Most Prompting Enables Complex Reasoning in Large Language Models" 中提出。这篇论文的核心观察是：即使 LLM 能够回答每一个单独的子问题，它在直接回答需要这些子问题组合的复杂问题时仍然表现不佳。

这种现象被称为"合成泛化失败"（compositional generalization failure）或"组合性差距"（compositional gap）——LLM 的能力在简单任务和它们的复杂组合之间存在显著落差。Least-to-Most 通过显式地将组合问题拆解为原子问题，弥合了这个差距。

## 之前针对这个问题的做法与结果

| 做法 | 描述 | 效果与局限 |
|------|------|-----------|
| 直接 CoT | 对整个复杂问题直接推理 | 在需要多步组合推理的 SCAN 数据集上几乎完全失败（准确率 < 5%）。模型无法处理"新组合" |
| Few-shot CoT | 提供复杂问题的推理示例 | 如果示例与测试问题的组合模式相似则有效，但一旦组合模式变化就失败——模型学到的是"模式匹配"而非"组合推理" |
| 分步指令 | Prompt 中写"请分步骤思考" | 对简单组合有效，但对新颖组合场景仍无法泛化。模型不懂得"先解子问题再组合"的结构化方法 |

## 核心矛盾

| 矛盾维度 | 整体推理 | 从简到繁分解 |
|---------|---------|-------------|
| 组合泛化 | 差（对新颖组合表现差） | 好（通过原子子问题实现泛化） |
| 推理可逆性 | 不可逆（一旦出错无法恢复） | 部分可逆（可检查哪个子问题出错） |
| 分解质量依赖 | 不依赖分解 | 高度依赖分解质量 |
| Token 消耗 | 低 | 较高（分解 + 逐步求解） |
| 实现复杂度 | 低 | 中等（需要分解和顺序执行逻辑） |
| 累进错误 | 晚期错误可能被早期步骤影响 | 子问题间的依赖可能导致错误级联 |

## 当前主流优化方向

1. **自适应分解粒度**——不是所有问题都需要分解到相同粒度。当前方向是让 LLM 自行判断"当前子问题是否仍然太复杂"，如果是则继续分解，实现递归分解
2. **验证驱动的分解**——在每个子问题求解后加一个验证步骤：检查答案是否合理、是否与上下文一致。如果验证失败，回到上一步重新分解或重新求解
3. **工具辅助分解**——在某些子问题中调用外部工具解决（如计算器解决算数子问题、搜索引擎解决事实查询子问题），而非完全依赖 LLM 的推理
4. **隐式分解 vs 显式分解**——研究表明，某些场景下不需要显式输出分解列表，而是通过轮次对话隐式地完成分解。Agent 的每次工具调用本质上就是一次"子问题求解"

## 实现的最大挑战

1. **分解的合理性**——如果分解出的子问题列表有遗漏、错误排序或与原始问题无关，后续所有推理都会偏离方向。分解质量直接决定最终结果质量
2. **子问题间的依赖管理**——并非所有子问题都是串行依赖的，有些子问题可能可以并行求解。如何自动识别依赖关系并安排最优执行顺序是工程难点
3. **上下文窗口压力**——随着子问题增多，累积的 Q/A 上下文可能超过模型的最大上下文长度。需要有效的上下文压缩或摘要策略

## 能力边界与结果边界

**能做什么：**
- 需要组合性泛化的任务（如语义解析、指令遵循、字符串操作）
- 多步骤推理任务（数学应用题、逻辑谜题）
- 结构化问题（需要先收集信息再得出结论的任务）
- 教学和解释场景（展示"一步步怎么想的"）

**不能做什么：**
- 子问题分解本身就很困难的任务（分解难度 ≈ 直接回答难度）
- 依赖隐式知识而非逐步推理的任务（如情感分析、风格判断）
- 子问题间依赖关系过于复杂的任务（可能需要 GoT 的图结构）

**产生的结果：**
- 结构化的推理链（每个子问题的 Q/A 对）
- 可追溯的推理步骤（知道最终答案依赖哪些子结论）
- 失败定位信息（哪个子问题出错了）

## 与其他技术的区别

| 对比技术 | 核心区别 |
|---------|---------|
| Chain-of-Thought | CoT 是"怎么算"的逐步展示；Least-to-Most 是"先算什么再算什么"的结构化分解 |
| Task Decomposition (Plan) | 传统任务分解是 Agent 规划阶段；Least-to-Most 将分解和求解融合在一个 Prompt 中 |
| ReAct | ReAct 是推理 + 行动交替；Least-to-Most 是推理 + 推理累积 |
| Tree-of-Thoughts | ToT 在同一层生成多个候选；Least-to-Most 是线性的顺序分解 |

## 核心优势

1. **组合泛化能力**——这是 Least-to-Most 最独特的优势。在需要"新组合"的任务上，它的表现远超 CoT 和传统 Few-shot 方法
2. **自然的教学结构**——分解 + 顺序求解的结构与人类教学方式一致，特别适合需要解释和教学的应用场景
3. **渐进式上下文构建**——每个子问题都建立在前一个子问题的答案上，信息逐步累积，无需一次性处理全部信息
4. **调试友好**——当最终答案错误时，可以回查每个子问题的答案，快速定位错误发生的位置

## 工程优化方向

1. **缓存子问题模板**——对于常见的问题类型（如"X 比 Y 多多少"、"如果 A 那么 B"），预定义子问题分解模板，避免每次重新分解
2. **并行子问题求解**——识别无依赖关系的子问题并行求解，利用并行 API 调用降低整体延迟。通过分析子问题间的依赖图来调度执行
3. **自适应停止**——当检测到子问题求解已经能直接回答原始问题时（如某子问题的答案已经足够支撑结论），提前结束后续子问题的求解

## 适合场景的判断标准

**应该使用 Least-to-Most 的场景：**
- 问题有明显的"前置知识"依赖关系
- 任务的组合模式是新颖的（模型训练中未见过）
- 需要可解释的逐步推理过程
- 问题可以自然拆分为独立子问题

**不应使用 Least-to-Most 的场景：**
- 问题本身已经是原子级别（无法进一步分解）
- 子问题之间的依赖关系不明确或过于复杂
- 需要快速响应的场景（分解 + 逐步求解的延迟较高）
- 模型已经能很好处理该问题的组合模式（CoT 就足够）
