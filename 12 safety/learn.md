# 12 safety — 安全、对齐与合规

**Agent 的安全不是"附加功能"，而是生存底线。** 一个不安全的 Agent 不仅无法产生价值，还会带来数据泄露、决策失控、合规风险等灾难性后果。如果说评估体系决定了你能看到 Agent 的天花板，那么安全体系决定了地板——最坏情况下 Agent 能造成多大的损害。

## Agent 安全与传统安全的核心区别

传统安全是"围墙模型"——修好边界墙，内部就安全。Agent 安全完全不同，因为 Agent 面对的是不可预测的 LLM 输出和动态的外部环境：

```
┌─────────────────────────────────────────────────────────────────┐
│  传统安全（边界防御）         Agent 安全（纵深防御）               │
│  ────────────────              ────────────────                  │
│  • 防火墙阻止外部访问          • Prompt 注入随时可能突破指令边界    │
│  • 权限固定、一次设置          • 权限需要动态评估上下文中调整       │
│  • 输入输出可严格校验          • LLM 输出非确定性，校验复杂         │
│  • 攻击面静态可枚举            • 攻击面随工具和知识扩展动态变化     │
│  • 漏洞可 patch 修复           • 对齐问题没有"修复"只有"缓解"    │
└─────────────────────────────────────────────────────────────────┘
```

## 安全设计原则（Security-First Design）

| 原则 | 说明 | 应用场景 |
|------|------|---------|
| **最小权限** | Agent 只拥有完成任务所需的最小权限 | 工具 API Key 作用域限制 |
| **纵深防御** | 多层防护，单层失效不意味全线崩溃 | 输入过滤 + 输出审核 + 权限控制 |
| **默认拒绝** | 未明确允许的操作一律拒绝 | 文件写入、高危操作审批 |
| **职责分离** | 关键操作需要多人/多组件协作 | 审批 Agent + 执行 Agent 分离 |
| **可审计性** | 所有决策和操作可追溯 | 完整日志链 + 不可篡改记录 |
| **优雅降级** | 安全机制失效时系统能安全降级 | 熔断后拒绝操作而非允许 |
| **人机协同** | 高风险操作需人工确认 | 数据删除、资金转账审批 |
