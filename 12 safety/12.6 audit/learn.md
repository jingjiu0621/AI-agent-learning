# 12.6 audit — 审计与合规

**在 Agent 系统中，"信任"必须建立在"可审计"的基础上。** 审计不仅是为了合规，更是为了发现异常、追溯问题和建立信任。当 Agent 犯了错误，你需要能回答：它做了什么？谁让它做的？为什么会这样做？

## Agent 审计 vs 传统审计

```
传统系统审计                        Agent 系统审计
─────────────────                  ─────────────────
• 记录谁在什么时间访问了什么          • 记录 Agent 的思考轨迹
• 审计重点是 API 调用和 DB 操作       • 审计重点是 LLM 推理 + 工具调用链
• 日志结构固定、容易分析              • 日志包含自然语言推理过程
• 操作确定性——可以重放验证            • 操作非确定性——重放可能不同结果
• 错误通常是代码 Bug                 • 错误可能是 LLM 幻觉 + 错误决策链

核心区别：Agent 审计需要同时记录 "思考过程" 和 "实际行动"，
         因为 "为什么做" 和 "做了什么" 同样重要。
```

## 合规框架映射

| 合规框架 | 适用范围 | 对 Agent 的关键要求 | 影响 |
|---------|---------|-------------------|------|
| GDPR | 欧盟用户数据 | 数据最小化、删除权、解释权 | 必须记录数据使用、支持遗忘 |
| SOC 2 | 通用安全 | 安全性、可用性、机密性 | 必须有审计日志、访问控制 |
| HIPAA | 医疗数据 | PHI 保护、访问审计 | 需要 PHI 脱敏、严格审计 |
| CCPA | 加州消费者 | 知情权、删除权、选择退出 | 数据分类 + 用户控制 |
| EU AI Act | AI 系统 | 风险管理、透明度、人工监督 | 高风险 Agent 需要评估 |
| ISO 42001 | AI 管理 | AI 管理体系 | 全生命周期管理 |

## 关键认知

- **没有审计就没有问责**：无法追溯的 Agent 行为等于无法管理
- **审计不是事后补救**：审计日志必须在系统设计阶段嵌入，无法后期添加
- **日志本身是攻击目标**：审计日志需要防篡改，否则失去可信度
- **隐私与审计的平衡**：详细日志有助于审计，但可能包含敏感信息（需要数据脱敏）
- **合规是底线不是目标**：满足合规要求是最低标准，好的审计超越合规要求
- **自动化审计**：Agent 数量多、操作快，人工审计不可行，需要自动化审计管道
