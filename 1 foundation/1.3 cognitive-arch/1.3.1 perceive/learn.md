# 感知模块：多源信息接收、解析与表示

## 简单介绍

感知（Perceive）是 Agent 认知循环的第一步——从环境接收原始信息，并将其转化为 Agent 可以理解和处理的结构化表示。LLM Agent 的感知远比传统程序复杂：它需要理解自然语言、解析工具返回结果、处理多模态输入，并在海量信息中提取关键内容。

## 基本原理

### 感知流程

```
原始输入 → 解析 → 结构化 → 过滤 → 上下文注入
    │         │         │        │         │
    ▼         ▼         ▼        ▼         ▼
用户消息    JSON解析   模式转换  重要性筛选  加入 Prompt
API返回    格式提取    Schema    Token 预算  消息列表
环境状态    多模态     统一表示   优先级排序   System Prompt
```

### 感知的类型

| 类型 | 来源 | 处理方式 |
|------|------|---------|
| 用户指令 | 自然语言 | 意图解析 + 关键信息提取 |
| 工具返回值 | JSON/结构化数据 | Schema 验证 + 字段提取 |
| 系统状态 | 运行环境 | 状态快照 + 变更检测 |
| 记忆检索 | 内部存储 | 相关性排序 + 结果融合 |
| 环境变化 | 外部事件 | 事件流 → 结构化事件 |

## 背景

传统程序的"感知"是**确定的**——输入格式固定（JSON/XML/表单），通过解析器精确提取字段。LLM Agent 的感知面临三个根本挑战：

1. **非结构化输入**：用户的自然语言指令没有固定格式
2. **多源异构**：工具返回结果格式各异（JSON/文本/表格/图片）
3. **信息过载**：上下文窗口有限，需要在海量信息中筛选关键内容

## 关键挑战

### 1. 解析的鲁棒性
- **问题**：工具返回格式可能不符合预期（错误/截断/非标准）
- **方案**：多级解析 → 首次用代码精确解析 → 失败后用 LLM 模糊解析

### 2. 信息的优先级
- **问题**：感知到的信息太多，上下文窗口装不下
- **方案**：重要性评分（相关性 × 时效性 × 权威性）

### 3. 多模态融合
- **问题**：文本/图像/音频等不同模态需要统一表示
- **方案**：多模态编码器 → 统一向量空间 → 交叉注意力融合

### 4. 上下文预算管理
- **问题**：感知的信息量超过 Token 预算
- **方案**：按优先级截断 + 摘要压缩 + 分层存储

## 工程实现

### 感知层设计模式

```
┌──────────────────────┐
│   感知管理器           │  ← 统一入口
├──────────────────────┤
│  ┌────┐ ┌────┐ ┌────┐ │
│  │解析器│ │验证器│ │转换器│ │  ← 可插拔处理器链
│  └─┬──┘ └─┬──┘ └──┬─┘ │
│    ▼      ▼        ▼   │
│  ┌──────────────────┐   │
│  │   优先级队列      │   │  ← 排序与过滤
│  └──────────────────┘   │
└──────────────────────┘
```

### 关键实现要点
1. **解析器链**：每种输入类型配备专门的解析器（Markdown/JSON/CSV/图像）
2. **格式验证**：解析后验证是否符合 Schema，不符合的尝试修复
3. **智能截断**：根据 Token 预算自动裁剪低优先级内容
4. **压缩策略**：对长文本自动摘要，保留关键信息

## 优化方向

1. **多模态感知扩展**：增加图像/音频/视频的感知能力
2. **流式感知**：Agent 运行中持续感知环境变化，而非一次性注入
3. **主动感知**：Agent 自主决定需要感知什么信息（查询数据库/发起搜索）
4. **感知质量控制**：检测感知信息的准确性（事实核查、冲突检测）
