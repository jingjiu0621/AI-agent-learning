# 思考模块：推理、规划、决策与反思

## 简单介绍

思考（Think）是 Agent 认知循环的核心——它决定了 Agent 将感知到的信息转化为什么行动。在 LLM Agent 中，思考模块由大语言模型驱动，涵盖了从简单的"下一步做什么"到复杂的多步推理、规划、决策制定和事后反思。

## 基本原理

### 思考的分层

```
决策层        在多个选项中选择
   ↑
规划层        制定达成目标的步骤序列
   ↑
推理层        从已知信息推导新结论
   ↑
响应层        生成即时回复（System 1 快速思考）
```

### LLM Agent 中的思考类型

| 思考类型 | 描述 | 触发条件 | 工程实现 |
|---------|------|---------|---------|
| 即时响应 | 直接回复，无需推理 | 简单问题/问候 | 直接 LLM 调用 |
| 单步推理 | 一步推理得出答案 | 中等复杂度 | CoT Prompt |
| 多步推理 | 分步推理链 | 复杂问题 | ReAct Loop |
| 规划 | 制定任务执行计划 | 多步骤任务 | Plan-and-Execute |
| 决策 | 在多个选项中做选择 | 需要判断 | 比较分析+评分 |
| 反思 | 回顾和评估已完成的行动 | 任务完成/出错 | Reflexion 模式 |

## 背景

在传统 AI 中，推理、规划和决策是由不同技术处理的：
- **推理**：逻辑推理器（Prolog/定理证明）
- **规划**：规划算法（STRIPS/HTN/MCTS）
- **决策**：决策树/效用理论

LLM 的强大之处在于它**统一了这些能力**——同一个模型可以执行推理、规划和决策。但同时这也带来了问题：LLM 的"思考"不可靠，有时会出错、产生幻觉、或陷入循环。

## 核心挑战

### 1. 推理可靠性
- **问题**：LLM 可能"推理错误"——看似合理但逻辑上有缺陷
- **原因**：LLM 是统计模型，不是严格逻辑推理器
- **缓解**：Self-Consistency（多次采样投票）、验证器

### 2. 规划深度与成本
- **问题**：规划步数越多，Token 消耗越大，失败概率越高
- **平衡**：详细的规划 vs 高效的执行
- **方案**：分层规划（高层粗粒度 + 低层细粒度）

### 3. 决策一致性
- **问题**：同一问题在不同上下文中可能做出不同决策
- **原因**：LLM 的不确定性和 Prompt 敏感性
- **方案**：决策框架 + 结构化输出约束

### 4. 反思的效率
- **问题**：反思机制虽然提高了质量，但消耗 Token
- **优化**：只在关键节点反思（错误/超时/用户投诉）

## 工程实现

### 思考模块的分层架构

```
┌─────────────────────────────┐
│     元认知层                  │  监控思考过程
│  (何时停止思考？需要反思？)    │
├─────────────────────────────┤
│     推理框架层                │  ReAct / CoT / ToT
│  (用什么方式思考？)           │
├─────────────────────────────┤
│     基础思考层                │  LLM 调用 + Prompt
│  (如何生成思考？)             │
└─────────────────────────────┘
```

### 关键实现要点
1. **思考过程结构化**：使用结构化输出定义 Thought 格式（代码块/markdown/JSON Schema）
2. **思考深度控制**：根据任务复杂度动态调整推理步骤数量
3. **思考链外部化**：将思考过程暴露给用户，增加透明度和信任
4. **思考与行动分离**：思考模块只生成决策，由执行模块具体执行

## 优化方向

1. **思考效率提升**：减少不必要的推理步骤（任务简单时走"快思考"路径）
2. **外部工具辅助推理**：复杂推理 → 调用代码执行器/计算器（不用 LLM 做数学）
3. **经验复用**：保存历史推理链，相似任务直接复用
4. **多路径推理**：Tree of Thoughts/Graph of Thoughts 探索多路径
5. **置信度校准**：Agent 能判断自己的推理可靠度，低置信度时请求帮助
