# Plan-and-Execute Agent：先规划后执行

## 简单介绍

Plan-and-Execute 模式将 Agent 的任务执行分为两个明确阶段：**规划阶段**——制定任务执行计划；**执行阶段**——按计划执行。与 ReAct 的"边想边做"不同，Plan-and-Execute 先想好再做，适合需要长途规划的复杂任务。

## 基本原理

### 两阶段架构

```
规划阶段（Planner）:
任务描述 → 分析 → 生成步骤序列 → 步骤依赖关系 → 计划

执行阶段（Executor）:
按计划执行 → 每步完成后检查 → 需要时调整计划 → 直到完成
```

```
用户：写一份关于新能源汽车市场分析的报告

Planner:
1. 搜索 2024-2025 新能源汽车销量数据 → 依赖：无
2. 搜索主要厂商（比亚迪/特斯拉/蔚来等）财报 → 依赖：无
3. 分析市场趋势 → 依赖：1, 2
4. 生成报告初稿 → 依赖：3
5. 检查报告完整性 → 依赖：4

Executor:
并行执行 1 和 2 → 等待完成 → 执行 3 → 执行 4 → ...
```

### 规划的类型

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| 顺序规划 | 步骤 A → B → C | 有明确依赖 |
| 并行规划 | 步骤 A \|\| B \|\| C | 独立任务 |
| 分层规划 | 高层子任务 → 细粒度步骤 | 复杂任务 |

## 核心价值

1. **全局视野**：先看到完整的"地图"，避免局部最优
2. **并行执行**：识别独立步骤，并行执行提升效率
3. **进度可追踪**：计划提供了可检查的"完成清单"
4. **可调整**：某个步骤失败时，只需要重规划受影响的部分

## ReAct vs Plan-and-Execute

| 维度 | ReAct | Plan-and-Execute |
|------|-------|-----------------|
| 规划时机 | 边做边规划 | 先全盘规划再执行 |
| 灵活性 | 高（随时调整） | 中（计划可能过时） |
| 长任务 | 容易迷失方向 | 更适合 |
| 并行性 | 无 | 可并行 |
| 规划开销 | 低 | 高（规划本身消耗 Token） |

## 限制

1. **规划可能不准**：LLM 生成的计划可能不完备或不合理
2. **计划过时**：执行中环境变化可能导致原计划无效
3. **规划开销**：制定详细计划本身消耗 Token 和时间
4. **规划-执行脱节**：Executor 可能偏离 Planner 的意图

## 工程实现

```python
def plan_and_execute(task):
    # 阶段 1：规划
    plan = planner.generate_plan(task)
    # plan = [
    #   {"step": 1, "action": "search", "args": {...}, "depends_on": []},
    #   {"step": 2, "action": "search", "args": {...}, "depends_on": []},
    #   {"step": 3, "action": "analyze", "args": {...}, "depends_on": [1, 2]},
    # ]
    
    # 阶段 2：执行（带依赖管理）
    results = {}
    for step_group in resolve_dependencies(plan):
        # 并行执行无依赖的步骤
        parallel_results = parallel_execute(step_group)
        results.update(parallel_results)
    
    return synthesize(results)
```

## 优化方向

1. **规划验证**：执行前让 LLM 检查计划质量（遗漏？重复？不合理？）
2. **动态重规划**：当关键步骤失败时，只重规划依赖该步骤的后续部分
3. **增量规划**：先规划第一个阶段，执行完再规划下一个（平衡 ReAct 和全盘规划）
4. **模板复用**：将成功任务的计划保存为模板，相似任务直接复用小调整
