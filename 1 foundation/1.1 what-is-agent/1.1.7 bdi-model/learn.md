# BDI 模型：信念-愿望-意图框架

## 简单介绍

BDI（Belief-Desire-Intention）模型是 Agent 理论中最具影响力的认知架构之一，由 Michael Bratman 的哲学工作（1987）发展而来，后被 Rao 和 Georgeff 形式化为计算模型。BDI 模型认为，一个理性 Agent 的行为可以用三个核心心智状态来解释：**信念（Belief）**——Agent 对世界的认知；**愿望（Desire）**——Agent 希望达到的状态；**意图（Intention）**——Agent 承诺要执行的行为路线。

## 基本原理

### 三要素定义

```
┌────────────────────────────────────────────────┐
│                    BDI Agent                     │
│                                                    │
│  信念 (Belief)      愿望 (Desire)     意图 (Intention) │
│  ┌──────────┐     ┌──────────┐     ┌──────────┐    │
│  │ 世界状态  │     │ 可选目标  │     │ 已承诺的  │    │
│  │ 的知识    │     │ 合集     │     │ 行动计划  │    │
│  └──────────┘     └──────────┘     └──────────┘    │
│       │                │                │          │
│       └────────┬───────┘               │           │
│                ▼                       │           │
│         愿望过滤 (Deliberation)         │           │
│                │                       │           │
│                ▼                       │           │
│         形成意图 (Means-End) ───────────┘           │
│                │                                    │
│                ▼                                    │
│          执行 (Execution)                           │
└────────────────────────────────────────────────┘
```

### 1. 信念（Belief）

Agent 对当前环境和自身状态的主观认知。

- **来源**：初始知识 + 感知输入 + 推理结果
- **特点**：
  - 可能不正确（Agent 的认知不等于客观事实）
  - 可以更新（新信息 → 信念修正）
  - 包含世界状态、工具能力、历史经验
- **在 LLM Agent 中的体现**：LLM 的上下文窗口内容 + 记忆中存储的信息

```
信念 = Agent 认为真实的信息集合
信念修正 = 新观察 → 更新旧信念（可能放弃或修改）
```

### 2. 愿望（Desire）

Agent 希望实现的目标状态，是对"世界应该怎样"的描述。

- **来源**：用户指令 + Agent 自身动机 + 系统任务
- **特点**：
  - 可能相互冲突（鱼和熊掌不可兼得）
  - 并非所有愿望都能同时追求
  - 不需要逻辑一致性（可以有不兼容的愿望）
- **在 LLM Agent 中的体现**：系统 Prompt 中的目标描述 + 用户请求

```
愿望 = Agent 希望实现的目标
冲突解决 = 根据优先级和可行性选择当前要追求的愿望
```

### 3. 意图（Intention）

Agent 承诺要执行的行为路线——被选中的愿望 + 实现它的行动计划。

- **来源**：愿望经过 deliberation（慎思）过滤后的结果
- **特点**：
  - 具有承诺性（一旦形成意图，Agent 会坚持执行）
  - 指导行动选择（只有与意图一致的行动才会被考虑）
  - 可以分层（高层意图 → 子意图 → 原子动作）
  - 可放弃（失败或环境变化时）
- **在 LLM Agent 中的体现**：Agent 制定的计划 + 正在执行的步骤

```
意图 = 已承诺的行动计划
承诺策略 = 盲目的（坚持到底） / 谨慎的（可重新考虑） / 开放的（随时调整）
```

## BDI 推理循环

```
初始化信念集
循环：
  1. 感知环境，更新信念
  2. 选项生成：基于信念和当前意图，生成可能的愿望
  3. 过滤：基于信念评估愿望的可行性，选择要追求的
  4. 手段-目的推理：为选定的愿望生成计划（形成意图）
  5. 执行计划中的下一步行动
  6. 如果计划完成或失败，回到步骤 1
  7. 如果信念发生重大变化，重新评估当前意图
```

## 背景

BDI 模型的理论基础：

- **哲学来源**：Bratman (1987) 《Intention, Plans, and Practical Reason》论证意图在实践推理中的独特作用——意图不仅指导规划，还约束后续推理
- **计算模型**：Rao & Georgeff (1995) 将 BDI 形式化为模态逻辑系统，定义了 BDI 框架的语义
- **工程实现**：AgentSpeak(L) 编程语言、JASON 平台、PRS（Procedural Reasoning System）
- **LLM 时代的复兴**：LLM Agent 的研究者开始重新审视 BDI，发现 LLM 天然适合实现 BDI 中的"信念维护"和"意图推理"

## 之前做法与结果

### 传统 BDI 实现
- 信念：知识库（Prolog 事实库、描述逻辑）
- 愿望：目标集合（goal formula）
- 意图：计划库（预定义的计划模板）
- **结果**：模型严谨但工程笨重——计划库手工编写、推理效率低、难以扩展

### LLM 时代的 BDI
- 信念：上下文窗口 + 向量数据库（动态、灵活）
- 愿望：用户指令 + 系统 Prompt（自然语言描述）
- 意图：LLM 生成的计划（动态、自适应）
- **结果**：BDI 概念框架得到了实用化的可能，但失去了理论的严格性

## 核心矛盾

| 矛盾 | 说明 |
|------|------|
| **承诺 vs 灵活** | 意图要求承诺（坚持执行），但过度承诺导致无法响应变化 |
| **信念一致性 vs 更新成本** | 维护信念的一致性需要推理代价，不一致会导致错误决策 |
| **愿望冲突 vs 选择困难** | 多个冲突愿望并存时，如何公平有效地选择 |
| **理论严格 vs 工程实用** | 严格 BDI 形式化难实现，简化版本失去理论保证 |

## BDI 对 LLM Agent 设计的影响

BDI 框架为 LLM Agent 提供了宝贵的**结构化设计思路**：

1. **信念管理与检索**：区分"永久信念"（常识、规则）和"临时信念"（对话中的新信息）；信念不是一视同仁的，而是有可信度等级
2. **意图承诺策略**：何时坚持原计划，何时重新规划——对 Agent 的稳定性和适应性都至关重要
3. **愿望过滤机制**：不是每个产生的目标都值得追求——需要评估可行性、与核心目标的一致性、所需资源

## BDI vs 其他认知架构

| 对比 | BDI | SOAR | ACT-R |
|------|-----|------|-------|
| 理论基础 | 意图哲学+模态逻辑 | 认知统一理论 | 认知心理学实验 |
| 核心概念 | 信念/愿望/意图 | 问题空间/算子/块状记忆 | 产生式系统/工作记忆/声明式记忆 |
| 学习机制 | 无内置学习 | 块状学习(chunking) | 产生式学习+参数调优 |
| 工程复杂性 | 中 | 高 | 高 |
| LLM 适配度 | 高（概念匹配好） | 中 | 中 |

## 工程实现建议

1. **BDI 作为设计模式而非严格实现**：借鉴 BDI 的组织理念，但不追求形式化正确性
2. **信念分层次**：短期信念（对话上下文）和长期信念（知识库/向量存储）分离管理
3. **愿望的优先级队列**：维护一个优先级排序的愿望队列，根据当前上下文动态调整
4. **意图检查点**：定期评估当前意图是否仍然有效（信念变化检测）
5. **承诺策略配置**：根据任务类型选择不同的承诺策略（简单任务"盲从"，复杂任务"谨慎"）

## 适用场景

| 场景 | BDI 适用性 | 理由 |
|------|-----------|------|
| 复杂推理任务 | ✅ 很适合 | 意图引导结构化推理 |
| 多目标权衡 | ✅ 很适合 | 愿望过滤和优先级管理 |
| 开放域对话 | ⚠️ 适度适用 | 信念维护有帮助但轻量化即可 |
| 实时控制 | ❌ 不适合 | 推理循环开销大 |
| 简单固定任务 | ❌ 过度设计 | 反应式就够用 |
