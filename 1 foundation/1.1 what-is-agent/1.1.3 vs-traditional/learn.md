# Agent vs 传统程序：状态驱动 vs 意图驱动

## 简单介绍

Agent 与传统程序最本质的区别在于 **控制范式**。传统程序是 **状态驱动（State-Driven）** 的：开发者预定义所有可能的输入状态和对应的处理逻辑。Agent 是 **意图驱动（Intent-Driven）** 的：用户指定"想要达成的目标"，由 Agent 自主决定如何实现。这个范式转换从根本上改变了软件的设计、开发和运维方式。

## 基本原理

### 状态驱动（传统程序）

```
输入 → [状态判断] → 预定义分支 A → 输出
                    → 预定义分支 B → 输出
                    → 预定义分支 C → 输出
```

- 开发者需要预先枚举所有可能的状态和转换
- 控制流是确定的、可预测的
- 行为通过代码精确指定

### 意图驱动（Agent）

```
意图/目标 → [LLM 推理] → 决定行动 1 → 观察结果 → 调整 → 行动 2 → ...
                          → 决定替代方案 → ...
```

- 用户只需要描述目标和约束
- 控制流由 LLM 动态决定
- 行为有不确定性，但适应性强

## 核心对比

| 维度 | 传统程序 | Agent |
|------|---------|-------|
| **控制逻辑** | if-else / switch / 状态机 | LLM Prompt + 推理 |
| **确定性** | 完全确定（相同输入→相同输出） | 概率性（相同输入可能不同输出） |
| **状态管理** | 显式变量/数据库 | 上下文窗口 + 记忆系统 |
| **错误处理** | try-catch / 错误码 | 重试 / 降级 / 替代路径 |
| **可测试性** | 单元测试精确覆盖 | 模糊评估（LLM-as-Judge） |
| **变更成本** | 改代码 → 测试 → 部署 | 改 Prompt → 验证 |
| **开发方式** | 写代码 | 写 Prompt + 配置工具 |
| **适应性** | 固定行为 | 动态适应 |
| **性能可预测性** | 高（执行路径确定） | 低（Token 消耗不确定） |
| **黑盒程度** | 透明（代码可读） | 半透明（推理过程可追踪但不可完全复现） |

## 背景

传统编程的核心信条是"程序 = 数据结构 + 算法"。这种方法在需求可穷举、输入输出明确的场景中表现完美。但随着软件应用场景变得越来越复杂和开放（例如自然语言交互、跨领域知识整合、动态环境自适应），传统编程的局限性开始显现：

1. 无法穷举所有分支路径
2. 维护条件判断的复杂度随规模指数增长
3. 难以处理非结构化输入（自然语言、图像等）
4. 无法从经验中改进行为

LLM Agent 的出现提供了一种替代范式：不写死控制流，而是让模型自行推理决策。

## 之前做法与核心矛盾

### 传统做法的困境

| 问题 | 表现 | 后果 |
|------|------|------|
| 分支爆炸 | 状态越多，if-else 呈组合增长 | 代码难以维护 |
| 封闭世界假设 | 无法处理未定义输入 | 低鲁棒性 |
| 硬编码逻辑 | 行为改变需修改代码 | 迭代周期长 |
| 缺乏常识 | 不理解"常识性"知识 | 需要精确指令 |

### 核心矛盾

**确定性需求 vs 开放世界适应**：传统程序追求确定性（可预测、可测试），这在封闭环境是优点；Agent 追求适应性（能处理未知情况），但牺牲了可预测性。两种范式在当前阶段需要混合使用，而非完全替代。

## Agent 的优势

1. **处理非结构化输入**：自然语言、图像等 Agent 可直接理解，传统程序需要复杂的解析逻辑
2. **组合创新能力**：Agent 可以将已有工具组合出新的解决方案，传统程序只执行预设路径
3. **快速原型验证**：修改 Prompt 比修改代码快得多，适合需求频繁变化的场景
4. **知识泛化能力**：LLM 的预训练知识使 Agent 能处理未专门训练过的任务

## Agent 的劣势（传统程序依然不可替代的场景）

1. **高可靠性场景**：要求 100% 确定性的系统（如银行交易、航空控制系统）
2. **性能敏感场景**：毫秒级响应要求
3. **资源受限环境**：边缘设备、嵌入式系统（Agent 需要 LLM 推理，计算成本高）
4. **合规审计**：要求完全可预测行为的监管场景
5. **精确计算**：数学计算、逻辑运算中 Agent 表现不如传统程序可靠

## 工程模式：混合架构

目前最佳的工程实践是 **混合架构**——Agent 和传统程序协同工作：

```
┌─────────────────────────────────────────┐
│              Agent Orchestrator          │  ← LLM 驱动，负责决策
├─────────────────────────────────────────┤
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │ 传统 API │ │ SQL 查询 │ │ 规则引擎 │   │  ← 传统代码，负责执行
│  └─────────┘ └─────────┘ └─────────┘   │
├─────────────────────────────────────────┤
│       工具层（Function Calling）         │  ← Agent 调用工具的桥梁
└─────────────────────────────────────────┘
```

Agent 负责"决定做什么"（意图推理），传统程序负责"具体怎么做"（精确执行）。

## 工程转换策略

从传统程序迁移到 Agent 架构的推荐路径：

1. **识别适合 Agent 的模块**：需要灵活决策、自然语言理解的部分
2. **保留确定性核心**：精确计算、安全关键逻辑保持传统代码
3. **构建工具 API**：将传统功能封装为 Agent 可调用的工具
4. **逐步替换**：从简单任务开始，逐渐扩展 Agent 的决策范围
5. **混合部署**：同时保留传统流程和 Agent 流程，通过 A/B 测试验证

## 适用场景判断

**传统程序更优的场景**：
- 输入输出结构明确、可穷举
- 对响应时间和吞吐量有严格要求
- 规则稳定、几乎不变
- 要求完全可审计和可追溯

**Agent 更优的场景**：
- 需要理解模糊或不完整的指令
- 任务涉及多种工具的组合调用
- 规则频繁变化、需要快速适应
- 需要在开放域中搜索和综合信息

**混合架构最优的场景**：
- 大部分场景——尤其是在企业应用中，Agent 做决策编排，传统代码做精确执行
