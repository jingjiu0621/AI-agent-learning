# 13.1 deployment — 部署方案

**部署是 Agent 从开发环境走向生产环境的第一道门槛。** 选错了部署方式，轻则运维成本飙升，重则在流量高峰直接宕机。与传统 Web 服务不同，Agent 部署的复杂之处在于：它需要同时管理 LLM API 通信、工具执行环境、记忆持久化和流式响应——这些都对部署基础设施提出了针对性要求。

## 部署维度分析

Agent 部署需要同时考虑以下六个维度：

```
传统部署只考虑：     Agent 部署额外考虑：
────────────────     ────────────────────
CPU / Memory         + LLM API 延迟与配额
网络带宽             + Token 消耗速率
磁盘 IO             + 上下文窗口大小
并发连接数           + 工具执行环境隔离
                   + 记忆存储访问速度
                   + 流式响应连接保持
```

### 横向对比

| 维度 | Docker | K8s | Serverless | Edge |
|------|--------|-----|------------|------|
| **运维负担** | 中低 | 高 | 低 | 中 |
| **弹性速度** | 分钟级 | 秒级 | 毫秒级 | 毫秒级 |
| **状态支持** | 好(本地) | 好(PVC) | 差(需外部) | 差(需外部) |
| **流式支持** | 原生 | 原生 | 有限(Max 30s) | 有限 |
| **GPU 支持** | 好 | 好 | 差 | N/A |
| **网络延迟** | 低 | 低 | 中 | 极低 |
| **单次调用成本** | 固定 | 固定+弹性 | 可变 | 可变 |
| **调试难度** | 低 | 中 | 高 | 高 |
| **Secret 管理** | 环境变量 | Secret 对象 | 平台密钥管理 | 环境变量 |

## 部署策略选择流程

```
开始部署方案选择
    │
    ▼
Agent 规模评估
    │
    ├── MVP / 单人使用 ──────────→ Docker Compose
    │
    ├── 团队使用 / 日均 < 1K 请求 ──→ Docker + 单机部署
    │
    ├── 日均 1K~100K 请求 ───────→ K8s 集群
    │
    ├── 事件驱动 / 调用频率低 ────→ Serverless (Lambda)
    │
    └── 全球用户 / 超低延迟需求 ──→ Edge (Cloudflare Workers)
```

## 部署反模式

### ❌ 把 Agent 当无状态服务部署
- **症状**：每次请求都可能路由到不同实例，对话历史丢失
- **修复**：会话亲和性（Sticky Session）+ 外部状态存储

### ❌ 使用单层部署
- **症状**：开发和生产环境不一致，CI 通过了上线却崩了
- **修复**：至少 dev / staging / prod 三层

### ❌ 忽略 LLM API 失败
- **症状**：LLM API 超时就返回 500，没有重试
- **修复**：指数退避重试 + 模型降级回退

### ❌ 透传 API Key 到 Agent 镜像
- **症状**：镜像泄露后 API Key 被盗用
- **修复**：运行时注入 (Secret / 环境变量)
