# 13.3 caching — 缓存策略

**Agent 系统中的缓存与传统 Web 缓存有本质不同：缓存不再是锦上添花的性能优化，而是决定系统能否盈利的核心基础设施。** 在一次典型的 Agent 交互中，LLM API 调用可能占总成本的 80-90%，而大量请求本质上是对相似或相同信息的重复查询。一个设计良好的缓存系统可以将 LLM 调用量降低 50-80%，同时将响应延迟从秒级降至毫秒级。关键在于，Agent 缓存的维度远比传统缓存丰富——我们不仅要缓存 LLM 响应本身，还要缓存工具执行结果、嵌入向量和知识检索结果，每一层的缓存策略都有不同的权衡。

## Agent 缓存的三维决策空间

Agent 缓存不是单一策略，而是在三个维度上的联合决策。每个维度的选择相互影响，共同决定缓存系统的效果。

```
Dimension 1: 什么需要缓存？
┌─────────────────────────────────────────────────────────────┐
│  Cacheable Layer        示例                    成本占比    │
│  ───────────────        ────                    ────────    │
│  LLM Response           完整生成文本              60-80%    │  ← 最大收益
│  Embedding Vector       文本转向量                5-10%     │
│  Tool Result            API 返回值、DB 查询        10-20%    │
│  Knowledge Chunk        检索到的上下文片段         5-15%     │
│  Agent Plan             ReAct 中间推理步骤        5-10%     │
└─────────────────────────────────────────────────────────────┘

Dimension 2: 缓存在哪里？
┌─────────────────────────────────────────────────────────────┐
│  层级        存储介质        延迟         容量     典型TTL   │
│  ─────        ────────       ────        ────      ──────   │
│  L1 (内存)    LRU Map /      <1μs        1-10GB   秒-分钟   │
│               in-memory DB                                  │
│  L2 (分布式)  Redis /        <5ms        10-100GB  分钟-小时  │
│               Memcached                                     │
│  L3 (持久)    PostgreSQL /   <50ms       100GB+    小时-天   │
│               S3 / Disk                                     │
└─────────────────────────────────────────────────────────────┘

Dimension 3: 如何失效？
┌─────────────────────────────────────────────────────────────┐
│  失效策略        适用场景          精确度     实现复杂度     │
│  ────────        ────────          ────      ────────      │
│  TTL (时间)      静态知识、工具结果   低        低            │
│  Semantic Diff   对话上下文         中        中            │
│  Event-Driven    数据变更          高        高            │
│  Manual Purge    系统维护          最高      最低           │
└─────────────────────────────────────────────────────────────┘
```

## 缓存策略决策矩阵

不同的缓存策略在效果、成本和复杂度上差异显著。以下矩阵帮助你在具体场景中选择合适策略：

| 策略 | 最佳场景 | 命中率 | 实现复杂度 | 成本节省 | 延迟改善 |
|------|----------|--------|------------|----------|----------|
| Response Cache | 完全相同的 prompt 重复请求 | 高 (70-90%) | 低 | 中 (30-50%) | 高 (ms vs s) |
| Semantic Cache | 语义相似但表述不同的请求 | 中高 (50-80%) | 中 | 高 (50-70%) | 中高 |
| Tool Result Cache | 外部 API 返回结果 | 高 (80-95%) | 低 | 中 (20-40%) | 高 |
| Embedding Cache | 同文本反复向量化 | 很高 (90%+) | 低 | 低 (5-10%) | 中 |
| Knowledge Cache | RAG 检索结果 | 中 (40-70%) | 中高 | 中 (20-40%) | 中 |
| Plan Cache | 固定工作流的推理路径 | 中低 (30-60%) | 高 | 高 (50-80%) | 高 |

**关键判断**：语义缓存的综合 ROI 通常最高，因为它直接减少了最贵的 LLM 调用。但它的实现复杂度也最高，需要引入嵌入模型和向量相似度搜索。

## Agent 缓存 vs 传统 Web 缓存

这是整个模块最重要的认知转换——Agent 缓存的思维模型与 Web 缓存完全不同：

```
维度               传统 Web 缓存                    Agent 缓存
────               ─────────────                    ──────────
缓存单元           HTML 页面 / API 响应              LLM 响应 / 工具结果 / 推理路径

缓存键             URL (精确匹配)                    Prompt 语义向量 (模糊匹配)

命中条件           完整 URL + 参数精确匹配            语义相似度超过阈值 (如 cosine > 0.92)

失效策略           TTL + 手动清除                    TTL + 语义漂移 + 上下文变化 + Event

未命中代价         一次后端请求 (通常 <100ms)         一次 LLM 调用 (通常 1-10s, 成本高 1000x+)

缓存价值           降低服务器负载                     降低 Token 成本 (直接节约 $$$)

预热方式           预生成热门页面                     预计算常见问题嵌入 + 预填充响应

一致性要求         强 (过时内容 = 错误)               弱 (过时回答 = 近似答案, 可接受)

大小评估           页面大小 × 页面数                   Token 数 × 向量维度 × 序列长度
```

**核心差异**：Web 缓存的未命中代价是一次数毫秒到数十毫秒的后端 DB 查询，而 Agent 缓存的未命中代价是一次 LLM API 调用——成本相差 3-5 个数量级。这决定了 Agent 缓存的**激进策略是合理的**：宁可缓存稍旧的数据，也要避免未命中。过时的缓存响应通常比完整的 LLM 重生成便宜得多。

## 多层级联缓存架构

生产级 Agent 系统的缓存应该是一个多层级的联合体，而不是单一缓存层。以下是一个经过验证的三层架构：

```
用户请求进入
     │
     ▼
┌──────────────────────────────────────────────┐
│  L1: In-Memory Cache (进程内)                 │
│  ┌────────────────────────────────────────┐   │
│  │  lru_cache / functools                 │   │  ← 零网络开销
│  │  Size: ~1GB, TTL: 30-60s              │   │    毫秒级响应
│  │  Key: prompt_hash                      │   │    入口级过滤
│  └──────────────┬─────────────────────────┘   │
│                 │ miss                          │
└─────────────────┼────────────────────────────┘
                  │
                  ▼
┌──────────────────────────────────────────────┐
│  L2: Distributed Cache (Redis Cluster)        │
│  ┌────────────────────────────────────────┐   │
│  │  Redis Hash + Vector Search            │   │  ← 跨实例共享
│  │  Size: ~10GB, TTL: 5-30min            │   │    精确 + 语义匹配
│  │  Key: prompt_hash / embedding_vector   │   │    批量响应缓存
│  │  Semantic: cosine_sim > 0.92           │   │
│  └──────────────┬─────────────────────────┘   │
│                 │ miss                          │
└─────────────────┼────────────────────────────┘
                  │
                  ▼
┌──────────────────────────────────────────────┐
│  L3: Persistent Cache (PostgreSQL / S3)       │
│  ┌────────────────────────────────────────┐   │
│  │  pgvector + Object Storage             │   │  ← 持久化存储
│  │  Size: 100GB+, TTL: 24h-7d            │   │    冷数据存档
│  │  Full content storage                  │   │    可离线分析
│  │  Write-through from LLM response       │   │
│  └──────────────┬─────────────────────────┘   │
│                 │ miss                          │
└─────────────────┼────────────────────────────┘
                  │
                  ▼
          ┌────────────────┐
          │  LLM API Call   │  ← 真正的成本发生点
          │  (成本极高)      │
          └───────┬────────┘
                  │ response
                  ▼
          写入所有缓存层 (Write-Through)
```

### 各级缓存的关键参数

```
L1 内存缓存:
  ├── 目的: 过滤短期内完全相同的重复请求
  ├── 命中率: ~20-30% (取决于流量模式)
  ├── 存储: functools.lru_cache 或 in-memory dict
  └── 典型代码:

      from functools import lru_cache

      @lru_cache(maxsize=10000)
      def get_llm_response(prompt: str, system: str) -> str:
          # 只在 L1 未命中时调用下游
          ...

L2 分布式缓存:
  ├── 目的: 跨实例共享精确匹配 + 语义匹配
  ├── 命中率: ~30-50% (取决于语义阈值)
  ├── 存储: Redis + redis-py
  └── 语义缓存示例:

      def semantic_cache_lookup(query: str, threshold: float = 0.92):
          query_emb = embed(query)
          # 在 Redis 中搜索相似缓存项
          results = redis.ft("cache_idx").search(
              f"@embedding:[VECTOR_RANGE {threshold}]",
              query_vector=query_emb
          )
          if results:
              return results[0].response
          return None

L3 持久缓存:
  ├── 目的: 冷数据回放 + 分析 + 模型训练数据
  ├── 命中率: ~10-20% (长尾请求)
  ├── 存储: PostgreSQL pgvector / S3
  └── 数据用于: Token 用量分析、缓存命中率优化、训练缓存模型
```

## Agent 缓存的核心挑战

### 1. 未命中代价极高

传统 Web 缓存的未命中意味着一次 DB 查询（毫秒级），Agent 缓存的未命中意味着一次 LLM 调用（秒级 + 金钱成本）。这个不对称性意味着：

```
未命中成本对比:
  Web 缓存 miss:   ~10ms  + $0.000001 (DB 查询)
  Agent 缓存 miss: ~3s    + $0.03-$0.30 (LLM 调用)
  ─────────────────────────────────────────────
  差距: 300x 延迟 + 30,000x 成本

推论:
  ┌─ 应该使用更宽松的缓存命中阈值 (宁可误中, 不要 miss)
  ├─ TTL 应该比传统缓存更长 (过时响应 < 昂贵生成)
  ├─ 缓存预热是必须的, 不是可有可无
  └─ 缓存层应该尽可能前置 (L1 优先于 L2)
```

### 2. 语义缓存键的模糊性

精确缓存 (exact match) 在 Agent 场景中只覆盖了重复请求这一小部分。大部分省钱潜力在于语义缓存，但语义缓存带来了根本性的挑战：

```
精确缓存:   prompt_hash == cached_prompt_hash  → 命中
            (简单、确定、安全)

语义缓存:   cosine_sim(embed(query), embed(cached)) > 0.92  → 命中
            (灵活, 但存在模糊边界)

边界问题:
  "Python 怎么读取 CSV 文件？"
  "如何用 pandas 加载 CSV 数据？"     → 语义相同 → 应该命中 ✓
  "Python 怎么写入 CSV 文件？"        → 语义不同 → 不应该命中 ✓
  "Python 读取 CSV 出现编码错误"      → 语义部分重叠 → 看阈值设定 ??

  ↓ 阈值 < 0.85: 高命中率但可能返回不相关的回答
  ↓ 阈值 > 0.95: 低命中率但返回准确
  ↓ 无完美阈值: 需要根据场景动态调整
```

### 3. 上下文依赖性

Agent 的 LLM 调用不是孤立的——同一个 prompt 在不同上下文中的期望输出可能不同。这使得缓存键设计更加复杂：

```
示例: "帮我总结一下"
  ┌─ 在对话开始时: 总结的是用户提供的文档？还是全局知识？
  ├─ 在对话中间: 总结的是刚才讨论的内容？还是历史消息？
  └─ 同样一句话, 但意图不同 → 缓存键必须携带上下文标识

解决方案: 缓存键 = f"{session_id}:{turn_id}:{prompt_hash}:{context_hash}"
          这显著降低了缓存命中率, 但保证了正确性
```

### 4. 冷启动问题

新部署的 Agent 系统缓存为空，前 N 个请求全部 miss，导致初始阶段的延迟和成本都不可接受。

```
冷启动阶段:
  缓存空 → 请求全部直通 LLM → 高延迟 + 高成本 → 用户体验差

缓解策略:
  ┌──────────────────────────────────────────┐
  │  策略                效果                │
  │  ──────              ────                │
  │  Seed Cache          消除冷启动           │
  │   (预填充高频Q&A)                        │
  │  Cache Warming       生产前预热           │
  │   (回放历史请求)                          │
  │  Gradual Rollout     分批放量             │
  │   (Canary Analysis)                      │
  │  P50/P95 Monitor     实时监控异常         │
  │   (慢缓存 vs 未缓存)                      │
  └──────────────────────────────────────────┘
```

## 缓存策略选择路径

不同的 Agent 类型和应用场景需要不同的缓存策略组合。以下提供针对三类典型场景的推荐配置：

```
场景 A: 客服问答 Agent (高频 + 高重复)
  ├── 核心策略: 语义缓存为主, 精确缓存为辅
  ├── 期望命中率: 70-85%
  ├── 缓存层级: L2 (Redis) + L3 (持久化用于分析)
  ├── TTL: 较长 (24h 以上, 因为知识变化慢)
  └── 预热: 基于历史问答库预填充

场景 B: 代码生成 Agent (低频 + 个性化)
  ├── 核心策略: 精确缓存 (prompt hash) + 工具结果缓存
  ├── 期望命中率: 10-30%
  ├── 缓存层级: L1 (内存) 为主, 不需要 L3
  ├── TTL: 较短 (5-10min, 代码上下文变化快)
  └── 预热: 预缓存常用库的 API 文档查询结果

场景 C: 数据分析 Agent (异构 + 工具密集型)
  ├── 核心策略: Tool Result Cache + Embedding Cache
  ├── 期望命中率: 40-60% (工具结果可复用性高)
  ├── 缓存层级: L2 (Redis) + L3 (持久存储)
  ├── TTL: 中等 (1-2h, 数据可能更新)
  └── 预热: 预缓存标准数据查询结果
```

## Agent 缓存的关键指标

衡量缓存效果需要建立完整的指标体系，而不仅仅是"命中率"：

```
核心指标:
  ├── 命中率 (Hit Ratio):       命中次数 / 总请求次数
  ├── 字节命中率 (Byte Hit):    缓存提供的 Token 数 / 总 Token 数
  ├── 成本节省率:               减少的 LLM 成本 / 总 LLM 成本
  ├── 延迟改善:                P50/P95/P99 响应延迟改善倍数
  └── 有效命中率:             命中且回答质量合格的请求比例

典型基准值 (经过优化的 Agent 缓存系统):
  ┌──────────────┬────────────┬────────────┬──────────────┐
  │ 指标          │ 差          │ 中          │ 优            │
  ├──────────────┼────────────┼────────────┼──────────────┤
  │ 整体命中率    │ < 30%      │ 30-60%     │ > 60%        │
  │ 成本节省率    │ < 20%      │ 20-50%     │ > 50%        │
  │ P50 延迟改善  │ < 2x       │ 2-10x      │ > 10x        │
  │ 有效命中率    │ < 80%      │ 80-95%     │ > 95%        │
  └──────────────┴────────────┴────────────┴──────────────┘
```

## 常见缓存反模式

```
❌ "用精确缓存就够了"
   精确缓存只覆盖完全相同的重复查询。在 Agent 场景中，
   用户同一意图的表述方式可能完全不同。
   → 必须引入语义缓存才能获得显著的成本节省。

❌ "缓存 TTL 设得越短越安全"
   短 TTL 意味着低命中率。Agent 缓存的未命中代价极高，
   过短 TTL 导致的额外 LLM 调用成本可能超过"数据过时"
   的风险。
   → 合理做法: 从长 TTL 开始, 用监控数据反向优化。

❌ "所有响应都缓存"
   有些响应是高度个性化的 (如"帮我写一封邮件给张三")，
   缓存这些响应几乎不会被复用。
   → 需要在缓存写入前做可缓存性判断。

❌ "L1/L2/L3 全部复用同一套缓存逻辑"
   每级缓存在延迟特性、容量、持久性上完全不同，
   同一策略难以适配所有层级。
   → 每级独立配置 TTL、淘汰策略和缓存键生成方式。

❌ "缓存命中率越高越好"
   过高的命中率可能意味着语义缓存阈值过于宽松，
   导致返回语义不相关的缓存结果。
   → 需要同时关注"有效命中率"——命中且回答正确的比例。
```

## 核心挑战总结

Agent 系统的缓存面临三个核心挑战：

1. **未命中代价不对称**：一次缓存未命中的成本比命中高出 3-5 个数量级，这要求缓存策略激进且前置。核心思路是"宁可给近似答案，不要付 LLM 全价"。

2. **语义不确定性**：Agent 请求的缓存键无法像 URL 一样精确匹配，语义相似度匹配引入了阈值选择困境——太宽松会返回错误答案，太严格会降低命中率。核心思路是动态阈值 + 多级回退。

3. **上下文耦合**：同一个 prompt 在不同对话上下文中需要不同的响应，限制了缓存复用。核心思路是将上下文信息编码到缓存键中，并用 L1-L2-L3 层级的 TTL 差异来平衡正确性与命中率。

后续的六个子主题将逐一深入这些维度的具体实现方案，从最简单的 Response Cache（13.3.1）开始，逐步深入到语义缓存（13.3.2）、多层架构设计（13.3.3）、缓存失效策略（13.3.4）、预热方案（13.3.5），以及分布式缓存（13.3.6）。
