# 13.5 reliability — 可靠性

**Agent 系统的可靠性问题比传统软件系统复杂一个维度：不确定性不仅来自基础设施故障（机器宕机、网络分区），更来自 LLM 自身的非确定性输出和行为多样性。** 一个 API 调用 99% 可靠的服务，包装成 5 步的 Agent 流程后，整体成功率可能只有 ~95%；而每一步中 LLM 的 5% 输出偏差率，会让 10 步 Agent 在 40% 的情况下出现至少一次异常。更关键的是，Agent 的故障模式与传统应用完全不同——LLM 不会"崩溃"，但它会"悄悄做错"：选择错误的工具、推理出错误结论、陷入无限循环。因此，Agent 可靠性工程的核心任务是构建一套从基础设施到行为层的**多级弹性体系**。

## Agent 可靠性 vs 传统系统可靠性

Agent 系统的故障模型与传统软件有本质差异。理解这些差异是设计可靠性策略的前提。

```
故障类型对比:

传统系统故障                        Agent 系统故障
─────────────────                   ─────────────────
• 进程崩溃 (Crash)                  • LLM 输出偏差 (语义错误)
• 网络分区 (Network Partition)       • 工具选择错误 (逻辑错误)
• 资源耗尽 (OOM/CPU 100%)           • 推理链断裂 (上下文丢失)
• 死锁 (Deadlock)                   • 无限循环 (Agent Loop)
• 数据损坏 (Corruption)             • 幻觉 (Hallucination)
• 超时 (Timeout)                    • 成本爆炸 (Cost Explosion)

可用性视角:
  ┌──────────────────────────────────────────────┐
  │ 传统: "服务是否在响应？"   → 二进制判断      │
  │  Agent: "输出了正确结果吗？" → 多维语义判断   │
  └──────────────────────────────────────────────┘
```

### 不确定性放大效应

Agent 的一个核心可靠性挑战在于不确定性是**乘性放大**的：

```
单步成功率模型:

  假设每一步成功概率: P_success = 0.95
  假设每一步输出正确概率: P_correct = 0.90

  5 步 Agent 流程:
    ├── 技术成功率: 0.95^5 = 0.774 (22.6% 概率至少一步技术失败)
    └── 语义正确率: 0.90^5 = 0.590 (41% 概率至少一步语义偏差)

  10 步 Agent 流程:
    ├── 技术成功率: 0.95^10 = 0.599 (40% 概率至少一步技术失败)
    └── 语义正确率: 0.90^10 = 0.349 (65% 概率至少一步语义偏差)

  核心洞察:
    ┌─ 步骤越多, 整体可靠性下降越快
    ├─ 语义错误率通常 > 技术错误率 (LLM 输出的不确定性 > API 调用的不确定性)
    ├─ 修复"语义错误"比重试"技术错误"困难得多
    └─ 每步都做验证 (Verification) 是可靠性设计的核心
```

"5 个 9" (99.999%) 可靠性在 Agent 系统中意味着什么？

```
传统 99.999% = 每年 5.26 分钟宕机
Agent 99.999% = 每 10 万次 Agent 执行中 < 1 次重大语义偏差

    传统可靠性衡量: 服务 UP/DOWN
    Agent 可靠性衡量: 服务正常 + 输出正确 + 成本可控 + 行为合规
```

## Agent 可靠性金字塔

可靠性设计需要在五个层面同时建立防护，自底向上形成金字塔结构：

```
                                    ╱╲
                                   ╱  ╲             行为层
                                  ╱    ╲            ──────
                                 ╱ 语义  ╲          幻觉检测、输出验证
                                ╱ 正确性  ╲         推理路径校验、事实对齐
                               ╱──────────╲
                              ╱              ╲     逻辑层
                             ╱    Agent       ╲    ──────
                            ╱     逻辑        ╲    循环检测、状态一致性
                           ╱                  ╲   工具调用验证、重试编排
                          ╱──────────────────────╲
                         ╱                        ╲  通信层
                        ╱     通信与                ╲  ──────
                       ╱    编排层                 ╲  超时控制、熔断器
                      ╱                            ╲  重试策略、背压
                     ╱──────────────────────────────╲
                    ╱                                ╲  资源层
                   ╱        资源隔离                  ╲  ──────
                  ╱         与分配                    ╲  隔舱模式、限流
                 ╱                                    ╲  配额管理、优先级
                ╱──────────────────────────────────────╲
               ╱                                        ╲  基础设施层
              ╱           基础设施                       ╲  ──────
             ╱            弹性                          ╲  多 AZ 部署
            ╱                                          ╲  自动扩缩容
           ╱──────────────────────────────────────────────╲
           └────────────────────────────────────────────────┘

  每层解决不同的问题:
  ┌─ 基础设施层: 机器不会挂 (传统 SRE)
  ├─ 资源层:      一个 Tenant 不会拖垮其他 Tenant
  ├─ 通信层:      调用不会无限等待或级联失败
  ├─ 逻辑层:      Agent 不会陷入死循环或状态混乱
  └─ 语义层:      输出内容本身是正确的 (Agent 特有!)
```

## 可靠性模式的分类与适用场景

不同的 Agent 场景需要不同的可靠性模式组合：

| 模式 | 解决的问题 | 适用场景 | 副作用 | 实现复杂度 |
|------|-----------|----------|--------|-----------|
| Retry | 瞬态故障 (网络抖动、LLM 限流) | 所有场景 | 增加延迟、放大成本 | 低 |
| Degradation | 持续故障时保持基本可用 | 高可用场景 | 功能受限的用户体验 | 中 |
| Circuit Breaker | 防止级联故障 | 跨服务/工具调用 | 需要降级兜底 | 中 |
| Bulkhead | 资源隔离避免全局影响 | 多租户/多任务 | 资源利用率降低 | 低 |
| Timeout | 防止无限等待 | 所有场景 | 可能过早终止长任务 | 低 |
| Chaos Engineering | 验证弹性设计有效性 | 生产级系统 | 可能引发真实故障 | 极高 |

### 模式组合方式

实际生产中这些模式是组合使用的。以下是已验证的组合模式：

## Agent 特有可靠性挑战

### 1. "静默失败" (Silent Failure)

传统系统失败时通常抛异常或返回错误码，但 Agent 失败时可能**看起来正常但事实错误**——

```
传统系统:
  DB 查询失败 → 抛 DatabaseError → 调用方明确知道失败

Agent 系统:
  LLM 生成错误结论 → 返回格式良好的回答 → 调用方认为成功
  但内容事实是错误的 (幻觉) 或推理是错误的 (逻辑断裂)

这就是"静默失败"——系统功能正常但结果错误。
传统监控 (HTTP 200 / 延迟 / 错误率) 完全检测不到。
```

**应对方案**：引入语义验证层，对 Agent 的输出做事实性校验（见 12.2 语义护栏）。

### 2. "循环陷阱" (Agent Loop)

Agent 陷入 Thought → Action → Observation 的无限循环，既不完成任务也不报错：

```
Agent Loop 示例:

  Thought: 我需要查询天气 → Action: get_weather()
  Observation: 返回北京天气数据
  Thought: 数据返回了, 但我还需要验证 → Action: get_weather() 又调用一次
  Observation: 返回同样的数据
  Thought: 两次结果一致, 但用户问的是"明天", 我需要再查 → Action: get_weather()
  ... 无限循环直到 Token 预算用尽
```

**应对方案**：
- 最大步骤数限制（硬性终止条件）
- 重复 Action 检测（相同工具+相同参数连续 N 次时中断）
- 循环深度自适应（每一步增加"循环检测"提示）

### 3. "成本爆炸" (Cost Explosion)

Agent 行为失控导致 Token 消耗指数级增长——

```
正常请求: 1 次 LLM 调用 + 2 次工具调用 = ~$0.05
失控请求: 15 次 LLM 调用 + 30 次工具调用 = ~$0.80 (16x 成本)

放大因素:
  ┌─ LLM 输出错误 → 重试 (2x)
  ├─ 重试再次错误 → 不同的错误模式 → 更多重试 (4x)
  ├─ Agent 进入循环 → 无限的 LLM 调用 (∞)
  └─ 上下文持续膨胀 → 每次调用成本递增 (O(n²))
```

**应对方案**：
- 每次调用的 max_tokens 限制
- 步骤级和会话级 Token 预算双重控制
- 成本异常检测（单次请求成本超过 3σ 时自动终止）

### 4. "异常传播" (Anomaly Propagation)

传统系统的异常通常局限于当前请求上下文，但 Agent 的异常可以通过记忆系统**跨请求传播**：

```
请求 A: Agent 执行出错, 将错误结论写入记忆
请求 B: Agent 从记忆中检索到请求 A 的错误结论作为参考
请求 C: Agent 基于 A 和 B 的错误信息做出进一步的错误决策

异常从"单次故障"变成"系统性偏差"——类似数据污染。
```

**应对方案**：
- 记忆写入前验证（只确认正确的信息进入长期记忆）
- 记忆溯源标记（每条记忆记录来源请求 ID）
- 异常记忆清除机制（发现错误时级联删除受污染的记忆）

### 5. "级联降质" (Cascading Quality Degradation)

多个 Agent 依次处理时，每个 Agent 引入的微小偏差会被后续步骤逐级放大：

```
Agent 1: 用户意图解析 (准确率 95%)
  → 输出传递给 Agent 2
Agent 2: 任务规划 (基于 Agent 1 的解析, 准确率 90%)
  → 输出传递给 Agent 3
Agent 3: 执行任务 (基于 Agent 2 的计划, 准确率 90%)
  → 最终准确率: 0.95 × 0.90 × 0.90 = 0.77 (23% 概率异常)

对于 5 级流水线: 0.95^5 = 0.77 (23% 异常)
对于 10 级流水线: 0.90^10 = 0.35 (65% 异常!)
```

**应对方案**：
- 每级 Agent 输出校验（校验通过才能传递给下一级）
- 压缩-验证循环（长流水线中定期做事实性对齐）
- 关键路径冗余（对决策节点使用多模型投票）

## 可靠性策略选择路径

不同阶段和场景需要不同的可靠性策略重心：

```
场景 A: MVP 验证期
  ├── 核心: 基本不崩溃
  ├── Retry (简单指数退避)
  ├── Timeout (防止无限循环)
  ├── 最大步骤数限制
  └── 简单的降级 (返回错误信息给用户)

场景 B: 增长期 (日均 1K-100K 请求)
  ├── 核心: 保证结果准确性
  ├── Circuit Breaker (工具调用层级)
  ├── Bulkhead (按 Tenant/任务类型隔离)
  ├── 语义验证层 (输出事实性校验)
  ├── 循环检测 (重复 Action 识别)
  └── 降级策略分层 (缓存 → 简化 → 人工)

场景 C: 规模化期 (日均 > 100K 请求)
  ├── 核心: 系统性弹性
  ├── Chaos Engineering (持续验证)
  ├── 自适应策略 (动态超时/重试/熔断阈值)
  ├── 跨 Agent 可靠性编排
  ├── 自动故障恢复 + 根因分析
  └── 可靠性 SLO 管理与错误预算
```

## 关键指标

衡量 Agent 系统可靠性需要比传统 SRE 更丰富的指标体系：

```
核心可靠性指标:
  ├── 任务成功率 (Task Success Rate):   Agent 成功完成任务的百分比    ← 核心 KPI
  ├── 语义正确率 (Semantic Accuracy):   输出内容事实正确的比例          ← Agent 特有
  ├── 技术成功率 (Technical Success):   无基础设施异常的请求比例
  ├── 零错误执行率:                     无重试、无降级、一次成功的比例
  ├── 循环发生率:                       Agent 陷入循环的请求比例
  ├── 平均 MTTR:                        从故障发现到恢复的平均时间
  └── 错误预算消耗率:                   可用错误预算的消耗速度

典型基准值 (生产级 Agent 系统):
  ┌─────────────────────┬──────────┬──────────┬──────────┐
  │ 指标                 │ 入门级    │ 标准级    │ 优秀级    │
  ├─────────────────────┼──────────┼──────────┼──────────┤
  │ 任务成功率           │ > 80%    │ > 90%    │ > 97%    │
  │ 语义正确率           │ > 85%    │ > 93%    │ > 97%    │
  │ 技术成功率           │ > 95%    │ > 99%    │ > 99.9%  │
  │ 循环发生率           │ < 5%     │ < 1%     │ < 0.1%   │
  │ 平均无人工干预率      │ > 70%    │ > 85%    │ > 95%    │
  └─────────────────────┴──────────┴──────────┴──────────┘
```

## 常见可靠性反模式

```
❌ "重试次数越多越可靠"
   无限重试 = 无限成本 + 无限延迟。在 Agent 系统中，
   LLM 调用重试需要 3-5 步渐进: 瞬时重试 → 退避重试 → 换模型 → 放弃。
   每次重试都增加 Token 成本和延迟。

❌ "Agent 和普通微服务用同一套重试策略"
   Agent 的故障模式 (5%-10% 语义偏差) 与微服务 (0.1% 网络抖动)
   完全不同。微服务重试 3 次能解决绝大多数瞬态故障，
   Agent 重试 3 次不一定能纠正 LLM 输出偏差。

❌ "超时设长一点更安全"
   超时太长 → Agent 陷入循环时成本爆炸
   超时太短 → 复杂任务被过早终止
   需要动态超时: 基于任务复杂度、历史执行时间、当前负载调整。

❌ "降级就是报错"
   降级不是简单地返回"系统繁忙"。好的降级提供替代方案:
   "我无法查询实时数据, 但这是昨天缓存的结果, 供参考。"

❌ "混沌工程只适合大公司"
   即使是小型 Agent 项目, 也可以做"故障注入单元测试"——
   测试 LLM 返回空值、工具超时、网络中断时的行为。
```

## 核心挑战总结

Agent 可靠性最大的挑战不是"如何让系统不崩溃"——传统 SRE 已经很擅长这个——而是**如何在不确定性的输出中保证结果的可靠**。这要求可靠性策略从"二进制思维"（成功/失败）转变为"连续光谱思维"（完全正确 → 部分正确 → 降级可接受 → 失败），并为之设计多级回退路径。

后续六个子主题将逐一深入每种可靠性模式的原理、实现和最佳实践：从最基础的重试模式（13.5.1）开始，逐步深入到降级策略（13.5.2）、熔断器（13.5.3）、隔舱模式（13.5.4）、超时控制（13.5.5），最后到系统性验证弹性的混沌工程（13.5.6）。
