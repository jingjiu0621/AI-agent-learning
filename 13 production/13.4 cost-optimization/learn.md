# 13.4 cost-optimization — 成本优化

**Agent 系统的成本结构与传统 Web 服务有本质差异：成本的大头不再是服务器和带宽，而是 LLM API 的 Token 消耗。** 一次 Agent 完整交互可能调用 3-10 次 LLM API，产生数万到数十万 Token，单次成本从几美分到几美元不等。在没有成本控制的情况下，一个日均 10 万请求的 Agent 服务每月 LLM 成本可能高达数十万美元。成本优化的核心目标是在**不显著降低 Agent 能力**的前提下，将每任务 Token 消耗降到最低。

## Agent 成本的三层结构

Agent 系统的成本呈现金字塔结构，不同层级的优化手段差异显著：

```
                    ╱╲
                   ╱  ╲                  LLM API 调用    占总成本 70-90%
                  ╱    ╲                 ────────────    ────────────────
                 ╱ LLM  ╲                Input Token     60-70%（含 System Prompt + 历史 + 工具描述）
                ╱  API   ╲               Output Token    20-30%（含思考链 + 最终回答 + 工具调用）
               ╱  Cost    ╲              Cache Miss      5-15%（语义缓存未命中导致的全价调用）
              ╱────────────╲
             ╱              ╲           基础设施         占总成本 10-20%
            ╱ Infrastructure ╲           ────────────    ────────────────
           ╱                  ╲          GPU/CPU 服务     40-50%（模型推理服务器）
          ╱     Cost          ╲         Vector DB        15-25%（索引存储 + 查询）
         ╱──────────────────────╲       Redis / Cache    10-15%
        ╱                        ╲     网络 / 带宽       5-10%
       ╱   Human & Operational    ╲    对象存储          10-15%（日志 + 轨迹）
      ╱         Cost              ╲
     ╱──────────────────────────────╲  人力运营         占总成本 5-15%
                                      ────────────    ────────────────
                                      Prompt 工程师     35-45%
                                      SRE / 运维        30-40%
                                      标注 / 评估        15-25%
```

**核心结论**：LLM API 费用是成本的大头，而 Input Token 又是 LLM 费用的大头。这决定了优化优先级：**先压输入、再管模型、最后补基础设施**。

## 成本优化的六维框架

```
Dimension 1: Reduce — 减少每次调用的 Token 量
  ├── 13.4.1 Token 压缩 (精简 Prompt)
  ├── 消息历史压缩 (滑动窗口 + 摘要)
  └── 工具描述精简 (只传递相关工具)

Dimension 2: Route — 让合适模型做合适的事
  ├── 13.4.2 模型路由 (大/小模型分层)
  ├── Fallback 策略 (高峰降级到便宜模型)
  └── Prompt 复杂度分类器

Dimension 3: Cache — 避免重复计算
  ├── 13.4.3 缓存命中率优化 (精确 + 语义)
  ├── 13.3 全模块缓存策略
  └── Provider 侧缓存 (Prompt Caching)

Dimension 4: Batch — 合并分摊成本
  ├── 13.4.4 批处理优化
  ├── Request 合并 (相似请求一起发)
  └── 异步结果轮询

Dimension 5: Budget — 量化与控制
  ├── 13.4.5 成本预算与告警
  ├── 每用户/每 Tenant 配额
  └── 成本归因与 Chargeback

Dimension 6: Distill — 自建低成本替代
  ├── 13.4.6 模型蒸馏
  ├── 专有小模型微调
  └── 本地部署替代 API 调用
```

## 成本优化决策矩阵

不同优化手段在效果、风险、实现难度上差异巨大：

| 优化手段 | 降低成本 | 影响 latency | 实现复杂度 | 风险 | 适用阶段 |
|----------|----------|-------------|------------|------|----------|
| System Prompt 精简 | 5-15% | 轻微改善 | 低 | 低 | 立即可做 |
| 消息历史压缩 | 20-40% | 轻微改善 | 中 | 中（可能丢失上下文） | 持续优化 |
| 工具描述精简 | 10-25% | 轻微改善 | 中 | 中（影响工具选择准确率） | 持续优化 |
| 模型路由 | 30-60% | 取决于分桶 | 高 | 中（路由错误可能降级质量） | 需充分测试 |
| 语义缓存 | 40-70% | 显著改善 | 高 | 中（过时缓存的准确性） | 流量稳定后 |
| 批处理 | 10-20% | 增加延迟 | 中 | 低 | 高吞吐场景 |
| 模型蒸馏 | 50-80% | 显著改善 | 极高 | 高（能力可能不足） | 长期投入 |
| Prompt Caching (Provider) | 10-50% | 轻微改善 | 低 | 低 | 立即可做 |

**关键判断**：
- **立即可做的**：精简 System Prompt + 启用 Provider Prompt Caching，这两项几乎零成本
- **ROI 最高的**：模型路由 + 消息历史压缩，通常是最快看到显著成本降低的组合
- **长期投入的**：模型蒸馏，前期投入大但边际成本最低

## Agent 成本优化的核心挑战

### 1. 质量-成本权衡是最核心的矛盾

Agent 成本优化的本质是在**质量**和**成本**之间找到最佳平衡点：

```
质量-成本帕累托前沿 (Pareto Frontier):

  高 ▲
     │           ● 最佳质量 (GPT-4, 全量上下文)
     │         ●   ↑
  质  │       ●     │ 可行区域
  量  │     ●       │ (减小上下文以换取成本)
     │   ●         │
     │●            ↓
     └──────────────────────────►
       低          成本          高

  阴影区域: 不可行 (质量和成本都不占优)
  ●: 帕累托最优解 (改善一个维度必然损害另一个维度)

  目标: 找到应用场景在帕累托前沿上的"膝盖点"
        —— 再降低成本就会显著影响质量的位置
```

### 2. 成本与延迟的耦合

Agent 的成本优化往往会增加延迟。例如批处理（等更多请求收集完才能发起）和模型路由（需要先做复杂度分类）。

```
            成本 ←───────────────→ 延迟
            ↓                      ↑
  优化手段:                     带来的代价:
  ├── 压缩 Prompt             →  更少的上下文 = 更少的思考依据
  ├── 批处理                  →  等待批次填满 = 额外延迟
  ├── 模型路由                →  分类开销 + 降级风险
  └── 语义缓存                →  向量搜索 + 阈值判断
```

### 3. 成本的可预测性差

传统基础设施成本相对稳定（固定实例费），而 LLM 成本取决于用户输入的 Token 量和任务复杂度，波动范围可达 10 倍。

```
成本波动来源:
  ┌─ 用户输入长度 (从 10 Token 到 100K Token)
  ├─ 任务复杂度 (简单问答 vs 多步推理)
  ├─ 工具调用次数 (0 到 10+ 次)
  ├─ 重试与纠错 (失败 → 重试 → 成本 2-3x)
  └─ 缓存命中率 (冷启动 vs 稳定态, 差异 3x+)
```

## 成本优化的三条黄金法则

1. **不降质量的原则**：任何成本优化措施都必须有质量护栏。制定明确的退化指标（如任务成功率下降 > 2% 则回滚），所有的成本优化 A/B 测试必须同时跟踪成本和质量。

2. **先测量再优化**：没有精确的成本归因，优化就是盲目的。建立每个请求、每个用户、每个工具的 Token 消耗追踪，找到"热点"再下手。常见的 80/20 分布：20% 的请求消耗 80% 的 Token。

3. **逐层递进**：从低成本、低风险的优化开始，用节省下来的钱来资助高成本、高收益的优化。典型顺序：Prompt 精简 → Provider 缓存 → 模型路由 → 语义缓存 → 蒸馏。

## 成本优化的灰度策略

成本优化是有风险的——压缩 Prompt 可能降低质量，模型路由可能选错模型。切忌一次性全量上线：

```
Phase 1: 可观测性 (第 1 周)
  └── 引入成本追踪，建立基准线

Phase 2: 低风险优化 (第 2 周)
  ├── System Prompt 精简 (A/B 测试 10% 流量)
  ├── 启用 Provider Prompt Caching (100%)
  └── 消息历史压缩 (A/B 测试 5% → 50%)

Phase 3: 中风险优化 (第 3-4 周)
  ├── 模型路由 (从最简单的二分类开始: 简单/复杂)
  ├── 语义缓存 (先对高频知识类问题启用)
  └── 批处理 (从非实时场景开始)

Phase 4: 高风险优化 (第 5-8 周)
  └── 模型蒸馏 (长期项目, 需要标注 + 训练 + 评估)

Phase 5: 持续迭代
  ├── 成本预算自动化
  ├── 动态路由优化
  └── 缓存策略自适应
```

## 关键指标

衡量成本优化效果需要建立多维度的指标看板：

```
核心指标:
  ├── 每任务成本 (Cost per Task):       完成任务的平均 LLM 费用          ← 核心 KPI
  ├── 每 Token 成本效率:                输出 Token / 输入 Token 比值    ← 衡量利用率
  ├── 缓存节省率:                      缓存命中节省的 Token / 总 Token
  ├── 路由效率:                        小模型处理的比例 (目标 > 60%)
  └── 成本-质量比:                    成本变化率 / 质量指标变化率       ← 优化健康度

典型基准值:
  ┌─────────────────────┬──────────┬──────────┬──────────┐
  │ 指标                 │ 未优化    │ 中等优化  │ 深度优化  │
  ├─────────────────────┼──────────┼──────────┼──────────┤
  │ 每任务成本 (GPT-4)   │ $0.50    │ $0.20    │ $0.08    │
  │ 缓存节省率           │ < 10%    │ 30-50%   │ > 60%    │
  │ 小模型占比           │ < 10%    │ 40-60%   │ > 70%    │
  │ Token 利用率         │ < 20%    │ 30-50%   │ > 50%    │
  │ 质量保持率           │ 100%     │ 98-99%   │ 95-98%   │
  └─────────────────────┴──────────┴──────────┴──────────┘
```

## 常见成本优化误区

```
❌ "只要缓存够多就能省钱"
   缓存只能处理完全相同的请求或高度相似的语义。
   对于个性化、上下文依赖强的 Agent 场景，
   缓存命中率天然受限。需要综合运用多种手段。

❌ "全用小模型能省很多钱"
   小模型在简单分类、格式化任务上表现良好，
   但在复杂推理、代码生成、多步规划上可能大幅降级。
   无差别的降级会导致用户体验下降，最终用户流失。

❌ "Prompt 压缩到最短就是最优"
   过短的 Prompt 可能导致 LLM 理解偏差，需要更多轮次纠错，
   反而增加了总体 Token 消耗和延迟。
   核心是减少"浪费的 Token"而非"所有的 Token"。

❌ "成本优化一次到位"
   LLM 价格在持续下降（GPT-4 在 2023-2025 年间降价 10x+），
   新的优化技术也不断涌现。成本优化是一个持续过程。
```

## 优化策略选择路径

不同阶段的 Agent 项目需要不同的成本优化重心：

```
场景 A: 初创 / 验证期 (日均 < 1000 请求)
  ├── 核心: 立即可做、零成本
  ├── System Prompt 精简
  ├── 启用 Provider Prompt Caching
  └── 基础消息历史滑动窗口

场景 B: 增长期 (日均 1K-100K 请求)
  ├── 核心: 系统性优化
  ├── 模型路由 (简单/复杂分桶)
  ├── 语义缓存 (针对高频知识查询)
  ├── 消息历史压缩 (摘要 + 滑动窗口)
  └── 成本监控与预算告警

场景 C: 规模化期 (日均 > 100K 请求)
  ├── 核心: 极限优化
  ├── 模型蒸馏 (专有小模型)
  ├── 动态路由 (基于实时质量反馈)
  ├── 多层缓存架构
  └── 成本自动化 (弹性预算 + 自动降级)
```
