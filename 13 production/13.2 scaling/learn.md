# 13.2 scaling — 扩展策略

**Agent 系统的扩展与传统 Web 服务有本质不同：瓶颈不在 CPU/内存，而在 LLM API 延迟、Token 成本和状态管理。** 一个生产级 Agent 从单用户扩展到百万用户时，会经历三个不同的瓶颈阶段——每个阶段面临的约束和优化方向完全不同。理解这些阶段，才能在正确的时间选择正确的扩展策略。

## Agent 扩展的三个阶段

Agent 系统从单用户到大规模生产，会经历三个截然不同的扩展阶段：

```
阶段                   瓶颈                     典型量级
────                    ────                      ────
Phase 1: 单实例扩展     LLM API 速率限制            1-100 用户
  (Scale Up)          单机资源上限

Phase 2: 水平扩展       状态管理复杂化              100-10K 用户
  (Scale Out)         数据一致性挑战

Phase 3: 规模化运营     Token 成本控制             10K+ 用户
  (Scale Smart)       多层级联延迟
                       系统可观测性
```

### 阶段一：Scale Up（单实例扩展，1-100 用户）

**这个阶段不需要分布式系统**——一个经过优化的单实例就能处理数百并发 Agent 会话。

```
典型配置:
┌────────────────────────────────────┐
│  1 台服务器 (4-8 CPU, 16-32GB RAM)  │
│                                    │
│  ┌──────────────────────────────┐  │
│  │  asyncio 事件循环             │  │  500+ 并发协程
│  │  ├── Agent Session 1 ── LLM  │  │
│  │  ├── Agent Session 2 ── LLM  │  │
│  │  ├── Agent Session 3 ── LLM  │  │  LLM API 调用是 I/O 密集的
│  │  └── ...                     │  │  等待响应时不占 CPU
│  └──────────────────────────────┘  │
│                                    │
│  瓶颈: LLM API 速率限制             │
│  优化: 连接复用、请求合并、语义缓存     │
└────────────────────────────────────┘
```

### 阶段二：Scale Out（水平扩展，100-10K 用户）

当单实例无法承载时，需要多实例水平扩展。**核心挑战是状态管理。**

```
                        ┌──────────────┐
                        │  负载均衡器    │
                        │  (Session    │
                        │   Affinity)  │
                        └──────┬───────┘
                               │
          ┌────────────────────┼────────────────────┐
          │                    │                    │
          ▼                    ▼                    ▼
   ┌──────────────┐   ┌──────────────┐   ┌──────────────┐
   │  Agent Pod 1  │   │  Agent Pod 2  │   │  Agent Pod N  │
   │  (Session A)  │   │  (Session B)  │   │  (Session C)  │
   └──────┬───────┘   └──────┬───────┘   └──────┬───────┘
          │                  │                  │
          └──────────────────┼──────────────────┘
                             │
                    ┌────────▼────────┐
                    │   Redis Cluster  │ ← 共享状态层
                    │  (会话/记忆/缓存) │
                    └─────────────────┘
```

### 阶段三：Scale Smart（规模化运营，10K+ 用户）

当用户量达到数万时，单纯扩展节点数量不再经济。需要智能优化。

```
核心目标：降低每用户的 Token 成本和延迟
策略：
├── 语义缓存: 相似问题命中缓存 (减少 50-70% LLM 调用)
├── 模型路由: 简单问题用小模型、复杂问题用大模型
├── 请求合并: 相同上下文的多请求批处理
├── 预测性预取: 预判用户下一步需求，提前加载
└── Agent 专用 CDN: 边缘节点缓存常用 Agent 响应
```

## Agent 扩展与传统扩展的核心区别

```
维度               传统 Web 服务               Agent 系统
────               ─────────────               ──────────
瓶颈                CPU/内存/IO                LLM API 延迟 + Token 成本

状态               无状态优先                   天然有状态

请求特征           短 (ms 级)                  长 (秒到分钟级)

负载特征           均匀 / 可预测                突发性强 (LLM 响应时间波动)

扩展信号           CPU > 80%                   队列深度 > 阈值

资源消耗           处理请求时消耗               等待时也消耗 (上下文保持)

降级策略           返回 503                     返回缓存 / 降级回答

A/B 测试           功能开关                     语义质量对比
```

## Agent 扩展的量化基准

### Token 预算模型

每个生产环境应该有明确的 Token 预算，指导扩展决策：

```
用户量       日均请求     每请求 Token    日均 Token     月成本 (GPT-4o)
──────       ────────     ──────────     ──────────     ──────────────
100          1,000        5,000          5M             ~$75
1,000        10,000       5,000          50M            ~$750
10,000       100,000      5,000          500M           ~$7,500
100,000      1,000,000    5,000          5B             ~$75,000

扩展不只是加机器，还需要梯队模型策略：
├── 80% 请求由 Haiku/GPT-4o-mini 处理 → 成本降至 1/10
├── 15% 请求由 Sonnet/GPT-4o 处理
└── 5% 请求由 Opus 处理
```

### 并发与资源估算

```
估算公式: 
  所需实例数 = (用户数 × QPS × 平均请求时长) / 单实例并发能力

Agent 特有修正:
  • 平均请求时长 = LLM 响应时间 + 工具执行时间 + 记忆检索时间
  • LLM 响应时间波动大 (P50: 1s, P95: 5s, P99: 15s)
  • 建议按 P75 延迟估算，留 50% 余量

示例:
  10,000 用户，每个用户每小时 2 次请求
  → 5.5 QPS
  → 平均请求时长 3s (LLM 1.5s + 工具 1s + 记忆 0.5s)
  → 每个 Pod 处理 50 并发 (asyncio)
  → 所需 Pods = ceiling(5.5 × 3 / 50) = 1 个 Pod
  → 考虑 HA + 余量 = 2-3 个 Pod

但这里的关键是：Agent 请求不是独立均匀的。
  一个用户在一次对话中可能连续发送 5 条消息，
  每条需要 3s，总耗时 15s。
  → 实际活跃用户并发数 = 用户数 × 平均对话并发因子
  → 对话并发因子通常 0.1-0.3
```

## 扩展策略选择指南

```
你的 Agent 当前处在哪个阶段？

▶ 1-100 用户 (Phase 1: Scale Up)
  └── 策略: 优化单实例
  └── 关注: asyncio 并发、API 速率限制、语义缓存
  └── 不需要: K8s、Redis 集群、自动扩缩容
  
▶ 100-10K 用户 (Phase 2: Scale Out)
  └── 策略: 水平扩展 + 共享状态
  └── 关注: Session Affinity、Redis 状态存储、KEDA 自动扩缩
  └── 不需要: 多区域部署、成本优化到极致

▶ 10K+ 用户 (Phase 3: Scale Smart)
  └── 策略: 分层架构 + 智能路由
  └── 关注: 模型路由、多级缓存、成本分析、全球部署
  └── 需要: 完整的可观测性 + 成本控制
```

### 常见扩展反模式

```
❌ "预先搞个微服务架构"
   99% 的 Agent 项目死在微服务的复杂度上。单体->水平拆分->微服务，逐步演进。

❌ "用 ThreadPoolExecutor 处理 Agent 请求"
   Agent 请求是 I/O 密集的，asyncio 比线程更高效、更可控。

❌ "CPU > 80% 就自动扩缩容"
   Agent Pod 可能在等待 LLM 响应时 CPU 接近 0，但队列已经堆满。
   
❌ "所有 Agent 共用一个 API Key"
   共享 API Key 导致全局限流，一个 Agent 的突发流量可能阻塞所有其他 Agent。

❌ "用数据库做会话状态"
   Agent 的会话状态读写频率极高，Redis 或内存是正确选择，数据库只做持久化。
```

## 核心挑战总结

Agent 系统的扩展面临三个核心挑战：

1. **状态与弹性矛盾**：Agent 天然有状态，而弹性扩展需要无状态。解决方案是外部化状态（Redis + 检查点）+ Session Affinity。

2. **延迟非线性**：LLM API 的延迟波动远超传统服务，P99/P50 比值可达 10x+，需要队列缓冲和自适应超时。

3. **成本随规模非线性增长**：Token 消耗随上下文增长和推理步数增加而膨胀，需要多层缓存 + 模型路由来控制。
