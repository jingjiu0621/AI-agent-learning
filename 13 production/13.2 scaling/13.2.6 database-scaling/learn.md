# 数据库扩展 (Database Scaling) —— AI Agent 系统的存储层扩展

## 概述

本文档聚焦于生产环境中 AI Agent 系统的**存储后端扩展**，特别是 Agent 记忆（Memory）与知识（Knowledge）存储层的扩展策略。与传统的 Web 应用不同，Agent 系统具有独特的数据存储画像 —— 没有任何单一数据库能高效满足所有需求。

---

## 1. 背景与问题

### 1.1 Agent 系统的"多语言持久化"现实

生产级 AI Agent 系统通常需要**三种截然不同的存储后端**：

```
+------------------------------------------------------------------+
|                     AI Agent 存储架构全景                           |
+------------------------------------------------------------------+
|                                                                    |
|  用户请求                                                          |
|     |                                                              |
|     v                                                              |
|  +--------+     +------------------+     +-------------------+    |
|  |  Agent  |---->|   Session State   |---->|   Redis Cluster    |  |
|  |  Runtime |     |  (会话状态)        |     |  (<5ms, 高吞吐)    |  |
|  +--------+     +------------------+     +-------------------+    |
|     |                                                              |
|     |--------->|   Long-term Memory  |---->|   Vector DB         |  |
|     |           |  (长期记忆)          |     |  (嵌入向量, 语义搜索) |  |
|     |                                                              |
|     |--------->|   Structured Know.  |---->|   SQL / Graph DB    |  |
|                 |  (结构化知识)        |     |  (实体关系, 用户画像) |  |
|                 +------------------+     +-------------------+    |
+------------------------------------------------------------------+
      存储类型         访问模式          典型延迟          扩展瓶颈
  ──────────────  ──────────────  ──────────────  ──────────────
  Session State   高频率读写       <5ms             内存容量
  Long-term Mem.  语义搜索读取     10-50ms           索引内存 & 召回率
  Structured Know. 事务性查询      20-100ms          数据量 & 连接数
```

### 1.2 为什么单一 PostgreSQL 无法满足 Agent 存储需求？

PostgreSQL 是优秀的通用数据库，但在 Agent 场景下面临三重矛盾：

| 存储需求 | PG 的能力 | 差距 |
|---------|----------|------|
| **向量搜索** (Agent 记忆检索) | pgvector 扩展，IVFFlat/HNSW 索引 | 超过 1000 万向量后性能急剧下降；索引构建期间阻塞写入；过滤条件与向量搜索组合时性能差 |
| **会话状态** (KV 存储) | 行存储，事务开销 | 每次会话读取需经过 SQL 解析、计划、执行全链路，延迟比 Redis 高 10-100 倍 |
| **高频写入** (流式记忆更新) | WAL 写入 + MVCC 开销 | 高并发写入导致 checkpoint 压力、 vacuum 膨胀、IO 瓶颈 |

> **核心问题**：Agent 系统需要同时在**毫秒级延迟**（会话状态）、**语义检索**（记忆）和**事务一致性**（知识）这三个维度上达到生产要求 —— 没有任何单数据库能做到。

### 1.3 三个扩展轴

```
扩展轴 1：读吞吐量 (Read Throughput)
  低  ●━━━━━━━━━━━━━━━━━━━━● 高
      单节点读        Redis 副本读    多区域多副本
      PG 主库             扩展        全球分布式

扩展轴 2：写吞吐量 (Write Throughput)
  低  ●━━━━━━━━━━━━━━━━━━━━● 高
      单主写入         分片写入      无冲突 CRDT
      PG 主库         (sharding)     (Dragonfly)

扩展轴 3：数据体量 (Data Volume)
  低  ●━━━━━━━━━━━━━━━━━━━━● 高
      GB 级           TB 级           PB 级
      单节点向量库      分片向量库     分层存储 + 量化
```

### 1.4 Agent 记忆的独特扩展特性

Agent 的记忆规模不仅随**用户数**增长，还随**单用户对话深度**和**记忆保留策略**变化：

```
单用户记忆量 ≈ 平均对话轮次 × 每轮 Token 数 × 嵌入维度 × 保留比例

示例计算：
  10K 用户 × 100 轮对话/用户 × 4K token/轮 × 768 维嵌入
  = 10,000 × 100 × 768 × 4 字节 (float32)
  = ~3TB 原始向量数据
  + HNSW 索引开销 (~1.5×) = ~4.5TB 总存储

加上对话历史文本：
  10,000 × 100 × 4K token × 2 字节/char ≈ 8GB (可压缩至 2-3GB)
```

## 2. 之前是怎么做的

### 2.1 单一 SQL 数据库方案

早期 Agent 系统将所有数据存入单个 PostgreSQL/MySQL：

```
+---------------------+
|    Agent Runtime     |
+---------------------+
          |
          v
+---------------------+
|   PostgreSQL (单库)   |
+---------------------+
|  - conversation      |  ← 文本对话历史
|  - session           |  ← 会话状态 (JSONB)
|  - user_memory       |  ← 记忆嵌入 (pgvector)
|  - user_profile      |  ← 用户画像
|  - task_queue        |  ← 任务队列
+---------------------+
```

**问题清单**：

| 问题 | 描述 | 影响 |
|------|------|------|
| 模式刚性 | 会话状态是动态的，JSONB 查询效率低 | 延迟 > 50ms |
| 向量搜索差 | pgvector 在 > 500万向量时索引构建慢 | 召回率下降，写入阻塞 |
| 连接耗尽 | Agent 实例数 * 连接池 > 数据库 max_connections | 连接拒绝 |
| 高延迟 | SQL 解析 + 事务开销对会话读取太重 | 不必要地增加了 10-20ms |
| 无原生 HA | 需要 Patroni / repmgr 等外部工具 | 运维复杂度高 |

### 2.2 Redis-only 方案

一些早期系统尝试将所有状态放入 Redis：

```
+---------------------+
|    Agent Runtime     |
+---------------------+
          |
          v
+---------------------+
|   Redis (单节点)      |
+---------------------+
|  session:xxx        |  ← 会话状态
|  memory:xxx         |  ← 记忆向量 (Redisearch)
|  history:xxx        |  ← 对话历史 (List)
|  user:xxx           |  ← 用户数据 (Hash)
+---------------------+
```

**问题清单**：

| 问题 | 描述 | 影响 |
|------|------|------|
| 数据丢失风险 | 未配置 AOF/RDB 时宕机丢数据 | 用户对话丢失 |
| 有限查询 | Redisearch 在 >100万向量时准确率低 | 记忆检索效果差 |
| 内存成本 | 所有数据在内存，TB 级场景成本爆炸 | 10TB 内存 ≈ 每月 $30K+ |
| 无复杂查询 | 无法做 JOIN、聚合、事务性更新 | 知识图谱无法实现 |
| 单节点瓶颈 | 单线程事件循环，多核利用率低 | CPU 瓶颈 |

### 2.3 单节点向量数据库

```
+---------------------+          +---------------------+
|    Agent Runtime     |--------->|   Milvus/Qdrant     |
+---------------------+          |   (单节点)           |
                                 |   collection/user    |
                                 |   1M-10M 向量        |
                                 +---------------------+
```

**瓶颈表现**：

- 超过 500 万向量后，查询延迟从 10ms 上升到 100ms+
- 索引重建期间（HNSW 构建）无法服务查询
- 无副本，单点故障
- 无水平扩展能力，达到内存上限后只能迁移到更大实例

### 2.4 手工分片对话历史

```
+---------------------+
|    Agent Runtime     |
+---------------------+
     |    |    |
     v    v    v
+----+  +----+  +----+
|shard1| |shard2| |shard3|  ← 文件名: history_{user_id % 3}.json
+----+  +----+  +----+
```

**问题清单**：
- 跨分片查询需要应用层聚合
- 重新分片需要停机迁移数据
- 无数据一致性保证
- 文件损坏恢复困难
- 备份策略混乱

## 3. 核心矛盾

### 3.1 向量索引的非线性扩展

高维向量的索引结构存在严重的非线性扩展特性：

```
HNSW 索引内存 = O(N × d × 4 + N × M × 4 × 2)

其中:
  N = 向量数量
  d = 向量维度 (常见 768 / 1024 / 1536)
  M = HNSW 图的最大连接数 (M=32 是常见配置)
  4 = float32 字节数

实际案例：
                   向量数      维度      HNSW 内存    查询延迟
                   ─────      ───      ─────────    ──────
                   100万       768        ~8 GB       5-10ms
                   1000万      768        ~80 GB      15-30ms
                   1亿         768       ~800 GB      50-150ms
                   10亿        768       ~8 TB        >500ms (需分片)

关键发现：
  HNSW 内存增长是 O(N) 但不是线性 ——
  图连通性的维护需要额外边，实际内存约为 N × d × 4 × 1.3~1.5
```

### 3.2 会话状态的矛盾需求

```
需求 A: 超低延迟 (<5ms)
  → 需要内存存储，最好在应用进程内
  → Agent 每次工具调用都要读取会话状态

需求 B: 持久性 (不丢数据)
  → 需要磁盘持久化，最好同步写入
  → Agent 崩溃后恢复会话上下文

矛盾：内存 vs 磁盘、速度 vs 持久化
           ↑
      ┌────┴────┐
      │  妥协方案  │
      └─────────┘
      ├─ 异步 AOF (每秒 fsync)
      ├─ 主从复制 + 快速故障转移
      └─ 会话压缩 + 写回策略
```

### 3.3 知识图谱的写入瓶颈

知识图谱（Graph）在 Agent 系统中用于存储实体关系和推理路径，但其写入性能是显著的瓶颈：

```
典型写入操作（单次 Agent 记忆更新）：

1. 创建实体:         INSERT INTO nodes (id, type, properties)
2. 创建关联:         INSERT INTO edges (source, target, relation)
3. 更新时序属性:     UPDATE node_properties SET value = $2 WHERE ...
4. 更新图索引:       REINDEX (如果使用图数据库)
5. 级联更新:         UPDATE related_nodes SET ... （可能波及数百节点）

单次 Agent 记忆写入涉及 3-10 条 SQL/图查询
1000 并发 Agent → 3000-10000 TPS 写入 → 事务冲突 → 延迟飙升
```

### 3.4 向量数据库的成本爆炸

```
10M 向量 (768维) 的月度成本对比：

┌──────────────────────┬──────────┬──────────┬──────────┐
│       方案           │  存储成本  │  计算成本  │  总成本/月 │
├──────────────────────┼──────────┼──────────┼──────────┤
│ Pinecone P2 (托管)   │  $0.10/GB │  $0.50/h │  ~$8,000  │
│ Qdrant Cloud         │  $0.08/GB │  $0.40/h │  ~$6,500  │
│ 自建 Milvus (3节点)   │  $0.04/GB │  $0.80/h │  ~$5,800  │
│ PGVector (RDS)       │  $0.10/GB │  $0.30/h │  ~$4,000  │
│ 自建 Qdrant + 量化    │  $0.02/GB │  $0.40/h │  ~$2,500  │
└──────────────────────┴──────────┴──────────┴──────────┘

100M 向量时，托管服务的月成本可达 $50K-$80K
这推动了对量化和分层存储的需求
```

## 4. 当前主流方案

### 4.1 会话状态扩展 (Session State Scaling)

#### 4.1.1 Redis Cluster 架构

Redis Cluster 使用基于哈希槽的分片机制，16384 个槽均匀分布在集群节点上：

```
+------------------------------------------------------------------+
|                      Redis Cluster 拓扑                              |
+------------------------------------------------------------------+
|                                                                      |
|   Agent 实例 1        Agent 实例 2        Agent 实例 3              |
|     |                   |                   |                        |
|     +--------+----------+--------+----------+                        |
|              |                   |                                   |
|              v                   v                                   |
|     +----------------+  +----------------+  +----------------+       |
|     |  Redis Node 1   |  |  Redis Node 2   |  |  Redis Node 3   |   |
|     |  slots: 0-5460   |  |  slots: 5461-10922 |  slots: 10923-16383 |
|     |  +-----------+   |  |  +-----------+   |  |  +-----------+   |
|     |  | Master A  |   |  |  | Master B  |   |  |  | Master C  |   |
|     |  +-----------+   |  |  +-----------+   |  |  +-----------+   |
|     |  | Replica A1|   |  |  | Replica B1|   |  |  | Replica C1|   |
|     |  +-----------+   |  |  +-----------+   |  |  +-----------+   |
|     +----------------+  +----------------+  +----------------+       |
|                                                                      |
|   Cluster Bus (端口 +10000, 点对点 Gossip 协议)                     |
+------------------------------------------------------------------+
```

**哈希槽路由原理**：

```
key = "session:user_12345:conv_67890"
slot = CRC16(key) % 16384
     = CRC16("session:user_12345:conv_67890") % 16384
     = 4281  → 路由到 Node 1 (slots 0-5460)

Agent 客户端通过 MOVED/ASK 重定向或 Cluster 代理找到正确节点
```

**会话亲和性 (Session Affinity)**：

```
+---------------------+
|   Load Balancer      |
+---------------------+
     |         |
     v         v
+----------+  +----------+
| Agent A   |  | Agent B   |  ← 同一用户的请求始终路由到同一 Agent
+----------+  +----------+     (减少跨节点会话读取)
     |         |
     +----+----+
          |
          v
+---------------------+
|   Redis Cluster      |  ← 会话数据固定在特定分片上
+---------------------+
```

**什么时候使用 Session Affinity + Replica Reads**：
- 读多写少的 Agent 场景（记忆检索 >> 记忆更新）
- 对最终一致性可接受的场景
- replica-read 可能读到过期数据 (<= 几毫秒的滞后)

#### 4.1.2 Redis 持久化策略权衡

```
┌──────────────────────────────────────────────────────────────┐
│                    Redis 持久化选项对比                         │
├──────────────┬──────────────┬──────────────┬──────────────────┤
│   策略        │ RDB (快照)    │ AOF (追加写)  │ RDB + AOF (混合)  │
├──────────────┼──────────────┼──────────────┼──────────────────┤
│ 数据完整性    │ 可能丢失最后一次  │ 每秒/每次都保证 │ 最可靠           │
│              │ 快照后的数据    │              │                  │
├──────────────┼──────────────┼──────────────┼──────────────────┤
│ 恢复速度      │ 快 (秒级)      │ 慢 (可能数分钟)│ 较快             │
├──────────────┼──────────────┼──────────────┼──────────────────┤
│ 写入性能影响  │ 低 (fork 子进程)│ 中等 (append) │ 中等             │
├──────────────┼──────────────┼──────────────┼──────────────────┤
│ 磁盘空间      │ 小 (压缩)      │ 大 (原始命令)  │ 中等             │
├──────────────┼──────────────┼──────────────┼──────────────────┤
│ Agent 推荐配置│ 跨会话场景      │ 关键会话场景   │ 生产环境默认       │
│              │ (可接受少量丢失) │ (不可丢数据)   │                  │
└──────────────┴──────────────┴──────────────┴──────────────────┘

生产推荐配置：
  save 900 1          # 900秒内至少1次变更 → RDB快照
  save 300 10         # 300秒内至少10次变更 → RDB快照
  save 60 10000       # 60秒内至少10000次变更 → RDB快照
  appendonly yes      # 开启 AOF
  appendfsync everysec # 每秒 fsync（性能与持久性的平衡）
  aof-use-rdb-preamble yes  # 混合持久化
```

#### 4.1.4 会话内存超限处理

当 Redis 内存达到 `maxmemory` 限制时，触发键驱逐策略：

```
maxmemory 32gb
maxmemory-policy allkeys-lru   # 或 volatile-lru

对 Agent 系统的影响：
──────────────────────

allkeys-lru:
  - Redis 会驱逐最久未使用的会话键
  - 长时间不活跃的 Agent 会话被驱逐
  - 用户回来时需要重新建立会话上下文
  - 方案：客户端缓存最近使用的会话 ID，提前预热

volatile-lru (设置 TTL 的键):
  - 只驱逐设置了过期时间的键
  - 会话键设置 TTL = 30min，记忆键不设 TTL
  - Agent 短期会话被驱逐，长期记忆保留
  - 更符合 Agent 使用模式

会话压缩（减少内存占用）:
  原始会话: 1024KB JSON (完整对话历史 + 工具调用记录)
  压缩后:   410KB (使用 zstd 压缩，压缩率 ~60%)
  
  适用场景：
  - 长时间不间断对话（>100 轮）
  - 携带大量工具调用结果的会话
  - 需要快速重建但不频繁访问的会话
```

### 4.2 向量数据库扩展 —— 最难的问题

#### 4.2.1 分片策略

```
┌──────────────────────────────────────────────────────────────────┐
│                    向量数据库分片策略对比                            │
├──────────────────────┬───────────────────────────────────────────┤
│       策略            │             适用场景                        │
├──────────────────────┼───────────────────────────────────────────┤
│ 按租户分片            │ 多租户 Agent 平台，每个租户的数据天然隔离       │
│ (partition_by_tenant) │ 优点：隔离性好，一个租户的索引重建不影响其他   │
│                      │ 缺点：租户间数据倾斜，小租户浪费资源           │
├──────────────────────┼───────────────────────────────────────────┤
│ 按文档集合分片         │ Agent 有多个知识库，每个知识库独立分片         │
│ (partition_by_collection) │ 优点：不同领域的记忆独立管理               │
│                      │ 优点：可按知识库大小分配不同资源               │
├──────────────────────┼───────────────────────────────────────────┤
│ 按嵌入命名空间分片      │ Agent 记忆按类型划分：短期记忆、长期记忆、技能  │
│ (partition_by_namespace)│ 每种记忆有不同的访问频率和生命周期            │
│                      │ 优点：可为不同命名空间配置不同的索引和资源       │
├──────────────────────┼───────────────────────────────────────────┤
│ 按向量 ID 哈希分片     │ 均匀分布，适合无法天然分片的场景               │
│ (hash_based)         │ 缺点：跨分片查询需要合并结果，精确度有损         │
└──────────────────────┴───────────────────────────────────────────┘
```

#### 4.2.2 HNSW 索引扩展的物理限制

```
HNSW (Hierarchical Navigable Small World) 索引是当前最流行的 ANN 算法之一。

┌──────────────────────────────────────────────────────────────────┐
│                    HNSW 参数与扩展影响                              │
├─────────────┬──────────┬──────────────┬──────────────────────────┤
│   参数       │  典型值   │  对性能影响     │  扩展时的考虑               │
├─────────────┼──────────┼──────────────┼──────────────────────────┤
│ M           │ 16-64    │  大 M → 高召回 │  M=32 时每向量需要          │
│ (每个节点的   │          │  大 M → 高内存 │  M×4bytes×2 = 256B 额外内存 │
│  最大连接数)  │          │  大 M → 较慢插入 │  10M 向量 → 2.56GB 图结构 │
├─────────────┼──────────┼──────────────┼──────────────────────────┤
│ ef_construction │ 100-500  │  大 → 更好的图质量 │  构建时间 O(N·logN)     │
│ (构建时的    │          │  大 → 构建更慢  │  ef=500 时构建时间是       │
│  动态列表大小)│          │              │  ef=100 的 3-5 倍          │
├─────────────┼──────────┼──────────────┼──────────────────────────┤
│ ef_search   │ 10-100   │  大 → 更高召回  │  召回率从 0.90 到 0.99     │
│ (搜索时的    │          │  大 → 更高延迟  │  延迟从 5ms 到 50ms        │
│  动态列表大小)│          │              │                          │
└─────────────┴──────────┴──────────────┴──────────────────────────┘

10M 向量 (768维) HNSW 内存占用明细：

  向量数据:       10,000,000 × 768 × 4 bytes = 28.6 GB (float32)
  HNSW 图:        10,000,000 × 32 × 4 × 2   =  2.6 GB (M=32, 双向边)
  原始向量:                                     28.6 GB (必须加载到内存)
  层级结构:       约 10% 额外                   3.1 GB
  ─────────────────────────────────────────────────────
  总计:                                         34.3 GB

如果没有 34GB 连续可用内存：
  → 必须使用 IVF (倒排索引) 将部分数据放在磁盘
  → 或使用量化 (PQ/SQ) 压缩向量
```

#### 4.2.3 IVF + PQ：更大规模的向量搜索

当 HNSW 所需内存超出可用资源时，IVF (Inverted File Index) + PQ (Product Quantization) 是替代方案：

```
IVF 原理示意图：

  向量空间                        Voronoi 图划分
  ┌──────────────────┐           ┌──────────────────┐
  │  .  .  .  .  .   │           │  +  +  |  x  x   │
  │  .  ●  .  .  .   │  聚类     │  +  +  ───  x  x  │
  │  .  .  .  .  .   │  ──────>  │  +  +  |  x  x   │
  │  .  .  .  ●  .   │   (K=4)   │  ──────────────────│
  │  .  .  .  .  .   │           │  o  o  |  △  △   │
  └──────────────────┘           │  o  o  ───  △  △  │
                                  │  o  o  |  △  △   │
                                  └──────────────────┘
  搜索: 只在最近的 1-2 个聚类内搜索
  复杂度: O(N/K + K) 而不是 O(N)

量化压缩：
  原始向量: [0.23, -0.45, 0.67, ..., 0.12]  ×768 (float32) = 3072 bytes
      ↓ 分割为 M 个子向量
  子向量1: [0.23, -0.45, ...]  × (d/M)
      ↓ 每个子向量量化到 Nbits
  量化后: [42, 127, 3, ..., 98]  × M (uint8) = M bytes
      ↓ M=96 时: 每个向量 96 bytes (压缩率 32x)
```

**IVF + PQ 与 HNSW 对比**：

```
┌───────────────────┬────────────────┬──────────────────────────┐
│     特性           │   HNSW (内存)   │   IVF + PQ (混合)        │
├───────────────────┼────────────────┼──────────────────────────┤
│ 10M 向量内存占用   │  ~34 GB        │  ~2-4 GB (压缩后)        │
│ 100M 向量可行性     │  需要 340GB     │  需要 20-40GB            │
│ 查询延迟           │  5-15ms        │  20-50ms                │
│ 召回率 (@10)       │  0.95-0.99     │  0.85-0.95              │
│ 写入速度           │  中等          │  快 (只需更新倒排索引)     │
│ 索引构建时间        │  小时级        │  分钟级                  │
│ 是否支持 >内存数据   │  否           │  是 (部分在磁盘)          │
│ Agent 推荐场景      │ 低延迟推理      │ 大规模记忆库              │
└───────────────────┴────────────────┴──────────────────────────┘
```

**跨分片查询的合并策略**：

```
用户查询向量 q (768维)

1. 广播 (Fan-out):
   请求 → Node 1 (shard 0):  search(q, top_k=20)
        → Node 2 (shard 1):  search(q, top_k=20)
        → Node 3 (shard 2):  search(q, top_k=20)

2. 每个分片返回各自的 top-20 结果:
   Node 1: [{id: 42, score: 0.95}, {id: 17, score: 0.91}, ...]
   Node 2: [{id: 88, score: 0.89}, {id: 35, score: 0.87}, ...]
   Node 3: [{id: 13, score: 0.93}, {id: 56, score: 0.90}, ...]

3. 协调节点合并:
   合并 3×20 = 60 个结果
   按 score 降序排列
   取 top-10

4. 精度问题:
   如果每个分片只返回 top-k，可能漏掉某个分片的 top-(k+1)
   实际中 top-k = 用户请求的 k × (分片数 × 2) 以获得足够召回
```

#### 4.3.2 水平分片 (Horizontal Sharding)

```
分片键选择: user_id % N 或 hash(user_id) % N

+------------------------------------------------------------------+
|                     应用层分片代理                                    |
|                                                                      |
|  分片路由函数: shard_id = user_id_hash % 8                           |
|                                                                      |
|  +----------------+  +----------------+  +----------------+         |
|  | Shard 0        |  | Shard 1        |  | ...  | Shard 7 |         |
|  | user_id % 8 = 0|  | user_id % 8 = 1|  |      |         |         |
|  +----------------+  +----------------+  +----------------+         |
|  | conversations   |  | conversations   |  | conversations   |       |
|  | memories        |  | memories        |  | memories        |       |
|  | user_profiles   |  | user_profiles   |  | user_profiles   |      |
|  +----------------+  +----------------+  +----------------+         |
|                                                                      |
|  每个分片内部可有自己的读写分离                                       |
+------------------------------------------------------------------+

分片原则:
  - 按 user_id 分片: Agent 数据天然按用户隔离
  - 每个用户的完整数据在同一分片 (避免跨分片 JOIN)
  - 分片数最好是 2 的幂，方便后续加倍
  - 热点用户: 单独分到一个分片

分片缺点:
  - 跨分片查询复杂 (需要应用层聚合)
  - 重新分片需要大量迁移
  - 不能使用数据库外键
```

#### 4.4.3 时序记忆数据库 (MemoriesDB / Temporal Memory)

```
Agent 记忆的时序模型：

  时间
   ↑
   |  记忆事件流
   |  [t5] 用户说"我喜欢蓝色" → 更新偏好
   |  [t4] Agent 完成任务 X   → 存储成果
   |  [t3] 工具调用返回错误    → 经验教训
   |  [t2] 用户问"我的订单"   → 对话上下文
   |  [t1] 会话开始           → 初始状态
   +------------------------------→ 语义空间

核心概念:
  - 每个记忆是时间线上的一个点 (temporal point)
  - 记忆检索 = 在时间和语义两个维度上做近邻搜索
  - 旧记忆自动衰减权重，但不会立即删除
  - 记忆合并: 相似记忆自动聚类压缩

与传统向量 DB 的差异:
  传统: 最近邻搜索 (只考虑语义相似度)
  时序: 最近邻 + 最近时 (语义 + 时间衰减)

  查询: "用户对颜色的偏好"
  传统: 返回所有颜色相关的记忆 (包括过时的)
  时序: 返回最新的颜色偏好 (时间加权)
```

```
|     | ← 返回缓存结果   | 缓存未命中                                    |
|     |                 v                                               |
|     |           +-----------+                                         |
|     |           |  Agent    |  ← 调用 Agent 处理                      |
|     |           |  Runtime  |                                         |
|     |           +-----------+                                         |
|     |                 |                                               |
|     |                 v                                               |
|     |           +-----------+                                         |
|     |           |  存储结果   |  ← 写入缓存 + 设置 TTL                 |
|     |           +-----------+                                         |
|     |                 |                                               |
|     +--- 返回结果 ----+                                               |
|                                                                      |
| 缓存策略:                                                              |
|  - 完全相同的请求: 缓存命中 (TTL=5min)                                 |
|  - 语义相似的请求: 近似缓存 (缓存 Agent 对相似问题的答案)               |
|  - Agent 主动失效: 当 Agent 学习新知识时清除相关缓存                   |
|                                                                      |
| 效果:                                                                 |
|  - 常见问题缓存命中率可达 40-60%                                       |
|  - Agent 调用成本降低 40-60%                                          |
|  - 响应延迟从 2-5s 降低到 50-100ms                                    |
```

### 5.1 Redis 会话存储 —— 故障转移处理

```python
import redis
import redis.cluster
import pickle
import zstd
import time
import logging
from typing import Optional, Any, Dict
from dataclasses import dataclass

logger = logging.getLogger(__name__)


@dataclass
class RedisSessionConfig:
    """Redis 会话存储配置"""
    # 主集群配置
    startup_nodes: list = None
    # 哨兵配置 (备选)
    sentinel_hosts: list = None
    sentinel_master_name: str = "mymaster"
    # 连接池
    max_connections: int = 50
    socket_timeout: int = 5
    socket_connect_timeout: int = 3
    retry_on_timeout: bool = True
    # 会话
    session_ttl: int = 1800  # 30分钟
    enable_compression: bool = True
    compression_threshold: int = 1024  # 1KB 以上才压缩


class RedisSessionStore:
    """
    生产级 Redis 会话存储

    特性:
    - 集群模式自动发现
    - 哨兵自动故障转移
    - 会话压缩 (zstd)
    - 带退避的重试机制
    - 健康检查与降级
    """

    def __init__(self, config: RedisSessionConfig):
        self.config = config
        self._client: Optional[redis.Redis] = None
        self._cluster_client: Optional[redis.cluster.RedisCluster] = None
        self._sentinel_client: Optional[redis.sentinel.Sentinel] = None
        self._mode: str = "direct"  # direct | cluster | sentinel
        self._connect()

    def _connect(self):
        """建立连接（自动探测模式）"""
        last_error = None

        # 尝试集群模式
        if self.config.startup_nodes:
            try:
                self._cluster_client = redis.cluster.RedisCluster(
                    startup_nodes=self.config.startup_nodes,
                    max_connections_per_node=self.config.max_connections,
                    socket_timeout=self.config.socket_timeout,
                    socket_connect_timeout=self.config.socket_connect_timeout,
                    retry_on_timeout=self.config.retry_on_timeout,
                    cluster_error_retry_attempts=3,
                )
                self._cluster_client.ping()
                self._mode = "cluster"
                logger.info("Connected to Redis Cluster mode")
                return
            except Exception as e:
                last_error = e
                logger.warning(f"Cluster mode failed: {e}, trying sentinel...")

        # 尝试哨兵模式
        if self.config.sentinel_hosts:
            try:
                self._sentinel_client = redis.sentinel.Sentinel(
                    self.config.sentinel_hosts,
                    socket_timeout=self.config.socket_timeout,
                )
                master = self._sentinel_client.master_for(
                    self.config.sentinel_master_name,
                    socket_timeout=self.config.socket_timeout,
                )
                master.ping()
                self._client = master
                self._mode = "sentinel"
                logger.info("Connected to Redis Sentinel mode")
                return
            except Exception as e:
                last_error = e
                logger.warning(f"Sentinel mode failed: {e}")

        # 直接连接 (兜底)
        try:
            self._client = redis.Redis(
                host="localhost",
                port=6379,
                socket_timeout=self.config.socket_timeout,
            )
            self._client.ping()
            self._mode = "direct"
            logger.info("Connected to Redis direct mode")
        except Exception as e:
            last_error = e
            raise ConnectionError(f"All Redis connection modes failed: {last_error}")

    def _get_client(self):
        """获取当前可用客户端"""
        if self._mode == "cluster":
            return self._cluster_client
        return self._client

    def _serialize(self, data: Dict[str, Any]) -> bytes:
        """序列化并压缩会话数据"""
        raw = pickle.dumps(data, protocol=pickle.HIGHEST_PROTOCOL)
        if self.config.enable_compression and len(raw) > self.config.compression_threshold:
            compressed = zstd.compress(raw, level=3)
            logger.debug(f"Session compressed: {len(raw)} → {len(compressed)} bytes")
            return b"z:" + compressed
        return b"r:" + raw

    def _deserialize(self, data: bytes) -> Dict[str, Any]:
        """解压并反序列化"""
        if not data:
            return {}
        if data.startswith(b"z:"):
            decompressed = zstd.decompress(data[2:])
            return pickle.loads(decompressed)
        elif data.startswith(b"r:"):
            return pickle.loads(data[2:])
        return pickle.loads(data)

    def save_session(
        self,
        session_id: str,
        data: Dict[str, Any],
        ttl: Optional[int] = None,
    ) -> bool:
        """保存会话数据"""
        client = self._get_client()
        if client is None:
            logger.error("No Redis client available")
            return False

        key = f"session:{session_id}"
        ttl = ttl or self.config.session_ttl

        for attempt in range(3):
            try:
                serialized = self._serialize(data)
                client.setex(key, ttl, serialized)
                return True
            except (redis.ConnectionError, redis.TimeoutError) as e:
                wait = 0.1 * (2 ** attempt)  # 指数退避: 0.1s, 0.2s, 0.4s
                logger.warning(f"Redis save failed (attempt {attempt+1}): {e}")
                time.sleep(wait)
                if attempt == 2:
                    self._reconnect()
        return False

    def load_session(self, session_id: str) -> Optional[Dict[str, Any]]:
        """加载会话数据"""
        client = self._get_client()
        if client is None:
            return None

        key = f"session:{session_id}"
        for attempt in range(3):
            try:
                data = client.get(key)
                if data is None:
                    return None
                # 刷新 TTL (滑动过期)
                client.expire(key, self.config.session_ttl)
                return self._deserialize(data)
            except (redis.ConnectionError, redis.TimeoutError) as e:
                wait = 0.05 * (2 ** attempt)
                logger.warning(f"Redis load failed (attempt {attempt+1}): {e}")
                time.sleep(wait)
                if attempt == 2:
                    self._reconnect()
        return None

    def delete_session(self, session_id: str) -> bool:
        """删除会话"""
        client = self._get_client()
        if client is None:
            return False
        try:
            client.delete(f"session:{session_id}")
            return True
        except redis.RedisError as e:
            logger.error(f"Redis delete failed: {e}")
            return False

    def health_check(self) -> Dict[str, Any]:
        """健康检查"""
        client = self._get_client()
        result = {"mode": self._mode, "connected": False}
        if client:
            try:
                info = client.info()
                result["connected"] = True
                result["used_memory_human"] = info.get("used_memory_human")
                result["total_connections_received"] = info.get("total_connections_received")
                result["uptime_in_seconds"] = info.get("uptime_in_seconds")
            except redis.RedisError:
                result["connected"] = False
        return result

    def _reconnect(self):
        """重新连接"""
        logger.info("Reconnecting to Redis...")
        try:
            self._connect()
        except Exception as e:
            logger.error(f"Reconnection failed: {e}")
```

### 5.2 多分片向量搜索客户端

```python
import asyncio
import httpx
import numpy as np
from typing import List, Dict, Tuple, Optional
from dataclasses import dataclass, field
import logging

logger = logging.getLogger(__name__)


@dataclass
class ShardConfig:
    """分片配置"""
    host: str
    port: int
    weight: float = 1.0  # 查询权重 (用于不均匀分片)


@dataclass
class SearchResult:
    """搜索结果"""
    id: str
    score: float
    payload: Dict = field(default_factory=dict)
    shard_id: str = ""


@dataclass
class VectorSearchConfig:
    """向量搜索配置"""
    shards: List[ShardConfig] = field(default_factory=list)
    dimension: int = 768
    top_k_per_shard: int = 40  # 每个分片返回 top-k (大于最终需要的 k)
    final_top_k: int = 10
    timeout_ms: int = 5000
    max_retries: int = 2
    concurrency: int = 4  # 同时查询的分片数


class DistributedVectorSearch:
    """
    多分片向量搜索客户端

    特性:
    - 并发查询所有分片
    - 结果合并与重排序
    - 分片级故障降级 (部分分片失败仍可返回)
    - 加权分片支持
    - 健康检查与自动降级
    """

    def __init__(self, config: VectorSearchConfig):
        self.config = config
        self._client = httpx.AsyncClient(
            timeout=httpx.Timeout(config.timeout_ms / 1000),
            limits=httpx.Limits(max_connections=config.concurrency * 2),
        )
        self._healthy_shards: List[ShardConfig] = list(config.shards)

    async def search(
        self,
        vector: List[float],
        filters: Optional[Dict] = None,
        namespace: Optional[str] = None,
    ) -> Tuple[List[SearchResult], Dict]:
        """
        多分片向量搜索

        Args:
            vector: 查询向量 (768维)
            filters: 元数据过滤条件
            namespace: 命名空间 (用于按命名空间分片)

        Returns:
            (合并后的结果列表, 搜索统计信息)
        """
        # 确定要查询的分片 (如有命名空间过滤)
        target_shards = self._select_shards(namespace)

        # 并发查询所有分片
        semaphore = asyncio.Semaphore(self.config.concurrency)
        async def query_shard(shard: ShardConfig) -> Tuple[str, List[SearchResult], bool]:
            async with semaphore:
                try:
                    results = await self._query_single_shard(
                        shard, vector, filters,
                        self.config.top_k_per_shard,
                    )
                    return shard.host, results, True
                except Exception as e:
                    logger.warning(f"Shard {shard.host}:{shard.port} failed: {e}")
                    return shard.host, [], False

        tasks = [query_shard(s) for s in target_shards]
        results = await asyncio.gather(*tasks)

        # 合并结果
        all_results: List[SearchResult] = []
        failed_shards = []

        for shard_host, shard_results, success in results:
            if success:
                all_results.extend(shard_results)
            else:
                failed_shards.append(shard_host)

        # 全局重排序 (按 score 降序)
        all_results.sort(key=lambda r: r.score, reverse=True)
        top_results = all_results[:self.config.final_top_k]

        # 返回搜索统计
        stats = {
            "total_shards": len(target_shards),
            "successful_shards": len(target_shards) - len(failed_shards),
            "failed_shards": failed_shards,
            "candidates_retrieved": len(all_results),
            "final_results": len(top_results),
        }

        return top_results, stats

    async def _query_single_shard(
        self,
        shard: ShardConfig,
        vector: List[float],
        filters: Optional[Dict],
        top_k: int,
    ) -> List[SearchResult]:
        """查询单个分片"""
        # 这里使用 Qdrant API 作为示例
        url = f"http://{shard.host}:{shard.port}/collections/agent_memory/points/search"

        payload = {
            "vector": vector,
            "limit": top_k,
            "with_payload": True,
        }
        if filters:
            payload["filter"] = self._build_filter(filters)

        for attempt in range(self.config.max_retries + 1):
            try:
                response = await self._client.post(url, json=payload)
                response.raise_for_status()
                data = response.json()

                return [
                    SearchResult(
                        id=point["id"],
                        score=point["score"],
                        payload=point.get("payload", {}),
                        shard_id=f"{shard.host}:{shard.port}",
                    )
                    for point in data["result"]
                ]
            except (httpx.TimeoutException, httpx.HTTPStatusError) as e:
                if attempt < self.config.max_retries:
                    wait = 0.5 * (2 ** attempt)
                    logger.warning(f"Retry {attempt+1} for shard {shard.host}: {e}")
                    await asyncio.sleep(wait)
                else:
                    raise

        return []

    def _build_filter(self, filters: Dict) -> Dict:
        """构建 Qdrant 过滤条件"""
        # 示例: {"user_id": "user_123", "type": "conversation"}
        conditions = []
        for key, value in filters.items():
            conditions.append({
                "key": key,
                "match": {"value": value},
            })
        return {"must": conditions}

    def _select_shards(self, namespace: Optional[str]) -> List[ShardConfig]:
        """选择要查询的分片"""
        if namespace:
            # 按命名空间映射到分片
            shard_idx = hash(namespace) % len(self._healthy_shards)
            return [self._healthy_shards[shard_idx]]
        return self._healthy_shards

    async def health_check_all(self) -> Dict[str, bool]:
        """检查所有分片健康状态"""
        status = {}
        for shard in self.config.shards:
            try:
                url = f"http://{shard.host}:{shard.port}/healthz"
                resp = await self._client.get(url, timeout=2.0)
                is_healthy = resp.status_code == 200
                status[f"{shard.host}:{shard.port}"] = is_healthy
            except Exception:
                status[f"{shard.host}:{shard.port}"] = False

        # 更新健康分片列表
        self._healthy_shards = [
            s for s in self.config.shards
            if status.get(f"{s.host}:{s.port}", False)
        ]
        # 如果全不可用，使用所有分片 (兜底)
        if not self._healthy_shards:
            self._healthy_shards = list(self.config.shards)

        return status
```

### 5.3 CQRS 模式实现

```python
import asyncio
import asyncpg
import json
from typing import Optional, Dict, List, Any
from datetime import datetime, timedelta
from dataclasses import dataclass, field
import logging

logger = logging.getLogger(__name__)


@dataclass
class CQRSConfig:
    """CQRS 配置"""
    write_dsn: str  # 写库连接字符串
    read_dsn: str   # 读库连接字符串
    min_connections: int = 5
    max_connections: int = 30
    sync_interval_seconds: int = 60  # 读库同步间隔
    enable_materialized_views: bool = True


class AgentMemoryCQRS:
    """
    Agent 记忆的 CQRS 模式实现

    写模型: Agent 记忆的原始记录 (规范化, 事务性)
    读模型: 预聚合的记忆视图 (反规范化, 查询优化)
    """

    def __init__(self, config: CQRSConfig):
        self.config = config
        self._write_pool: Optional[asyncpg.Pool] = None
        self._read_pool: Optional[asyncpg.Pool] = None
        self._sync_task: Optional[asyncio.Task] = None

    async def start(self):
        """初始化连接池"""
        self._write_pool = await asyncpg.create_pool(
            dsn=self.config.write_dsn,
            min_size=self.config.min_connections,
            max_size=self.config.max_connections,
            command_timeout=30,
        )
        self._read_pool = await asyncpg.create_pool(
            dsn=self.config.read_dsn,
            min_size=self.config.min_connections * 2,  # 读库需要更多连接
            max_size=self.config.max_connections * 2,
            command_timeout=10,  # 读库超时更短
        )

        if self.config.enable_materialized_views:
            self._sync_task = asyncio.create_task(self._periodic_sync())
            logger.info(f"CQRS sync task started (interval={self.config.sync_interval_seconds}s)")

        logger.info("CQRS initialized: write pool + read pool")

    async def stop(self):
        """关闭连接池"""
        if self._sync_task:
            self._sync_task.cancel()
        if self._write_pool:
            await self._write_pool.close()
        if self._read_pool:
            await self._read_pool.close()

    # ========== WRITE MODEL ==========

    async def record_conversation_turn(
        self,
        user_id: str,
        session_id: str,
        turn_number: int,
        user_message: str,
        agent_response: str,
        tool_calls: List[Dict],
        metadata: Dict = None,
    ) -> int:
        """
        写入: 记录一次对话轮次 (写模型)

        这是原始的、完整的对话记录，用于审计和重建。
        优化: 批量写入，使用预备语句。
        """
        async with self._write_pool.acquire() as conn:
            row_id = await conn.fetchval("""
                INSERT INTO conversation_turns (
                    user_id, session_id, turn_number,
                    user_message, agent_response, tool_calls,
                    metadata, created_at
                ) VALUES ($1, $2, $3, $4, $5, $6, $7, NOW())
                RETURNING id
            """,
                user_id, session_id, turn_number,
                user_message, agent_response,
                json.dumps(tool_calls) if tool_calls else "[]",
                json.dumps(metadata) if metadata else "{}",
            )
            return row_id

    async def update_user_profile(
        self,
        user_id: str,
        updates: Dict[str, Any],
    ) -> bool:
        """
        写入: 更新用户画像 (写模型)

        使用 UPSERT 模式，保留历史变更。
        """
        async with self._write_pool.acquire() as conn:
            # 先获取现有画像
            existing = await conn.fetchval("""
                SELECT profile FROM user_profiles
                WHERE user_id = $1
            """, user_id)

            if existing:
                profile = json.loads(existing)
            else:
                profile = {}

            # 合并更新
            profile.update(updates)
            profile["_updated_at"] = datetime.utcnow().isoformat()

            # UPSERT
            await conn.execute("""
                INSERT INTO user_profiles (user_id, profile, updated_at)
                VALUES ($1, $2, NOW())
                ON CONFLICT (user_id)
                DO UPDATE SET profile = $2, updated_at = NOW()
            """, user_id, json.dumps(profile))

            return True

    async def store_memory_extraction(
        self,
        user_id: str,
        source_turn_id: int,
        memory_type: str,  # 'fact', 'preference', 'relationship'
        content: Dict[str, Any],
        importance: float = 0.5,
    ) -> int:
        """
        写入: 存储从对话中提取的记忆

        memory_type:
          - fact: 提取的事实信息 (用户生日: 1990-01-01)
          - preference: 用户偏好 (喜欢蓝色主题)
          - relationship: 实体关系 (用户是管理员)
        """
        async with self._write_pool.acquire() as conn:
            row_id = await conn.fetchval("""
                INSERT INTO memory_extractions (
                    user_id, source_turn_id, memory_type,
                    content, importance, created_at
                ) VALUES ($1, $2, $3, $4, $5, NOW())
                RETURNING id
            """,
                user_id, source_turn_id, memory_type,
                json.dumps(content),
                importance,
            )
            return row_id

    # ========== READ MODEL ==========

    async def get_user_context(
        self,
        user_id: str,
        max_history: int = 20,
        include_summary: bool = True,
    ) -> Dict[str, Any]:
        """
        读取: 获取用户的完整上下文 (读模型)

        这是读模型的核心查询 —— 使用物化视图和缓存加速。
        Agent 每次处理请求前调用此方法重建上下文。
        """
        async with self._read_pool.acquire() as conn:
            # 1. 最近的对话摘要 (从物化视图读取)
            if include_summary:
                summary = await conn.fetchrow("""
                    SELECT * FROM mv_user_conversation_summary
                    WHERE user_id = $1
                """, user_id)
            else:
                summary = None

            # 2. 用户画像
            profile = await conn.fetchrow("""
                SELECT profile FROM mv_user_profiles
                WHERE user_id = $1
            """, user_id)

            # 3. 最近对话轮次 (从读副本读取)
            recent_turns = await conn.fetch("""
                SELECT turn_number, user_message, agent_response,
                       tool_calls, created_at
                FROM conversation_turns
                WHERE user_id = $1
                ORDER BY turn_number DESC
                LIMIT $2
            """, user_id, max_history)

            # 4. 重要记忆
            important_memories = await conn.fetch("""
                SELECT memory_type, content, importance, created_at
                FROM mv_important_memories
                WHERE user_id = $1
                ORDER BY importance DESC
                LIMIT 10
            """, user_id)

        return {
            "user_id": user_id,
            "profile": json.loads(profile["profile"]) if profile else {},
            "summary": dict(summary) if summary else None,
            "recent_turns": [
                {
                    "turn": t["turn_number"],
                    "user": t["user_message"][:200],  # 截断以节省 token
                    "agent": t["agent_response"][:200],
                    "tools": json.loads(t["tool_calls"]) if isinstance(t["tool_calls"], str) else t["tool_calls"],
                    "time": t["created_at"].isoformat(),
                }
                for t in reversed(recent_turns)
            ],
            "important_memories": [
                {
                    "type": m["memory_type"],
                    "content": json.loads(m["content"]) if isinstance(m["content"], str) else m["content"],
                    "importance": m["importance"],
                    "time": m["created_at"].isoformat(),
                }
                for m in important_memories
            ],
        }

    async def search_memories(
        self,
        user_id: str,
        query: str,
        memory_type: Optional[str] = None,
        limit: int = 5,
    ) -> List[Dict]:
        """
        读取: 全文搜索记忆条目 (读模型)

        这里使用读库的全文索引 (相比写库的规范表性能更好)
        """
        async with self._read_pool.acquire() as conn:
            if memory_type:
                rows = await conn.fetch("""
                    SELECT memory_type, content, importance, created_at
                    FROM memory_extractions
                    WHERE user_id = $1
                      AND memory_type = $2
                      AND content::text ILIKE $3
                    ORDER BY importance DESC
                    LIMIT $4
                """, user_id, memory_type, f"%{query}%", limit)
            else:
                rows = await conn.fetch("""
                    SELECT memory_type, content, importance, created_at
                    FROM memory_extractions
                    WHERE user_id = $1
                      AND content::text ILIKE $2
                    ORDER BY importance DESC
                    LIMIT $3
                """, user_id, f"%{query}%", limit)

            return [
                {
                    "type": r["memory_type"],
                    "content": json.loads(r["content"]),
                    "importance": r["importance"],
                    "time": r["created_at"].isoformat(),
                }
                for r in rows
            ]

    # ========== 读模型维护 ==========

    async def _refresh_materialized_views(self):
        """刷新物化视图 (异步执行)"""
        async with self._read_pool.acquire() as conn:
            try:
                await conn.execute("REFRESH MATERIALIZED VIEW CONCURRENTLY mv_user_conversation_summary")
                await conn.execute("REFRESH MATERIALIZED VIEW CONCURRENTLY mv_important_memories")
                logger.debug("Materialized views refreshed")
            except Exception as e:
                logger.error(f"Materialized view refresh failed: {e}")

    async def _periodic_sync(self):
        """定期同步读模型"""
        while True:
            try:
                await asyncio.sleep(self.config.sync_interval_seconds)
                await self._refresh_materialized_views()
            except asyncio.CancelledError:
                break
            except Exception as e:
                logger.error(f"Sync error: {e}")
```

```python
# config/connection_manager.py
# 连接管理器 —— 使用上述配置的 Python 端

import asyncio
import asyncpg
from typing import Optional, Dict, Any
import aioredis
import logging

logger = logging.getLogger(__name__)


class PoolExhaustedError(Exception):
    """连接池耗尽异常"""
    pass


class ConnectionManager:
    """
    统一连接管理器

    管理所有数据库连接池的生命周期:
    - 连接池初始化
    - 健康检查
    - 优雅关闭
    - 指标暴露
    """

    def __init__(self, config: Dict):
        self.config = config
        self._pools: Dict[str, Any] = {}
        self._health_tasks: Dict[str, asyncio.Task] = {}
        self._metrics = {
            "pool_hits": 0,
            "pool_misses": 0,
            "pool_timeouts": 0,
            "retry_attempts": 0,
            "retry_successes": 0,
            "retry_failures": 0,
        }

    async def initialize_all(self):
        """初始化所有连接池"""
        tasks = []

        if "redis" in self.config:
            tasks.append(self._init_redis())
        if "postgresql" in self.config:
            tasks.append(self._init_postgresql())
        if "vector_db" in self.config:
            tasks.append(self._init_vector_db())

        await asyncio.gather(*tasks)
        logger.info(f"All connection pools initialized: {list(self._pools.keys())}")

    async def _init_redis(self):
        """初始化 Redis 连接池"""
        redis_cfg = self.config["redis"]["session_store"]
        pool = aioredis.ConnectionPool(
            max_connections=redis_cfg["connection_pool"]["max_connections"],
            timeout=redis_cfg["connection_pool"]["timeout"],
            socket_keepalive=redis_cfg["connection_pool"]["socket_keepalive"],
        )
        self._pools["redis_session"] = pool

    async def _init_postgresql(self):
        """初始化 PostgreSQL 连接池"""
        pg_cfg = self.config["postgresql"]["agent_memory"]

        # 写池
        write_pool = await asyncpg.create_pool(
            dsn=pg_cfg["write_pool"]["dsn"],
            min_size=pg_cfg["write_pool"]["min_size"],
            max_size=pg_cfg["write_pool"]["max_size"],
            max_inactive_connection_lifetime=pg_cfg["write_pool"]["max_lifetime"],
            command_timeout=pg_cfg["write_pool"]["command_timeout"],
        )
        self._pools["pg_write"] = write_pool

        # 读池
        read_pool = await asyncpg.create_pool(
            dsn=pg_cfg["read_pool"]["dsn"],
            min_size=pg_cfg["read_pool"]["min_size"],
            max_size=pg_cfg["read_pool"]["max_size"],
            max_inactive_connection_lifetime=pg_cfg["read_pool"]["max_lifetime"],
            command_timeout=pg_cfg["read_pool"]["command_timeout"],
        )
        self._pools["pg_read"] = read_pool

    async def _init_vector_db(self):
        """初始化向量数据库连接"""
        # 实际实现使用 Qdrant/Milvus 客户端
        # 这里展示占位
        self._pools["vector_db"] = {
            "url": self.config["vector_db"]["qdrant"]["url"],
            "pool_size": self.config["vector_db"]["qdrant"]["connection_pool"]["max_size"],
        }

    async def graceful_shutdown(self):
        """优雅关闭所有连接池"""
        logger.info("Shutting down connection pools...")
        for name, pool in self._pools.items():
            try:
                if hasattr(pool, "close"):
                    await pool.close()
                elif hasattr(pool, "wait_closed"):
                    await pool.wait_closed()
                logger.info(f"Pool '{name}' closed")
            except Exception as e:
                logger.error(f"Error closing pool '{name}': {e}")
        logger.info("All pools shut down")

    def get_metrics(self) -> Dict:
        """获取连接池指标"""
        return {
            **self._metrics,
            "active_pools": len(self._pools),
            "pool_details": {
                name: {
                    "type": type(pool).__name__,
                }
                for name, pool in self._pools.items()
            },
        }
```

## 6. 能力边界与限制

### 6.1 CAP 定理与 Agent 状态

Agent 系统的状态管理在 CAP 定理下需要针对性取舍：

```
+------------------------------------------------------------------+
|               Agent 存储的 CAP 权衡                                  |
+------------------------------------------------------------------+
|                                                                      |
|  存储类型        | 一致性 (C) | 可用性 (A) | 分区容忍 (P) | 选择     |
|  ─────────      | ───────── | ───────── | ─────────── | ──────── |
|  会话状态       |  弱       |  强       |  强         | AP       |
|  (Redis)        | (最终一致) | (自动故障转移) | (集群模式)  |          |
|  长期记忆       |  弱       |  强       |  强         | AP       |
|  (向量DB)       | (异步索引) | (多副本)   | (分片)      |          |
|  结构化知识     |  强       |  中       |  强         | CP       |
|  (SQL DB)       | (事务)     | (故障转移窗口) | (分片)    |          |
|                                                                      |
|  说明:                                                                 |
|  - 会话状态和记忆可以接受最终一致性 (Agent 不要求读到最新写入)          |
|  - 结构化知识 (用户画像、偏好设置) 需要强一致性                          |
|  - 同一系统内不同组件对 CAP 的需求不同                                  |
+------------------------------------------------------------------+
```

```
        精度 (Precision)
             ↑

             |  / \
            /   \
           /     \

  (Recall)         (Latency)

├─────────────┼────────┼────────┼────────┤

└─────────────┴────────┴────────┴────────┘

Agent 场景建议:
  - 记忆检索: 召回 > 0.95 (漏掉重要记忆会导致 Agent 做出错误决策)
  - 知识检索: 召回 > 0.90 (知识有冗余, 可接受少量遗漏)
  - 会话检索: 延迟 < 10ms (用户体验敏感)
```

### 6.3 "记忆墙" (Memory Wall)

当对话历史 + 嵌入向量超过热存储容量时，出现"记忆墙"：

```
用户活跃度
    ↑
  高 │ ┌──────────────────────────────┐
     │ │    记忆墙                      │
     │ │  (热存储容量上限)               │
  中 │ │  ┌──── 开始压缩和迁移 ────┐     │
     │ │  │                        │     │
  低 │ │  │   冷存储 (S3/磁盘)      │     │
     │ │  └────────────────────────┘     │
     └─┴──────────────────────────────→ 时间
        1月   2月   3月   4月   5月

记忆墙的影响:
  1. 超出热存储 → 部分记忆被驱逐 → Agent "失忆"
  2. 旧记忆检索延迟从 5ms 上升到 200ms+ (S3 读取)
  3. 冷数据加载到热存储需要 500ms-2s (影响 Agent 响应)
  4. 量化后的向量检索精度下降

突破记忆墙的策略:
  - 分层存储 (hot/warm/cold)
  - 智能预加载 (预测用户即将需要的记忆)
  - 语义压缩 (将多轮对话融合为摘要)
  - 遗忘机制 (降低旧记忆的重要性权重)
```

### 6.4 跨分片查询问题

```
向量搜索跨分片查询的精度问题:

真实 top-10 在各分片的分布 (理想):
  Shard A: [0.95, 0.92, 0.88, 0.85]
  Shard B: [0.94, 0.91, 0.87, 0.84]
  Shard C: [0.93, 0.90, 0.86, 0.83]
  合并: [0.95, 0.94, 0.93, 0.92, 0.91, 0.90, 0.88, 0.87, 0.86, 0.85] ✓

但每个分片只返回 top-k 时可能遗漏:
  Shard A: [0.95, 0.92, 0.88]
  Shard B: [0.94, 0.91, 0.87]
  Shard C: [0.93, 0.90, 0.89]  ← 0.89 被截断
  合并: [0.95, 0.94, 0.93]  ← 0.89 丢失!

解决方案:
  - top_k_per_shard = final_k × 2 (过度获取)
  - 基于历史分布动态调整每个分片的 top_k
  - 使用全局 IDF/重要性权重调整
```

### 6.5 多语言持久化的备份恢复复杂性

```
协调备份多个数据库的复杂性:

时间点 A:           时间点 B:
  Redis 快照 RDB ───→ t1              Redis: 10:00 RDB
  PG 快照 ──────────→ t2              PG: 10:02 snapshot
  向量库快照 ───────→ t3              向量: 10:05 snapshot

问题:
  t1 ≠ t2 ≠ t3 ─── 三个快照不在同一时间点
  恢复后数据不一致: Redis 回退到 10:00，但 PG 是 10:02

解决方案:
  1. 协调快照: 使用 CDC (Debezium) 记录变更序列
  2. 逻辑恢复: 写模型记录全局序列号，恢复时按序列号重放
  3. 降低要求: Agent 场景下，各存储的不一致是可接受的 (最终一致性)
  4. 定期"对账": 异步对比各存储的关键数据
```

### 8.1 智能数据分层 (Hot/Warm/Cold)

```

+------------------------------------------------------------------+
|                                                                      |

|   └──────────────────────────────────────────────────┘                |

|                          | 自动迁移 (基于访问频率 + 最后访问时间)        |
```

### 8.3 向量量化 (PQ/SQ)

Product Quantization (PQ) 和 Scalar Quantization (SQ) 是减少向量存储的关键技术：

```
┌─────────────────────────────────────────────────────────────────────┐
│                          向量量化方法对比                              │
├──────────────┬──────────────┬──────────────┬─────────────────────────┤
│   方法         │ 压缩比        │ 内存节约       │ 召回率影响                │
├──────────────┼──────────────┼──────────────┼─────────────────────────┤
│ 无量化         │ 1x (32-bit)   │ 0%           │ 基准 (召回=0.98)          │
│ (float32)     │              │              │                          │
├──────────────┼──────────────┼──────────────┼─────────────────────────┤
│ Scalar Quant  │ 4x (8-bit)    │ 75%           │ -0.5% ~ -1.0%            │
│ (SQ8)         │              │              │                          │
├──────────────┼──────────────┼──────────────┼─────────────────────────┤
│ PQ 4bit       │ 8x (4-bit)    │ 87.5%          │ -1.0% ~ -2.0%            │
│ (M=64, 4bit)  │              │              │                          │
├──────────────┼──────────────┼──────────────┼─────────────────────────┤
│ PQ 2bit       │ 16x (2-bit)   │ 93.75%         │ -2.0% ~ -3.0%            │
│ (M=64, 2bit)  │              │              │                          │
├──────────────┼──────────────┼──────────────┼─────────────────────────┤
│ OPQ (优化 PQ) │ 8-16x        │ 87-94%       │ -0.5% ~ -1.5%            │
│              │              │              │ (比标准 PQ 好 0.5-1%)      │
└──────────────┴──────────────┴──────────────┴─────────────────────────┘

内存对比 (10M 向量, 768维):

  float32:  10,000,000 × 768 × 4 bytes = 28.6 GB
  SQ8:      10,000,000 × 768 × 1 byte  =  7.2 GB
  PQ 4bit:  10,000,000 × 384 × 1 byte  =  3.6 GB (M=64, 每组6维平均)
  PQ 2bit:  10,000,000 × 192 × 1 byte  =  1.8 GB

实现注意:
  - PQ 需要在索引构建时训练码本 (codebook)
  - 码本大小 = K × M × (d/M) × 4 bytes (K=256, M=64 → 64×256×12×4 = 768KB)
  - OPQ 先做旋转再量化，效果更好但训练时间 2-3 倍
```

### 8.5 跨数据库 Saga 模式

当 Agent 操作需要跨多个数据库的原子性时，使用 Saga 模式：

```
+------------------------------------------------------------------+
|                Agent 记忆更新的 Saga 流程                            |
+------------------------------------------------------------------+
|                                                                      |
|  故事: Agent 学习到用户的新的重要信息                                    |
|       "用户说他是 VIP 客户，需要优先处理"                               |
|                                                                      |
|  步骤:                                                                 |
|                                                                      |
|  1. [SQL DB] 更新用户画像: vip_status = true                        |
|     └── 成功 ✓                                                       |
|                                                                      |
|  2. [Vector DB] 插入新的记忆向量: "用户是 VIP"                        |
|     └── 成功 ✓                                                       |
|                                                                      |
|  3. [Redis] 更新用户会话: 标记为 VIP 会话                              |
|     └── 失败 ✗ (Redis 超时)                                          |
|                                                                      |
|  补偿事务:                                                             |
|     └── [SQL DB] 回滚: 将 vip_status 恢复为 false (补偿事务)          |
|     └── [Vector DB] 删除刚插入的向量                                   |
|                                                                      |
|  最终状态: 所有数据库回滚到操作前的一致状态                              |
|                                                                      |
|  Python 实现:                                                         |
|                                                                      |
|  @saga                                                                |
|  async def update_vip_user(user_id: str):                             |
|      # 步骤 1                                                         |
|      await sql_exec("UPDATE profiles SET vip=true WHERE uid=$1", uid) |
|      yield compensation("UPDATE profiles SET vip=false WHERE uid=$1") |
|                                                                      |
|      # 步骤 2                                                         |
|      vector_id = await vector_insert(embedding, {"uid": uid})         |
|      yield compensation(lambda: vector_delete(vector_id))             |
|                                                                      |
|      # 步骤 3                                                         |
|      await redis_set(f"vip:{uid}", "true")                            |
|      yield compensation(lambda: redis_del(f"vip:{uid}"))             |
+------------------------------------------------------------------+
```

```
|  ─────────────────────────────────────────────────────────────────    |
|   200K+      │ 极深 (500+) │ >90天 │ <20ms │ $30K-100K+/月           |
|   选择: CockroachDB + Dragonfly + 自建 Milvus + S3 冷存储             |
|   说明: 分布式 SQL, 高性能 KV, 存算分离向量库, 全链路分层存储            |
|  ─────────────────────────────────────────────────────────────────    |
|   Database-per-  │ N/A │ N/A │ <10ms │ 预算充足                        |

```

### 9.2 何时合并 vs 何时分离

```
合并为单一存储 (简化)          分离为多存储 (优化)
─────────────────────         ─────────────────────

适合合并的场景:                  适合分离的场景:
  - 用户数 < 5K                   - 用户数 > 50K
  - 对话深度 < 50 轮               - 对话深度 > 200 轮
  - 记忆保留 < 7 天                - 记忆保留 > 30 天
  - 延迟要求 < 200ms               - 延迟要求 < 20ms
  - 团队 < 5 人 (运维能力有限)       - 有专业 SRE/DBA 团队
  - 预算有限                       - 预算充足
  - 原型验证阶段                    - 生产级规模

合并的风险:                       分离的风险:
  - 单一组件的故障影响所有功能        - 多组件间的数据一致性问题
  - 无法针对每种负载独立扩展         - 运维复杂度 N 倍增长
  - 向量搜索在 PG 上性能有限         - 备份恢复复杂
  - PG 写入压力影响向量检索         - 需要更多开发和调试

推荐演进路径:
  单 PG (创业) → PG + Redis (增长期) → PG + Redis + 向量DB (成长期)
  → 分布式 SQL + 高性能KV + 分布式向量库 (成熟期)
  → Database-per-Agent (超大规模定制)
```

### 9.3 从简单到分布的迁移策略

```
阶段 1: 单体 (1-5000 用户)
  ┌──────────────────────────────────────────────┐
  │  应用层                                      │
  │    ↓                                         │
  │  PostgreSQL (pgvector)                       │
  │    ├─ conversations                          │
  │    ├─ sessions (非频繁访问)                   │
  │    └─ embeddings (pgvector)                  │
  └──────────────────────────────────────────────┘
  迁移信号: PostgreSQL 连接数 > 80%, 查询延迟 > 100ms

阶段 2: 分离会话 (5000-50000 用户)
  ┌──────────────────────────────────────────────┐
  │  应用层                                      │
  │    ├─────→ Redis (会话缓存)                   │
  │    └─────→ PostgreSQL (历史 + 知识)           │
  │                 └─ pgvector (向量)            │
  └──────────────────────────────────────────────┘
  迁移信号: 向量查询队列增长, pgvector 索引构建阻塞写入

阶段 3: 分离向量 (50000-200000 用户)
  ┌──────────────────────────────────────────────┐
  │  应用层                                      │
  │    ├─────→ Redis Cluster (会话)               │
  │    ├─────→ Qdrant/Milvus (向量记忆)           │
  │    └─────→ PostgreSQL (结构化知识 CQRS)       │
  └──────────────────────────────────────────────┘
  迁移信号: 单节点向量库 > 80% 内存, 备份时间 > 4h

阶段 4: 全分布式 (200000+ 用户)
  ┌──────────────────────────────────────────────┐
  │  应用层                                      │
  │    ├─────→ Dragonfly/Redis Cluster (会话)     │
  │    ├─────→ Milvus 分布式 (向量记忆)           │
  │    ├─────→ CockroachDB/Yugabyte (结构化知识)  │
  │    └─────→ S3 冷存储 (归档)                   │
  └──────────────────────────────────────────────┘

关键原则:
  - 每次只迁移一个组件
  - 迁移期间双写 (旧 + 新) 以便回滚
  - 使用特性开关控制流量切换
  - 每个阶段至少运行 2 周稳定期
```

## 总结

生产 AI Agent 系统的数据库扩展是一个**多维度、多存储引擎**的系统工程问题。核心要点：

1. **多语言持久化是必然**：没有单一数据库能满足 Agent 的会话、记忆和知识存储需求，三引擎架构是当前的最佳实践。

2. **向量数据库扩展最难**：HNSW 索引的内存非线性增长、跨分片查询的精度损失、量化带来的召回率下降，都需要在架构设计时仔细权衡。

3. **分层存储是成本控制关键**：Hot/Warm/Cold 三层架构 + 向量量化可以在保持 95%+ 召回率的同时降低 60-80% 的存储成本。

4. **CQRS 模式特别适合 Agent 系统**：Agent 记忆的写入（记忆提取）和读取（上下文恢复）有不同的优化方向，CQRS 可以让两方面都做到极致。

5. **渐进式迁移胜于大爆炸重构**：从单体 PG 逐步演进到全分布式架构，每个阶段只迁移一个组件，用双写确保可回滚。

6. **新兴模式值得关注**：Database-per-Agent (Yugabyte) 提供了前所未有的隔离性，时序记忆数据库将时间维度引入 Agent 记忆，这些方向可能在 2026-2027 年成为主流。
