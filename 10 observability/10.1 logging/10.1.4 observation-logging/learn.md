# 10.1.4 Observation Logging — 观察结果与异常标记

## 简单介绍

Observation Logging（观察日志）记录 Agent 在 ReAct 循环中接收到的"外部反馈"——工具调用的返回结果、环境的响应、以及系统异常。它位于 Action 之后、Thought 之前，是连接 Agent 内部推理与外部世界的桥梁。如果说 Thought 是 Agent 的"内心独白"，那 Observation 就是 Agent 的"感官输入"。

## 基本原理

在 ReAct 循环中，Observation 是 Action 的直接产物：

```
Thought → Action → Observation → Thought → ...
                        ↑
                这步的结果作为
                下一轮思考的输入
```

Observation 日志的核心职责是记录**发生了什么**，而非"为什么发生"或"意味着什么"：

```python
observation_log = {
    "type": "observation",
    "trace_id": "tr-xxx",
    "turn_number": 3,
    "action_id": "act-3",          # 关联触发了这个 observation 的 action
    "origin": "tool_result",        # 来源：tool_result | system | human_feedback
    "content_summary": "天气 API 返回 200 OK，数据内容略",
    # 质量标记
    "is_error": False,              # 是否是错误
    "is_empty": False,              // 结果是否为空
    "is_truncated": True,           // 结果是否被截断
    "error_type": None,             // 错误类型（如果 is_error=True）
    # 数据特征
    "result_size_bytes": 8192,
    "content_type": "application/json",
    "data_hash": "sha256-xxx",      // 内容哈希，用于去重
    # 处理标记
    "parsing_status": "success",    // 是否成功解析
    "validation_status": "passed",  // 校验是否通过
}
```

## 背景

传统软件中，"函数调用结果"通常被视作确定性的——给同样的参数，得到同样的结果。但在 Agent 系统中，Observation 具有高度的不确定性：

1. **工具可能返回意料之外的格式**——API 改了、字段没了、结果被截断了
2. **工具可能返回错误但非异常**——HTTP 200 但 body 中是 `{"error": "rate limit"}`
3. **结果可能过于庞大**——搜索返回 1000 条结果，但 Agent 的上下文窗口无法容纳
4. **结果可能包含敏感信息**——工具返回了本不该暴露给 Agent 的数据

Observation 日志捕捉了所有这些边缘情况和异常，为 Agent 的错误恢复和人类审计提供原始数据。

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 不记录 Observation | 只记录 Agent 调用了什么工具 | 无法知道工具实际返回了什么，调试时只能"猜" |
| 将结果直接注入 Prompt 而不记录 | Observation 只在 LLM 的上下文中存在 | 上下文被压缩或窗口滚动后，中间结果永久丢失 |
| 原样记录全部结果 | 把工具的所有返回数据都写入日志 | 存储爆炸，尤其对于大型文档、代码库扫描等场景 |
| 靠 LLM 自己总结 Observation | 让 Agent 在回复中描述"我收到了什么" | LLM 可能错误描述或遗漏关键信息 |

## 核心矛盾

| 矛盾维度 | 不记录 | 完整记录 | 结构化摘要 |
|---------|-------|---------|-----------|
| 存储开销 | 0 | 极高 | 低 |
| 可追溯性 | 无 | 高 | 中（缺细节） |
| 隐私风险 | 无 | 高 | 可控 |
| 调试价值 | 无 | 高 | 高 |
| LLM 上下文影响 | 无 | 可能过大 | 无影响 |

**核心矛盾**：Observation 是 Agent 下一轮推理的输入，在日志中它是"输入数据"；但同时它可能非常大、包含敏感信息、或者格式异常。解决方案是**摘要 + 哈希 + 异常标记**：日志中只记录摘要和文件指纹，完整数据存外部，异常情况特殊标记。

## 当前主流优化方向

1. **结果质量分类**——自动对 Observation 进行分类：正常（green）、异常但可处理（yellow）、致命错误（red），每类对应不同的日志详细度和告警级别
2. **自动结果摘要**——使用 LLM 或规则自动对工具返回结果生成结构化摘要（类型、大小、关键字段），而非直接拼接原始数据
3. **异常模式识别**——积累常见的 Observation 异常模式（API 返回了错误码、搜索无结果、文件不存在等），自动打标签辅助故障定位
4. **增量 Observation**——当结果很大时，只记录"增量差异"：本轮结果与上轮相似时只记录变化部分
5. **Observation 校验器**——在写入日志前使用 JSON Schema 或 Pydantic 校验结果格式，自动标记格式异常

## 实现的最大挑战

1. **大规模结果的描述**——当工具返回 10MB 的搜索结果时，如何在不记录全部内容的同时保留足够的可追溯信息？哈希可以验证一致性但不能提供信息
2. **流式 Observation**——工具调用可能通过 SSE 返回持续的数据流，需要实时记录"事件流"而非仅记录最终结果
3. **多源 Observation 合并**——并行工具调用的多个结果需要关联记录，但 Agent 在处理时可能会无序消费这些结果
4. **敏感信息自动检测**——不在预期 schema 中的敏感字段难以用规则检测，需要结合 ML 或正则表达式进行启发式判断

## 能力边界

**能做什么：**
- 追踪 Agent"接收到"了什么信息
- 检测工具的异常返回模式（空结果、错误码、格式错误）
- 为每步推理提供输入的可验证记录
- 支持事后审计：验证 Agent 的决策是否基于正确的输入

**不能做什么：**
- 不能验证 Observation 的"正确性"——日志记录了工具返回了什么，但不判断返回的内容是否合理
- 不能解决 Observation 过大的问题——"截断"只是一种优化策略，可能丢失关键信息
- 不能自动修复异常的 Observation——异常检测是诊断能力，修复需要 fallback 机制

## 与其他的区别

| 对比项 | Action 日志 | Observation 日志 |
|--------|------------|----------------|
| 记录时机 | 调用前 | 调用后 |
| 内容 | "我打算调用什么" | "我收到了什么" |
| 焦点 | 工具、参数 | 结果、状态、异常 |
| 错误处理 | 记录错误发生 | 记录错误内容 |
| 数据流向 | 出站 | 入站 |

Action 日志 + Observation 日志构成了 Agent 与外部世界交互的完整双向记录。

## 最终工程优化

1. **分级的 Observation 存储**——小结果（< 1KB）直接嵌入日志；中结果（1KB-100KB）摘要 + 外部存储引用；大结果（> 100KB）仅记录元数据和外部存储引用
2. **自动异常标记流水线**——规则引擎 + ML 分类器双路判断：规则引擎覆盖已知模式（空数组、error 字段等），ML 分类器覆盖异常模式识别
3. **Observation 缓存**——对相同工具、相同参数的返回结果使用内容哈希去重，如果多轮调用返回了相同结果，只记录一次
4. **与正常流程解耦**——Observation 的日志写入使用单独的异步队列，即使日志系统故障也不影响 Agent 获取工具调用结果
