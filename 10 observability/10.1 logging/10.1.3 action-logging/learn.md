# 10.1.3 Action Logging — 工具调用日志

## 简单介绍

Action Logging（行动日志/工具调用日志）记录 Agent 执行的所有外部操作——调用工具、执行代码、发送 API 请求等。它回答了 **"Agent 做了什么"** 这一核心问题，是可观测性中最具工程价值的部分。通过 Action 日志，开发者可以精确追踪每次工具调用的输入参数、执行结果、耗时和错误。

## 基本原理

Agent 在 ReAct 循环中发出 Action，Action 日志在调用前后分别记录：

```
调用前日志（Call Log）: 记录了 Agent "打算"调用什么工具
调用后日志（Result Log）: 记录了工具"实际"返回了什么结果
```

一条完整的 Action 日志包含三个阶段的信息：

```python
# 阶段1：Action 声明——Agent 决定调用工具
action_call = {
    "event": "action_call",
    "trace_id": "tr-xxx",
    "turn_number": 3,
    "timestamp_call": "2026-07-17T10:30:00.123Z",
    "tool_name": "get_weather",
    "arguments": {
        "city": "Beijing",
        "units": "metric"
    },
    "thought_ref": "th-3",  # 指向产生的这个 action 的 thought
}

# 阶段2：Action 执行——工具实际执行
action_exec = {
    "event": "action_exec",
    "trace_id": "tr-xxx",
    "action_id": "act-3",
    "execution": {
        "host": "api-server-01",
        "start_time": "2026-07-17T10:30:00.130Z",
        "end_time": "2026-07-17T10:30:00.450Z",
        "duration_ms": 320,
        "status": "success",  # success | error | timeout
    }
}

# 阶段3：Action 结果——工具返回的数据
action_result = {
    "event": "action_result",
    "trace_id": "tr-xxx",
    "action_id": "act-3",
    "result_summary": "200 OK, temperature=22, condition=sunny",
    "result_size_bytes": 1024,
    "truncated": False,      # 结果是否被截断
    "error": None,           # 错误信息（如果有）
}
```

## 背景

在传统软件中，"做了什么"是通过代码路径来理解的——开发者可以读代码、看日志、加断点来追踪执行流程。但在 Agent 系统中，工具调用是**动态决定**的：

- 开发者无法预知 Agent 会调用哪些工具以及调用的顺序
- 工具调用可能出错、超时或返回意料之外的结果
- Agent 可能反复调用同一工具（如在循环中搜索—评估—再搜索）

Action 日志提供了 Agent 行为的"外部可观测记录"，是所有后续分析（调试、审计、评估）的基础。

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 仅记录工具名称 | "调用了 get_weather" | 完全缺少参数和结果信息，无法排查错误 |
| 记录完整参数和结果 | 原样记录所有输入输出 | 结果可能非常大（如文件读取），导致存储爆炸；且可能包含敏感信息 |
| 在工具函数内部打日志 | 在每个工具函数里写日志 | 日志分散在代码各处，格式不统一，丢失与 Agent 推理链的关联 |
| 依赖 LLM 自己总结 | 让 Agent 在回复中说明"我调用了XX" | 不可靠，Agent 可能遗漏或描述错误 |

## 核心矛盾

| 矛盾维度 | 不记录 | 完整记录 | 结构化日志 |
|---------|-------|---------|-----------|
| 存储开销 | 0 | 极高（大结果） | 可控 |
| 隐私风险 | 无 | 高 | 脱敏可控 |
| 调试能力 | 无 | 好 | 最好 |
| 开销影响 | 无 | 影响 Agent 性能（IO） | 异步，低影响 |

**最核心的工程矛盾**：工具调用的结果可能非常大（读取整个文件、返回大量数据库记录），但想要完整记录所有信息。解决方案是**摘要 + 完整分离**——日志中只记录结果摘要，完整结果按需存储到外部系统（如 S3 或数据库）。

## 当前主流优化方向

1. **自动参数脱敏**——基于规则或 ML 自动检测工具参数中的敏感字段（密码、API key、身份证号），在日志写入前自动替换为 `***`
2. **结果摘要 + 分级存储**——日志中记录结果摘要（首行/前 1KB/自动摘要），完整结果关联存储到对象存储，按需检索
3. **流式 Action 日志**——对于流式工具调用（如 SSE、WebSocket），实时记录事件流，而非等到调用结束才写一条日志
4. **Action 依赖图构建**——自动分析工具调用之间的数据依赖关系（B 工具的入参来自 A 工具的输出），构建执行 DAG
5. **成本分摊标签**——为每个 Action 关联 Token 消耗和 API 费用，实现精确的成本核算

## 实现的最大挑战

1. **参数序列化**——工具参数可能是复杂的嵌套对象，需要在序列化时处理好循环引用、None 值、大数据结构等问题
2. **结果截断策略**——截断位置的选择（简单截断前 N 字符 vs 智能截断保留关键结构）直接影响调试效率
3. **二进制结果的记录**——如图像生成、文件输出等二进制结果，不能直接 JSON 序列化，需要特殊处理（base64 或外部引用）
4. **Action 与 Thought 的关联**——一个 Action 可能对应多个 Thought（Agent 思考了多次才决定调用），日志需要正确维护这种多对多关系

## 能力边界

**能做什么：**
- 完整还原 Agent 执行了哪些操作以及操作的顺序
- 精确分析每次工具调用的延迟和成功率
- 发现 Agent 在哪个步骤出错并理解出错原因
- 审计 Agent 的行为是否符合预期

**不能做什么：**
- 不能解释"为什么"调用工具——这需要结合 Thought 日志
- 不能保证工具的调用结果正确——日志记录的是"返回了什么"而不是"返回的是否正确"
- 不能自动修复工具的错误——需要结合 retry 和 fallback 机制

## 与其他的区别

| 对比项 | Action 日志 | 工具本身日志 |
|--------|------------|------------|
| 焦点 | Agent 如何调用工具 | 工具内部执行细节 |
| 记录者 | Agent 框架 | 工具实现代码 |
| 内容 | 入参 + 出参 + 耗时 | 内部处理步骤 + 资源使用 |
| 关联键 | trace_id + action_id | 可能没有关联 |

两者互补：Action 日志记录"黑盒调用"，工具日志记录"内部实现"。联合查询可以精确定位问题。

## 最终工程优化

1. **统一 Action Schema**——定义 Action 日志的 JSON Schema，所有工具调用都遵循相同结构，便于统一分析和告警
2. **Action 索引优化**——按 (agent_id, tool_name, timestamp) 建立复合索引，支持高效查询"某个 Agent 都调用了哪些工具"
3. **预计算 Action 指标**——在日志写入时同步更新计数器（工具调用次数、平均延迟、错误率），避免事后全量扫描计算
4. **智能采样**——对于高频、低风险的 Action（如 `search_web`），按 10% 采样；对于高风险 Action（如 `execute_sql`、`delete_file`），100% 记录
5. **Action 回溯重放**——将 Action 日志设计为可重放格式，开发调试时可以重放 Agent 的某次执行序列，观察是否修复了问题
