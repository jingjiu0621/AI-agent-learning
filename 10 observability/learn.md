# 10 Observability — 可观测性、监控与调试

Agent 系统的可观测性是将"黑盒推理"变为"透明管道"的核心工程能力。没有可观测性，Agent 就是一个不可调试、不可评估、不可信任的随机输出器。

## 概述

传统软件的可观测性关注请求延迟、错误率、系统资源；Agent 可观测性在此基础上增加了**推理过程追踪、Token 消耗分析、工具调用链路、记忆检索路径、多 Agent 通信**等 AI 特有维度。Agent 的"输出"不再是简单的 HTTP 响应，而是一个包含多步推理、工具调用、知识检索的复杂过程。

### 为什么 Agent 可观测性更难？

| 维度 | 传统系统 | Agent 系统 |
|------|---------|-----------|
| 决策路径 | 固定代码路径，确定性 | LLM 生成路径，非确定性 |
| 错误来源 | 代码 bug / 网络 / 资源 | LLM 幻觉 + 工具失败 + 规划错误 |
| 状态空间 | 有限、可控 | 无限、开放 |
| 可复现性 | 相同输入 → 相同输出 | 相同输入 → 可能不同输出 |
| 性能瓶颈 | CPU/内存/IO | Token 消耗 + LLM 延迟 + 工具调用 |
| 调试手段 | 断点 / 堆栈 / 日志 | Prompt 审查 / 推理轨迹 / 回放 |

## 核心设计原则

1. **日志是基础，追踪是骨架，监控是眼睛**——三层各自解决不同粒度的问题，缺一不可
2. **Agent 行为的非确定性要求更强的可观测性**——传统系统"出了 bug 能复现"，Agent 系统"同样的输入可能不出同样的错"，必须靠完整轨迹记录才能事后分析
3. **Token 是最昂贵的资源，必须度量**——Agent 的 Token 消耗直接等于成本，可观测性系统首先要回答"钱花在哪了"
4. **可观测性本身不能成为性能瓶颈**——日志/追踪的 IO 开销可能影响 Agent 的响应延迟，需要异步、采样、批处理
5. **从开发到生产的可观测性一致性**——开发阶段的调试手段应该平滑过渡到生产监控，避免"开发时能看到一切，生产时是黑盒"

## 与其他模块的关系

- ← **5 agents-core**：Agent 主循环是日志和追踪的埋点位置，loop-engine 的每一步都需要可观测性插桩
- ← **6 memory**：记忆检索路径需要追踪，记忆操作的耗时/成功率需要监控
- → **11 evaluation**：可观测性数据是评估的输入源，轨迹日志用于评分和回归测试
- → **12 safety**：审计日志和异常检测是可观测性和安全的交叉领域
- → **13 production**：生产部署离不开监控、告警、SLO 等可观测性基础设施
