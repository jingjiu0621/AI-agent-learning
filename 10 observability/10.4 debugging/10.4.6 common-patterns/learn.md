# 10.4.6 Common Patterns — 常见 Bug 模式与调试策略

## 简单介绍

Common Patterns 总结了 Agent 系统开发中反复出现的 Bug 模式和对应的调试策略。与通用软件不同，Agent 的 Bug 模式有显著的 AI 特性——很多"Bug"不是代码写错了，而是**LLM 的行为与预期不符**。

## Agent Bug 的总体分类

```
Agent Bug 分类树：

1. Prompt 问题（占比 ~40%）
   ├── 指令模糊/冲突
   ├── 工具描述不准确
   ├── 少样本示例不合适
   └── System Prompt 被用户输入覆盖

2. LLM 行为问题（占比 ~25%）
   ├── 工具选择错误
   ├── 参数幻觉
   ├── 推理跳跃
   └── 过度自信

3. 状态/上下文问题（占比 ~20%）
   ├── 历史消息被错误压缩
   ├── 上下文窗口耗尽
   ├── 状态不一致
   └── Agent 间状态冲突

4. 工具/外部问题（占比 ~15%）
   ├── 工具返回格式变更
   ├── 工具超时/限流
   ├── 工具认证过期
   └── 外部 API 结果错误
```

## 10 种常见 Agent Bug 及调试策略

### 1. 死循环（Agent keeps repeating）

**症状**：Agent 反复调用同一个工具、思考同一个问题、输出相似内容。

**根因**：
- 终止条件不足（Agent 不知道"什么时候够了"）
- Observation 格式 Agent 无法解析 → 反复尝试
- Agent 对工具结果不满意 → 重试但参数不变

**调试策略**：
```
1. 检查 Agent 的环境配置，是否有"必须完成任务"等绝对指令
2. 检查 Agent 收到工具调用结果后是否解析到了正确的格式
3. 检查 Agent 对工具返回的结果是否使用正确
4. 设置最大步数限制作为兜底保护
```

### 2. 忘记上下文（Context Amnesia）

**症状**：Agent 在几轮对话后"忘记"之前已经获取的信息，重复询问相同问题。

**根因**：上下文窗口滚动截断了早期信息、或被摘要压缩丢失了细节。

**调试策略**：
```
1. 使用 Prompt Inspection 检查历史消息是否被截断
2. 检查 Agent Prompt 中的上下文管理规则的设定
3. 检查 Agent Prompt 中"引用早期信息"的引导提示是否充足
4. 查看"早期信息"是否被保存到长期记忆或其他地方
```

### 3. 工具选择混乱（Tool Confusion）

**症状**：Agent 使用完全不相关的工具来处理用户请求。

**根因**：工具名称或描述不清晰，工具之间的区分度不足。

**调试策略**：
```
1. 检查 Agent 实际发送给 LLM 的所有工具定义
2. 对比"预期应该调用什么工具"和"LLM 实际选择的工具"
3. 检查是否有"默认工具"在无参数时被多次选中
4. 检查工具的描述是否太相似导致 LLM 难以区分
```

### 4. 参数幻觉（Parameter Hallucination）

**症状**：Agent 在工具调用中传入工具定义中没有的字段。

**根因**：LLM 根据训练数据中的"常见 API 模式"自行补全了工具定义中没有的参数。

**调试策略**：
```
1. 在工具调用执行前使用 JSON Schema 验证参数
2. 检查 Agent 对超出定义范围的参数返回的报错信息
3. 如果某个 Agent 反复出现参数幻觉，检查对应参数的描述是否有"误导"的嫌疑
```

### 5. 推理跳跃（Reasoning Leap）

**症状**：Agent 的 Thought 中跳过了关键的推理步骤，直接得到结论。

**根因**：LLM 倾向于"快速回答"而非"完整推理"，特别在时间压力下。

**调试策略**：
```
1. 检查 Agent Prompt 中是否有"请完整思考"等要求
2. 检查 Agent 是否在每一轮调用前都给出足够详细的 Thought
3. 添加"请逐步推理，每一步都要确认之后再继续"等引导
```

### 6. 过度工具依赖（Over-tooling）

**症状**：Agent 调用工具获取自己已经知道的信息（如调用计算器算 2+2）。

**根因**：Agent/LLM 被训练得过于依赖工具，丧失了自主判断能力。

**调试策略**：
```
1. 检查 Agent 的"行动计划"，看是否每一步都依赖工具
2. 检查 Agent 的系统提示中是否添加了"使用工具的规范/策略"
3. 根据实际执行数据，优化 Agent 对简单工具和自身知识的判断策略
```

### 7. 错误累积（Error Cascade）

**症状**：早期一个微小错误在后续步骤中被不断放大。

**根因**：Agent 的 ReAct 循环中，每步的输出作为下一步的输入，错误像滚雪球一样扩大。

**调试策略**：
```
1. 使用回放调试定位"第一个错误"出现在哪一步
2. 检查 Agent 检查上一步结果的方式，确认其判断依据是否正确
3. 在 Agent Prompt 中明确要求"检查之前的信息是否正确"
```

### 8. 敏感数据泄露（Data Leak）

**症状**：Agent 在回复或日志中暴露了不应该暴露的敏感信息。

**根因**：工具返回了敏感数据，Agent 没有过滤就直接使用。

**调试策略**：
```
1. 检查 Agent 在调用前是否检查了工具的返回结果
2. 检查敏感信息相关的日志记录配置
3. 检查 Agent 的工具调用记录中是否包含用户隐私字段
```

### 9. 计划执行不一致（Plan-Execute Mismatch）

**症状**：Agent 制定了计划但不按计划执行，或者执行到一半自己改了计划。

**根因**：Plan 阶段的输出在 Execute 阶段没有被正确约束。

**调试策略**：
```
1. 检查 Agent 的 Plan 和 Execute 阶段的衔接是否准确
2. 检查 Plan-and-Execute Agent 中 Execute 阶段 Agent 的 Prompt
3. 添加"严格按照计划执行"等相关约束
```

### 10. JSON 格式错误（JSON Parse Failure）

**症状**：工具调用返回结果时格式不一致导致 Agent 无法解析。

**根因**：不同工具返回的 JSON 格式不一致，或者 LLM 的 JSON Mode 产生了无效 JSON。

**调试策略**：
```
1. 检查 Agent 对 JSON 错误的处理逻辑是否合理
2. 检查是否 JSON Schema 约束不足，允许了不明确的字段
3. 检查 Agent 对错误格式 JSON 的提示信息是否有助于修复格式问题
```

## Bug 调试快速排查表

```python
QUICK_CHECKLIST = [
    # Prompt 层
    "✅ 系统 Prompt 清晰定义了 Agent 的角色和行为边界？",
    "✅ 工具描述清晰且互相区分度高？",
    "✅ Agent 有正确的少样本示例？",
    "✅ 安全边界和拒绝策略明确声明？",

    # 执行层
    "✅ Agent 的最大步数限制设置合理？",
    "✅ 工具有超时和重试机制？",
    "✅ 工具有合理的错误处理？",
    "✅ 工具调用参数经过 Schema 验证？",

    # 状态层
    "✅ Agent 的上下文窗口管理配置正确？",
    "✅ 历史消息不会意外截断？",
    "✅ Agent 的状态有定期的快照/备份？",

    # 观测层
    "✅ Agent 的日志记录了完整的 Trace？",
    "✅ 关键步骤有触发告警？",
    "✅ 有回放调试的能力？",
]
```

## 调试策略总结

1. **从外部到内部**——先检查日志和监控（外部现象），再检查 Trace，最后才是 Prompt 和状态（内部原因）
2. **从简单到复杂**——先排除"代码 bug"（最常见的），再检查"配置问题"，最后才是"LLM 行为问题"
3. **从最近到最早**——对于长链问题，从最后一步的错误开始回溯，找到最早的"第一个错误触发点"
4. **隔离法**——禁用部分工具或简化 System Prompt，看问题是否复现，逐步缩小排查范围
