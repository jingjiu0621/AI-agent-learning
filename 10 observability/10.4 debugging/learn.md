# 10.4 Debugging — Agent 调试技术

Debugging（调试）是 Agent 可观测性的"行动层"。日志、追踪、监控回答的是"发生了什么"，调试回答的是 **"为什么会这样"** ——在日志和追踪不足以诊断问题根源时，主动的调试手段介入。

## Agent 调试的特殊性

Agent 调试与传统的函数断点调试有本质区别：

| 维度 | 传统调试 | Agent 调试 |
|------|---------|-----------|
| 执行路径 | 确定性、可复现 | 非确定性、难以复现 |
| 状态空间 | 有限（变量+堆栈） | 无限（LLM 上下文 + 历史消息） |
| 错误来源 | 代码 bug / 数据异常 | LLM 幻觉 + 工具错误 + 规划问题 |
| 断点方式 | 代码行断点 | "Prompt 级"断点（思考前/行动后） |
| 修改效果 | 即改即生效 | Prompt 修改后效果不确定 |
| 复现方法 | 相同输入复现 | 相同输入可能不同输出 |

## Agent 调试的核心问题

```
Agent 调试要回答的 5 个核心问题：
1. Agent 看到了什么？        → 实际发送给 LLM 的 Prompt 是什么
2. Agent 想了什么？          → LLM 返回的 Thought 内容是什么
3. Agent 做了什么？          → 调用了什么工具、传了什么参数
4. 工具返回了什么？          → 观察结果是什么
5. Agent 最后输出了什么？    → 最终回复是什么
```

## 调试工具链

```
┌────────────────────────────────────────────────────────────┐
│                  Agent 调试工具链                            │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  问题发现 ← 监控/告警/用户反馈                              │
│     ↓                                                      │
│  检查 Trace ← LangSmith / Jaeger / OTel                    │
│     ↓                                                      │
│  检查 Prompt ← Prompt Inspector（实际发出的完整消息）        │
│     ↓                                                      │
│  检查状态 ← State Snapshot（Agent 当前状态快照）             │
│     ↓                                                      │
│  回放执行 ← Replay Engine（按记录重放每一步）               │
│     ↓                                                      │
│  修改验证 ← 修改 Prompt/逻辑 → 重新测试                     │
│     ↓                                                      │
│  回归测试 ← 将问题用例加入回归测试套件                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

## 关键设计原则

1. **"可调试"是 Agent 框架的基本要求**——一个不可调试的 Agent 系统在生产环境中是"定时炸弹"
2. **调试信息应比日志更详细**——日志关注关键事件，调试需要记录所有细节（包括完整的 Prompt 内容）
3. **调试模式不影响正常模式**——开启调试功能不应改变 Agent 的行为（Heisenberg 效应）
4. **复现是关键**——Agent 的非确定性使得复现成为最大挑战，需要完整的执行轨迹记录
