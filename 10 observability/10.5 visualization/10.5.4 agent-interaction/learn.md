# 10.5.4 Agent Interaction — 多 Agent 交互可视化

## 简单介绍

Agent Interaction Visualization（多 Agent 交互可视化）展示多个 Agent 之间的通信、协作、任务委派和结果汇报过程。在多 Agent 系统中，理解"谁在什么时候和谁说了什么"是调试的基础。

## 基本原理

多 Agent 交互可视化的四种核心视图：

### 1. 泳道图（Swimlane Diagram）

展示各 Agent 在时间线上的活动：

```
时间 ──────────────────────────────────────────────────────►
                                                          
┌─ Orchestrator ─────────────────────────────────────────┐
│  [分配任务]─────────[收到结果]──[汇总]──[最终回复]       │
└────────────────────────────────────────────────────────┘
                                                          
┌─ Researcher ───────────────────────────────────────────┐
│           [搜索]──[分析]──[返回结果]                     │
└────────────────────────────────────────────────────────┘
                                                          
┌─ Coder ────────────────────────────────────────────────┐
│                 [写代码]──[测试]──[返回代码]             │
└────────────────────────────────────────────────────────┘
                                                          
┌─ Reviewer ─────────────────────────────────────────────┐
│                           [审查代码]──[反馈修改意见]     │
└────────────────────────────────────────────────────────┘
```

### 2. 消息流图（Message Flow Graph）

展示 Agent 间的消息传递和状态：

```python
class AgentInteraction:
    """Agent 交互事件"""
    from_agent: str
    to_agent: str
    message_type: str           # task_assignment / result_report / question / feedback
    content_summary: str
    timestamp: float
    duration: float              # 消息传递耗时
    status: str                  # sent / received / acknowledged / failed
```

### 3. 协作拓扑图（Collaboration Topology）

展示 Agent 系统中的角色分工和组织结构：

```
              Orchestrator
              │   │   │
        ┌─────┘   │   └─────┐
        ▼         ▼         ▼
    Researcher   Coder   Reviewer
        │         │
        └────┬────┘
             ▼
        KnowledgeBase (共享)
```

## 背景

当 Agent 系统从单 Agent 扩展到多 Agent（Module 8），最大的调试挑战是"谁做了什么"——在 5 个 Agent、50 轮交互、200 条消息中定位问题，完全靠读文本日志是不可能的。多 Agent 交互可视化应运而生。

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 文本日志 | 每 5 个 Agent 各自输出日志 | 信息分散，难以关联 |
| 单 Agent 视角 | 只跟踪 Orchestrator 的视角 | 丢失了子 Agent 的详细信息 |
| 消息时间线 | 所有消息按时间排列 | 没有分组，混杂在一起无法区分角色 |
| 不专门做可视化 | 把多 Agent 当单 Agent 看 | 完全丢失了交互的全貌 |

## 当前主流优化方向

1. **实时泳道图**——Agent 执行过程中实时更新泳道图，新消息出现时在对应泳道弹入，支持暂停/回放
2. **交互模式分析**——自动识别多 Agent 的交互模式（如"Orchestrator 分配任务 → Worker 执行 → 结果汇总"），标注常见模式
3. **协作瓶颈检测**——在可视化中自动标注"等待时间最长的 Agent"、"消息最多的 Agent"、"响应最慢的 Agent"，发现协作瓶颈
4. **多 Agent 会话回放**——将多 Agent 的完整交互记录保存为可回放的交互会话，支持快进/后退/跳转到特定消息

## 实现的最大挑战

1. **Agent 数量扩展**——当 Agent 从 3 个扩展到 30 个，泳道图的宽度不足以容纳所有 Agent，需要分组和折叠
2. **消息量爆炸**——多 Agent 系统每秒可能产生数百条消息，所有消息都展示会导致视觉过载
3. **异步交互的时序**——Agent 之间的消息可能是异步的（发送后隔一段时间才处理），时间线上的"因果链"需要特殊标注
4. **跨会话交互**——不同会话的 Agent 交互混在一起，需要按 session/trace 隔离

## 能力边界

**能做什么：**
- 直观了解多 Agent 系统的协作模式
- 快速定位"哪个 Agent 在等待"、"哪个 Agent 出错了"
- 分析多 Agent 系统的"通信开销"——花了多少时间在 Agent 间通信而非实际工作
- 发现协作瓶颈和不平衡（某个 Agent 负载特别高）

**不能做什么：**
- 不能解释 Agent 间消息的"语义"——谁和谁说了什么，但"为什么这么说"需要结合 Agent 的 Thought
- 不能自动优化多 Agent 拓扑——可视化只展示现状，优化需要架构调整
- 不能展示 Agent 内部的推理过程——只展示 Agent 间的交互，不展示 Agent 内的 Thought

## 最终工程优化

1. **Agent 分组与聚合**——当 Agent > 5 时，同类型 Agent 自动折叠为一个组（如"3 个 Worker Agents"），展开时显示内部详细交互
2. **消息重要性过滤**——自动过滤"ACK"、"心跳"等低价值消息，只展示业务相关的关键交互
3. **热力图+泳道图叠加**——在泳道图上叠加 Agent 的活动热力图，颜色越深表示该 Agent 越忙，一眼看出负载不均
4. **交互统计面板**——在可视化旁边展示统计面板：总消息数、平均响应时间、活跃 Agent 数、各 Agent 的消息占比
