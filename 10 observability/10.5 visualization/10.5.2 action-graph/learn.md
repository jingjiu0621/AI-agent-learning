# 10.5.2 Action Graph — 行动图谱

## 简单介绍

Action Graph（行动图谱）可视化 Agent 的工具调用过程——调用了哪些工具、调用的顺序、参数和结果、工具之间的依赖关系。与 Thought Map 展示"推理"不同，Action Graph 展示的是"执行"——Agent 与外部世界交互的完整图谱。

## 基本原理

```python
class ActionNode:
    action_id: str
    tool_name: str
    status: str                    # pending / running / success / error
    
    # 参数摘要
    params_summary: str           # 关键参数摘要，非完整参数
    result_summary: str           # 结果摘要
    duration_ms: float
    
    # 依赖关系
    depends_on: list[str]         # 依赖哪些工具的结果
    parallel_group: str           # 并行分组

class ActionGraph:
    nodes: list[ActionNode]
    edges: list[tuple[str, str, str]]  # (from_id, to_id, relation)
    # relation: "data_flow" / "trigger" / "retry" / "fallback"
```

工具调用 DAG 可视化示例：

```
get_user_location ──── get_weather ──── generate_response
        │                   │
        │             ┌─────┴─────┐
        │             │           │
        ▼             ▼           ▼
   [Beijing]    [22°C/sunny]   [回复用户]
        │             │
        └─────┬───────┘
              │
              ▼
   [复杂研究Agent的调用拓扑:]
   
   search_web ──→ read_page ──→ search_web (new query)
        │            │               │
        │            ▼               ▼
        │      extract_facts    read_page
        │            │               │
        └──────┬─────┘               │
               │                     │
               ▼                     │
         synthesize_answer ◄─────────┘
               │
               ▼
          final_response
```

## 背景

在文本日志中，工具调用是独立的 JSON 记录。但在实际运行中，工具之间往往存在数据依赖（B 工具的入参来自 A 工具的输出）、时序关系（必须先 A 后 B）、并行关系（A 和 B 可以同时调用）。Action Graph 将这些关系以图形方式呈现，回答了"Agent 究竟是怎么一步步完成任务的"。

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 日志表格 | 将工具调用列成一个序号表格 | 丢失了依赖和并行关系 |
| 文本描述 | "Agent 先调用了 A，然后调用了 B" | 顺序关系清楚，但并行关系和依赖关系不直观 |
| 静态架构图 | 画一张"Agent 的所有可用工具"图 | 静态的，无法展示实际执行路径 |
| 不展示 | 用户看不到工具调用过程 | 黑盒，用户不知道 Agent "在做什么" |

## 当前主流优化方向

1. **实时工具调用流**——Agent 每次调用工具时，可视化界面实时展示：工具名称正在闪烁、参数展示、进度条表示等待结果、结果返回更新
2. **依赖关系自动推断**——从工具调用的参数和结果中自动推断数据依赖关系（B 的参数中引用了 A 的结果字段），无需手动标注
3. **执行时间可视化**——在每个 Action 节点上用宽度表示耗时（类似火焰图的横向条），一眼看出"最慢的工具"
4. **异常路径高亮**——错误的工具调用用红色高亮，超时的用黄色闪烁，自动缩略异常的工具调用分支

## 实现的最大挑战

1. **动态 DAG 布局**——工具调用图是实时增长的（Agent 可能随时决定调用新工具），图谱布局需要支持动态添加节点而不破坏已有布局
2. **并行调用展示**——当 Agent 并行调用 5+ 工具时，图谱的宽度骤增，如何在有限界面中展示
3. **深层调用链管理**——工具调用链可能很深（A→B→C→D→E），线性展示占用空间，需要折叠和展开

## 能力边界

**能做什么：**
- 直观展示 Agent 的工具调用顺序和依赖关系
- 快速定位最慢的工具（性能瓶颈）
- 发现异常调用模式（不应该被调用的工具出现在了调用链中）
- 展示 Agent 的"执行效率"（并行了多少、串行了多少）

**不能做什么：**
- 不能展示工具内部的具体处理过程——Action Graph 关注"调用"而非"实现"
- 不能自动判断工具结果的正确性——只展示"调用了什么"和"返回了什么"
- 不能解释"为什么"选择了这个工具——需要结合 Thought Graph

## 最终工程优化

1. **分层展示**——高层面板展示"Agent 级"执行状态（当前在第几步），下面可展开"工具级"DAG 细节
2. **延迟热力图**——用节点颜色表示工具调用的延迟（绿 < 1s，黄 1-5s，红 > 5s），视觉上快速发现延迟异常
3. **搜索/过滤**——支持按工具名称搜索、按状态过滤（只看失败的调用）、按耗时排序
4. **Gantt 图集成**——将工具调用展示为 Gantt 图（横轴时间、纵轴工具），直观展示并行和串行关系
