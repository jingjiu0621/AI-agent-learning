# 10.3.2 Latency Monitoring — 延迟监控

## 简单介绍

延迟监控（Latency Monitoring）追踪 Agent 系统的响应速度。在 Agent 系统中，延迟是一个复杂的多维问题——"一次请求花了多久"不是单一数字，而是由 LLM 调用延迟、工具调用延迟、内部处理延迟等多个部分叠加而成。

## 基本原理

Agent 延迟的构成分解：

```
总延迟 = LLM 调用延迟 + 工具调用延迟 + 内部延迟

LLM 调用延迟 = 网络传输 + 服务端排队 + 推理时间 + Token 流式传输
工具调用延迟 = 网络传输 + 工具服务端处理时间
内部延迟 = 上下文序列化 + Prompt 构建 + 结果解析 + 状态更新
```

多步 Agent 的延迟累积：

```
用户请求 ─────────────────────────────────────────────────────┐
  │                                                            │
  ├── Step 1: Thought (LLM) ─ 800ms                           │
  ├── Step 1: Action (Tool) ─ 300ms                           │
  ├── Step 1: Parse ────────── 50ms                           │
  ├── Step 2: Thought (LLM) ─ 600ms                           │
  ├── Step 2: Action (Tool) ─ 1200ms ← 瓶颈                  │
  ├── Step 2: Parse ────────── 50ms                           │
  ├── Step 3: Final (LLM) ──── 500ms                          │
  │                                                            │
  └── 总延迟: 3.5s                                             │
```

## 关键延迟指标

| 指标 | 定义 | 典型值 | 告警阈值 |
|------|------|-------|---------|
| End-to-End Latency | 用户请求到最终回复的总时间 | 2-10s | > 30s |
| LLM Call Latency | 单次 LLM 调用到返回的时间 | 0.5-3s | > 10s |
| Tool Call Latency | 单次工具调用时间 | 0.1-2s | > 5s |
| Time-to-First-Token | 从请求到收到第一个 Token 的时间 | 0.5-2s | > 5s |
| Inter-Step Gap | 思考结束到工具调用开始的时间 | 10-50ms | > 200ms |

## 背景

相比传统 API（延迟通常 < 200ms），Agent 的端到端延迟在秒级甚至分钟级。这使得延迟监控面临不同的挑战：Agent 用户预期等待时间较长，但**不确定性**比绝对值更糟糕——如果用户不知道 Agent "是还在思考还是已经卡住了"，体验会很差。

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 只监控总延迟 | 记录从请求到响应的总时间 | 无法定位瓶颈——"总延迟高，但不知道哪一步慢" |
| 监控单个 LLM 调用 | 只记录 LLM API 的响应时间 | 忽略了工具调用和内部处理的延迟 |
| 仅 P50 监控 | 只关注中位数延迟 | 忽略了长尾——P99 可能是 P50 的 10 倍 |
| 无步骤级延迟 | 不知道每个 ReAct 步骤的耗时 | 无法优化"哪一步效率低" |

## 当前主流优化方向

1. **步骤级延迟分解**——在 Agent 主循环的每个步骤（Thought/Action/Observation）中埋点，自动计算每步耗时，生成延迟瀑布图
2. **流式 TTFB 监控**——监控从请求到首次流式 Token 返回的延迟，这是用户感知延迟的核心指标
3. **异常延迟模式检测**——自动检测 Agent 的延迟异常模式：如"思考时间异常长"（可能陷入推理循环）、"工具调用超时"（下游服务问题）
4. **延迟预算管理**——为 Agent 的每步分配延迟预算（如 "LLM 调用 < 3s，工具调用 < 2s"），超出预算时自动告警或触发降级
5. **长尾延迟的根因分析**——自动关联 P99 延迟 Trace 与对应的 Agent 配置/工具版本/LLM 模型，识别长尾的模式

## 实现的最大挑战

1. **并行调用的延迟归因**——Agent 同时调用多个工具时，总延迟不等于各延迟之和（部分并行），需要正确的并行延迟计算模型
2. **LLM 流式延迟的测量**——SSE 流式响应中，TTFB 和 total streaming time 需要分别测量，前者反映 LLM 服务端处理速度，后者反映响应长度
3. **与外部服务延迟的区分**——工具调用延迟慢是外部服务的问题还是 Agent 的问题？需要区分"Agent 处理时间"和"等待外部服务时间"
4. **延迟采样偏差**——监控系统通常在请求完成时记录延迟，但长时间运行的请求（特别是挂起的请求）在采样窗口内可能被忽略

## 能力边界

**能做什么：**
- 精确定位 Agent 系统的延迟瓶颈（LLM vs 工具 vs 内部处理）
- 监控延迟的长期趋势（延迟是否在逐渐增加）
- 识别异常的延迟模式（如某类请求突然变慢）
- 为 SLO 提供延迟数据支撑

**不能做什么：**
- 不能自动减少延迟——监控只暴露问题，优化需要缓存/量化/模型路由等手段
- 不能代表用户体验——P50 延迟好不代表所有用户都快
- 不能区分"正常慢"和"异常慢"——需要结合业务上下文判断

## 最终工程优化

1. **延迟指标的 HISTOGRAM 存储**——使用 Prometheus Histogram 存储延迟分布（0-1s、1-2s、2-5s、5-10s、10s+），支持 P50/P90/P99 的高效计算
2. **自动延迟基线**——基于过去 7 天的延迟数据自动建立基线，检测"相对于昨天同期"的延迟变化
3. **延迟瀑布图自动生成**——每次 Agent 请求完成后自动从 Trace 中提取延迟数据，生成延迟瀑布图，直接在监控仪表盘展示
4. **慢请求采样**——当请求延迟超过 P99 时，自动保留该请求的完整 Trace 和日志，用于后续根因分析
5. **延迟预算传播**——在 Agent 调用链中传播延迟预算（如 "总预算 10s"），每步执行后更新剩余预算，接近超时时主动降级（如跳过非必需步骤）
