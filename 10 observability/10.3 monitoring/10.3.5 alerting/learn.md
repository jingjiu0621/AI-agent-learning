# 10.3.5 Alerting — 告警规则设计

## 简单介绍

告警（Alerting）是监控系统的"主动通知"能力——当指标超过预设阈值或出现预定义的异常模式时，系统自动通过邮件、短信、IM（Slack/钉钉/飞书）等方式通知相关负责人。在 Agent 系统中，合理的告警机制是确保服务质量和成本可控的最后一道防线。

## 基本原理

告警管道的标准流程：

```
指标数据 → 告警规则评估 → 告警事件 → 通知路由 → 升级策略 → 告警闭环
    │            │            │           │           │
    ▼            ▼            ▼           ▼           ▼
Prometheus   Rule Engine   Alertmanager  Slack/   Escalation
                          /PagerDuty     Email    → 30min/1h/2h
```

```python
# Prometheus 告警规则示例
groups:
  - name: agent_alerts
    rules:
      # 1. 高错误率告警
      - alert: HighLLMErrorRate
        expr: |
          rate(llm_errors_total[5m]) / rate(llm_calls_total[5m]) > 0.05
        for: 2m
        labels:
          severity: critical
          team: agent-infra
        annotations:
          summary: "LLM 错误率超过 5%"
          description: "最近 5 分钟 LLM 错误率为 {{ $value | humanizePercentage }}"

      # 2. Token 消耗异常告警
      - alert: TokenSpike
        expr: |
          sum(rate(token_consumption_total[1h])) > 2 * avg(
            sum(rate(token_consumption_total[1h])) offset 24h
          )
        labels:
          severity: warning
        annotations:
          summary: "Token 消耗异常增加"

      # 3. Agent 死循环检测
      - alert: AgentLoopDetected
        expr: |
          agent_turns_current_session > 20
        labels:
          severity: critical
        annotations:
          summary: "Agent {{ $labels.agent_id }} 可能陷入死循环"
```

## 告警设计原则

Agent 系统中关键的告警维度：

| 告警类型 | 触发条件 | 严重级别 | 响应要求 |
|---------|---------|---------|---------|
| LLM 错误率 > 5% | 5 分钟内 LLM API 返回错误超过 5% | Critical | 立即处理 |
| 工具错误率 > 10% | 5 分钟内工具调用失败率超过 10% | Warning | 工作时间处理 |
| Token 日消耗 > 预算 80% | 当日 Token 消耗超过日预算的 80% | Warning | 关注 |
| Agent 延迟 P99 > 15s | P99 端到端延迟超过 15 秒 | Warning | 调查 |
| Agent 死循环 | 单会话超过 20 步未结束 | Critical | 立即处理 |
| 任务成功率 < 90% | 1 小时内任务完成率低于 90% | Critical | 立即处理 |
| 缓存命中率 < 20% | 1 小时内语义缓存命中率低于 20% | Info | 优化参考 |

## 背景

传统告警系统（如 Prometheus Alertmanager）设计用于监控基础设施和微服务。Agent 系统在此基础上面临新挑战：

1. **告警阈值难以固定**——Agent 的正常延迟范围很大（简单问题 2s，复杂研究 60s），固定阈值不可行
2. **错误率暴涨可能不是故障**——用户突然大量使用 Agent（如新产品上线）会导致错误计数增加但比例正常
3. **"静默失败"最危险**——Agent 可能回复得非常礼貌但内容完全错误，这种错误没有任何告警指标可以捕捉

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 无告警 | 完全靠人工定期查看监控 | 发现问题时已经产生了大量错误和成本 |
| 基础设施告警 | 只告警服务器 CPU/内存/磁盘 | Agent 服务运行正常但"输出错误结果"完全漏掉 |
| 固定阈值告警 | 延迟 > 5s 告警 | 误报极多（复杂研究请求天然需要更长时间） |
| 告警疲劳 | 太多了告警规则，大部分是噪音 | 工程师忽略告警，真出了问题也被忽略 |

## 核心矛盾

| 矛盾维度 | 告警太少 | 告警太多 |
|---------|---------|---------|
| 风险暴露 | 发现时已造成损失 | 误报导致注意力分散 |
| 响应及时性 | 差 | 好 |
| 信任度 | N/A | 低（狼来了效应） |
| 运维质量 | 低 | 低（噪音淹没信号） |

**核心矛盾**：Agent 系统的非确定性使得告警阈值设计成为一门"艺术"——必须平衡敏感度和特异性。**推荐策略：从少到多**——先加 3-5 个核心告警（LLM 错误率、工具错误率、任务成功率），运行一个月稳定后逐渐增加。

## 当前主流优化方向

1. **基于基线的动态告警**——使用过去的指标数据建立动态基线（同时间对比、同业务量对比），而非固定阈值。如"今天的错误率比昨天同期高了 3 倍"而非"错误率 > 5%"
2. **多指标复合告警**——单个指标的波动可能是噪音，但多个指标同时异常是真正的信号。如"延迟上升 + 错误率上升 + 成功率下降"三合一告警
3. **告警分级与升级策略**——Warning 级别走 IM 通知，Critical 级别走电话/PagerDuty，且长时间未确认的告警自动升级到下一级负责人
4. **静默期与维护窗口**——在已知的 Agent 版本迭代期、模型升级期、业务高峰期，自动调整告警阈值的敏感度
5. **行为告警而非指标告警**——监控 Agent 的"行为模式"（如连续调用同一工具 5 次 = 可能循环），而非仅仅是指标

## 实现的最大挑战

1. **"正确"但"不好"的检测**——Agent 回复了语法正确、格式规范的错误答案（如"北京市的温度是 100°C"），这类错误需要语义理解才能发现，传统告警系统无能为力
2. **延迟告警的昼夜模式**——Agent 在白天和夜晚的使用模式不同，对应的延迟基线也不同，单一的阈值策略必然误报
3. **告警风暴**——一个 LLM 服务故障会影响所有 Agent → 所有上游 Agent 都报错 → 告警风暴。需要设计依赖链路的告警抑制规则
4. **多租户告警隔离**——一个租户的 Agent 出问题不应触发全局告警，告警的路由需要与租户隔离策略匹配

## 能力边界

**能做什么：**
- 在 Agent 系统出现异常时第一时间通知相关人员
- 通过告警升级机制确保问题不会被遗漏
- 记录告警历史，用于事后回顾和系统改进
- 通过告警抑制避免告警风暴

**不能做什么：**
- 不能修复 Agent 的内容错误——告警只能检测到可量化的异常
- 不能判断告警的"业务影响"——告警告诉你"错了"，但不告诉你"多严重"
- 不能替代人工判断——告警是辅助工具，最终决策需要人工

## 最终工程优化

1. **告警分组和抑制规则**——如果 LLM 提供商故障导致所有 Agent 都报错，只发一条告警而非 N 条。通过 Alertmanager 的 group_by 和 inhibit_rules 实现
2. **告警与 Runbook 绑定**——每条告警规则都关联一个 Runbook 文档，说明"这个告警意味着什么、应该如何处理"，告警通知中附带 Runbook 链接
3. **告警的预演和测试**——使用 Fault Injection（故障注入）测试告警规则是否真正有效：模拟 Agent 死循环、LLM 超时等场景，验证告警是否触发
4. **告警事后分析（Postmortem）**——每次告警事件后自动生成分析报告：告警原因、响应时间、修复措施、后续改进项
5. **告警疲劳度管理**——追踪每个工程师上周收到告警数量，如果某工程师收到告警太多，自动 Review 其负责系统的告警规则是否需要优化
