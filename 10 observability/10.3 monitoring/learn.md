# 10.3 Monitoring — 实时监控

Monitoring（监控）是 Agent 可观测性的第三层。日志回答"当时发生了什么"，追踪回答"这次请求经历了什么"，监控回答 **"系统整体表现如何"** ——通过聚合指标（Metrics）反映 Agent 系统的健康状态、性能趋势和资源使用。

## 什么是 Agent 监控？

监控关注的是聚合数据而非单个事件：

```
监控指标示例：
  token_consumption_total{model="claude-sonnet-5"}   = 1,250,000 tokens
  agent_latency_p95{agent_type="research"}            = 12.3 seconds
  tool_error_rate{tool="search_web"}                  = 2.1%
  active_agents{environment="production"}              = 47
  agent_turns_per_task{agent_type="assistant"}         = 5.2 (avg)
```

## 监控 vs 日志 vs 追踪

| 维度 | 日志 (Logs) | 追踪 (Traces) | 监控 (Metrics) |
|------|------------|---------------|----------------|
| 数据结构 | 事件 | Span 树 | 时序指标 |
| 粒度 | 单条事件 | 单次请求 | 聚合（窗口/直方图） |
| 查询方式 | 搜索 | 按 trace_id | 按时间 + 标签 |
| 存储量 | 最大 | 中 | 最小 |
| 查询延迟 | 秒级 | 秒级 | 毫秒级 |
| 典型用途 | 调试 | 延迟分析 | 告警、容量规划 |
| 保留时间 | 天~周 | 周~月 | 月~年 |

## 核心设计原则

1. **从指标反推告警**——先定义"我们需要什么告警"，再反推需要收集哪些指标，而不是先收集所有指标再考虑告警
2. **RED 方法**——Rate（请求率）+ Errors（错误率）+ Duration（延迟），三个指标覆盖大部分监控需求
3. **USE 方法**——Utilization（利用率）+ Saturation（饱和度）+ Errors（错误率），用于系统资源监控
4. **服务的概念引入**——将 Token 消耗视为"成本服务"、将 LLM 延迟视为"性能服务"，与传统基础设施指标统一管理
5. **以 SLO 为中心**——指标不是目的，达到 SLO 才是。每个指标都应有对应的 SLO 目标

## 关键工程启示

- Agent 的延迟远比传统 API 高（秒级 vs 毫秒级），监控系统的采集周期应该调整（15s → 60s）
- Token 监控需要按模型维度拆分，因为不同模型的成本差异巨大（如 Claude Haiku vs Opus 可差 10x）
- 工具错误率需要单独追踪——Agent 系统的"下游依赖"比传统系统更多样
- 监控仪表盘应按不同角色设计：开发看延迟/错误，运维看资源，管理者看成本
