# 10.3.4 Dashboard Building — 监控仪表盘构建

## 简单介绍

监控仪表盘（Dashboard）将 Agent 的原始指标转化为**可视化的系统状态视图**。好的仪表盘让运维人员"一眼"就能判断系统健康状态，而不需要逐条查看日志或查询数据库。

## 基本原理

仪表盘的核心设计原则是 **GLCE**：Goal（目标）- Level（水平）- Color（颜色）- Exceptions（异常）。

```python
# 仪表盘指标体系设计示例
DASHBOARD_LAYOUT = {
    "Row 1: 高层面板（Goal）": [
        {"title": "Token 消耗", "metric": "total_tokens", "type": "timeseries"},
        {"title": "活跃 Agent 数", "metric": "active_agents", "type": "stat"},
        {"title": "任务成功率", "metric": "task_success_rate", "type": "gauge"},
    ],
    "Row 2: 性能面板（Level）": [
        {"title": "端到端延迟 P50/P95/P99",
         "metric": "end_to_end_latency", "type": "timeseries"},
        {"title": "LLM 调用延迟", "metric": "llm_latency", "type": "timeseries"},
        {"title": "工具调用延迟", "metric": "tool_latency", "type": "timeseries"},
    ],
    "Row 3: 错误面板（Color）": [
        {"title": "LLM 错误率", "metric": "llm_error_rate", "type": "timeseries"},
        {"title": "工具错误率", "metric": "tool_error_rate", "type": "timeseries"},
        {"title": "TOP 5 错误类型", "metric": "errors_by_type", "type": "table"},
    ],
    "Row 4: 深入面板（Exceptions）": [
        {"title": "慢 Trace 采样", "metric": "slow_traces", "type": "list"},
        {"title": "异常 Agent 列表", "metric": "failing_agents", "type": "table"},
    ],
}
```

## 背景

Agent 监控仪表盘不同于传统的 IT 监控仪表盘。传统的 CPU/内存/磁盘面板对 Agent 运维的价值有限——Agent 运维需要的是"Token 消耗趋势"、"Agent 是否陷入循环"、"哪个工具响应最慢"等 Agent 原生的指标。

## 之前的做法与结果

| 做法 | 描述 | 结果 |
|------|------|------|
| 通用 Grafana + K8s 面板 | 标准的 CPU/内存/网络面板 | 无 Agent 特定指标，对 Agent 运维帮助有限 |
| 单一大屏 | 把所有指标放在一个大屏上 | 信息过载，无法快速定位问题 |
| 纯技术指标 | 只有 LLM 延迟和 Token 数 | 缺少业务视角的指标（如任务成功率） |
| 无仪表盘 | 全靠日志搜索 | 缺乏全局视图，发现问题的时机严重滞后 |

## 核心设计原则

1. **从上到下的信息降噪**——顶部是高层"健康状态"（系统是否正常），中间是"性能趋势"（是否在变慢），底部是"异常详情"（有哪些具体问题）
2. **红色/黄色/绿色编码**——每个核心面板都配置告警阈值，值在正常范围 = 绿色，接近阈值 = 黄色，超阈值 = 红色
3. **静态基线 vs 动态基线**——固定阈值（如 "延迟 > 10s = 红"）适合有明显界限的指标；动态基线（如 "错误率比过去 7 天均值高 3σ"）适合无明显界限的指标
4. **默认 5 分钟窗口**——默认展示最近 5 分钟的实时数据，支持切换到 1h/6h/24h/7d 查看趋势

## 当前主流优化方向

1. **AIOps 集成**——将 Agent 监控仪表盘与 AIOps 系统集成，自动检测异常指标并生成根因分析报告
2. **角色定制面板**——开发团队关注延迟/错误/调试信息，运维团队关注资源/部署/稳定性，管理者关注成本/SLA/趋势，不同角色看到不同的仪表盘
3. **事件驱动更新**——仪表盘不是"手动刷新"的静态页面，而是通过事件推送实时更新（如 Agent 每次完成一步就更新一步的指标）
4. **Trace-to-Dashboard 一键跳转**——在仪表盘上点击一个高延迟数据点，直接跳转到对应请求的完整 Trace 视图
5. **自动面板发现**——Agent 框架自动注册新的指标和维度，仪表盘自动发现并展示，无需手动配置

## 实现的最大挑战

1. **指标的基数爆炸**——Agent 的指标维度很多（agent_id、model、tool、task_type），每个维度组合都生成面板会导致面板数量爆炸
2. **有意义的阈值设置**——Agent 系统的正常指标范围比传统系统更宽（延迟从 1s 到 60s 都可能正常），阈值设置需要经验积累
3. **多 Agent 拓扑图**——多 Agent 协作系统的拓扑图在仪表盘中展示很困难，Agent 之间的调用关系动态变化，难以静态展示
4. **仪表盘与 AB 测试的关联**——当团队同时测试多个 Agent 版本时，仪表盘需要支持"分别展示不同版本 Agent 的指标"

## 能力边界

**能做什么：**
- 一目了然地展示 Agent 系统的整体健康状态
- 支持不同角色的视角（开发/运维/管理者）
- 快速识别异常趋势和模式
- 提供从指标到 Trace 的排查路径

**不能做什么：**
- 不能自动根因分析——仪表盘告诉你"出了问题"，但不能告诉你"为什么出问题"
- 不能替代告警——仪表盘需要人主动去看，告警是主动通知
- 不能解决"不知道看什么"的问题——仪表盘的设计反映了团队对系统的理解，设计不好就没用

## 最终工程优化

1. **Grafana + Prometheus 标准栈**——使用 Prometheus 存储 Agent 指标（每秒可处理百万级数据点），Grafana 构建仪表盘（丰富的面板类型和告警支持）
2. **面板变量化**——在仪表盘中使用变量（agent_id、环境、模型），支持下拉筛选和动态查询，一个面板覆盖所有 Agent
3. **重点指标 + 探索区域 分离**——将面板分为"必须关注"（核心 RED 指标）和"按需探索"（细分维度分析），前者自动刷新常驻，后者按需打开
4. **定期仪表盘 Review**——每月检查仪表盘的使用率：无人查看的面板删除，不够用的维度新增，保持仪表盘的"瘦身"
5. **告警与仪表盘联动**——在告警通知中附带对应仪表盘的链接和当前快照，告警接收人点击链接直接看到相关面板的上下文
