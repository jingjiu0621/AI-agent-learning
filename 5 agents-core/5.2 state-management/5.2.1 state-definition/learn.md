# 5.2.1 state-definition — 状态构成：会话状态、任务状态、环境状态

## 简单介绍

Agent 的状态不是一个简单的 JSON 对象，而是由多个层次、不同生命周期的状态共同组成。理解状态的分层结构是设计可维护 Agent 的基础。

## 基本原理

### 三层状态模型

```
┌─────────────────────────────────────┐
│  会话状态（Session State）           │ ← 用户连接级别
│  - 认证信息                          │
│  - 会话配置 / 偏好                   │
│  - 整个会话的生命周期                 │
├─────────────────────────────────────┤
│  任务状态（Task State）              │ ← 单个任务级别
│  - 任务目标 / 子任务列表              │
│  - 当前进度 / 已完成步骤              │
│  - 中间结果缓存                      │
├─────────────────────────────────────┤
│  环境状态（Environment State）        │ ← 外部环境级别
│  - 当前时间 / 时区                    │
│  - 可用工具 / 资源状态               │
│  - 外部系统连接状态                   │
└─────────────────────────────────────┘
```

### 状态生命周期

| 状态类型 | 生命周期 | 存储位置 | 示例 |
|---------|---------|---------|------|
| 会话状态 | 会话期间 | Redis / 内存 | user_id, locale |
| 任务状态 | 任务期间 | Agent State | step, results |
| 环境状态 | 执行期间 | 动态获取 | tool_status |

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 早期 Agent | 全混在一个 dict | 不可维护、难以调试 |
| LangGraph | TypedDict State 模式 | Schema 变更麻烦 |
| 现代实践 | 分层解耦 + 不可变事件 | 实现复杂度高 |

## 核心矛盾

**状态完整性 vs 状态体积**：
- 保留所有状态 → 决策信息充分 → 上下文膨胀、Token 消耗大
- 精简状态 → 体积小、速度快 → 可能丢失关键信息

## 主流优化方向

1. **最小必要状态原则**：只保留当前决策必需的状态
2. **状态分片**：不同层级的状态分开存储，按需加载
3. **延迟加载**：大块状态（如日志）仅在需要时加载
4. **状态快照**：关键节点打快照用于恢复和审计

## 实现挑战

1. **状态一致性**：三层状态之间可能冲突（如环境状态已变但任务状态未感知）
2. **状态序列化**：复杂对象（如 numpy array、ML model）的序列化
3. **状态版本管理**：代码升级后状态 Schema 不兼容

## 能力边界

- 无法 100% 捕获所有与环境相关的状态（外部系统总有 Agent 未知的状态）
- 状态定义需要在通用性和效率间做取舍

## 核心优势

清晰的三层状态划分让 Agent 状态 **可理解、可调试、可恢复**，是生产级 Agent 的基础保障。

## 工程优化

1. 使用 Pydantic / dataclass 定义强类型状态
2. 会话和任务状态分表存储
3. 环境状态动态获取而非缓存（避免过期）
4. 状态变更记录日志用于审计

## 场景判断

- 客服 Agent：会话状态最重要（用户身份、历史记录）
- 数据分析 Agent：任务状态最重要（数据处理管线进度）
- 监控 Agent：环境状态最重要（当前系统指标）
