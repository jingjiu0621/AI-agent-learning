# 5.2.4 state-recovery — 状态恢复与故障转移

## 简单介绍

当 Agent 进程崩溃、节点宕机或网络中断时，能从持久化的状态中恢复并继续执行是生产级 Agent 的必备能力。状态恢复决定了 Agent 系统的可用性（Availability）和可靠性（Reliability）。

## 基本原理

### 恢复流程

```
检测故障
    ↓
识别恢复点 ← 读取最近的检查点
    ↓
加载状态快照
    ↓
重放未提交的事件
    ↓
验证状态一致性
    ↓
继续执行
```

### 检查点策略

| 策略 | 描述 | 恢复时间 | 实现复杂度 |
|------|------|---------|---------|
| 每步检查点 | 每一步都保存 | 快（秒级） | 低 |
| 定期检查点 | 每 N 步保存一次 | 中（需重放） | 低 |
| 异步检查点 | 后台线程保存 | 快（可能丢少量） | 中 |
| 分布式检查点 | 多节点协调保存 | 取决于协调协议 | 高 |

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 无恢复 | 崩溃即丢失 | 无法用于生产 |
| 简单 JSON 保存 | 每步写文件 | 性能差、并发问题 |
| LangGraph Checkpoint | 内置检查点机制 | 限定于 LangGraph |
| 事件溯源 | 完整事件日志可回放 | 存储成本高 |

## 核心矛盾

**恢复粒度 vs 性能开销**：
- 每步检查 → 快速恢复 → 写入开销大
- 少检查点 → 写入快 → 恢复慢（需要重放多步）

## 主流优化方向

1. **增量检查点**：只保存变化部分而非全量状态
2. **多级检查点**：内存缓冲 + 磁盘快照 + 远程备份
3. **预写日志（WAL）**：先写日志再执行操作
4. **自动故障转移**：主节点崩溃后从节点自动接管
5. **幂等恢复**：确保重放操作不会产生副作用

## 实现挑战

1. **"幽灵"状态**：恢复后的状态与真实外部状态不一致（如已发邮件但恢复后会再发一次）
2. **分布式一致性**：多 Agent 协作场景的恢复一致性
3. **非幂等工具**：调用外部 API 的操作无法安全重放
4. **存储依赖**：恢复过程中存储系统本身不可用

## 能力边界

- 无法恢复外部系统的副作用（如已发送的邮件）
- 恢复时间随状态大小线性增长
- 无法保证在各类故障下 100% 恢复

## 核心优势

自动状态恢复让 Agent 系统达到 **99.9%+ 可用性**，是现代 Agent 基础设施的必备组件。

## 工程优化

1. 对非幂等操作使用"预检-执行-确认"模式
2. 检查点合并策略：N 个增量检查点后合并为一个全量
3. 恢复后进行完整性自检，发现不一致主动报错
4. 为长时间计算任务实现进度条恢复

## 场景判断

- 非关键任务：定期检查点（每 5 步）即可
- 关键交易任务：每步检查点 + 预写日志
- 长时间执行任务：异步增量检查点
