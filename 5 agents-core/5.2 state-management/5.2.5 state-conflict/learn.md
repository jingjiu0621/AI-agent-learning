# 5.2.5 state-conflict — 状态冲突检测与解决

## 简单介绍

在分布式或多 Agent 场景中，不同 Agent 或进程可能同时修改同一状态，导致冲突。状态冲突检测与解决机制确保状态的一致性和正确性。

## 基本原理

### 冲突类型

```
┌─────────────────────────────────────────┐
│  写写冲突（Write-Write）                 │
│  Agent A 和 Agent B 同时修改同一字段     │
│  → 最后一个写入者覆盖前一个               │
├─────────────────────────────────────────┤
│  读写冲突（Read-Write）                  │
│  Agent A 读到旧值并基于它做决策          │
│  → 决策基于过时信息                       │
├─────────────────────────────────────────┤
│  语义冲突（Semantic）                    │
│  操作本身在语义上冲突                    │
│  → A 说"发送邮件"，B 说"取消发送"        │
└─────────────────────────────────────────┘
```

### 解决策略

| 策略 | 描述 | 适用场景 |
|------|------|---------|
| Last-Write-Wins | 最后写入者获胜 | 非关键状态 |
| 乐观锁（版本号） | 写入时检查版本 | 高竞争状态 |
| 悲观锁 | 写入前锁住状态 | 关键状态 |
| CRDT | 无冲突数据类型 | 协作场景 |
| 人工解决 | 标记冲突让人决策 | 高风险操作 |

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 单进程 | 无冲突（同步执行） | 无法扩展 |
| 多进程无协调 | 最后写入者获胜 | 数据丢失 |
| 乐观锁 | 版本号检查 | 重试成本 |
| CRDT 合并 | 自动无冲突合并 | 实现复杂 |

## 核心矛盾

**一致性 vs 可用性**：
- 强一致性 → 系统变慢（需要协调）
- 最终一致性 → 快 → 可能读到不一致数据

## 主流优化方向

1. **Operation Transform**：基于操作而非状态的合并
2. **Three-way Merge**：类似 Git 的合并策略
3. **声明式冲突解决**：用规则声明如何解决冲突
4. **自动 CRDT**：为常用数据结构自动化 CRDT 转换

## 实现挑战

1. **冲突检测的完整覆盖**：发现所有可能的冲突模式
2. **合并的正确性**：合并后的状态在语义上是否正确
3. **用户透明性**：冲突解决对用户无感
4. **性能开销**：版本检查和冲突检测增加延迟

## 能力边界

- CRDT 无法覆盖所有数据类型
- 自动合并无法解决语义冲突（需要人类判断）
- 冲突检测增加了系统的整体复杂度

## 核心优势

系统性的冲突处理让 Agent 系统从"偶尔出错"变为"优雅处理并发"。

## 工程优化

1. 尽量设计无状态操作减少冲突可能
2. 使用 Agent 专用状态空间避免交叉
3. 乐观锁失败时自动重试（带指数退避）
4. 对高风险冲突预设降级策略

## 场景判断

- 单 Agent：冲突极少，Last-Write-Wins 足够
- 多 Agent 写同一状态：需要乐观锁或 CRDT
- 金融/交易：必须悲观锁
