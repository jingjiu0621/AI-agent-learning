# 5.2.2 state-transition — 状态机与状态转移图

## 简单介绍

状态转移定义了 Agent 在不同阶段之间如何转换。使用状态机模型可以将 Agent 的行为形式化，使其可预测、可验证。从"初始"到"执行"到"完成"，每一步转移都有明确的触发条件和动作。

## 基本原理

### Agent 状态机模型

```
          start
            ↓
      ┌──────────┐
      │  IDLE    │ ← 等待任务
      └────┬─────┘
           │ task_assigned
           ↓
      ┌──────────┐
      │ THINKING │ ← LLM 推理中
      └────┬─────┘
     ┌─────┼──────┐
     ↓     │      ↓
  ACTING   │   COMPLETED
     ↓     │
  ┌──────┐ │
  │ TOOL │ │
  │ WAIT │──┘
  └──┬───┘
     ↓ error
  ┌──────────┐
  │ HANDLING │ ← 错误处理
  │  ERROR   │
  └────┬─────┘
       │ recovered
       └→ THINKING
```

### 转移表表示

| 当前状态 | 事件 | 条件 | 下一状态 | 动作 |
|---------|------|------|---------|------|
| IDLE | task_assigned | - | THINKING | 初始化任务上下文 |
| THINKING | thought_ready | has_action | ACTING | 提取行动参数 |
| THINKING | thought_ready | is_final | COMPLETED | 生成最终输出 |
| ACTING | tool_result | - | THINKING | 注入观察结果 |
| ACTING | tool_error | retryable | HANDLING_ERROR | 执行重试逻辑 |
| HANDLING_ERROR | recovered | - | THINKING | 恢复推理 |
| HANDLING_ERROR | fatal | - | FAILED | 终止并报告 |

## 背景与演进

| 时期 | 方式 | 问题 |
|------|------|------|
| 无状态机 | if-else 判断 | 状态爆炸、维护灾难 |
| 有限状态机 | 固定状态集 | 场景变化需改代码 |
| LangGraph | 图状 StateGraph | 灵活但调试困难 |
| 事件溯源 | 不可变事件流 | 实现复杂、查询慢 |

## 核心矛盾

**表达力 vs 简单性**：
- 简单状态机状态少、易理解 → 无法表达复杂行为
- 图状状态机表达力强 → 状态路径太多，验证困难

## 主流优化方向

1. **层次状态机（HSM）**：状态嵌套，高层状态包含子状态
2. **事件驱动**：基于事件发布-订阅驱动状态转移
3. **图数据库存储**：用图数据库存储状态机定义
4. **形式化验证**：模型检查确保无死锁/陷阱状态

## 实现挑战

1. **并发状态**：多个子任务同时进行时状态如何表示
2. **状态爆炸**：条件组合导致的状态数指数增长
3. **中间状态**：转移过程中的"半状态"处理

## 能力边界

- 任何有限状态机都无法覆盖所有真实世界的边界情况
- 状态转移的决策依赖于正确的观察，而观察可能出错

## 核心优势

显式状态机让 Agent 的执行路径 **可预测、可测试、可形式验证**。

## 工程优化

1. 用状态图绘制工具可视化 Agent 状态机
2. 单元测试覆盖所有合法转移
3. 记录每次状态转移（审计日志）
4. 为状态机设计 fallback 状态（如 UNKNOWN）

## 场景判断

- 简单 Agent：3-5 个状态即可
- 复杂编排 Agent：需要层次状态机
- 生产级 Agent：必须有 FAILED 和 UNKNOWN 兜底状态
