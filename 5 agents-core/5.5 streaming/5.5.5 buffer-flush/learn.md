# 5.5.5 buffer-flush — Buffer 与 Flush 策略

## 简单介绍

Buffer-Flush 策略决定 Agent 流式输出中何时将缓存的数据推送给客户端。过早推送增加网络开销，过晚推送增加用户感知延迟。合理的 Buffer-Flush 策略是流式体验的关键优化点。

## 基本原理

### 触发 Flush 的条件

```
┌────────────────────────────────────────┐
│  定时 Flush                            │
│  每 200ms 刷新一次缓存                 │
├────────────────────────────────────────┤
│  边界 Flush                            │
│  遇到句子边界（句号、换行、感叹号）    │
├────────────────────────────────────────┤
│  事件 Flush                            │
│  遇到重要事件（工具调用开始/结束）     │
├────────────────────────────────────────┤
│  强制 Flush                            │
│  Agent 循环结束 / 缓存超过最大大小     │
└────────────────────────────────────────┘
```

### Buffer 大小 vs 延迟权衡

| 策略 | 平均延迟 | 网络开销 | 用户体验 |
|------|---------|---------|---------|
| 每 Token 推送 | 极低 | 极高 | 断断续续 |
| 200ms 定时推送 | 低 | 中 | 流畅 |
| 句子边界推送 | 中 | 低 | 自然 |
| 全量推送 | 高 | 极低 | 等待时间长 |

## 背景与演进

早期流式实现采用"每 Token 推送"，网络开销大，且客户端渲染频繁导致卡顿。后逐步演进到使用 Buffer-Flush 策略平衡实时性和效率。

## 核心矛盾

**实时性 vs 网络效率**：
- 频繁 Flush → 延迟低 → 网络包多，开销大
- 少 Flush → 网络效率高 → 延迟高

## 主流优化方向

1. **自适应 Flush**：根据网络状况和客户端处理能力动态调整
2. **内容感知 Flush**：在语义完整边界 Flush 而非固定间隔
3. **优先级队列**：高优先级消息（如状态更新）立即推送，低优先级（如详细日志）攒批
4. **Nagle 算法类似策略**：在延迟和效率间动态平衡

## 实现挑战

1. **Flush 延迟累积**：多个 Flush 条件同时触发时消息顺序保障
2. **网络拥塞**：频繁 Flush 在小带宽网络上造成拥塞
3. **客户端渲染压力**：高频率 Flush 导致客户端 UI 线程过载
4. **边界识别准确性**：语义边界（完整句子）的可靠检测

## 能力边界

- 没有一种 Buffer-Flush 策略在所有场景都最优
- 语义边界的检测受限于语言和 Agent 输出格式
- 自适应策略需要足够的观测数据才能生效

## 核心优势

好的 Buffer-Flush 策略在不改变网络和硬件条件下 **显著提升流式体验**。

## 工程优化

1. 默认组合策略：200ms 定时 + 句子边界 Flush
2. Token 积累到 50 个以上强制 Flush
3. 工具调用开始/结束事件强制立即 Flush
4. 在 agent loop 结束时无条件 Flush

## 场景判断

- 实时聊天：偏向低延迟（100ms 定时 + 句子边界）
- 长文本生成：偏向效率（500ms 定时 + 段落边界）
- 代码生成：行边界 Flush（代码每行推送）
