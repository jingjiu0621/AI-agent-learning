# 5.5.6 streaming-error — 流式传输中的错误处理

## 简单介绍

流式传输中的错误处理比批处理复杂得多——错误可能在部分输出已发送后发生，需要处理"已发送的内容怎么办"的问题。

## 基本原理

### 流式错误类型

```
┌────────────────────────────────────────┐
│  连接断开                              │
│  客户端断开 / 网络中断 / 超时          │
│  → 连接中断时已发送部分内容            │
├────────────────────────────────────────┤
│  LLM 中途错误                          │
│  生成过程中 API 返回错误 / 内容过滤    │
│  → 部分 Token 已发送，突然中断         │
├────────────────────────────────────────┤
│  工具执行错误                          │
│  工具调用失败但之前已推送"调用中"状态  │
│  → 状态更新和纠错需要补偿              │
├────────────────────────────────────────┤
│  逻辑错误                              │
│  Agent 发现之前的推理有误              │
│  → 需要撤回或更正已发送的内容          │
└────────────────────────────────────────┘
```

### 错误恢复策略

```python
# 纠错事件
{"type": "correction", "replace_range": {"start": 100, "end": 250}, "new_content": "..."}

# 错误事件
{"type": "error", "code": "TOOL_FAILED", "message": "天气服务暂时不可用", "severity": "warning"}

# 重试事件
{"type": "retry", "step": "get_weather", "attempt": 2, "strategy": "fallback_api"}
```

## 背景与演进

批处理模式下，错误是一个"要么全有要么全无"的问题。流式下错误发生时部分内容可能已经推送给用户，需要正向处理"不完整"的状态。

## 核心矛盾

**及时反馈 vs 错误隔离**：
- 尽快推送结果 → 用户体验好 → 错误暴露在用户面前
- 错误验证后再推送 → 避免展示错误 → 增加延迟

## 主流优化方向

1. **纠错事件协议**：定义 replace/delete/insert 事件模型
2. **内容确认机制**：重要内容确认无误后再推送
3. **错误回滚 UI**：客户端支持删除/替换已展示的错误内容
4. **优雅错误提示**：在流中插入错误提示而非终止流

## 实现挑战

1. **编辑冲突**：客户端已渲染的内容被替换时的 UI 过渡
2. **错误扩散**：一个错误导致后续多个错误，纠正链复杂
3. **幂等纠错**：同一条纠错指令被多次执行的问题
4. **客户端存储**：客户端需要保留已渲染内容的索引才能支持替换

## 能力边界

- 无法撤回客户端已展示且用户已阅读的内容
- 纠错事件需要客户端配合实现，增加前端复杂度
- 某些错误（如生成了完整但错误的回答）无法增量修复

## 核心优势

流式错误处理避免了"一错全废"——**错误可以就地修复**，而非终止整个流。

## 工程优化

1. 使用 replace 操作而非 append-only 的流式协议
2. 工具调用结果先验证再推送给用户
3. 错误信息以非破坏性方式插入（如通知条而非替换内容）
4. 客户端支持内容版本号，便于追踪变更

## 场景判断

- 聊天 Agent：错误时插入提示，继续尝试
- 代码生成 Agent：错误时需要撤回错误代码
- 数据分析 Agent：图表生成错误可单独替换图表区域
