# 5.5.4 cancellation — 流式取消与中断

## 简单介绍

流式取消（Cancellation）允许用户在 Agent 执行过程中停止正在进行的操作。这是交互式 Agent 体验的关键组成部分——用户应该能随时说"停，换个方向"。

## 基本原理

### 取消信号传播路径

```
用户点击"停止"按钮
    ↓
客户端发送取消请求（HTTP DELETE / WebSocket 关闭帧）
    ↓
服务端接收取消信号
    ↓
┌─ 传播到各层 ──────────────────────┐
│  → LLM 调用：中止当前生成          │
│  → 工具执行：中断正在运行的调用    │
│  → Agent 循环：跳出循环            │
│  → 状态保存：保存已完成的中间结果  │
└────────────────────────────────────┘
    ↓
返回已完成的（部分）结果给用户
```

### 取消类型

| 类型 | 行为 | 用户感知 |
|------|------|---------|
| 软取消 | 完成当前步骤后停止 | 当前步骤完成，后续不执行 |
| 硬取消 | 立即中止所有操作 | 可能留下不完整的状态 |
| 优雅取消 | 保存检查点后停止 | 可后续恢复 |

## 背景与演进

在没有取消机制的早期 Agent 中，用户一旦提交了请求就只能等待完成。对于长时间运行的 Agent，用户可能等待 30 秒以上而无法中断，体验极差。

## 核心矛盾

**取消及时性 vs 状态一致性**：
- 立即取消 → 响应快 → 状态可能不一致（工具调用半途中止）
- 优雅取消 → 状态一致 → 取消有延迟

## 主流优化方向

1. **取消令牌（CancellationToken）**：类似 .NET 的协作式取消模式
2. **分层取消**：不同层面的取消策略不同（LLM 取消 vs 工具取消）
3. **取消确认**：对于不可逆操作，取消前询问确认
4. **部分结果提交**：取消时保存已完成的成果，而非全部丢弃

## 实现挑战

1. **非可中断操作**：某些 LLM 调用或工具操作不支持中断
2. **资源泄漏**：取消后未正确释放资源（如数据库连接）
3. **竞态条件**：取消信号到达时操作刚好完成
4. **级联取消**：一个 Agent 的取消需要传播到子 Agent 或下游服务

## 能力边界

- 非可中断的外部 API 调用无法被取消
- 取消不能完全消除已产生的副作用（如已发送的日志）
- 取消操作的传播有时间延迟

## 核心优势

取消机制给了用户 **"控制权"**——不是被动等待 Agent 完成，而是可以随时调整方向。

## 工程优化

1. 使用 CancellationToken 模式实现协作式取消
2. 取消时执行清理：关闭连接、释放句柄
3. 取消后返回已完成的中间结果而非空响应
4. 超时自动取消作为兜底

## 场景判断

- 交互式 Agent：必须支持取消（用户等待的不耐烦阈值约 5 秒）
- 批处理 Agent：取消非必需（但需要超时）
- 编排型 Agent：取消需要级联传播到子任务
