# 5.5.2 intermediate-state — 中间状态暴露

## 简单介绍

中间状态暴露（Intermediate State Exposure）是 Agent 将执行过程中的中间状态（思考过程、工具调用进度、部分结果）主动推送给客户端的技术。这解决了"Agent 在想什么？它在做什么？"的用户疑问。

## 基本原理

### 事件类型

```json
// 思考事件
{"type": "thought", "content": "用户想知道今天的天气，我需要先获取位置信息"}

// 工具调用事件
{"type": "tool_call", "tool": "get_location", "status": "executing"}

// 工具结果事件 
{"type": "tool_result", "tool": "get_location", "content": "北京", "status": "success"}

// 进度事件
{"type": "progress", "current": 2, "total": 5, "message": "分析数据中..."}

// 决策事件
{"type": "decision", "choice": "use_weather_api", "reason": "已获取位置信息"}
```

### 客户端渲染示例

```
用户：今天天气怎么样？

Agent 正在思考...
  └→ 思考：用户想知道天气，先获取位置
Agent 正在执行...
  ├→ 🔍 正在获取位置... ✅ 北京
  ├→ 🌤️ 正在查询天气... ✅ 25°C 晴
  └→ 📋 正在生成回答...

今天北京天气晴朗，气温 25°C，适合户外活动。
```

## 背景与演进

无中间状态的 Agent 就像一个黑盒——用户输入问题，等了几秒后看到结果。如果 Agent 在默默重试失败的工具，用户完全不知情。中间状态暴露使 Agent 的行为透明化。

## 核心矛盾

**透明度 vs 信息噪音**：
- 暴露越多 → 越透明 → 用户被过多技术细节淹没
- 暴露越少 → 越简洁 → 用户不知道 Agent 在做什么

## 主流优化方向

1. **分级暴露**：普通用户看简洁版，开发者看详细版
2. **语义化事件**：用自然语言描述中间状态而非技术 log
3. **可交互状态**：用户可点击中间状态查看详情
4. **预测下一步**：根据中间状态预渲染可能的下一步 UI
5. **状态摘要**：多步状态汇总为一句简洁描述

## 实现挑战

1. **状态到描述的映射**：技术状态（如"调用 API get_weather"）→ 用户友好的描述（如"正在查询天气"）
2. **事件顺序一致性**：并行执的行为事件顺序要在 UI 上合理展示
3. **状态更新频率控制**：太频繁导致 UI 闪烁，太少失去实时感
4. **敏感信息过滤**：中间状态可能包含 API Key、Token 等敏感信息

## 能力边界

- 中间状态的事件定义需要在设计时确定，无法覆盖所有 Agent 行为
- 过多的中间事件可能使用户困惑
- 状态暴露增加了 API 设计和客户端的复杂度

## 核心优势

中间状态暴露将 Agent 从"黑盒"变为"玻璃盒"，**用户信任度显著提升**。

## 工程优化

1. 定义标准事件类型枚举（type + payload 结构）
2. 敏感信息自动过滤（关键词匹配 + 正则）
3. 事件降采样：高频事件合并为一条汇总
4. 为开发者保留详细模式（debug=1 参数）

## 场景判断

- 终端用户视角：暴露"思考中→执行中→完成"三个状态即可
- 开发者调试：暴露完整技术细节
- 监控系统：暴露所有事件流用于日志和分析
