# 5.5.3 partial-output — 部分输出渲染与增量更新

## 简单介绍

部分输出渲染（Partial Output Rendering）是指 Agent 在生成过程中逐步展示可用的部分结果，而非等待所有内容生成完毕。例如 Agent 生成了图表代码后立即渲染，而不是等所有文本也生成完再展示。

## 基本原理

### 增量渲染模式

```
时间线 →
Agent 生成: [思考] → [写代码] → [执行] → [分析结果] → [生成报告]

传统方式:              ═══════════ 全部完成后渲染 ════════════→
增量方式: [展示思考] → [展示代码] → [展示图表] → [展示分析] → [展示报告]
```

### 示例：数据分析 Agent

```python
# 流式事件
async for event in agent.stream("分析销售数据"):
    match event.type:
        case "code_generated":
            # 代码一生成就展示，用户可以提前看到分析思路
            ui.show_code_block(event.code)
        case "execution_result":
            # 结果一出来就渲染图表
            ui.render_chart(event.chart_data)
        case "analysis":
            # 逐步追加文本分析
            ui.append_text(event.text)
```

## 背景与演进

传统 UI 模式是"全有或全无"——要么看不到，要么全部看到。增量渲染改变了这个模式，让用户可以"看着 Agent 工作"。

## 核心矛盾

**渲染完整性 vs 响应实时性**：
- 立即展示部分结果 → 响应快 → 可能展示不完整/错误的内容
- 等待完整结果 → 内容准确 → 用户等待时间长

## 主流优化方向

1. **可更新占位**：先展示占位/骨架屏，结果就绪后填充
2. **渐进式增强**：先展示文本，再逐步嵌入图表/图片
3. **结果锁定**：一旦某部分结果确认，内容不再变化（避免用户困惑）
4. **差异更新**：只发送变化的部分而非全量替换

## 实现挑战

1. **部分结果的状态管理**：已展示的部分结果可能在后续发现是错误的，如何处理？
2. **UI 一致性**：增量更新过程中 UI 的过渡要平滑
3. **前端复杂度**：客户端需要处理多种增量更新模式
4. **交互状态**：用户可能在增量渲染过程中进行交互

## 能力边界

- 某些类型的结果无法增量展示（如完整的 PDF 报告）
- 部分结果可能误导用户（如展示了部分分析但后续结果推翻它）
- 增量渲染增加前后端复杂度

## 核心优势

部分输出渲染将用户的等待时间从"焦虑的空白"转变为**"有内容可看"**，极大改善用户体验。

## 工程优化

1. 先渲染确定性内容（如已完成的代码），再渲染分析性内容
2. 用骨架屏或加载动画填充渲染间隙
3. 结果状态标记：final（最终结果）vs streaming（临时结果，可能变化）
4. 客户端维护一个可更新的结果树，增量合并

## 场景判断

适合增量渲染的场景：
- 数据分析（逐步加载图表和结果）
- 代码生成（先显示代码，再显示运行结果）
- 长报告生成（先显示大纲，再逐步填充）

不适合增量渲染的场景：
- 短文本回复（增量意义不大）
- 法律/金融文档（需要确认完整内容无误）
