# 5.3.4 lossy-compression — 有损压缩：关键信息抽取

## 简单介绍

有损压缩（Lossy Compression）在 Agent 上下文中指主动丢弃低价值信息，只保留关键信息。与摘要不同，有损压缩不生成新的文本，而是从原始内容中提取最有价值的部分。

## 基本原理

### 信息价值分层

```
每条消息的价值取决于：
1. 任务相关性：与当前任务目标的关联度
2. 信息新颖性：是否包含新信息（vs 重复已有信息）
3. 决策影响力：是否影响后续决策
4. 时间衰减：旧消息的价值随时间递减
```

### 抽取流程

```
原始工具结果（如 API 返回 5000 tokens）
    ↓
关键信息抽取：
  - 提取核心数据点（数字、状态码、摘要句）
  - 丢弃冗余信息（日志、重复字段、格式化内容）
    ↓
保留：{"status": "ok", "count": 142, "errors": ["timeout: job_3"]}
丢弃：详细的请求日志、每个数据项的全部字段
```

## 背景与演进

早期的 Agent 框架倾向于保留完整的消息历史，因为"多一点总比少一点好"。但实践发现大量低价值信息不仅浪费 Token，还稀释了 LLM 对关键信息的注意力。

## 核心矛盾

**信息完整性 vs Token 效率**：
- 保留越少 → 越省 Token → 丢失重要信息的风险越大
- 保留越多 → 信息越全 → 噪音干扰决策

## 主流优化方向

1. **工具感知压缩**：为每个工具定制关键信息提取器
2. **差分记录**：只记录状态变化而非全量快照
3. **层次化信息评级**：自动给信息打重要性标签，低分丢弃
4. **LLM-as-Judge 压缩**：用 LLM 判断哪些信息值得保留

## 实现挑战

1. **"重要"定义困难**：重要与否高度依赖上下文
2. **异常值的处理**：99% 情况可丢弃的信息在 1% 情况中极其重要
3. **透明性**：压缩过程对用户不可见，可能隐藏问题
4. **压缩时机**：在每步都压缩还是批量压缩？

## 能力边界

- 有损压缩永远无法"无损复原"
- 对于安全敏感场景（如金融交易记录），不应做有损压缩
- LLM 做压缩决策时本身可能出错

## 核心优势

有损压缩是 Token 优化最有效的手段之一——通常能以 10:1 的压缩比保留 90% 的关键信息。

## 工程优化

1. 为每个工具定义 result_summarizer 函数
2. 压缩后保留 metadata（如原始大小、压缩比）
3. 支持可配置的压缩级别（激进 / 均衡 / 保守）
4. 对数值和时间类信息精确保留（这些最有价值）

## 场景判断

适合有损压缩的场景：
- 工具返回大量结构化数据
- 长日志/文档分析
- 重复轮次的信息（Agent 多次调用同一工具）

不适合有损压缩的场景：
- 法律/医疗/金融原始数据
- 用户提供的精确指令
- 需要完整审计轨迹的任务
