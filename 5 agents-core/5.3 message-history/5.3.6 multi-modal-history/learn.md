# 5.3.6 multi-modal-history — 多模态消息历史管理

## 简单介绍

多模态 Agent（处理文本、图像、音频、视频的 Agent）的消息历史管理比纯文本 Agent 复杂得多——一个图像可能消耗数千 Token，但"值得"保留。多模态消息管理需要同时考虑不同模态的价值、大小和关联性。

## 基本原理

### 多模态 Token 成本

| 模态 | 典型 Token 成本 | 信息密度 | 压缩方法 |
|------|---------------|---------|---------|
| 文本 | 1 token ≈ 4 chars | 高 | 摘要 |
| 图像（详细） | ~2000 tokens/张 | 极高 | 降低分辨率 |
| 图像（低分辨率） | ~500 tokens/张 | 高 | - |
| 音频 | ~6000 tokens/min | 中 | 文本转录 |
| 视频 | ~12000 tokens/min | 低-中 | 关键帧提取 |

### 管理策略

```
多模态消息到达
    ↓
┌─────────────────────────────────┐
│ 模态感知的管理决策               │
│                                 │
│ 文本：标准摘要/裁剪策略          │
│ 图像：是否保留原始图？降低分辨率？│
│        → 如果不保留，保留描述文本 │
│ 音频/视频：→ 转录为文本后管理    │
└─────────────────────────────────┘
```

## 背景与演进

随着 GPT-4V、Claude 3 Vision、Gemini 等多模态模型的发展，Agent 开始支持多模态输入。但多模态内容的 Token 消耗大，且不同模态的"信息价值"难以直接比较。

## 核心矛盾

**视觉信息 vs Token 成本**：
- 图像包含大量非结构化信息（布局、颜色、空间关系），文本描述会丢失这些
- 保留原始图像占用大量 Token，限制上下文长度

## 主流优化方向

1. **模态降级**：原始图像 → 低分辨率图像 → 文本描述
2. **关键帧提取**：视频/动图中只保留关键帧
3. **模态感知预算**：为不同模态分配不同的 Token 预算
4. **关联性缓存**：如果多张图属于同一场景（如电商商品图），合并处理
5. **文本替代优先**：只有在文本描述不够时才保留原始多模态内容

## 实现挑战

1. **模态间引用**：Agent 说"看这张图里的数据"，需要同时保留图和引用的上下文
2. **跨模态摘要**：将"含图片的对话"摘要为纯文本，信息损失大
3. **图像选择**：10 张截图中哪些该保留、哪些可以丢弃？
4. **流式多模态**：视频流中何时截取关键帧？

## 能力边界

- 当前多模态模型对图像的"理解"仍然有限（如精确读图、数据提取）
- 文本描述永远无法完全替代图像信息（尤其是空间关系）
- 音频/视频的实时处理延迟高于文本

## 核心优势

多模态消息管理让 Agent **不再局限于文本交互**，可以"看见"用户分享的截图、图表、文档扫描件。

## 工程优化

1. 默认将图像降低到低分辨率（715×715 或更低）
2. 保持图像和引用它的文本消息的关联性
3. 预提取图像中的文字（OCR）作为文本备份
4. 使用描述性 alt text 作为图像的"可丢弃版本"
5. 对截图类图像设置更高的压缩率（大部分区域不关键）

## 场景判断

适合保留原始图像的场景：
- 用户分享设计稿、UI 截图
- 数据可视化图表
- 文档扫描件（包含复杂格式）

适合文本替代的场景：
- 用户头像、装饰性图片
- 重复的缩略图
- 视频中大部分近似的帧
