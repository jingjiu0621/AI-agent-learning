# 5.3.3 hierarchical-arch — 分层消息架构

## 简单介绍

分层消息架构是 Agent 消息管理的高级模式，将消息分为多个层次（核心层、压缩层、长期记忆层），每层使用不同的管理策略。这种架构平衡了 Token 效率和信息完整性。

## 基本原理

### 三层架构

```
┌────────────────────────────────────────────┐
│  核心层（Core Context）                      │
│  内容：系统指令、工具定义、当前任务目标      │
│  大小：固定（2-4K tokens）                   │
│  策略：从不丢弃、从不压缩                    │
├────────────────────────────────────────────┤
│  工作层（Working Context）                   │
│  内容：最近 N 轮交互、当前工具调用链        │
│  大小：动态（4-8K tokens）                   │
│  策略：滑动窗口 + 摘要                      │
├────────────────────────────────────────────┤
│  长期层（Long-term Context）                 │
│  内容：历史摘要、关键决策记录、用户画像      │
│  大小：按需加载                              │
│  策略：向量存储 + 检索                      │
└────────────────────────────────────────────┘
```

### 消息路由流程

```
新消息到达
    ↓
归类：属于哪一层？
    ↓
核心层？→ 直接追加（始终保留）
工作层？→ 追加，如超限则触发窗口策略
长期层？→ 提取关键信息 → 向量化存储
```

## 背景与演进

早期 Agent 不分层，所有消息混在一起。随着 Agent 复杂度提升，发现不同信息的生命周期和重要性不同，分层成为了自然的设计选择。

## 核心矛盾

**统一管理 vs 差异化策略**：
- 所有消息同一策略 → 简单但不高效
- 分层差异化 → 高效但设计复杂

## 主流优化方向

1. **动态分层**：消息的层级不是固定的，根据重要性动态升降
2. **跨层关联**：各层之间的消息保持关联索引
3. **层级感知 Prompt**：在 system prompt 中告知 LLM 消息的分层结构
4. **自动分层**：用分类器自动将消息分配到合适的层

## 实现挑战

1. **层间一致性**：同一信息在不同层的表示要保持一致
2. **检索精度**：长期层检索不到关键信息会影响质量
3. **层边界管理**：消息从工作层移到长期层的时机
4. **调试复杂度**：三层架构的调试比单层复杂

## 能力边界

- 分层不能完全消除信息损失
- 层数越多，系统越复杂，收益递减
- 对于超长任务（100轮+），三层仍可能不够

## 核心优势

分层架构让 Agent 的消息管理从"一刀切"变为"按需分配"，在 Token 效率和信息完整性之间找到了优雅的平衡。

## 工程优化

1. 三层是经过实践检验的最佳层数（过少不够用，过多太复杂）
2. 核心层内容在部署时就确定，运行时不变
3. 工作层和长期层使用不同的存储后端
4. 提供统一查询接口屏蔽分层细节

## 场景判断

- 简单 Agent（<10轮）：不需要分层
- 标准 Agent（10-50轮）：两层（核心+工作）够用
- 复杂 Agent（50轮+）：需要完整三层
- 企业级 Agent：三层 + 外部知识库
