# 5.3.2 summarization — 历史摘要

## 简单介绍

历史摘要（Summarization）是将旧的对话历史压缩为一段简短的摘要，替代原始消息传入 LLM。这是解决长上下文问题的核心策略，也是 LangGraph 等框架的内置机制。

## 基本原理

### 摘要流程

```
原始历史（10轮对话，5000 tokens）
    ↓
LLM 摘要调用："请将以上对话总结为 200 tokens 以内的摘要"
    ↓
摘要结果："用户询问了天气和交通信息，Agent 调用了天气API和地图API"
    ↓
新的消息序列：[摘要(200t) + 最近2轮原始消息(800t) = 1000t]
```

### 增量摘要

```python
class IncrementalSummarizer:
    def __init__(self):
        self.summary = ""
    
    def add_messages(self, new_messages):
        combined = self.summary + "\n" + new_messages
        if token_count(combined) > THRESHOLD:
            self.summary = llm.summarize(combined)
        else:
            self.summary = combined
```

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 无摘要 | 全量消息 | Token 爆炸 |
| 简单摘要 | 所有旧消息一条摘要 | 丢失细节 |
| 增量摘要 | 逐步累积 | 摘要偏差累积 |
| 分层摘要 | 按重要性分层 | 实现复杂 |

## 核心矛盾

**压缩比 vs 信息保真度**：
- 压缩率越高 → Token 越省 → 信息丢失越多
- 保留细节 → 信息完整 → Token 节省有限

## 主流优化方向

1. **选择性摘要**：只摘要工具调用结果，保留用户意图
2. **多视角摘要**：从不同维度（目标、进度、结果）分别摘要
3. **结构化摘要**：用固定 Schema 的输出替代自由文本摘要
4. **摘要偏差检测**：检查摘要与原文的一致性

## 实现挑战

1. **摘要偏差**：LLM 生成的摘要可能扭曲原意
2. **增量累积误差**：多次增量摘要后偏差放大
3. **摘要成本**：摘要本身也需要 LLM 调用，额外 Token
4. **时机选择**：何时触发摘要？每次？达到阈值时？

## 能力边界

- 摘要总会有信息损失，只是多少的问题
- 对细节敏感的任务（如法律、医疗）摘要风险较高
- LLM 自身的摘要质量波动会影响下游 Agent 行为

## 核心优势

相比滑动窗口的"丢弃"，摘要是"压缩保留"，在节省 Token 的同时保留了更多关键信息。

## 工程优化

1. 使用结构化摘要格式（JSON/YAML）确保解析可靠
2. 关键数据点（数字、名称、日期）在摘要中显式保留
3. 定期用原始消息校验摘要的正确性
4. 重要任务不做摘要（如关键交易步骤）

## 场景判断

- 工具调用密集的 Agent：工具结果可高度压缩
- 客户对话 Agent：需保留客户原话，不适合摘要
- 长文档处理 Agent：文档摘要是必需功能
