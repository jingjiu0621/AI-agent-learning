# 5.3.5 priority-pruning — 按重要性裁剪消息

## 简单介绍

优先级裁剪（Priority Pruning）不是简单地保留"最近的消息"，而是为每条消息计算重要性分数，优先丢弃低分消息。这使得 Agent 可以保留"最重要"的消息，即使它们在时间上较早。

## 基本原理

### 重要性评分模型

```python
def compute_importance(message, task_context):
    score = 0
    
    # 1. 消息类型权重
    type_weights = {
        "system_instruction": 10.0,   # 系统指令：最高优先级
        "user_goal": 9.0,            # 用户目标：很高
        "critical_tool_result": 8.0, # 关键工具结果：高
        "tool_call": 5.0,            # 工具调用
        "observation": 4.0,          # 观察
        "chat": 2.0,                 # 闲聊
    }
    score += type_weights.get(message.type, 1.0)
    
    # 2. 引用计数（被后续消息引用的次数）
    score += message.reference_count * 2.0
    
    # 3. 任务相关性
    score += compute_similarity(message, task_context) * 5.0
    
    # 4. 时间衰减
    age_factor = math.exp(-message.age / HALF_LIFE)
    score *= age_factor
    
    return score
```

### 裁剪流程

```
所有消息 → 计算重要性分数 → 按分数降序排列
    → 取出 top-K 直到 Token 预算耗尽
    → 丢弃剩余的低分消息
```

## 背景与演进

滑动窗口假设"越新的越重要"，但这对 Agent 不一定成立——第 1 轮的用户核心需求可能比第 20 轮的普通工具结果重要得多。优先级裁剪突破了时间顺序的限制。

## 核心矛盾

**重要性判断的准确性 vs 判断成本**：
- 越准确的重要性判断需要越多的上下文分析
- 为判断重要性而消耗的 Token 不能超过裁剪节省的

## 主流优化方向

1. **混合策略**：时间窗口内用全量，窗口外用优先级
2. **基于引用的重要性**：被后续消息提及/引用的消息重要
3. **分层评分**：先用轻量规则评分，关键消息用 LLM 深度评分
4. **重要性固化**：在消息写入时就计算好重要性，而非整理时批量计算

## 实现挑战

1. **重要性计算的"鸡生蛋"问题**：要判断消息重要性需要理解全部上下文，但全部上下文正是要压缩的对象
2. **评分标准一致性**：不同轮次的重要性评分标准要统一
3. **重要性悬崖**：分数只差 0.1，一个被保留一个被丢弃
4. **跨消息依赖**：两条低分消息合在一起才完整

## 能力边界

- 重要性计算本质上是启发式的，无法保证最优
- 引用计数法无法覆盖"隐式引用"（语义相关但未显式提及）
- 用户目标和上下文变化后，原有重要性判定失效

## 核心优势

优先级裁剪突破了"时间即重要性"的简单假设，更适合 **信息价值密度不均匀** 的 Agent 场景。

## 工程优化

1. 始终保留系统指令、用户原始需求、关键决策点
2. 消息重要性在生成时打标签，避免事后批量计算
3. 设置最低重要性阈值确保不会丢失所有上下文
4. 定期用 LLM 检验裁剪后的上下文是否仍能正确处理任务

## 场景判断

适合优先级裁剪的场景：
- 用户在第 1 轮给了详细的任务说明，后续全是工具调用
- 工具调用链中有一些关键结果（如错误信息）非常重要
- 多轮对话中用户需求可能变化

不适合优先级裁剪的场景：
- Token 预算宽松
- 所有消息的重要性相对均衡
