# 5.3.1 window-strategy — 滑动窗口策略

## 简单介绍

滑动窗口是最基础的消息历史管理策略——只保留最近 N 轮的消息，丢弃旧消息。实现简单且效果可预测，但对需要长期上下文的场景不适用。

## 基本原理

```
完整消息序列:
  [msg₁, msg₂, msg₃, ..., msg₉, msg₁₀]

滑动窗口 (window=5):
  [msg₆, msg₇, msg₈, msg₉, msg₁₀]
            ↑
       丢弃的部分
```

### 两种窗口类型

**固定轮次窗口**：
```python
# 保留最近 10 轮交互
messages = conversation[-10:]  # 丢弃前面的
```

**Token 预算窗口**：
```python
# 在 Token 预算内保留尽可能多的消息
budget = 4000  # tokens
messages = []
for msg in reversed(conversation):
    if sum(m.token_count for m in messages) + msg.token_count > budget:
        break
    messages.insert(0, msg)
```

## 背景与演进

早期 Agent 将所有消息都传给 LLM，随着上下文增长 Token 消耗线性增加。滑动窗口是最简单直接的解决方案，也是各大框架的默认策略。

## 核心矛盾

**记忆长度 vs Token 消耗**：
- 窗口大 → 记忆长 → Token 消耗大
- 窗口小 → Token 省 → 丢失早期上下文

## 主流优化方向

1. **自适应窗口**：根据消息重要性动态调整窗口大小
2. **分层窗口**：用不同的窗口大小管理不同层级的信息
3. **重要性保留**：窗口外的关键消息仍保留（如系统指令、任务定义）
4. **智能压缩前置**：在窗口边界先压缩再丢弃

## 实现挑战

1. Token 计数不精确（不同模型的 Tokenizer 不同）
2. 系统消息和工具定义需要始终保留，不能计入窗口
3. 窗口边界可能截断正在进行中的工具调用链

## 能力边界

- **无法保留跨窗口的长期依赖**：第 1 轮的用户需求可能在 50 轮后才需要引用
- 对需要长期记忆的 Agent，单纯滑动窗口不够

## 核心优势

实现极其简单、性能开销极小、行为可预测，适合大多数场景的基线策略。

## 工程优化

1. 系统消息和工具定义不在窗口限制之内（始终保留）
2. 使用 tiktoken 等库精确计数 Token
3. 设置软性告警线：当窗口频繁截断时提醒需要摘要策略

## 场景判断

- 短对话（<10轮）：不需要窗口
- 中等对话（10-50轮）：固定窗口
- 长对话（>50轮）：Token 预算窗口
