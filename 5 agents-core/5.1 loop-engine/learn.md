# 5.1 loop-engine — Agent 主循环：Thought-Action-Observation

## 概念定义

Agent 主循环（Loop Engine）是 Agent 最核心的执行架构，定义了 Agent 如何在"思考 → 行动 → 观察"之间循环迭代，直到完成任务。它是 Agent 区别于传统程序的关键特征——传统程序是静态的指令序列，而 Agent 是一个动态的感知-推理-执行闭环。

## 核心角色

| 角色 | 说明 | 对应模块 |
|------|------|---------|
| 思考器（Thought） | 生成推理轨迹，决定下一步做什么 | LLM 调用层 |
| 选择器（Action Selection） | 从候选行动中选择一个执行 | 决策策略层 |
| 执行器（Action Execution） | 执行选定的行动（工具调用/输出） | 工具执行层 |
| 观察器（Observation） | 解析执行结果，更新状态 | 结果处理层 |

## 为什么需要 Loop Engine

没有主循环的 Agent 只是一个"一次性的 LLM 调用"：

- **单轮无记忆**：LLM 生成一次回复就结束，无法根据结果调整行为
- **无错误恢复**：第一次调用出错，整个任务就失败
- **无法处理多步任务**：任何需要多个推理步骤的任务都无法完成

Loop Engine 从根本上解决了三个核心矛盾：
1. **动态性 vs 静态程序**：让程序在运行时根据中间结果动态决策下一步
2. **容错性 vs 单次调用**：允许在循环中纠正错误，而非一次定生死
3. **组合性 vs 简单输出**：将多个工具/推理步骤组合成完整的任务解决流程

## 设计范式演进

| 时代 | 方案 | 问题 |
|------|------|------|
| 早期 Agent | 简单 if-else 循环 | 死板、无法泛化 |
| ReAct 范式 | Thought-Action-Observation 循环 | 可能陷入循环、效率波动 |
| 现代框架 | 带状态管理和条件终止的增强循环 | 复杂度提升、调试困难 |

## 核心设计维度

1. **循环结构**：单轮 vs 多轮、线性 vs 分支
2. **思考生成**：如何让 LLM 产生高质量推理轨迹
3. **行动选择**：多个候选行动时如何决策
4. **观察解析**：如何将工具结果有效注入下一轮推理
5. **终止条件**：何时结束循环

## 工程最佳实践

- 设置最大迭代步数防止无限循环
- 每一步记录完整的 Thought-Action-Observation 轨迹
- 关键节点（工具调用失败/连续相同行动）触发降级或终止
- 按实际场景选择合适的循环变体（ReAct / Reflexion / Plan-Execute）
