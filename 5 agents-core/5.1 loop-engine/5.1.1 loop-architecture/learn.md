# 5.1.1 loop-architecture — 主循环架构：单轮 vs 多轮迭代

## 简单介绍

Agent 主循环架构是整个 Agent 系统的执行骨架，定义了"思考→行动→观察"如何组织为可执行的循环。不同的架构选择直接决定了 Agent 是"快准狠"还是"慢而稳"。

## 基本原理

### 单轮架构（Single-turn）

Agent 只执行一轮"思考→行动"就返回结果。适用于简单、明确的任务。

```
输入 → 思考 → 行动（工具调用）→ 观察结果 → 输出
```

**特点**：
- 只调用一次 LLM
- 无状态、无记忆
- 结果不可修改

### 多轮架构（Multi-turn / Iterative）

Agent 在一个循环中反复执行"思考→行动→观察"，直到满足终止条件。

```
输入 → 循环 { 思考 → 行动 → 观察 }* → 输出
                           ↑_________↓
```

**特点**：
- 多次调用 LLM
- 带中间状态跟踪
- 可根据观察结果调整后续行为

## 背景与演进

| 时期 | 架构 | 代表 |
|------|------|------|
| 早期 (2022) | 单轮 I/O | 简单 LLM 封装 |
| ReAct 提出 (2022) | 多轮循环 | ReAct 论文实现 |
| 框架化 (2023) | 图编排循环 | LangGraph StateGraph |
| 生产级 (2024+) | 混合架构 | 条件路由 + 子循环 |

## 核心矛盾

**单轮 vs 多轮的根本权衡**：

| 维度 | 单轮 | 多轮 |
|------|------|------|
| 响应速度 | 快（一次 LLM 调用） | 慢（多次调用） |
| 任务复杂度 | 低 | 高 |
| 错误恢复 | 无 | 有 |
| Token 消耗 | 少 | 多（含累积上下文） |
| 状态管理 | 不需要 | 必需 |

## 主流优化方向

1. **条件路由**：简单任务走单轮，复杂任务走多轮，避免一刀切
2. **提前终止**：通过置信度评分在达到足够好的结果时提前退出
3. **并行子循环**：多轮中并行执行独立子任务
4. **上下文预算**：多轮中动态管理 Token 消耗，防止超限

## 实现挑战

1. **状态一致性**：多轮中保持状态不丢失、不冲突
2. **循环控制**：防止无限循环或无效循环
3. **上下文膨胀**：每轮积累的对话历史导致 Token 消耗线性增长
4. **调试困难**：多轮交互的调试比单轮复杂得多

## 能力边界

- 单轮架构无法处理需要多步推理或工具调用的任务
- 多轮架构在超过 10-20 轮后质量明显下降（上下文稀释）
- 没有一种循环架构在所有任务上都最优

## 与相关概念的区别

- **vs 消息管理（5.3）**：循环架构定义"怎么循环"，消息管理定义"循环中怎么管历史"
- **vs 状态管理（5.2）**：架构是执行框架，状态管理是数据层

## 核心优势

多轮架构的核心优势是 **动态适应性**——能在执行过程中根据新信息调整策略，这是传统程序无法做到的。

## 工程优化

1. 为不同复杂度任务选择不同架构（策略模式）
2. 设置最大迭代步数和 Token 预算
3. 记录每轮延迟和 Token 消耗用于优化决策
4. 对超过 N 轮的任务发出告警

## 场景判断

适合多轮架构的场景：
- 需要多步工具调用链
- 结果依赖前一步的执行输出
- 任务需要纠错和重试
