# 5.1.4 observation-parse — 观察结果解析与归因

## 简单介绍

Observation Parse 是 Agent 主循环中将工具执行结果转化为下轮推理可用信息的关键环节。原始的工具返回值（JSON、文本、错误信息）需要经过解析、过滤、归因后才能成为有效的"观察"输入到下一轮思考中。

## 基本原理

观察解析的核心流程：

```
工具执行结果（原始）
    ↓
格式解析（JSON/XML/文本）
    ↓
信息提取（过滤噪音、提取关键字段）
    ↓
归因分析（结果与哪个行动的哪部分相关）
    ↓
状态更新（写入 Agent 状态）
    ↓
结构化的 Observation 消息
```

### 三类典型观察

| 类型 | 示例 | 处理方式 |
|------|------|---------|
| 结构化数据 | `{"temp": 25, "humidity": 60}` | JSON 解析、字段提取 |
| 文本结果 | 日志、文档片段 | 摘要、关键句提取 |
| 错误信息 | 超时、权限拒绝、404 | 错误分类、降级标记 |

## 背景与演进

| 时期 | 方式 | 问题 |
|------|------|------|
| 早期 | 原始结果直接塞回 prompt | 噪音信息稀释推理 |
| 框架化 | 结果注入 tool_result 消息 | 格式标准化了但信息未提炼 |
| 高级处理 | 自动摘要 + 关键字段提取 | 额外 LLM 调用成本 |
| 现代实践 | 依工具类型定制处理器 | 维护成本高 |

## 核心矛盾

**信息保真度 vs 信息密度**：
- 保留原始结果 → 信息完整但 Token 消耗大，噪音多
- 摘要压缩 → Token 节省但可能丢失关键细节

## 主流优化方向

1. **工具感知解析**：为每个工具定制 ObservationHandler，只提取相关字段
2. **差异观察**：只记录与上一轮不同的信息，忽略重复结果
3. **分层观察**：完整结果入库（供审计），摘要结果入 prompt
4. **错误模式识别**：自动识别常见错误模式并分类（工具故障 / 权限问题 / 数据不存在）
5. **多模态观察**：处理图像、音频等非结构化工具输出

## 实现挑战

1. **长结果截断**：工具返回超长内容时如何截断而不失关键信息
2. **错误 vs 有效结果的区分**：部分错误包含有效信息（如"只返回了前10条"）
3. **流式观察**：多个并行工具的结果如何按顺序整合
4. **幻觉传播**：Agent 可能"看错"观察结果（LLM 错误理解工具返回值）

## 能力边界

- 解析逻辑无法覆盖所有工具的所有返回格式
- 对于不透明工具（黑盒 API），无法验证观察是否正确
- 从观察中提取的信息质量直接影响后续推理——GIGO 原则

## 相关概念区别

- **vs 工具结果注入（4.1.3）**：那是 API 层把结果放回 messages，这是 Agent 层理解结果
- **vs 状态更新（5.2）**：观察解析是"理解发生了什么"，状态更新是"记录下来"

## 核心优势

高质量的观察解析能将原始工具输出转化为 **Agent 可直接消费的结构化知识**，减少 LLM 的解析负担和误解概率。

## 工程优化

1. 为每个工具注册专门的 result handler
2. 对常见错误模式建立分类映射表
3. 大结果分页观察，避免截断丢失信息
4. 使用结构化格式（JSON）统一工具返回值

## 场景判断

需要复杂观察解析的场景：
- 工具返回大量非结构化文本
- 多工具并行执行，结果需要合并归因
- 工具可能返回多种类型的错误
