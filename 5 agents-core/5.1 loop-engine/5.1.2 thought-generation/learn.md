# 5.1.2 thought-generation — 思考生成与内部推理轨迹

## 简单介绍

Thought Generation 是 Agent 主循环中"思考"环节的核心——让 LLM 生成推理轨迹（Reasoning Traces），解释当前状态、分析可用信息、决定下一步行动。思考质量直接决定了 Agent 的决策质量。

## 基本原理

思考生成的核心机制是 **Chain-of-Thought**——引导 LLM 在回答问题前先进行逐步推理。在 Agent 循环中，这个推理过程被结构化为：

1. **状态回顾**：回顾当前任务进度、已有信息
2. **分析推理**：基于观察结果进行推理分析
3. **行动决策**：决定下一步行动并给出理由

### 典型思考结构

```
Thought: 目前我已经获取了用户的历史订单信息，发现最近三个订单都出现了配送延迟。
我需要查看物流系统的状态来确认是否是普遍性问题。
Action: check_logistics_status
```

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 无思考 | 直接输出行动 | 不可解释、无法调试 |
| 简单 CoT | "Let's think step by step" | 思考太浅、容易跑偏 |
| 结构化思考 | 固定模板（状态-分析-决策） | 灵活性受限 |
| ReAct 格式 | Thought-Action-Observation 交织 | 需要多轮训练对齐 |

## 核心矛盾

**思考深度 vs Token 消耗**：
- 思考越深入，Token 消耗越大，延迟越高
- 思考太浅则决策质量低，可能需要更多轮次来纠错

**思考一致性 vs 多样性**：
- 结构化的思考模板可保证输出格式一致，但限制创造力
- 自由思考更灵活，但解析不可靠

## 主流优化方向

1. **结构化思维模板**：为不同类型任务设计固定的思考模板，保证输出可靠性
2. **思考预算控制**：根据任务复杂度动态分配思考 Token 预算
3. **分层思考**：先高层战略思考，再细化到具体行动
4. **思考压缩**：压缩长推理轨迹为关键摘要
5. **Few-shot 思考示例**：在 system prompt 中嵌入高质量思考示例

## 实现挑战

1. **思考与行动分离**：LLM 有时会跳过推理直接输出行动，需要在 prompt 层面强制分离
2. **幻觉扩散**：思考阶段的错误推理会污染后续所有决策
3. **过度思考**：简单任务上花大量 Token "分析"明明显而易见的结论
4. **语言漂移**：多轮中思考风格和语言一致性难以保持

## 能力边界

- 思考质量受限于 LLM 本身的推理能力
- 无法保证思考过程与最终决策的因果一致性（事后合理化）
- 超过 LLM 上下文窗口的复杂推理链无法处理

## 相关概念区别

- **vs CoT Prompting（3.1.1）**：那是 prompt 层面的技术，这里是 Agent 循环中思考环节的工程实现
- **vs ReAct Pattern（7.2）**：ReAct 定义了思考与行动的宏观交织方式，Thought Generation 关注思考本身

## 核心优势

结构化的思考生成让 Agent 的行为变得 **可解释、可调试、可审计**——这是生产部署的必备条件。

## 工程优化

1. 为不同类型任务设计定制的思考模板
2. 使用 token 计数器监控思考部分的消耗
3. 实现思考缓存：相同状态下的推理结果可复用
4. 在调试模式中暴露完整思考轨迹

## 场景判断

需要深度思考的场景：
- 多步推理任务（数学、逻辑）
- 涉及多种信息源的综合分析
- 需要解释决策原因的场景

不需要深度思考的场景：
- 简单信息查询
- 确定性工具调用
