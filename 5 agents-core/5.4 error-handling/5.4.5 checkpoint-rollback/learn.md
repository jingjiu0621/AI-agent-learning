# 5.4.5 checkpoint-rollback — 检查点与回滚

## 简单介绍

检查点（Checkpoint）是 Agent 在关键步骤保存状态快照的机制，回滚（Rollback）是在错误发生后恢复到之前检查点的能力。这是 Agent 从错误中恢复的最激进也最有效的方式。

## 基本原理

### 检查点保存

```
步骤 1: Thought → Action → Observation
                              |
                       [检查点 1: 保存全部状态]
                              |
步骤 2: Thought → Action → Observation
                              |
                       [检查点 2: 保存全部状态]
                              |
步骤 3: Action → ❌ 错误!
    ↓
回滚到检查点 2 → 更换策略重试
```

### 回滚策略

| 策略 | 描述 | 适用场景 |
|------|------|---------|
| 全部回滚 | 恢复到检查点的完整状态 | 严重错误 |
| 部分回滚 | 只回滚特定维度（如工具结果） | 局部错误 |
| 选择性回滚 | 保留某些进展，回滚其他部分 | 混合场景 |
| 带重放的回滚 | 回滚后回放某些步骤 | 需要保留部分历史 |

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 无检查点 | 失败后从头开始 | 浪费已完成的工作 |
| 内存检查点 | 在内存中保存快照 | 进程崩溃即丢失 |
| 持久化检查点 | 写入磁盘/数据库 | 写入性能开销 |
| LangGraph Checkpointer | 内置检查点管理 | LangGraph 生态绑定 |

## 核心矛盾

**检查点粒度 vs 性能开销**：
- 每步都检查 → 恢复精确 → 写入开销巨大
- 少写检查点 → 性能好 → 恢复丢失较多工作

## 主流优化方向

1. **增量检查点**：只保存变化的状态部分
2. **分层检查点**：内存级 + 磁盘级 + 远程级
3. **智能检查点频率**：高重要性操作前检查，低重要性操作后延迟检查
4. **压缩检查点**：合并历史检查点，节省存储
5. **可中断 Agent**：Agent 被中断后可在检查点恢复

## 实现挑战

1. **外部副作用处理**：回滚不能撤回已经发送的邮件或已执行的 API 调用
2. **检查点存储成本**：长时间运行的 Agent 产生大量检查点
3. **序列化完整性**：所有状态必须可序列化（包括 LLM 调用中间状态）
4. **恢复后一致性**：恢复后的状态与外部实际状态可能存在偏差

## 能力边界

- 无法回滚外部系统的副作用（已支付的订单不会自动退款）
- 检查点占用额外存储
- 恢复操作也不能保证 100% 成功

## 核心优势

检查点回滚让 Agent 可以 **"从错误中安全恢复"**，避免因一次失败就导致整个任务从头开始。

## 工程优化

1. 对非幂等操作的回滚实施"补偿操作"而非状态回滚
2. 检查点保留策略：保留最近 N 个 + 关键里程碑检查点
3. 使用异步写入减少检查点对主流程的延迟影响
4. 恢复后自动执行一致性检查

## 场景判断

- 短任务（<5步）：不需要检查点
- 长任务（>20步）：需要检查点
- 关键任务：每步都检查 + 外部补偿
