# 5.4.2 graceful-degradation — 优雅降级策略

## 简单介绍

优雅降级（Graceful Degradation）是指在 Agent 部分功能不可用时，不是直接失败，而是以降低的能力继续运行。这是生产级 Agent 保障可用性的关键策略。

## 基本原理

### 降级层次

```
功能完全正常 → 核心功能正常（非核心降级）→ 只读模式 → 人工接管
```

### 降级决策流程

```
检测到功能不可用
    ↓
评估影响范围
    ├── 非关键功能？→ 禁用该功能，通知用户替代方案
    ├── 核心功能但可替代？→ 切换替代方案
    ├── 核心功能且不可替代？→ 进入只读/有限模式
    └── 所有功能不可用？→ 通知用户系统不可用
```

### 示例场景

| 故障 | 降级策略 | 用户感知 |
|------|---------|---------|
| 天气 API 超时 | 返回"天气服务暂时不可用" | 看不到天气，其他正常 |
| 搜索服务降级 | 从语义搜索降级到关键词搜索 | 搜索结果质量略降 |
| LLM API 限流 | 切换到备用模型 | 回复质量可能有差异 |
| 数据库不可用 | 使用缓存数据 + 只读模式 | 无法写入新数据 |

## 背景与演进

传统软件中，功能要么可用要么不可用（二进制）。Agent 在复杂环境中运行时，部分资源不可用是常态而非异常。优雅降级使得 Agent 可以在不完美的环境中仍然提供价值。

## 核心矛盾

**服务质量 vs 可用性**：
- 坚持高质量但降级决策保守 → 部分场景不必要地失败
- 过于容忍降级 → 用户获得低质量服务

## 主流优化方向

1. **分级降级**：定义多个降级等级，逐级尝试
2. **用户偏好**：让用户选择"宁可失败也不要降级"或"尽可能完成"
3. **透明降级**：降级时明确通知用户"当前 XX 功能使用备用方案"
4. **自动恢复**：定时检测故障是否恢复，自动升级回正常模式

## 实现挑战

1. **降级感知**：Agent 需要知道自己处于降级模式，调整行为
2. **级联降级**：一个功能降级可能触发其他功能也降级
3. **降级测试**：降级路径通常是最少测试的路径
4. **恢复时机**：何时从降级模式恢复？过度恢复导致抖动

## 能力边界

- 降级不能应对所有故障（某些关键路径的故障无法降级）
- 降级越多，系统行为越偏离正常，不确定性越大
- 某些降级方案需要额外的基础设施（如备用模型、缓存数据）

## 核心优势

优雅降级让 Agent 的可用性从"要么全有要么全无"变为**"总有价值"**。

## 工程优化

1. 为每个功能定义降级方案（如果不可用怎么办）
2. 降级路径也需要测试
3. 降级决策和恢复逻辑自动化
4. 降级指标纳入监控（降级率和时长）

## 场景判断

- 在线客服 Agent：任何降级都优于完全不可用
- 金融交易 Agent：宁可失败不可降级
- 信息查询 Agent：大多数场景可降级运行
