# 5.4.3 retry-logic — 智能重试

## 简单介绍

重试是 Agent 最基础但最重要的错误恢复策略。与简单的心跳重试不同，Agent 的智能重试需要考虑错误类型、退避策略、替代路径和 Token 成本之间的平衡。

## 基本原理

### 重试决策树

```
工具调用失败
    ↓
是否可重试？
  ├── 否（如 400 Bad Request：参数错误）→ 不重试，报告参数问题
  └── 是（如 429/5xx/超时）
        ↓
    重试策略选择
      ├── 指数退避：等待 1s → 2s → 4s → 8s
      ├── 固定间隔：每 5s 重试
      ├── 立即重试：仅一次
      └── 自适应退避：根据错误码动态调整
            ↓
    达到最大重试次数？
      ├── 是 → 执行替代路径或报告失败
      └── 否 → 继续重试
```

### 退避算法

```python
import random
import time

def exponential_backoff(attempt, base_delay=1.0, max_delay=60.0):
    """指数退避 + jitter"""
    delay = min(base_delay * (2 ** attempt), max_delay)
    # 添加 jitter 防止 thundering herd
    jitter = random.uniform(0, delay * 0.5)
    return delay + jitter

# 使用
for attempt in range(max_retries):
    try:
        return call_tool()
    except RetryableError:
        wait = exponential_backoff(attempt)
        time.sleep(wait)
```

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 无重试 | 失败即报错 | 临时故障也失败 |
| 固定重试 | 固定 N 次 | 没有退避、浪费资源 |
| 指数退避 | 带 jitter 退避 | 通用但不能针对错误类型优化 |
| 智能重试 | 错误感知 + 自适应 | 实现复杂 |

## 核心矛盾

**重试收益 vs 重试成本**：
- 重试可能成功（收益），但消耗时间和 Token（成本）
- 有些错误重试 100 次也不会成功

## 主流优化方向

1. **错误感知重试**：根据错误类型决定重试策略
2. **自适应最大次数**：根据任务紧急程度动态调整
3. **幂等性检查**：重试前确认操作是幂等的（不会产生副作用）
4. **快速失败**：对不可恢复错误立即失败，不浪费重试

## 实现挑战

1. **非幂等操作**：重试可能导致重复发送邮件、重复扣款
2. **超时累积**：每次重试的超时时间累积导致总等待过长
3. **LLM 重试的特殊性**：LLM 调用失败后重试，但 LLM 的无状态性可能导致同样的错误
4. **资源耗尽**：大量 Agent 同时重试可能耗尽系统资源

## 能力边界

- 重试无法解决根本性故障（如数据库宕机、模型下线）
- 重试以增加延迟为代价提高成功率
- 不能无限重试，必须有终止条件

## 核心优势

智能重试是性价比最高的错误恢复策略——**少量代码即可显著提高成功率**。

## 工程优化

1. 区分 RETRYABLE 和 NON-RETRYABLE 错误
2. 指数退避 + jitter 是默认最佳实践
3. 幂等 key 机制防止重复执行
4. 重试次数和延迟纳入监控

## 场景判断

- API 限流（429）：指数退避重试，最多 5 次
- 网络超时：立即重试一次，然后退避
- 参数错误（400）：不重试
- 认证失败（401）：不重试，触发重新认证
