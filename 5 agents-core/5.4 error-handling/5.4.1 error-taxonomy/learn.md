# 5.4.1 error-taxonomy — Agent 错误分类

## 简单介绍

系统性的错误分类是有效错误处理的前提。将 Agent 错误按来源、严重程度、可恢复性等维度分类，可以针对不同类型采用不同的处理策略。

## 基本原理

### 三维分类模型

```
维度 1：错误来源
├── LLM 层：推理错误、幻觉、格式异常、指令理解偏差
├── 工具层：超时、网络错误、权限拒绝、限流、数据异常
├── Agent 层：状态不一致、循环检测、选择失误
└── 基础设施层：配置错误、依赖缺失、资源耗尽

维度 2：严重程度
├── CRITICAL：任务无法继续，需要人工介入
├── ERROR：当前步骤失败，可尝试替代方案
├── WARNING：非关键操作失败，不影响主线任务
└── INFO：轻微异常，记录但无需处理

维度 3：可恢复性
├── RETRYABLE：重试可能成功（网络超时、限流）
├── ALTERNATIVE：可用替代方式完成
├── DEGRADED：降级后继续
└── FATAL：无法恢复
```

### 分类决策树

```
检测到错误
    ↓
来源判断：LLM / 工具 / Agent / 基础设施
    ↓
严重程度判断
    ↓
可恢复性判断 → 选择处理策略
```

## 背景与演进

早期 Agent 简单地将所有错误视为"调用失败"统一处理。随着 Agent 在关键场景的部署增加，需要更精细的错误分类来差异化处理。

## 核心矛盾

**分类粒度 vs 分类成本**：
- 分类越细 → 处理策略越精准 → 分类系统越复杂
- 粗分类 → 简单 → 有些场景策略不当

## 主流优化方向

1. **自动错误分类器**：用 LLM 或分类模型自动对错误分类
2. **动态严重度**：根据当前上下文动态调整严重级别
3. **错误模式库**：积累常见错误模式并自动匹配
4. **可扩展分类体系**：允许为特定工具注册自定义错误分类

## 实现挑战

1. **边界模糊**：某些错误难以明确分类（如工具返回空结果——是正常情况还是错误？）
2. **复合错误**：一个操作可能触发多个不同类型的错误
3. **分类漂移**：随时间推移，错误模式变化导致分类标准失效

## 能力边界

- 没有分类体系能覆盖所有可能的错误
- 自动分类器本身也会出错
- 分类标准的有效性需要持续验证和调整

## 核心优势

好的错误分类让 Agent 能够 **"知道自己在面对什么类型的错误"**，从而选择最合适的处理策略。

## 工程优化

1. 定义清晰的错误类型枚举（enum）而非自由字符串
2. 每种错误类型关联处理策略（Strategy Pattern）
3. 错误日志包含分类信息（便于事后分析）
4. 定期 review 错误分布，优化分类标准

## 场景判断

- 开发阶段：粗分类（3-5 类）足够
- 生产初期：中等分类（5-10 类）
- 成熟系统：细分类（10+ 类）+ 自动分类器
