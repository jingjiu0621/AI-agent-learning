# 5.4.4 human-escalation — 人工介入与升级机制

## 简单介绍

Human Escalation 是 Agent 在无法自主决策或处理时将问题升级给人类的机制。这是 Agent 安全的最后一道防线，也是将"由 AI 完成"变为"在人类监督下由 AI 完成"的关键设计。

## 基本原理

### 升级触发条件

```
┌────────────────────────────────────────┐
│  自动检测触发                          │
│  → 低置信度（LLM 对结果不确定）        │
│  → 高风险操作（删除/修改/支付）        │
│  → 超出能力边界                        │
│  → 多轮重试失败                        │
├────────────────────────────────────────┤
│  用户主动触发                          │
│  → 用户要求转人工                      │
│  → 用户对 Agent 回复不满意            │
│  → 用户提供了新指令需要确认            │
└────────────────────────────────────────┘
```

### 升级流程

```
                       Agent 工作流
                            │
               ┌──── 遇到需要升级的情况 ────┐
               │                            │
          自动处理                     需要人工
          (继续执行)                       │
                                           ↓
                               ┌─────────────────────┐
                               │  准备升级上下文包    │
                               │  - 当前状态摘要      │
                               │  - 遇到的问题        │
                               │  - 已尝试的方案      │
                               │  - 需要人类做的决定  │
                               └─────────┬───────────┘
                                         ↓
                               ┌─────────────────────┐
                               │  通知人类            │
                               │  - 异步（邮件/工单） │
                               │  - 同步（实时通知）  │
                               └─────────┬───────────┘
                                         ↓
                               ┌─────────────────────┐
                               │  等待人类响应        │
                               │  → 获得指令后继续    │
                               │  → 超时后降级/重试  │
                               └─────────────────────┘
```

## 背景与演进

| 时期 | 方案 | 问题 |
|------|------|------|
| 纯自主 | Agent 自行决定一切 | 安全风险 |
| 简单升级 | 抛出异常转人工 | 上下文丢失 |
| 结构化升级 | 带完整上下文的升级 | 需要等待人类 |
| Human-in-the-Loop | 预定义审批节点 | 设计复杂 |

## 核心矛盾

**自主性 vs 安全性**：
- 需要人工越少 → 效率越高 → 风险越大
- 人工介入越多 → 越安全 → 越慢、越贵

## 主流优化方向

1. **分级升级**：普通问题异步（邮件），紧急问题同步（即时通讯）
2. **智能上下文打包**：自动整理人类做决策需要的信息
3. **预测性升级**：在问题发生前预测是否需要人类介入
4. **半自动化审批**：相似的人工审批结果可自动化处理
5. **升级反馈闭环**：人工决策被学习复用，减少未来相同问题的升级

## 实现挑战

1. **上下文完整性**：升级时提供的上下文必须足够人类做决策
2. **等待超时**：人类可能长时间不响应，Agent 需要超时策略
3. **人工决策的机器学习**：将人工决策转化为可复用的规则
4. **升级频率控制**：太频繁失去 Agent 价值，太少安全风险高

## 能力边界

- 人工并不能保证比 Agent 做得更好（人也会犯错）
- 人工响应有延迟，影响端到端效率
- 24/7 运营需要人工值班，增加运营成本

## 核心优势

Human Escalation 是 Agent 安全的 **"最后一道保险"**，也是向"完全自主"过渡的必经阶段。

## 工程优化

1. 升级时打包完整的上下文：当前状态 + 已尝试 + 决策需求
2. 升级设置超时 + 自动降级预案
3. 记录人工决策，构建知识库减少后续相同升级
4. 提供批量审批接口提高人工效率

## 场景判断

必须升级的场景：
- 涉及金钱、法律、医疗决策
- 删除用户数据
- 系统级变更

可选升级的场景：
- Agent 置信度低于阈值
- 用户多次要求修改
- 跨领域请求超出 Agent 知识边界
