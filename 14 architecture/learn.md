# 14 系统架构设计 — System Architecture Design

> Agent 系统架构设计与传统软件架构有本质区别：Agent 不是简单的请求-响应处理单元，而是具备自主推理、工具调用和多步规划能力的"智能体"。这要求架构师在设计时需要同时考虑 **确定性基础设施**（服务发现、消息队列、负载均衡）和 **非确定性 AI 组件**（LLM 推理、工具选择、行为一致性）。本章从架构模式、企业级设计、微服务化和真实案例四个维度，全方位解析 Agent 系统的架构设计方法。

## Agent 系统架构 vs 传统系统架构

Agent 系统在架构层面与传统的 Web 后端/微服务系统有多个关键差异点，理解这些差异是正确选型的前提。

### 核心差异矩阵

| 维度 | 传统系统 | Agent 系统 |
|------|----------|-----------|
| **处理模式** | 确定性：相同输入 → 相同输出 | 非确定性：相同输入 → 概率分布输出 |
| **状态管理** | 无状态优先，有状态显式设计 | 天然有状态：会话/记忆/推理轨迹 |
| **调用模式** | 同步请求-响应 | 同步+异步混合：推理→工具→推理→工具 |
| **故障模型** | 崩溃/超时/资源耗尽 | LLM 幻觉/工具选择错/循环陷阱/语义漂移 |
| **扩展粒度** | CPU/内存/连接数 | Token 预算/上下文窗口/推理深度 |
| **可观测性** | 请求延迟/错误率/吞吐量 | 推理轨迹/工具调用链/语义正确性 |
| **安全边界** | 输入验证/权限控制 | Prompt 注入/工具劫持/记忆污染 |
| **缓存策略** | 响应缓存 (精确匹配) | 语义缓存 (相似性匹配) |

### Agent 系统的架构复杂度公式

```
Agent 系统总复杂度 ≈ 基础服务复杂度 × 推理步骤数 × 工具数量 × 协作 Agent 数

传统系统: C_base = f(API 数, 服务数, 数据量)     → 线性增长
Agent 系统: C_agent ≈ C_base × Steps × Tools × N  → 多项式/指数增长

示例对比:
  一个 10 个 API 的传统系统:   C_base ≈ 10 × 1 × 1 × 1 = 10
  一个 5 步 × 3 工具的 Agent: C_agent ≈ 10 × 5 × 3 × 1 = 150
  一个 5 步 × 3 工具 × 3 Agent: C_agent ≈ 10 × 5 × 3 × 3 = 450
```

## 架构设计核心原则

### 1. 推理与执行分离

将 LLM 推理层与工具执行层从架构层面分离，这是 Agent 系统最重要的架构原则。

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  推理层 (Brain)  │────►│  编排层 (Cortex) │────►│  执行层 (Muscle) │
│                 │     │                 │     │                 │
│ • LLM 推理       │     │ • 工具选择       │     │ • 工具调用       │
│ • 规划与决策     │     │ • 步骤编排       │     │ • API 执行       │
│ • 反思与修正     │     │ • 状态管理       │     │ • 代码执行       │
│ • 知识整合       │     │ • 错误处理       │     │ • 数据操作       │
└─────────────────┘     └─────────────────┘     └─────────────────┘
         │                       │                       │
   可替换 LLM            可插拔编排策略             可扩展工具集
```

**为什么重要**：LLM 模型更新频繁（GPT-4 → GPT-5 → ...），工具也在不断增加。如果推理与执行紧耦合，任何一个维度的变更都需要重构整个系统。

### 2. 有状态设计

Agent 天生有状态，架构必须原生支持状态管理，而不是事后添加。

```
状态层次:
  ┌──────────────────────────────────────────────┐
  │  会话状态: 当前对话上下文、用户信息、权限       │
  │  推理状态: 当前思考链、待验证假设、部分结果     │
  │  任务状态: 任务进度、子任务列表、已完成步骤     │
  │  记忆状态: 历史交互、学习到的知识、偏好         │
  │  工具状态: 已建立连接、认证 token、游标位置     │
  └──────────────────────────────────────────────┘

架构影响:
  • 需要状态持久化 (Redis/PostgreSQL) — 无状态重启会丢失推理进度
  • 需要状态恢复机制 — 节点故障后能从中断处继续
  • 需要状态快照 — 用于调试、回放、审计
```

### 3. 可观测性内建

Agent 系统的行为比传统系统复杂得多，可观测性不能是事后附加，必须从 Day 1 设计。

```
Agent 可观测性三支柱:
  ┌─ 日志 (Logs):    Thought/Action/Observation 完整轨迹
  ├─ 指标 (Metrics):  推理步数、Token 消耗、工具调用延迟
  └─ 追踪 (Traces):   多步推理链路、多 Agent 交互链路
  
  每个 Agent 的输入/输出都必须能够完整回溯
```

### 4. 渐进式复杂度

从最简单的架构开始，按需演进。不要一开始就设计完美的分布式架构。

```
演进路径:
  单体 Agent ──► 事件驱动 Agent ──► 管道架构 ──► 微服务 Agent ──► 网格架构
   (MVP 验证)    (异步解耦)       (职责分离)    (独立扩缩)       (灵活协作)
```

## 架构选型流程

### Step 1: 需求分析

| 问题 | 影响维度 |
|------|----------|
| Agent 数量？1 个 vs N 个 | 单体 vs 多 Agent 架构 |
| 工具数量？<5 vs 5-20 vs >20 | 工具管理复杂度 |
| 推理步数？<10 vs 10-50 vs 无限制 | 状态管理/上下文策略 |
| 是否多租户？ | 隔离/配额/计费 |
| 延迟要求？实时 vs 可接受秒级等待 | 同步 vs 异步 |
| 部署环境？云端 vs 边缘 vs 混合 | 基础设施选择 |
| 团队规模？1-3 人 vs 10+ 人 | 架构复杂度承受力 |

### Step 2: 选择架构模式

根据需求分析结果，使用上面的决策树选择初始架构模式。

### Step 3: 设计关键链路

画出核心用户场景的完整调用链路：
```
用户输入 → [API 网关] → [编排器] → [LLM 推理] → [工具选择]
  → [工具执行] → [结果注入] → [LLM 推理] → ... → [最终响应]
```

在每个环节标注：
- 状态管理需求（什么数据需要持久化）
- 错误处理策略（失败时怎么办）
- 可观测性数据（什么需要记录）

### Step 4: 迭代演进

架构不是一次成型的。随着系统复杂度的增长逐步演进：
- **Phase 1**: 单体 Agent + 基本工具调用
- **Phase 2**: 加入记忆系统 + 多工具支持
- **Phase 3**: 多 Agent 协作 + 事件驱动
- **Phase 4**: 微服务化 + 完整可观测性
- **Phase 5**: 生产级可靠性 + 企业级特性
