# 14.2 企业级架构 — Enterprise Agent Architecture

> 企业级 Agent 架构关注的是在大规模、多用户、高安全要求的组织环境中，如何设计 Agent 系统的多租户隔离、身份认证、数据合规、安全防护、服务等级协议和系统集成。与传统的企业软件架构相比，Agent 系统引入了 LLM 推理、工具调用、知识检索等非确定性组件，这些组件在企业环境中的治理和合规成为架构设计的核心挑战。

---

## 企业级 Agent 架构的核心矛盾

```
传统企业系统:                         Agent 系统:
────────────────                      ────────────────
确定性逻辑处理                       非确定性 LLM 推理
静态权限模型                         动态工具调用权限
固定数据流                           自主决策与规划
人为错误可追溯                       Agent "幻觉" 难以追溯
可控的执行路径                       不可控的推理路径
```

企业引入 Agent 时，最核心的问题不是"能否实现功能"，而是"能否控制在企业边界内"。

## 各子模块概览

### 14.2.1 多租户隔离 (Multi-Tenant Isolation)

| 层面 | 隔离内容 | 隔离策略 |
|------|----------|----------|
| 身份 | 租户 ID 贯穿所有请求 | 从 JWT/SSO 令牌解析，服务端注入 |
| 数据 | 向量库 / 知识库 / 对话历史 | 元数据过滤 / 独立 Collection / 独立实例 |
| 模型 | 模型配额 / API Key | 租户级 Token 池 + Rate Limit |
| 工具 | 工具权限 / 注册表 | 租户级工具白名单 + 参数级权限 |
| 配置 | Prompt / 知识库 / 安全策略 | 配置版本化 + 租户作用域 |
| 缓存 | 响应缓存 / 语义缓存 | Cache Key 包含 Tenant ID |
| 日志 | 审计日志 / 监控指标 | 日志可按租户检索 + 隔离 |

**关键设计原则**：
- 租户边界必须在**基础设施层**强制执行，不能仅依赖 Prompt 约束
- 每次跨层调用（网关→推理→检索→工具）都必须携带并校验租户上下文
- 租户隔离是架构问题，不是代码问题——必须有可测试的隔离边界

### 14.2.2 身份认证与 SSO 集成 (SSO/OAuth Integration)

| 组件 | 企业标准 | Agent 适配 |
|------|----------|------------|
| 认证协议 | OAuth 2.0 / OIDC / SAML | 扩展 Token 携带 Agent 上下文 |
| 目录服务 | LDAP / Active Directory | Agent 权限派生自 AD 组 |
| 身份传播 | JWT / Token Exchange | 链式传播：用户→Agent→工具→API |
| 委托授权 | OAuth 范围 (Scopes) | 细粒度工具授权 (Tool-level Scopes) |
| 会话管理 | SSO Session / Refresh | Agent 长任务 vs 短 Token 的矛盾 |

**核心挑战**：Agent 的自主操作时长可能远超 SSO Token 的有效期（秒级 vs 分钟级），需要设计 Token 刷新和委托授权机制。

### 14.2.3 数据驻留合规 (Data Residency & Compliance)

| 合规要求 | 对 Agent 的影响 | 架构应对 |
|----------|----------------|----------|
| GDPR | 用户数据可删除、可导出 | 记忆系统必须支持级联删除 |
| SOC 2 | 安全控制、审计追踪 | 不可篡改审计日志 |
| HIPAA | PHI 数据保护 | LLM 推理不能离开合规区域 |
| EU AI Act | AI 系统风险分级 | Agent 行为透明度报告 |
| 数据主权 | 数据不能出境 | Region 级部署 + 数据本地化 |

### 14.2.4 企业安全 (Enterprise Security)

Agent 系统在企业中面临**四类特有安全风险**：

1. **间接提示注入**：Agent 读取的文档/网页可能包含恶意 Prompt
2. **工具劫持**：攻击者诱导 Agent 调用危险工具
3. **数据泄露**：Agent 将敏感信息写入外部系统
4. **权限提升**：Agent 的自主操作被用于越权

### 14.2.5 SLA 设计 (SLA Design)

| 维度 | 传统 SLA | Agent SLA | 挑战 |
|------|----------|-----------|------|
| 可用性 | 99.9%+ uptime | 99.9%+ + LLM API 可用性 | LLM API 不在控制范围内 |
| 延迟 | P99 < 200ms | 复杂任务可能 > 30s | 推理时间不可预测 |
| 准确性 | 100% 确定 | 依赖 LLM 推理质量 | 非确定性输出 |
| 新鲜度 | N/A | 知识截止日期保证 | 需要 RAG 系统配合 |

### 14.2.6 企业系统集成 (Enterprise System Integration)

| 集成模式 | 协议/标准 | 典型场景 |
|----------|-----------|----------|
| API 集成 | REST / gRPC | CRM、ERP 数据查询与操作 |
| 事件集成 | Kafka / RabbitMQ | 异步工作流触发与通知 |
| 数据库集成 | JDBC / ODBC | 内部报表与数据仓库查询 |
| 文件集成 | SFTP / S3 | 批量数据处理与导入导出 |
| 遗留系统 | Mainframe / SOAP | 核心银行、保险系统对接 |

## 企业级 Agent 架构的核心原则

### 1. 隔离优先
```
所有企业级 Agent 设计的出发点：租户之间在数据、权限、配置、日志层面
必须做到"不可能意外交叉"，而不是"不容易交叉"。
```

### 2. 可审计默认
```
默认记录所有 Agent 决策和行动（包括推理过程），不记录需要显式配置。
审计日志必须满足"法庭可接受"标准（防篡改、链式哈希、完整性校验）。
```

### 3. 渐进式信任
```
Agent 的权限不是静态分配的，而是基于风险动态评估的：
  读操作 → 自动授权
  写操作 → 条件授权（敏感级别 + 上下文）
  删除/高危 → 人工审批
```

### 4. 非确定性补偿
```
在确定性基础设施层补偿 LLM 的非确定性：
  • Schema 强制输出（不是自由文本）
  • 工具调用的参数验证（不是模型自行决定）
  • 检索结果的事后校验（不是完全信任）
  • 敏感操作的二次确认（不是单步执行）
```

## 工程决策树

```
问题: 需要多大程度的企业级架构？
                   │
                   ▼
        租户数量 > 1 ？
        ├── 否 ──→ 跳过多租户隔离
        └── 是 ──→ 
                   │
        需要外部用户访问？
        ├── 否 ──→ 内部 SSO 即可
        └── 是 ──→ OAuth 2.0 + OIDC
                   │
        数据类型包含敏感信息？
        ├── 否 ──→ 标准日志 + 基础安全
        └── 是 ──→
                   │
        受监管行业（金融/医疗/政务）？
        ├── 否 ──→ 数据加密 + 访问控制
        └── 是 ──→ 合规认证 + 审计 + 数据驻留
                   │
        预期 Agent 调用量？
        ├── < 1K/天 → 基础监控 + 手动扩缩
        ├── 1K-10K/天 → SLA 监控 + 自动扩缩
        └── > 10K/天 → 完整 SLO + 错误预算 + 多活
```
