# 工具描述优化：让 LLM 准确选择工具

## 简单介绍

工具描述（description）是影响 LLM 工具选择准确率的最关键因素。工具描述优化（Description Engineering）是一门专门的技术，它研究如何编写让 LLM 能够准确理解和选择的工具描述。好的描述可以显著提升工具选择准确率，差的描述是 Agent 系统失败的最常见原因之一。

## 基本原理

LLM 选择工具的过程本质上是一个**语义匹配**问题：

```
用户输入: "帮我查北京天气"
               ↓ 语义匹配
工具描述: "获取指定城市的天气数据，包括温度、湿度和风力"
```

LLM 将用户意图与每个工具的描述进行隐式比较，选择语义最匹配的工具。描述的质量直接影响这个匹配过程。

## 编写原则

### 1. 描述用户意图，而非实现细节

```
❌ 差: "调用 weather_api v3 的 GET /v3/weather 接口，返回 JSON 格式的气象数据"
✅ 好: "获取指定城市的当前天气数据，包括温度、天气状况、湿度和风力"
```

### 2. 包含关键触发词

用户在说"发邮件"时，LLM 在工具描述中看到"发送邮件"时匹配度最高：

```
❌ 差: "消息投递功能（基于 SMTP 协议）"
✅ 好: "发送电子邮件给指定收件人，支持主题、正文和附件"
```

### 3. 明确区分边界条件

```json
{
  "name": "delete_user",
  "description": "⚠️ 不可逆操作！从系统中永久删除指定用户及其所有关联数据。请先调用 search_user 确认用户身份后再删除",
  "parameters": { ... }
}
```

### 4. 描述参数的使用场景

```json
{
  "name": "search_products",
  "parameters": {
    "properties": {
      "category": { "description": "商品分类过滤，例如 '电子产品'、'服装'。留空表示搜索所有分类" },
      "sort_by": { "description": "排序方式。'price_asc'（价格从低到高）、'price_desc'（价格从高到低）、'rating'（按评分）" }
    }
  }
}
```

## 结构模板

```
{工具的核心功能}，包括{主要能力1}、{主要能力2}等。
{边界条件/限制}。{与其他工具的区分说明}。
```

例如：
```
"发送营销邮件给已订阅用户，支持 HTML 模板和个性化变量。
不会发送给已退订用户。与 send_transactional_email 的区别在于：
本工具用于营销推广，发送频率受反垃圾邮件策略限制。"
```

## A/B 测试方法论

工具描述也需要像 Prompt 一样进行 A/B 测试：

1. 构造一组测试用例（20-50 条用户请求）
2. 用工具集 A（原描述）和工具集 B（优化描述）分别测试
3. 比较工具选择准确率（工具命中率）

## 核心挑战

- **描述长度 vs Token 消耗**：长描述更准确，但消耗更多 Token
- **多工具区分**：当多个工具功能相近时（如 send_email 和 send_sms），描述需要精确区分场景
- **抽象工具**：像 search_knowledge_base 这样的泛化工具很难用简短描述说清楚

## 工程启示

- 将工具描述视为代码：定期审核、版本控制、同行评审
- 监控"从未被选中的工具"——可能是描述有问题
- 监控"频繁被误选中的工具"——描述可能与其他工具冲突
