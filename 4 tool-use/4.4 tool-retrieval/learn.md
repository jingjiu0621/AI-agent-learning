# 4.4 tool-retrieval — 工具检索（海量工具场景）

## 概念定义

当 Agent 可用的工具数量超过 LLM 上下文窗口的容量时（如企业级平台拥有数百个 API），就需要工具检索来动态选择当前任务最相关的工具子集。工具检索本质上是一个**面向 Agent 的搜索系统**——从工具库中检索出最可能被当前任务用到的工具。

## 为什么需要工具检索

| 场景 | 工具数 | 处理方式 |
|------|--------|----------|
| 简单 Agent | 3-10 个 | 全量注入 Prompt 即可 |
| 中级 Agent | 10-30 个 | 按类别分组注入 |
| 复杂 Agent | 30-100 个 | 需要选择性注入 |
| 企业平台 | 100-10000+ 个 | **必须检索** |

LLM 的上下文窗口虽然不断增长（128K、200K），但**给 LLM 塞入 1000 个工具定义不仅浪费 Token，更会严重稀释注意力**——模型在海量信息中难以精准选择。

## 核心挑战

1. **语义匹配**：用户说"发邮件给张三"，需要检索到 send_email 工具而非 send_sms
2. **意图关联**：工具名和参数名不一定包含用户用词，需要语义理解
3. **动态场景**：哪些工具可用可能取决于用户权限、上下文状态
4. **实时性**：工具增删改需要即时反映在检索结果中
