# 4.1 function-calling — Function Calling 原理深度剖析

## 概念定义

Function Calling（函数调用）是 LLM 提供商设计的一种协议，允许模型在生成回复时输出结构化的工具调用请求，而非纯文本。它不是让 LLM "执行"代码，而是让 LLM "声明"需要调用哪个函数和参数，真正的执行由客户端完成。

## 核心角色

| 角色 | 说明 | 谁承担 |
|------|------|--------|
| 声明者 | 将可用工具告知 LLM | 开发者（通过 API） |
| 决策者 | 决定何时调用、调用哪个工具 | LLM |
| 执行者 | 实际运行工具代码 | 客户端（你的代码） |
| 反馈者 | 将工具结果注入对话上下文 | 客户端（通过 API） |

## 为什么需要 FC

在 Function Calling 出现之前，让 LLM 使用工具的方法极其脆弱：
- **Prompt 指令**：在 prompt 中写"如果你想查天气，请输出 GET_WEATHER:北京"，然后用正则解析——解析一错就全盘崩溃
- **JSON 模式**：要求 LLM 输出 JSON，但格式不稳定，字段名经常漂移

FC 从根本上解决了两个核心矛盾：
1. **格式稳定性**：模型被训练为在特定条件下输出结构化的函数调用 Token，而不是自由文本
2. **意图清晰性**：通过 Schema 声明让模型"知道"哪些工具可用、参数是什么，而非靠 prompt 暗示

## 设计范式演进

| 时代 | 方案 | 问题 |
|------|------|------|
| 早期 | 正则解析自然语言 | 不稳定、无法泛化 |
| Prompt 工程期 | 结构化 Prompt + 后处理解析 | 格式漂移、调试困难 |
| FC 协议期 | 原生结构化输出（当前） | 绑定特定模型 |
| 未来 | Grammar-constrained 解码 | 标准化但未广泛商用 |
