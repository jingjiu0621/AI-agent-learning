# 函数调用局限性与边界情况

## 简单介绍

Function Calling 是一个强大的协议，但并非万能。理解它的局限性同等重要——很多 Agent 故障的根源不是因为 LLM 不够聪明，而是因为开发者对 FC 的能力边界认知不足。

## 核心局限性

### 1. 工具选择不可靠——工具越多越不准确

当可用的工具数量增加时，LLM 选择正确工具的准确率显著下降。

| 工具数 | 选择准确率（典型值） | 影响 |
|--------|-------------------|------|
| 1-5 | ~95%+ | 几乎没有误选 |
| 5-15 | ~85-95% | 偶有误选，可接受 |
| 15-30 | ~70-85% | 有明显误选率 |
| 30+ | <70% | 频繁误选，需要检索 |

**原因**：LLM 需要在一次前向传播中"注意"到所有工具定义，工具越多，注意力越分散。

### 2. 参数幻觉

LLM 可能生成在工具定义中不存在，但语义上"看起来合理"的参数。例如：

```
工具定义参数: {location, unit}
LLM 输出: {"location": "北京", "unit": "celsius", "date": "2024-01-15"}
// "date" 并未在定义中出现
```

**频率**：较新模型（GPT-4, Claude 3）有明显改善，但仍非零。

### 3. 工具描述消耗上下文

每个工具的定义（name + description + parameters schema）平均消耗 50-200 Token。注入 50 个工具可能消耗 2500-10000 Token——这既影响成本，又压缩了实际对话内容的空间。

### 4. 精确匹配 vs 语义理解

FC 是**精确匹配**机制：工具名必须完全匹配，参数名必须完全匹配。LLM 需要精确地"记住"工具名并生成正确的字符串。这与人类的工作方式不同——人类擅长模糊匹配。

### 5. 结果长度不可控

工具返回的结果可能非常长（如搜索返回 50 条结果、数据库返回数千行）。Agent 必须截断或摘要这些结果，但摘要本身可能丢失关键信息。

## 边界情况

| 场景 | 问题 | 缓解策略 |
|------|------|----------|
| 空参数 | 某些参数在不传递时有不同含义 | 明确标注默认值 |
| enum 越界 | 参数值不在 enum 中 | 后验校验 + 自动修正 |
| 可选参数遗漏 | LLM 遗漏了重要可选参数 | 提高参数重要性描述 |
| 循环调用 | Agent 反复调用同一个工具 | 检测循环 + 终止 |
| 错误递归 | Agent 根据错误输出反复重试 | 限制最大重试次数 |

## 能力边界

- FC 不能**保证**工具调用正确——它只是"使正确更容易"
- FC 不能理解工具的内部逻辑——它只看 Schema，不看实现
- FC 不能处理需要实时数据流的场景（需要额外设计）

## 工程启示

1. **总是做后验校验**：不信任 LLM 输出的工具名和参数，始终在你的代码中校验
2. **设计优雅降级路径**：工具调用失败时，Agent 应该有备选方案
3. **监控工具选择率**：如果某个工具从未被选中，可能是描述有问题或根本不需要
4. **不要滥用 FC**：不是所有交互都需要 FC——有时候纯文本对话就足够了
