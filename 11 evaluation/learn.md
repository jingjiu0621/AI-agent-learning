# 11 evaluation — 评估体系与测试

评估是 Agent 系统从"能跑"到"好用"的必经之路。没有评估，就不知道 Agent 是不是真的变好了、退步了、还是只是凑巧对了。如果说架构设计决定 Agent 的天花板，那么评估体系就决定了你能不能看到这个天花板。

## 为什么 Agent 评估比传统软件测试更难

传统软件测试是确定性验证：输入 X，期望输出 Y，不等就报错。Agent 评估面对三个根本性挑战：

**1. 非确定性输出**
同一个 Prompt、同一个模型，两次输出可能不同。temperature > 0 时更是如此。这意味着"通过/不通过"的二元判定不适合 Agent——需要统计意义上的评估。

**2. 多步骤链式依赖**
Agent 不是一次回答完事，而是思考→行动→观察→再思考的循环。中间步骤正确但最终结果错误（或反过来），让归因变得困难。如面试题所说："中间任何一步错了，最终结果可能还是对的，也可能是凑巧对的。评测必须覆盖过程，不能只看结果。"

**3. 环境耦合**
Agent 的表现在很大程度上依赖外部工具、知识库状态、API 响应。同样的代码在不同的环境状态下表现可能截然不同。评测必须绑定环境快照。

## 核心挑战

| 挑战 | 描述 | 缓解策略 |
|------|------|----------|
| 评估成本高 | 每次评测需要大量 LLM 调用 + 人工标注 | 分层采样、自动评估(LM-as-Judge)、渐进式评测 |
| 指标不可靠 | LLM-as-Judge 有偏差、评分不一致 | 多评委投票、校准评分标准、定期评测评委质量 |
| 评测集过拟合 | 反复调试同一套评测集导致分数虚高 | 开发集/回归集/隐藏集三区分立 |
| 环境漂移 | 外部 API/知识库变化导致评测不可比 | 绑定环境快照、并行对比新旧版本 |
| 长尾覆盖不足 | 常见场景表现好但边界情况差 | 意图聚类、主动补充长尾、模糊测试 |

## 评估与可观测性的协同

评估和可观测性（Module 10）是一体两面：
- **评估**回答"做得好不好"——主动的、周期性的质量检查
- **可观测性**回答"发生了什么"——被动的、实时的运行监控

实践中，评估发现的问题靠可观测性定位根因，可观测性发现的异常靠评估确认影响范围。两者共用同一套日志和链路追踪基础设施。

## 与其他模块关联

- **10 Observability**: 评估和可观测性共享数据管道，评估结果写入监控看板
- **12 Safety**: 安全评估是 Agent 评估的重要子集，特别是注入攻击和输出护栏测试
- **5 Agents-core**: 评估需要 Agent 主循环暴露中间状态（思考、工具调用、观察）
- **9 RAG**: RAG 评估（检索质量、生成忠实度）是 Agent 评估的基础能力
- **15 Projects**: 实战项目中评估体系是上线门禁的核心门控
