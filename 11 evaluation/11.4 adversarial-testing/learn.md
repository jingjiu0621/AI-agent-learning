# 11.4 adversarial-testing — 对抗测试与压力测试

## 简单介绍

对抗测试（Adversarial Testing）是 Agent 评估体系中"测底线"的部分。之前的章节关注"Agent 正常运行时表现如何"，而对抗测试回答的是另一个问题：**当有人恶意攻击、输入异常、环境恶化时，Agent 还能不能正常工作？**

```
正常评估 vs 对抗测试的区别：

正常评估                         对抗测试
─────────                       ────────────
"最好的情况下表现如何"            "最坏的情况下会怎样"
Agent 尽力完成任务的场景          有人故意让 Agent 出错的场景
用户是善意的                    攻击者是恶意的
输入在预期范围内                 输入在预期范围外
环境稳定                        环境被干扰或篡改
```

对抗测试在 Agent 安全体系中扮演"压力测试+渗透测试"的双重角色。它既是安全评估（找漏洞），也是可靠性评估（测极限）。

## 攻击面全景

Agent 系统的攻击面比传统软件广泛得多——攻击者可以通过自然语言直接与 Agent 交互，这是传统系统没有的攻击向量。

## 背景

### 对抗测试的演进

```
2022 年及以前
  传统软件对抗测试（Fuzzing、渗透测试）
  攻击面：网络协议、文件解析、Web 应用
  方法：输入变异、代码覆盖引导

2023 年 — LLM 对抗测试兴起
  Prompt 注入成为新攻击向量
  "越狱"Prompt 在社区传播
  攻击面扩展到自然语言层面
  方法：手动构造对抗 Prompt、基于规则的注入检测

2024 年 — Agent 对抗测试专业化
  攻击面扩展到工具调用、记忆检索、多 Agent 通信
  自动化对抗测试工具涌现
  OWASP 发布 LLM Top 10 安全风险
  企业开始建立红队团队
  方法：自动化注入生成、Agent 行为模糊测试

2025-2026 年 — 体系化对抗测试
  Agent-vs-Agent 对抗测试成主流
  对抗训练与防御前置到 Agent 训练阶段
  持续红队（Continuous Red Teaming）成标准实践
  多 Agent 协作攻击的防御研究
  方法：对抗性 Agent 自动生成攻击、进化式攻击策略、防御蒸馏
```

### 为什么 Agent 比传统系统更容易被攻击

```
传统系统                            LLM Agent 系统
─────────                          ────────────────
输入格式受限（JSON/XML/二进制）      输入是自然语言（无限可能）
攻击需要理解协议                    攻击只需要会说话
语法错误会被拒绝                    语义"正确"的恶意内容难以拦截
攻击面可枚举                        攻击面不可枚举
权限模型明确                        Prompt 级别的"权限"模糊
安全边界是代码级的                   安全边界是语义级的
```

### Agent 对抗测试的关键挑战

| 挑战 | 描述 | 影响 |
|------|------|------|
| 攻击面无限 | 自然语言输入空间本质上不可穷举 | 无法像传统软件一样证明"没有漏洞" |
| 攻击与正常使用模糊 | 恶意 Prompt 和正常 Prompt 没有明确边界 | 过度防御会影响正常用户体验 |
| 环境依赖 | 同一攻击在不同环境下效果不同 | 复现性差，修复验证困难 |
| 多步骤攻击 | 攻击可以跨多步实施，每一步单独看都正常 | 传统单步检测无效 |
| 间接注入 | 攻击隐藏在 Agent 读取的外部内容中 | 用户无法控制 Agent 读取的内容 |
| 进化式攻击 | LLM 可以自动生成新的攻击变体 | 静态防御很快失效 |

## 对抗测试方法论

```
┌─────────────────────────────────────────────────────────────────┐
│                     Agent 对抗测试方法论                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. Threat Modeling（威胁建模）                                   │
│     → 谁可能攻击、用什么方式、攻击什么目标                         │
│     输出：攻击树 + 风险优先级                                    │
│                                                                 │
│  2. Attack Generation（攻击生成）                                 │
│     → 手动构造 / 模板生成 / LLM 自动生成 / 进化搜索               │
│     输出：对抗测试集                                             │
│                                                                 │
│  3. Test Execution（测试执行）                                    │
│     → 在受控环境中运行对抗测试                                    │
│     输出：攻击成功率 + 攻击影响分析                               │
│                                                                 │
│  4. Defense Evaluation（防御评估）                                │
│     → 检测率 / 拦截率 / 误报率 / 对正常功能的影响                  │
│     输出：防御覆盖矩阵                                           │
│                                                                 │
│  5. Iterative Hardening（迭代加固）                               │
│     → 修复漏洞 → 重新测试 → 验证修复                              │
│     输出：安全改进记录                                            │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 核心矛盾

**Agent 越有用、越开放，攻击面就越大；越安全、越封闭，能力就越受限。**

```
可用性-安全性权衡曲线：

安全 ←──────────────────────────────────── 开放
  │                                          │
  │  完全封闭                        │        │  完全开放
  │  (无工具/无外部信息/            │        │  (任意工具/任意输入
  │   输出严格过滤)         最佳平衡点 │        │   无内容过滤)
  │                          ↑       │        │
  │                          └── 这是做对抗测试    │
  │                              要找到的点        │
  └──────────────────────────────────────────────┘
  
  对抗测试的目标不是消灭所有漏洞（不可能），而是：
  1. 知道漏洞在哪里
  2. 知道最坏情况下会损失什么
  3. 把风险控制到可接受水平
```

## 如何评估对抗测试本身的质量

| 指标 | 定义 | 目标 |
|------|------|------|
| 攻击覆盖率 | 覆盖了多少种攻击类型 | 覆盖所有已知攻击面 |
| 攻击成功率 | 攻击成功突破防御的比例 | 越低越好 |
| 防御检测率 | 防御机制成功检测攻击的比例 | > 95% |
| 误报率 | 正常行为被误判为攻击的比例 | < 1% |
| 对抗鲁棒性 | 攻击变种对成功率的影响 | 攻击变种不应大幅提高成功率 |
| 修复有效性 | 修复后同一攻击是否仍有效 | 100% 修复 |
| 攻击效率 | 构造有效攻击需要的尝试次数 | 越多次越好（攻击成本高） |
