# 11.1 metrics — 评估指标

**Agent 的性能不能靠"感觉"来衡量。** 指标是把主观质量转化为客观数字的标尺——但选择错误的指标比没有指标更危险，因为它会让你朝着错误的方向优化。

## 内容结构

| 子主题 | 核心问题 | 难度 |
|--------|----------|------|
| 11.1.1 Success Rate | Agent 完成任务的成功率怎么算？ | ★★★☆☆ |
| 11.1.2 Efficiency | 完成任务花了多少步/Tokens/时间？ | ★★★☆☆ |
| 11.1.3 Robustness | 面对干扰和异常输入时表现如何？ | ★★★★☆ |
| 11.1.4 Adaptability | 在新场景下能否快速适应？ | ★★★★☆ |
| 11.1.5 Reliability | 多次运行结果是否一致和可复现？ | ★★★☆☆ |
| 11.1.6 Composite Score | 多个指标如何加权融合成综合评分？ | ★★★★☆ |

## 指标分类体系

```
                      Agent 评估指标树
                            │
        ┌───────────────────┼───────────────────┐
        │                   │                   │
    ┌───┴───┐         ┌─────┴─────┐       ┌────┴────┐
    │ 有效性 │         │   效率     │       │  稳定性  │
    ├───────┤         ├───────────┤       ├─────────┤
    │成功率  │         │ 步数      │       │ 鲁棒性   │
    │目标达成度│        │ Token 消耗 │       │ 适应性   │
    │完成质量 │         │ 响应延迟   │       │ 可靠性   │
    │用户满意度│        │ 成本      │       │ 一致性   │
    └────────┘         └───────────┘       └─────────┘
```

## 核心指标速查表

| 指标 | 衡量的方面 | 典型值参考 | 陷阱 |
|------|-----------|-----------|------|
| 任务成功率 | 是否达到目标 | Agent 任务通常 60-85% | 简单任务天花板效应；复杂任务定义模糊 |
| 平均步数 | 效率 | 简单 1-3 步，复杂 5-15 步 | 步数少不一定好（可能跳过必要步骤） |
| Token 消耗 | 成本 | 简单 1-3K，复杂 10-50K | 只看总量忽略输入/输出比例 |
| 端到端延迟 | 响应速度 | P50 < 5s, P95 < 15s | 长思考 vs 快速响应需要平衡 |
| 鲁棒性得分 | 抗干扰 | 取决于干扰强度 | 需要标准化干扰模板 |
| 一致性 | 可复现 | 同 prompt 同结果率 > 80% | 部分创造性任务天然低一致性 |
| 适应性 | 泛化能力 | 新场景保持率 > 70% | 数据获取困难 |

## 指标间的权衡关系

```
                       高成功率
                          ▲
                          │
          ┌────────────────┼────────────────┐
          │                │                │
    高一致性 ◄─────────────┼──────────────► 高适应性
          │                │                │
          │          ┌─────┴─────┐          │
          │          │  最优区域  │          │
          │          └─────┬─────┘          │
          │                │                │
          ▼                ▼                ▼
     低 Token 消耗     低延迟          高鲁棒性
```

**实际工程中的取舍：**
- 追求高成功率 → 增加推理步数和 Token 消耗（成本上升）
- 追求高效率 → 可能降低复杂任务的完成质量
- 追求高鲁棒性 → 额外的验证和回退机制降低速度
- 追求高适应性 → 泛化通常以牺牲特定场景的最优性能为代价

## 分层次指标设计

```
L1: 单步指标
    ┌──────────────────────────────────────────┐
    │ 工具选择准确率 │ 参数正确率 │ 单步完成率 │
    └──────────────────────────────────────────┘
                             │
L2: 任务级指标               │
    ┌──────────────────────────────────────────┐
    │ 任务完成率 │ 目标达成度 │ 用户满意度     │
    │ 步数效率 │ Token 效率 │ 延迟指标        │
    └──────────────────────────────────────────┘
                             │
L3: 系统级指标               │
    ┌──────────────────────────────────────────┐
    │ 整体成功率(聚合) │ P50/P95 延迟         │
    │ 错误率 │ 成本 / 请求 │ SLO 达成率       │
    └──────────────────────────────────────────┘
```

## 指标设计的常见误区

1. **单一指标崇拜**：只看成功率会鼓励 Agent 在不确定时拒绝任务
2. **幸存者偏差**：只统计完成的请求，忽略中途放弃的
3. **指标透支**：反复调优同一个指标导致过拟合
4. **忽略成本**：100% 成功率但成本翻 10 倍不可持续
5. **忽略过程**：只看最终结果，中间步骤的 bug 被掩盖

## 最佳实践建议

1. **至少选择 3 个指标构成评估体系**：1 个有效性 + 1 个效率 + 1 个稳定性
2. **区分离线指标和线上指标**，两者可能不一致
3. **指标要可分解**：能在模块级和步骤级定位问题
4. **定期校准**：检查指标是否仍然反映真实质量
5. **多维度对比**：A/B 测试时要同时对比所有核心指标

## 与其他模块关联

- **11.3 Automated Eval**: LLM-as-Judge 本质上是自动化指标计算
