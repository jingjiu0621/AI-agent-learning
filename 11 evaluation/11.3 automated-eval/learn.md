# 11.3 automated-eval — 自动评估：让 LLM 评估 LLM

## 简单介绍

自动评估（Automated Evaluation）是 Agent 评估体系中"怎么测"的答案。当测试集很大、人工评审成本太高时，我们需要自动化手段来评判 Agent 输出的质量。核心问题只有一个：**如何在没有人参与的情况下，可靠地判断一个 Agent 做得好不好？**

最主流的方法就是 **LLM-as-Judge**——用一个 LLM（通常是更强或同等的模型）来评审另一个 Agent 的输出。这听起来有点递归（用 AI 评估 AI），但在实践中被证明是成本和质量的最佳平衡点。

```
评估方式的性价比光谱：

人工评审        LLM-as-Judge        自动化脚本
─────────       ────────────        ───────────
最准确           准确性中高          局限性大
成本最高          成本可控            成本最低
不可扩展          可扩展              可完全自动化
有主观偏差        有系统性偏差         没有偏差（但面窄）
适合小批量精评     适合大批量评估        适合确定性任务
```

## 自动评估的三种基本模式

```
模式一：参考答案匹配 (Reference-based)
  Agent 输出 ──→ 与参考答案比较 ──→ 得分
  适用：答案确定的场景（SQL 生成、实体抽取）
  原理：精确/模糊/语义匹配

模式二：无参考答案评判 (Reference-free)
  Agent 输出 ──→ 直接评判质量 ──→ 得分
  适用：开放式任务（对话、摘要、创意写作）
  原理：LLM-as-Judge、质量评分

模式三：行为轨迹评判 (Trajectory-based)
  Agent 轨迹 ──→ 评判过程质量 ──→ 得分
  适用：多步 Agent 任务（工具调用、网页导航）
  原理：过程分析、无效行为检测
```

## 背景

### 为什么需要自动评估？

在 Agent 开发的早期阶段（1-5 人团队），评估靠人工看——开发者在终端里跑几个测试，肉眼检查输出是否正确。但当团队扩大、迭代加快、评测集增长到数百上千条时，纯人工评估就不可行了：

```
人工评估瓶颈：
  • 100 条 Agent 响应 × 3 分钟评审 × 3 轮迭代 = 15 小时
  • 评审者疲劳 → 评分不一致 → 数据不可靠
  • 每天合并多个 PR → 每个 PR 都要评估 → 根本评审不过来
  • 优化周期拉长 → 不敢频繁修改 → 迭代速度下降
```

自动评估解决了"扩展性"问题，但又带来了新的挑战——LLM 评审本身是否可靠？

### 自动评估的演化史

```
2022 年及之前
  传统 NLP 指标：
    BLEU / ROUGE / METEOR / BERTScore
    问题：与人类判断的相关性有限

2023 年
  LLM-as-Judge 兴起：
    GPT-4 作为评审 → 相关性远超传统指标
    MT-Bench, Chatbot Arena 等基准使用 LLM 评审
    问题：LLM 评审有偏差（位置偏好、长度偏好、自增强）

2024 年
  校准技术发展：
    多评委投票、评分标准细化、偏差检测
    JudgeBench 提出评审元评估
    G-Eval 等结构化评分框架普及

2025-2026 年
  成熟化与工具化：
    Rubric 引导评分成主流——结构化标准降低方差
    偏好感知自适应奖励建模 (PaTaRM)
    Pairwise vs Pointwise 争议趋于融合方案
    评审校准从"事后修补"前置到"评估设计"阶段
```

## 核心矛盾

**自动评估的可靠性与成本的矛盾。**

```
用更强的模型（如 GPT-5 / Claude 5）评判 → 更可靠，但成本高
用同等级模型评判 → 成本低，但可能有"同模型偏好"偏差
用开源小模型评判 → 成本极低，但可靠性存疑
用确定性脚本评判 → 零成本但面很窄

核心问题：
  "更强模型的评判"是否真的比"被评判模型"更可靠？
  当法官模型和被评判模型能力接近时，评估是否还有意义？
```

研究表明（JudgeBench, ICLR 2025），**法官模型的能力确实直接影响评估质量**——能力更强的模型在判别准确率上显著优于弱模型。但好消息是：即使使用同等级模型，通过多评委投票和结构化评分标准，也能获得可用的评估结果。

## 当前挑战

### 1. 法官偏差 (Judge Bias)

LLM 作为评审者存在多种系统性偏差，这是自动评估最核心的挑战：

```python
JUDGE_BIASES = {
    "position_bias": {
        "description": "偏好先出现/后出现的答案",
        "典型表现": "Pairwise 比较中位置在前的选项得分更高",
        "缓解方法": "交换位置取平均、对称评估"
    },
    "leniency_bias": {
        "description": "偏好更长/更短的输出",
        "典型表现": "长篇大论的输出即使质量一般也获得更高分",
        "缓解方法": "长度校准、控制输出长度范围"
    },
    "self_enhancement_bias": {
        "description": "偏好自己模型家族的输出",
        "典型表现": "GPT-4 评判 GPT-3.5 比评判 Claude 更宽松",
        "缓解方法": "使用不同模型家族作为法官"
    },
    "verbatim_copying_bias": {
        "description": "偏好输出中复用了提示词的内容",
        "典型表现": "简单复述指令的 Agent 得分更高",
        "缓解方法": "检测和惩罚复制行为"
    },
    "format_bias": {
        "description": "偏好特定格式的输出",
        "典型表现": "结构清晰但内容差的输出得分高于内容好但结构乱的",
        "缓解方法": "分项评分、内容先于格式"
    },
    "severity_bias": {
        "description": "评分范围的系统性偏移",
        "典型表现": "有些法官倾向于给 3-4 分（中心偏移），有些给 1-5 分",
        "缓解方法": "评分标准化、已知参考点校准"
    }
}
```

### 2. 评委间一致性 (Inter-rater Reliability)

不同法官模型之间的一致性可能很差：

```
同一批 Agent 输出：
  法官 GPT-4:  平均分 4.2/5
  法官 Claude:  平均分 3.6/5
  法官 Gemini:  平均分 4.5/5

问题：谁的评分是对的？
如果法官之间都不一致，评分还有参考价值吗？
```

缓解方案：使用**校准参考**——在评估批中加入已知质量的"锚点"样本，将各法官的评分映射到统一尺度。

### 3. 评估的评估（元评估）

谁能评判法官？这就是 JudgeBench 解决的问题。

```
元评估问题：
  我们需要评估 LLM-as-Judge 本身的质量。
  但如果用另一个 LLM 来评估，又回到了同样的问题。

目前的解决思路：
  1. 构建人工标注的"黄金标准"判断集
  2. 用黄金标准衡量法官的准确率
  3. 选择准确率最高的法官或法官组合
  4. 长期追踪法官性能是否退化
```

## 2025-2026 年自动评估的推荐实践

基于最新研究和社区实践，以下是当前自动评估的最佳配置建议：

1. **Rubric 引导优先**：结构化评分标准比自由提示的 LLM 评判方差低 30-50%，2025 年后成为主流共识
2. **多法官集成**：使用 3+ 不同模型家族的法官投票，比单一强法官更可靠
3. **校准锚点**：在每个评估批次中插入已知质量的参考样本，将各法官评分映射到同一尺度
4. **位置随机化**：Pairwise 比较时随机化 A/B 位置，多轮取平均消除位置偏差
5. **不同场景不同方法**：答案确定用脚本匹配，开放式用 LLM Judge，连续监控用 Rubric 评分
6. **定期元评估**：跟踪法官一致性趋势，如果一致性下降说明评估标准需要重新校准
7. **成本分层**：90% 的例行评估用轻量方案（单 Judge + 简单 Rubric），10% 的关键评估用重方案（多 Judge + 人工抽检）
