# 6.1 short-term — 短期记忆：Agent 的工作台面

## 定义

短期记忆（Short-term Memory）是 Agent 的"工作台面"——它在物理上直接对应 LLM 的上下文窗口（Context Window），是当前对话中所有可见信息的总和。包括系统指令、用户消息、助手回复、工具调用记录、检索到的外部内容等所有进入 LLM 推理范围的数据。

## 核心角色

| 角色 | 说明 | 谁承担 |
|------|------|--------|
| 容器 | 承载当前对话所有可见信息 | 上下文窗口 |
| 分配器 | 决定什么信息进入/留在上下文 | Token 预算管理模块 |
| 淘汰者 | 上下文超限时决定丢弃什么 | 截断/压缩策略 |
| 刷新者 | 确保关键信息不会过期 | 过期与刷新机制 |

## 为什么需要短期记忆

### 没有短期记忆的 Agent

如果不管理短期记忆，Agent 要么"失忆"（上下文被截断丢失关键信息），要么"撑爆"（Token 超限导致调用失败），要么"注意力稀释"（无关信息太多降低推理质量）。

### 与人类短时记忆的类比

| 人类短时记忆 | Agent 短期记忆 |
|-------------|---------------|
| 约 7±2 个组块（Miller's Law） | 约 4K-200K Tokens（取决于模型） |
| 未经复述的信息在 15-30 秒内遗忘 | 超限后被截断/压缩移除 |
| 注意力决定什么进入短时记忆 | Token 预算决定什么保留 |
| 复述可维持记忆 | 重新注入可恢复上下文 |

### 核心矛盾

**上下文窗口是有限资源，怎么分配？**

一个 128K 窗口的模型，看起来很大，但实际可用空间远小于理论值：
- 系统指令通常占据 2K-8K
- 工具定义可能占据 10K-100K+
- 检索到的文档片段可能占据 30K+
- 历史对话记录不断累积
- 模型生成回复本身也需要空间

实际留给"有效交互"的上下文，通常只有总量的 30%-60%。
