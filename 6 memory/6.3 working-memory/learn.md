# 6.3 工作记忆 (Working Memory)

## 定义

工作记忆是 AI Agent 架构中的"草稿纸与任务板"——它是系统当前正在积极处理的信息的临时存储与操作空间。与长期记忆（存储事实、知识、程序性技能）不同，工作记忆跟踪的是**此刻正在进行中的任务状态**：当前的目标是什么、已经完成了哪些步骤、还有哪些子任务待处理、推理过程中产生了哪些中间结论。

工作记忆的本质是 **Agent 的瞬时计算上下文**，它直接决定了 Agent 能够处理的问题复杂度和多步骤推理的可靠性。

## 核心角色

| 角色 | 说明 |
|------|------|
| **状态追踪** | 维护当前任务的进行状态、已完成步骤、待办项 |
| **中间计算** | 存储推理过程中产生的临时变量和中间结论 |
| **上下文粘合** | 连接不同模块/工具调用的输入输出，保证信息流不断裂 |
| **优先级裁决** | 决定在资源受限时哪些信息应该保留、哪些可以丢弃 |
| **焦点控制** | 确定 Agent "此刻应该关注什么" |

## 为什么需要专门的工作记忆层？

1. **长期记忆不适合频繁读写**：长期记忆（如向量数据库、知识图谱）的读写延迟通常在毫秒到秒级，而 Agent 的推理步之间需要微秒到毫秒级的信息交换。用长期记忆做中间存储会带来不可接受的性能开销。

2. **长期记忆不适合临时数据**：将中间推理步骤写入长期记忆会造成不可控的记忆污染——后续检索可能误将这些临时状态当作"知识"返回，导致幻觉。

3. **上下文窗口不是工作记忆**：LLM 的上下文窗口（context window）本质是只读的输入缓冲区，Agent 无法在推理过程中高效地对其内容进行增删改查。工作记忆是 Agent 层自己维护的读写数据结构。

4. **任务结构化需要明确的分层**：复杂的多步骤任务天然具有层次结构（主任务 -> 子任务 -> 原子步骤），需要一种机制来管理这种嵌套关系，而工作记忆就是这种机制的核心实现。

## 核心矛盾：容量有限 vs 任务复杂

工作记忆面临的根本矛盾是**有限的容量与无限增长的任务状态之间的持续冲突**：

- **容量硬约束**：工作记忆的物理载体（LLM 的上下文窗口、Agent 的内存数据结构、token 预算）存在严格的容量上限。
- **任务复杂度增长**：真实世界任务往往需要跟踪大量状态——多步骤推理链、多工具调用历史、分支探索、异常处理路径等。
- **信息衰退**：随着新的信息不断涌入，旧信息如果不被显式维护，会被自然覆盖或衰减。

这个矛盾是所有工作记忆设计的核心驱动力。大部分优化技术本质上都是在回答同一个问题：**当装不下的时候，保留什么、丢弃什么、如何压缩？**

## 与 CPU 寄存器的类比

工作记忆与 CPU 寄存器在架构哲学上有深刻的相似性：

| 维度 | CPU 寄存器 | Agent 工作记忆 |
|------|-----------|----------------|
| **位置** | 最靠近 CPU 计算单元 | 最靠近 LLM 推理核心 |
| **速度** | 纳秒级，远快于主存 | 即时可用，无需检索 |
| **容量** | 极少（通常 16-32 个） | 极小（受上下文窗口严格限制） |
| **用途** | 暂存中间计算结果 | 暂存推理中间状态 |
| **溢出策略** | 寄存器溢出到栈/堆 | 溢出到短期/长期记忆 |
| **手动 vs 自动** | 编译器决定寄存器分配 | Agent 架构决定工作记忆管理策略 |

正如寄存器分配是编译器后端最关键的优化之一，工作记忆管理是 Agent 架构中最核心的设计挑战。

## 工作记忆的信息生命周期

```
  信息产生       写入工作记忆        使用/消费        清理/归档
   [推理] ────> [分配空间] ────> [被后续步骤引用] ────> [溢出/丢弃/压缩]
                    │                     │
                    ▼                     ▼
              容量检查并触发         引用计数跟踪
              溢出策略选择           依赖关系维护
```

## 关键设计维度

1. **容量策略**：固定大小 vs 动态扩展 vs 自适应调节
2. **淘汰策略**：LRU、FIFO、优先级、语义重要性
3. **持久化策略**：何时溢出、溢出到什么层、如何重建
4. **一致性策略**：如何保证工作记忆中的状态与外部世界一致
5. **可见性策略**：哪些内容暴露给 LLM、哪些只在 Agent 框架内部维护

## 典型工作记忆结构

```
WorkingMemory {
    scratchpad:          // 推理过程的中间草稿
    taskStack:           // 子任务调用栈
    progressTracker:     // 任务进度状态
    intermediateCache:   // 中间结果缓存
    focusManager:        // 注意力焦点控制
    overflowHandler:     // 溢出处理
}
```
