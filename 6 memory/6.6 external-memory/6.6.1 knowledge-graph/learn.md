# 6.6.1 知识图谱构建（Knowledge Graph Construction）

## 简单介绍

知识图谱是 Agent 外部记忆中最富语义表达能力的形式。它将现实世界中的实体（如人物、地点、概念、事件）以及它们之间的关系编码为图结构，使 Agent 能够从关系网络中理解世界运行逻辑。

在 AI Agent 体系内，知识图谱不是静态的百科全书，而是动态演化的语义存储器——Agent 每处理一次交互，都可能抽取新的三元组充实图谱，使其成为一个自生长的知识体。

## 基本原理

知识图谱的核心是 **RDF 三元组**（Subject-Predicate-Object）：

```
(实体A) ---[关系]---> (实体B)
```

例如：
```
(Claude) ---[created_by]---> (Anthropic)
(Anthropic) ---[headquartered_in]---> (San Francisco)
```

多条三元组连接形成网络，Agent 可在网络上执行多跳推理。每个三元组可以附带属性（属性图模型）或上下文元数据（时间戳、置信度、来源 trace）。

## 背景与演进

| 阶段 | 时间 | 代表系统 | 特征 |
|---|---|---|---|
| 早期符号AI | 1970s-1980s | Cyc、Frame-based systems | 手工编码的符号知识库，推理能力强但构建成本极高 |
| 语义网 | 1990s-2000s | RDF、OWL、SPARQL | 标准化知识表示语言，面向 Web 的开放互联 |
| 互联网图谱 | 2000s-2010s | Google Knowledge Graph、Freebase、DBpedia | 大规模自动构建，搜索引擎驱动 |
| NER+关系抽取 | 2010s-2020s | 深度学习驱动的信息抽取 | 从非结构化文本中自动抽取实体和关系 |
| LLM+KG | 2020s-今 | 大语言模型增强的知识图谱构建 | 利用 LLM 进行零样本/少样本三元组抽取和推理 |

## 核心矛盾：精确性 vs. 覆盖度

知识图谱构建面临的根本矛盾是：**精确性要求知识高度结构化且人工验证，覆盖度要求自动大规模抽取，两者互为代价**。

- 人工构建的图谱（如 Cyc）精度可达 95% 以上，但覆盖范围极其有限，构建周期以年计
- 自动抽取的图谱（如从 Web 文本抽取）覆盖广泛，但噪音高，关系准确率可能低于 70%

Agent 需要在这两者之间寻找平衡：核心领域知识采用人工审核的高质量图谱，长尾知识采用自动抽取并标注置信度。

## 主流优化方向

### 1. LLM 辅助三元组抽取
利用 LLM 从非结构化文本中提取结构化三元组。优化点：
- **Prompt 工程**：设计 Chain-of-Thought prompt 引导 LLM 逐步识别实体和关系
- **Schema 约束**：用预定义的 Schema 限制输出格式和关系类型
- **Self-Consistency**：多次采样并通过投票机制消除 LLM 幻觉引入的错误三元组

```python
# 伪代码：LLM 辅助三元组抽取
def extract_triples(text, schema):
    prompt = f"""
    从以下文本中抽取(all 实体, relation)三元组。
    仅使用 Schema 中定义的关系类型。
    
    文本: {text}
    Schema: {schema}
    
    输出格式: (实体1, 关系, 实体2)
    """
    candidates = []
    for _ in range(5):  # Self-Consistency 采样
        candidates.append(llm.generate(prompt))
    return majority_vote(candidates)
```

### 2. 实体链接与消歧
同一实体可能有多种表述（"乔布斯"、"Steve Jobs"、"Steven Paul Jobs"），实体链接负责将不同表述映射到同一图谱节点：
- **指代消解**（Coreference Resolution）：将代词和别名归并到同一实体
- **实体归一化**（Entity Normalization）：通过 embedding 相似度判断是否为同一实体
- **模糊匹配**：利用字符级相似度 + 知识库别名索引处理拼写差异

### 3. 关系推理与补全
基于已有三元组推断缺失关系。典型方法：
- **TransE / RotatE**：将实体和关系映射到向量空间，通过平移/旋转变换建模关系
- **Node2Vec**：图上的随机游走生成节点 embedding，保留图结构信息
- **Graph Neural Networks**：通过邻居节点聚合更新实体表示

### 4. 动态图谱更新
Agent 运行过程中，知识图谱需要持续演进：
- **增量抽取**：每次 Agent 交互后，仅对新信息进行抽取而非全量重建
- **冲突检测**：新三元组与已有知识冲突时，依据时间戳、置信度、来源权威性裁决
- **淘汰机制**：过时的三元组被降权或删除，避免知识污染

## 产生的结果

一个建设良好的 Agent 知识图谱会产生：

- **关系推理能力**：Agent 可以通过图路径发现隐含关联（如 A 的同事的上级 = A 的间接上级）
- **事实一致性**：图谱中每条知识都有明确的来源 trace，Agent 可追溯验证
- **跨场景复用**：同一图谱可被多个 Agent 会话共享，知识积累不丢失
- **可解释推理链**：Agent 可展示从初始实体到结论的完整关系路径

## 最大挑战

知识图谱构建的最大挑战是 **知识版本控制与冲突解决**。

现实世界知识具有时间维度（"苹果公司 CEO 在2011年是乔布斯，在2024年是库克"），但 Agent 在不同时间获取的信息可能相互冲突。设计一个支持时间版本化、多源投票、置信度衰减的知识图谱存储引擎，是工程上的巨大难题。

## 能力边界

-   Agent 能利用图谱进行多跳推理（2-5 跳），精度随跳数增加而衰减
-   图谱查询的延迟通常在 10-100ms（本地图数据库），远程查询可达 200ms+
-   实体数量通常在百万级以内时查询性能最优，数十亿级需分布式图数据库
-   知识图谱仅能存储显式编码的实体-关系知识，无法存储隐性知识（如技能熟练度、情感反馈）

## 区别对比：知识图谱 vs. 向量数据库

| 维度 | 知识图谱 | 向量数据库 |
|---|---|---|
| 数据结构 | 图（节点+边） | 向量 embedding |
| 查询方式 | 结构化查询（Cypher/SPARQL） | 相似度搜索（ANN） |
| 推理能力 | 支持多跳关系推理 | 不支持关系推理 |
| 精确性 | 精确匹配，关系可解释 | 语义近似，结果模糊 |
| 更新成本 | 关系维护成本高 | 重新 embedding 成本高 |
| 适用场景 | 需要精确关系推理 | 需要语义相似度匹配 |

## 核心优势

知识图谱的核心优势是 **精准的关系推理能力**。它不像向量数据库那样基于统计相似性做模糊匹配，而是基于精确的逻辑关系进行多跳推导。当 Agent 需要回答问题"A 公司与 B 公司是否存在供应链关系"时，只有知识图谱能够给出精确的"是/否/通过路径 C 间接关联"的回答。

## 工程优化

### 1. 缓存层级
- **L1 缓存**：最近访问的实体和关系（内存 LRU cache）
- **L2 缓存**：频繁访问的子图（Redis / Memcached）
- **L3 缓存**：全量图谱（图数据库持久化存储）

### 2. 查询优化
- **关系预计算**：将常见多跳路径预计算为虚拟关系（如"推荐链"）
- **查询裁剪**：基于实体热度、关系类型对搜索空间进行剪枝
- **批量查询**：将多个 Agent 的图谱查询合并为一个 batch 减少 I/O

### 3. Neo4j 实战要点
```cypher
// 创建实体和关系
CREATE (a:Agent {name: 'travel_agent', version: '1.0'})
CREATE (d:Destination {name: 'Tokyo', country: 'Japan'})
CREATE (a)-[:RECOMMENDS {confidence: 0.95}]->(d)

// 多跳查询 - 查找推荐链上第三跳的目的地
MATCH (a:Agent {name: 'travel_agent'})-[:RECOMMENDS*3]->(d:Destination)
RETURN d.name

// 带权重的路径查询
MATCH path = (user:User)-[:INTERESTED_IN]->(c:Category)
              <-[:BELONGS_TO]-(d:Destination)
WHERE user.id = 123
RETURN d ORDER BY d.score DESC
```

### 4. 自研图引擎优化
如果是自研轻量级图引擎（如 NetworkX 基础上的优化）：
- 邻接表压缩：使用 CSR（Compressed Sparse Row）格式存储图的邻接信息
- 并行 BFS：利用 SIMD 指令加速图遍历
- 属性按需加载：查询路径时先只加载节点 ID 和边类型，确认路径有效后再加载完整属性

## 场景判断：何时使用知识图谱

| 场景特征 | 适合程度 | 原因 |
|---|---|---|
| 需要精确关系匹配 | 高 | 知识图谱的关系是显式的，不存在模糊性 |
| 需要多跳逻辑推理 | 高 | 图遍历自然支持多跳 |
| 实体之间关联密集 | 高 | 图谱在关系密集型数据上优势最大 |
| Agent 需要解释推理链 | 高 | 图谱路径天然构成可解释链 |
| 知识变化频繁 | 低 | 图谱 Schema 变更成本高 |
| 主要是非结构化文本 | 低 | 需要先做信息抽取，增加复杂度和精度损失 |
| 对延迟极其敏感（<1ms） | 低 | 图查询通常比向量检索慢，多跳查询延迟累积 |
