# 6.6.2 图谱遍历（Graph Traversal）

## 简单介绍

图谱遍历是 Agent 在知识图谱上执行路径搜索、关系推理和信息聚合的核心操作。如果说知识图谱是"知识的地图"，图谱遍历就是"在地图上导航的算法"。Agent 通过遍历操作，从初始实体出发，沿关系边探索关联实体，最终到达目标信息节点。

## 基本原理

图谱遍历的本质是在图数据结构上的搜索问题。一个知识图谱 G 定义为 (V, E)，其中 V 是节点集（实体），E 是边集（关系）。遍历算法从起始节点 v0 出发，按照特定策略访问可达节点。

遍历的核心参数三元组：
- **起始节点**：遍历的入口点（如用户的问题中提及的实体）
- **关系路径**：遍历所沿的关系类型序列（如 `[:EMPLOYED_BY]->[:LOCATED_IN]`）
- **终止条件**：何时停止遍历（如达到目标实体、最大深度、累计置信度阈值）

## 背景与演进

| 阶段 | 时间 | 代表技术 | 特征 |
|---|---|---|---|
| 经典图算法 | 1960s-1970s | BFS、DFS、Dijkstra | 基础搜索算法，理论基础完备 |
| 图数据库查询 | 2000s-2010s | Cypher、SPARQL、Gremlin | 声明式/命令式图查询语言，面向应用开发者 |
| 大规模图分析 | 2010s | Pregel、GraphX、TinkerPop | 分布式图处理框架，支持十亿级节点 |
| ML 增强遍历 | 2020s | GNN、RL-based traversal | 利用机器学习指导遍历方向，减少搜索空间 |
| Agent 图推理 | 2020s-今 | LLM + Graph Agent | 大模型辅助的语义化图遍历决策 |

## 核心矛盾：搜索广度 vs. 推理深度

图谱遍历的核心矛盾是：**搜索广度越大，找到目标的可能性越高，但计算开销和延迟也随之剧增；搜索深度越深，推理能力越丰富，但路径噪音和错误累积也越严重**。

Agent 必须在每一步遍历中权衡：是否继续拓展到更多邻居节点（广度）？是否在当前路径上走得更深（深度）？这一决策直接影响查询质量和系统性能。

## 主流优化方向

### 1. 有偏遍历（Biased Traversal）

经典 BFS/DFS 对所有节点一视同仁。有偏遍历引入优先级机制，引导搜索朝高价值区域前进：

- **PageRank 引导**：优先访问 PageRank 分数高的节点，这些节点通常是知识图谱中的中心实体
- **关系类型加权**：不同关系类型具有不同重要程度（如 `[:EMPLOYED_BY]` 可能比 `[:MENTIONED_IN]` 携带更多信息）
- **时序衰减**：较新的三元组获得更高的遍历优先级，反映知识的时效性

```python
# 有偏 BFS 伪代码
def biased_bfs(start_node, target_type, max_depth=3):
    queue = PriorityQueue()
    queue.push((start_node, 0), priority=1.0)  # (node, depth), priority
    visited = set()
    
    while not queue.empty() and depth <= max_depth:
        node, depth = queue.pop()
        if node.type == target_type:
            return construct_path(node)  # 找到目标
        
        for neighbor, relation in get_neighbors(node):
            if neighbor.id in visited:
                continue
            priority = compute_priority(neighbor, relation)
            queue.push((neighbor, depth+1), priority)
            visited.add(neighbor.id)
    
    return None  # 未找到目标
```

### 2. 元路径引导（Meta-Path Guided）

元路径是预定义的关系序列模板，例如 `[Person]--[:WORKS_AT]-->[Company]--[:LOCATED_IN]-->[City]`。利用元路径，Agent 可以将遍历限制在语义上合理的路径上，避免探索无意义的组合。

常见元路径示例：
```
# 知识图谱中的典型元路径
[User] -[:BOUGHT]-> [Product] -[:BELONGS_TO]-> [Category] -[:RECOMMENDED_FOR]-> [User]
[Agent] -[:TRAINED_ON]-> [Dataset] -[:RELATES_TO]-> [Domain] -[:QUERIED_BY]-> [User]
[Entity] -[:MENTIONED_IN]-> [Document] -[:AUTHORED_BY]-> [Person] -[:SPECIALIZES_IN]-> [Topic]
```

元路径的核心价值是**将搜索空间从指数级压缩到线性级**——Agent 不需要尝试所有可能的关系组合，只需沿着预定义的语义路径前进。

### 3. 条件停止策略

决定何时停止遍历与决定如何遍历同样重要。过度遍历浪费计算资源，过早停止则可能遗漏关键信息。

条件停止策略包括：
- **信息增益阈值**：当新访问节点不再显著增加已知信息量时停止（基于熵的度量）
- **置信度衰减**：路径上的置信度分数累积衰减，低于阈值时剪枝
- **Timeout 机制**：设定硬性的遍历时间上限（如 200ms），超时返回当前最优结果
- **结果数量阈值**：收集到足够数量的候选结果后停止探索

### 4. LLM 辅助的语义化遍历

传统图遍历依赖预定义的关系类型匹配，LLM 可以增强遍历的语义理解：

- **自然语言到遍历路径**：将用户问句转换为图遍历计划
  - 用户问："东京最好的寿司店"
  - LLM 解析：`[City:Tokyo]--[:HAS_RESTAURANT]-->[Restaurant]--[:SERVES]-->[Cuisine:Sushi]--[:RATED]-->[Rating > 4.5]`
- **模糊关系匹配**：当查询的关系类型与图谱中的不完全匹配时，LLM 进行语义对齐
- **动态元路径合成**：LLM 根据问题上下文实时生成非预定义的元路径

### 5. 并行遍历与结果融合

大型图谱中，从多个起点同时发起遍历可以显著加速：

- **双向 BFS**：从起点和潜在目标同时遍历，在中点相遇
- **多源发散**：从用户问题中提取的多个实体同时出发，多路并行遍历
- **结果融合排序**：将多个遍历路径的结果合并，按置信度、路径长度、信息量综合排序

## 产生的结果

一个高效的图谱遍历系统使 Agent 具备：

- **多跳关系推理能力**：从用户问题中的单个实体出发，沿关系路径到达答案实体
- **路径可解释性**：可以展示从输入到输出的完整关系链
- **上下文扩展能力**：在对话中主动扩展相关实体（"你提到了东京，我还知道东京有..."）
- **不确定条件下的近似推理**：当精确路径不存在时，返回相似度最高的近似路径

## 最大挑战

图谱遍历的最大挑战是 **搜索空间组合爆炸**。

在密集连接的知识图谱中，一个实体的平均度（degree）可能达到 50-100。如果 Agent 尝试 3 跳遍历，最坏情况下的搜索空间达到 50^3 = 125,000 个节点。对于 6 跳推理，搜索空间膨胀到 50^6 = 156 亿，完全是天文数字。

实际系统中必须综合使用元路径剪枝、有偏遍历、提前停止等多种策略来控制在计算预算内。

## 能力边界

-   单次遍历的可靠深度：在高质量图谱上为 3-5 跳，超过 5 跳路径噪音显著增加
-   搜索空间限制：元路径引导下通常限制在 1,000-10,000 个候选节点内
-   时间开销：3 跳本地遍历约 5-50ms，远程图数据库查询约 50-500ms
-   跨图遍历：跨不同知识图谱的遍历需要实体对齐层，对齐精度通常低于 90%

## 区别对比：BFS vs. DFS vs. 有偏遍历

| 维度 | BFS | DFS | 有偏遍历 |
|---|---|---|---|
| 搜索策略 | 逐层扩展 | 沿一条路径走到底 | 按优先级扩展 |
| 空间复杂度 | O(b^d) | O(b*d) | O(b^d)，但受优先级限制 |
| 能找到最短路径 | 是 | 否 | 不保证 |
| 适合场景 | 需要最近邻信息 | 深度推理、路径存在性验证 | 高价值信息导向型查询 |
| A gent 适用性 | 低（浪费资源在低价值节点） | 中（可能陷入无意义的深路径） | **高**（最符合 Agent 的信息需求） |

## 核心优势

图谱遍历相比其他检索方式的核心优势是 **可组合的语义推理能力**。向量检索只能做"语义相似度匹配"，给出与查询语义上近似的结果；图遍历可以做"通过关系链推导"，给出需要多步逻辑推理才能得到的答案。前者是统计相似性，后者是逻辑推演——这决定了 Agent 在面对需要精确推理的问题时，图遍历是不可替代的。

## 工程优化

### 1. 预计算视图
- **物化视图**：将高频查询的遍历路径预计算结果存储为物化视图（Materialized View）
- **路径索引**：对常见的 2 跳、3 跳路径模式建立倒排索引，O(1) 时间完成遍历
- **热点节点缓存**：被频繁访问的节点及其邻居的拓扑结构保持在热缓存中

### 2. 查询计划优化
```python
# 基于代价的遍历计划选择
def plan_traversal(query, graph_stats):
    plans = [
        ("bidirectional_bfs", estimate_cost_bfs(query, graph_stats)),
        ("meta_path_follow", estimate_cost_metapath(query, graph_stats)),
        ("biased_beam", estimate_cost_beam(query, graph_stats))
    ]
    # 选择预估代价最低的计划
    return min(plans, key=lambda x: x[1])[0]
```

### 3. 批量路径推理
多个 Agent 或同一 Agent 的多个查询可以合并为批量图遍历：
- 共享 BFS 的前几层探索结果
- 利用 GPU 并行计算多条路径的置信度分数
- 批处理结合向量化的邻接矩阵乘法加速大规模遍历

### 4. 增量遍历
对于流式查询（如实时对话中 Agent 持续追问）：
- 保留上一次遍历的 frontier 节点集
- 新查询只从上一次的 frontier 继续推进，而非从头开始
- 维护遍历状态的 checkpoint，支持回溯和分支

## 场景判断：何时使用图谱遍历

| 场景特征 | 适合程度 | 原因 |
|---|---|---|
| 需要多跳逻辑推理 | 高 | 遍历是图推理的基本操作 |
| 关系路径需要可视化 | 高 | 遍历路径天然可展示 |
| 查询关系类型明确 | 高 | 遍历算法依赖关系类型的精确匹配 |
| Agent 需要解释推理过程 | 高 | 每条遍历路径都是一个解释链 |
| 需要全图范围的分析 | 中 | 遍历可以但成本较高，更适合局部搜索 |
| 实时性要求极高（<10ms） | 低 | 多跳遍历的延迟累积难以满足 |
| 图谱非常稀疏 | 低 | 稀疏图中遍历没有扩展方向，效果差 |
