# 6.4.1 语义检索（Semantic Search）

## 简介

语义检索通过将自然语言查询和记忆内容映射到高维向量空间，计算它们之间的语义相似度来实现检索。与关键词匹配不同，语义检索能够理解"苹果公司"和"库克领导的科技企业"之间的语义等价关系，即使它们没有共享任何词汇。

## 基本原理

语义检索的核心流程分为三个步骤：

```
查询文本 → 编码器(Embedding Model) → 查询向量 q
记忆记录 → 编码器(Embedding Model) → 记忆向量库 D = {d₁, d₂, ..., dₙ}
                                    ↓
                     相似度计算: sim(q, dᵢ) → Top-K 结果
```

其中编码器通常是一个预训练的 Transformer 模型（如 BERT、Sentence-BERT、E5 等），将变长文本映射为固定维度的稠密向量（通常 384-1536 维）。

## 相似度度量

### 余弦相似度（Cosine Similarity）

最常用的语义相似度量，衡量两个向量方向的夹角余弦值：

```
cos(q, d) = (q · d) / (||q|| × ||d||)
```

- **取值范围**：[-1, 1]（实际文本场景通常为 [0, 1]）
- **优点**：不受向量长度影响，适合比较不同长度的文本
- **缺点**：对向量各维度一视同仁，无法区分维度重要性

### 最大内积搜索（MIP, Maximum Inner Product）

在某些 embedding 空间中，向量未被归一化，内积大小同时受方向和模长影响：

```
MIP(q, d) = q · d = ||q|| × ||d|| × cos(q, d)
```

- **与余弦相似度的关系**：当所有向量被归一化为单位向量时，MIP 等价于余弦相似度
- **适用场景**：衡量匹配程度时还需考虑向量的"强度"或"置信度"
- **挑战**：MIP 无法通过传统 L2 空间索引高效检索，需要专门的 MIP 索引结构

## 近似最近邻搜索（ANN）索引原理

面对百万级以上的记忆向量，暴力遍历（精确最近邻）的计算复杂度 O(N×D) 不可接受。近似最近邻搜索通过牺牲少量精度换取数量级的性能提升。

### HNSW（Hierarchical Navigable Small World）

HNSW 是当前工业界最流行的 ANN 索引算法，基于可导航小世界图的多层结构。

#### 结构原理

```
Layer 2:    ○ ─── ○        (稀疏层，长程连接)
              ↙ ↓ ↘
Layer 1:  ○ ── ○ ── ○ ── ○  (中间层)
            ↙ ↘ ↙ ↘ ↙ ↘
Layer 0:  ○─○─○─○─○─○─○─○─○  (底层，全部节点+短程连接)
```

- **多层图索引**：上层是下层的粗粒度抽象，节点数逐层指数递减
- **搜索过程**：从最顶层开始贪心搜索，找到当前层局部最近邻后进入下一层继续搜索，逐层细化
- **插入复杂度**：O(log N)，新节点通过多层链接策略连接到近邻节点
- **搜索复杂度**：O(log N)，远优于暴力搜索的 O(N)

#### 核心参数

| 参数 | 作用 | 调优方向 |
|------|------|----------|
| M | 每个节点每层最大连接数 | M 越大，检索精度越高，内存消耗越大 |
| ef_construction | 构建时的动态候选集大小 | 越大构建质量越高，但构建时间增加 |
| ef_search | 搜索时的动态候选集大小 | 越大召回率越高，但延迟增加 |

#### 优缺点

- **优点**：召回率高、查询延迟低（毫秒级）、支持增量插入
- **缺点**：内存占用高（每个节点需存储多层连接关系）、构建时间较慢
- **最适场景**：对延迟和精度要求都高的在线系统

### IVF（Inverted File Index）

IVF 通过聚类将向量空间划分为多个 Voronoi 单元，搜索时只在与查询最近的几个单元中搜索。

#### 结构原理

```
向量空间
    │
    ├── 聚类 (K-means, K=1024)
    │
    ├── Cell 0: [d₁, d₅, d₉, ...]   ← 质心 c₀
    ├── Cell 1: [d₂, d₇, d₁₂, ...]  ← 质心 c₁
    ├── Cell 2: [d₃, d₆, d₁₁, ...]  ← 质心 c₂
    └── ...
            │
搜索流程: 查询 q → 找到最近的 nprobe 个质心 → 在这些单元内暴力搜索
```

#### 核心参数

| 参数 | 作用 | 调优方向 |
|------|------|----------|
| nlist | 聚类中心数量 | 越大每个单元越小，搜索越快，但精度可能下降 |
| nprobe | 搜索时探访的单元数 | 越大召回率越高，但延迟增加 |

#### 优缺点

- **优点**：内存效率高（只需存储聚类中心和倒排列表）、构建速度快
- **缺点**：召回率低于 HNSW（尤其在 nprobe 较小时）、不支持高效的增量插入（需要周期性重建索引）
- **最适场景**：内存受限、对构建速度有要求的批处理或近线系统

### HNSW vs IVF 对比

| 维度 | HNSW | IVF |
|------|------|-----|
| 搜索延迟 | 低（O(log N)） | 中（O(nprobe × N/nlist)） |
| 内存占用 | 高 | 低 |
| 构建速度 | 慢 | 快 |
| 增量插入 | 支持（在线友好） | 不友好（需重建） |
| 召回率@高精度 | 极高 | 中高 |
| 典型场景 | 在线实时检索 | 大规模批量检索 |

## 主流优化方向

### 1. 量化压缩

- **PQ（Product Quantization）**：将向量分割为子空间，每个子空间独立量化，大幅降低内存
- **SQ（Scalar Quantization）**：将 float32 降为 int8，内存减少 75%，精度损失可控
- **OPQ（Optimized PQ）**：通过旋转矩阵优化子空间划分，提升量化质量

### 2. 混合检索

- 稠密检索 + 稀疏检索（BM25）融合，兼顾语义匹配和精确关键词匹配
- 典型代表：ColBERT 的延迟交互架构

### 3. 分片与分布式

- 将索引分片到多台机器，每台机器维护部分数据的独立索引
- 查询时广播到所有分片，合并各分片 Top-K 结果

## 工程优化

- **向量数据库选型**：Milvus（分布式）、Faiss（单机高性能）、Qdrant（Rust 实现低延迟）、Pinecone（托管服务）
- **Embedding 模型选型**：bge-large（中文强）、gte-large（通用）、e5-mistral（长文本）、text-embedding-3（OpenAI）
- **缓存策略**：高频查询向量缓存、最近查询结果缓存
- **预过滤**：在向量搜索前通过元数据标签过滤缩小候选集

## 能力边界

- 对极度稀疏或罕见的语义关系理解有限
- 无法处理需要多跳推理的复杂查询（如"昨天提到的那个和 A 项目有关但 B 曾修改过的计划"）
- Embedding 模型的上下文窗口限制了单次处理的文本长度
- 对高度领域特定的术语可能表现不佳（通用 embedding 未见过）

## 最大挑战

语义鸿沟（Semantic Gap）：查询与记忆之间的语义差异可能非常大——一个简短的查询需要匹配到包含丰富细节的长篇记忆，embedding 在这一过程中的信息压缩损失是不可避免的。

## 场景判断

| 场景 | 推荐策略 |
|------|----------|
| 对话历史检索 | 语义检索 + 时间衰减 |
| 知识库问答 | 语义检索为主 |
| 代码片段查找 | 语义检索 + 结构特征匹配 |
| 精确事实查询 | 语义检索 + 关键词 BM25 混合 |
| 实时响应系统 | HNSW 索引 + 量化 |
| 内存受限设备 | IVF + PQ 量化 |
