# 6.5.1 情节记忆摘要 (Episodic Summary)

## 简单介绍

情节记忆摘要是将 Agent 与用户之间的一段完整交互经历（从开始到结束的对话序列）压缩为结构化摘要的过程。它保留了交互的核心要素——谁、什么时间、做了什么、结果如何——同时丢弃对长期推理无意义的对话噪声。

## 基本原理

情节记忆遵循 Tulving 定义的三个核心维度：

- **时间性 (Temporal)**：事件发生的顺序有时间戳锚定
- **空间上下文 (Spatial Context)**：交互发生的环境、场景、任务领域
- **因果结构 (Causal Structure)**：事件之间的因果链，包括前置条件、行动序列、结果反馈

Agent 的情节摘要必须同时保留这三个维度的关键信息，因为长期记忆检索的核心需求就是"在什么情况下遇到过什么，当时的做法和结果是什么"。

## 背景与演进

| 时期 | 方法 | 局限性 |
|------|------|--------|
| 早期 (Pre-LLM) | 关键词提取 + 固定模板填充 | 无法理解语义，摘要质量差 |
| 第一代 (2022-2023) | 简单 Prompt 全文摘要 | Token 消耗大，摘要粒度不可控 |
| 第二代 (2023-2024) | 分块提取 + 层次化合并 | 引入了段落间的连贯性问题 |
| 当前 (2024-2026) | 事件驱动的结构化摘要管线 | 仍有信息压缩偏差问题 |

## 核心技术：LLM 摘要 + 关键事件提取

### 1. LLM 驱动的结构化摘要

利用 LLM 的语义理解能力，从原始对话中提取结构化信息。典型的提示词设计如下（伪代码）：

```
system_prompt = """
你是一个记忆摘要系统。将下面 Agent-用户对话转换为结构化情节记忆。

输出格式：
- 标题: 一句话概括
- 用户意图: 用户的核心需求
- 关键事件链: [事件1, 事件2, ...]
- 最终结果: 成功/失败/待定
- 涉及知识领域: [领域1, 领域2]
- 重要决策点: 交互中做出的关键决策
- 情感/满意度: 用户情绪变化曲线
"""
```

### 2. 关键事件提取策略

事件提取是情节摘要的精髓。以下几种策略可组合使用：

**基于语义转折点 (Semantic Shift Detection)**：
- 检测对话中话题或意图的显著变化点
- 使用向量嵌入的余弦距离突变检测
- 例如：从"解释 Python 装饰器"跳转到"比较 Flask vs FastAPI"就是语义转折

**基于行动-结果对 (Action-Result Pairs)**：
- 提取 Agent 的每个行动及其对应的用户反馈
- 关键指标：行动是否达到预期效果
- 失败的行动比成功的行动更有记忆价值

**基于重要性评分 (Importance Scoring)**：
- 每段对话片段计算一个重要性分数
- 评分因素：信息新颖度、对任务目标的贡献度、用户明确强调程度
- 低于阈值的片段被丢弃或压缩到最低层级

### 3. 分层摘要架构

```
原始对话 (full context)
    │
    ├── Token 窗口分割
    │        │
    ├── Chunk 级摘要 (保持局部连贯性)
    │        │
    ├── 段落级摘要 (合并相关 chunk)
    │        │
    └── 全局摘要 (最终结构化记忆)
```

此架构的关键在于：**底层摘要保留细节，上层摘要提炼核心**，形成一棵摘要树。检索时可以根据需求选择不同粒度的摘要节点。

## 核心矛盾

**压缩率 vs 信息完整性**：

已知原始对话长度 L_tokens，目标摘要长度 S_tokens，压缩率 R = S/L。

实验表明：
- R > 0.3：摘要几乎保留所有重要信息，但压缩效果有限
- 0.1 < R < 0.3：保留主要事件链，但细节和语气信息开始丢失
- R < 0.1：仅能保留最核心的结论，因果链可能断裂

实践中推荐 R 控制在 0.1~0.2 之间，且根据重要性动态调整——重要交互保留更多细节。

## 主流优化方向

1. **要点蒸馏 (Key Points Distillation)**：不再生成连贯摘要，而是提取离散的关键事实点，每个事实点是一个独立记忆单元
2. **结构感知压缩 (Structure-Aware Compression)**：识别对话中的代码块、数据表格等重要结构并优先保留
3. **多视角摘要 (Multi-Perspective Summarization)**：从不同角度（技术、业务、用户偏好）同时生成多个维度的摘要
4. **增量更新 (Incremental Update)**：当对话追加新内容时，只对新增部分做摘要增量更新，避免全文重新压缩

## 最大挑战

**关键信息丢失 (Critical Information Loss)**：
LLM 在摘要过程中可能会丢失对整体上下文来说看似不重要、但后续检索时至关重要的信息。这是有损压缩的固有问题，没有彻底解决方案，只能通过冗余存储和多粒度检索来缓解。

例如：用户随口说了一句"下个月要去日本出差"，在当前交互中无关紧要，但三天后 Agent 需要根据这条信息推荐日本旅行攻略时，如果这句话已经被摘要过滤掉，就出现了信息丢失。

## 能力边界

- **无法理解隐含意图**：如果用户没有明确表达的需求，基于文本的摘要无法提取
- **跨会话关联需要更高层模块**：情节摘要只在单次交互内工作，跨会话的模式识别需要语义抽象层处理
- **实时性限制**：高质量摘要需要等待交互完成才能生成（因为需要完整的事件链），无法在对话中间生成完整的结构化摘要

## 区别对比

| 维度 | 情节摘要 | 完整日志存储 | 语义抽象 |
|------|---------|------------|---------|
| 粒度 | 事件级 | Token 级 | 概念级 |
| 压缩率 | 高 (0.1-0.2) | 极低 (1.0) | 极高 (<0.05) |
| 可回溯性 | 中 | 高 | 低 |
| 检索效率 | 高 | 低 | 极高 |
| 噪声水平 | 低 | 高 | 极低 |

## 核心优势

- **与人类认知对齐**：人类记忆本身也是摘要式的，Agent 的情节摘要机制与人类自然交互
- **高度结构化**：便于后续的检索、关联和推理
- **可并行处理**：多个交互的摘要可以独立生成，适合分布式处理

## 工程优化

1. **滑动窗口增量摘要**：维护一个不断更新的摘要缓存，新对话追加时只对新增内容做增量摘要，然后与历史摘要合并
2. **重要性触发的全量重摘要**：当增量摘要累积到一定程度后，触发一次全量重摘要以修复增量合并带来的信息扭曲
3. **摘要树稀疏化**：定期清理底层的低层级摘要节点，只保留高层结构化记忆
4. **异步批处理**：将多个交互的摘要任务打包为 batch，利用 LLM 的批处理接口降低开销

## 场景判断

**适合使用情节摘要的情况**：
- 用户与 Agent 的每次交互都需要记录核心要点
- 需要从历史交互中快速检索"之前是怎么做的"
- 在有限的存储空间中保留尽可能多的交互记录

**不适合使用情节摘要的情况**：
- 需要对交互细节进行精确审计（如金融交易记录）
- 交互内容涉及法律或合规要求必须保留原始对话
- 需要从大量交互中挖掘跨会话的模式和趋势（此时应使用语义抽象）
