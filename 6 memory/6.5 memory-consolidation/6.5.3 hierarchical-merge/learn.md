# 6.5.3 分层合并 (Hierarchical Merge)

## 简单介绍

分层合并是一种多级渐进式的记忆整合策略，将记忆按照时间粒度和重要性分为多个层级，从短期工作记忆开始，经历中期缓冲区，最终沉淀到长期知识库。每一级都有独立的容量限制、合并策略和提升条件。

## 基本原理

分层合并受人类记忆系统启发——人类的记忆不是一步到位的，而是经历感觉记忆 → 短期记忆 → 长期记忆的三级渐进过程。Agent 的分层合并在设计上遵循相同的原理：**不是所有信息都值得立即进入长期记忆，也不是所有信息都应该被丢弃**。

核心设计理念：

```
短期层 (L1)          中期层 (L2)          长期层 (L3)
──────────           ──────────           ──────────
容量: 高             容量: 中             容量: 有限
延迟: 实时           延迟: 秒级           延迟: 批次
精度: 完整           精度: 过滤后         精度: 结构化
驻留: 单次会话       驻留: 跨会话        驻留: 永久(衰减)
          │                  │
          └──── 提升 ────────┘
                (达到阈值)
```

## 背景与演进

| 时期 | 方法 | 局限性 |
|------|------|--------|
| 固定窗口 | 只保留最近 N 轮对话 | 没有长期记忆能力 |
| 单级缓存 | 一个统一缓冲区，定期转存 | 没有区分信息重要性 |
| 两级架构 | 短期 + 长期，按重要性判断 | 缺乏中间过渡，突变太大 |
| 多级分层 | 3+ 级渐进式整合 | 系统复杂度显著增加 |

## 典型的三级架构设计

### 第一级：交互缓冲区 (L1 - Working Memory)

```
特点:
  - 容量: ~10K tokens 或 ~50 轮对话
  - 保留完整原始内容
  - 不压缩，不抽象
  - 随会话结束自动降级或清理

提升条件:
  - 当 L1 容量使用超过 80%
  - 或者会话自然结束
  - 或者检测到高重要性信息
```

L1 的核心职责是"不丢信息"。在此阶段，原始对话的所有细节都被保留，不做任何有损压缩。

### 第二级：情节缓存区 (L2 - Medium-Term Memory)

```
特点:
  - 容量: ~100 个情节摘要
  - 每个情节已做摘要压缩 (压缩率 0.1-0.2)
  - 跨会话持续累积
  - 为语义抽象提供足够样本

提升条件 (满足任一):
  - 同一主题的情节数量 ≥ N (通常 3-5)
  - 重要性累积值超过阈值
  - 在后续推理中被反复引用
  - 在 L2 中驻留超过时间阈值 (老化提升)
```

L2 可以理解为"等待处理的待办队列"。这里的情节摘要尚未被抽象，但已经做了初步清洗。L2 的主要作用是累积足够的样本量，供后续的语义抽象层使用。

### 第三级：长期知识库 (L3 - Permanent Memory)

```
特点:
  - 容量: 理论无上限，实际按存储策略管理
  - 存储形式: 语义抽象 + 关键情节锚点
  - 高度结构化，已建立关联网络
  - 可以设置衰减和遗忘机制

触发合并条件:
  - 从 L2 提升而来:
    - 语义抽象置信度 ≥ 0.75
    - 或情节重要性评分 ≥ 阈值
  - 直接注入 (紧急高优):
    - 用户明确要求"请记住这一点"
    - 涉及安全/配置的关键信息
```

L3 不是简单存储更多数据，而是形成了一个**知识网络**——新进入的记忆会与已有知识建立链接，更新相关抽象，并可能触发已有知识结构的重组。

## 触发条件设计

分层合并的触发条件是系统设计的核心决策点：

### 时间阈值触发

```python
def check_time_promotion(memory, tier_config):
    elapsed = now() - memory.creation_time

    # 每个层级有不同的最大驻留时间
    if tier == L2 and elapsed > MAX_L2_DWELL_TIME:
        # 强制提升或降级
        if memory.importance > PROMOTE_IMPORTANCE_THRESHOLD:
            promote_to_L3(memory)
        else:
            degrade_and_discard(memory)
    return
```

### 容量阈值触发

```python
def check_capacity_promotion(tier):
    usage = tier.current_size / tier.max_size
    if usage > CAPACITY_WATERMARK:  # 例如 80%
        # 触发批量压缩和提升
        candidates = tier.sort_by_importance()
        # 低重要性: 丢弃
        # 中重要性: 留在本级
        # 高重要性: 提升到上级
```

### 重要性累积触发

```python
def compute_importance_accumulation(memories):
    """
    同一主题的多个相关记忆累积重要性
    """
    topics = cluster_by_semantic_similarity(memories)
    for topic, cluster in topics.items():
        cumulative_importance = sum(
            m.importance_score * m.age_weight()
            for m in cluster
        )
        if cumulative_importance > PROMOTE_THRESHOLD:
            trigger_promotion(cluster)
```

### 引用频率触发

```python
def check_reference_promotion(memory):
    """
    在推理中被频繁引用的记忆自动提升
    """
    if memory.reference_count > REFERENCE_THRESHOLD:
        # 被频繁引用说明有用，提升层级
        increase_retention_priority(memory)
        # 如果还未在 L3，提升到 L3
        if memory.current_tier == L2:
            promote_to_L3(memory)
```

## 降级与遗忘策略

分层合并不是单向的"只升不降"。同样重要的还有降级机制：

```python
def apply_decay(memory):
    """
    所有长期记忆都有衰减曲线
    """
    # 遗忘曲线公式 (Ebbinghaus 变体)
    retention = exp(-time_since_access / memory.strength)

    if retention < DISCARD_THRESHOLD:
        if memory.current_tier == L3:
            # 降级到 L2 (保留摘要，丢弃详细内容)
            demote_to_L2(memory)
        elif memory.current_tier == L2:
            # 降级到 L1 或丢弃
            if memory.importance > RELEVANCE_FLOOR:
                demote_to_L1(memory)
            else:
                discard_memory(memory)
```

## 核心矛盾

**响应延迟 vs 整合深度 (Latency vs Depth)**：

分层越深，整合效果越好（信息更结构化、关联更丰富），但延迟越大。实时交互场景要求快速响应，深度整合在时间上不可接受。

```
实时响应要求 ──→ 浅层快速处理 ──→ 信息可能不够结构化
深度整合要求 ──→ 深层慢速处理 ──→ 响应延迟不可接受
        │
        ▼
  异步分层处理：
  实时层只做最轻量的存储，
  深度整合在后台异步执行
```

## 主流优化方向

1. **自适应层级数量**：根据系统负载动态调整分层深度，负载高时减少层级节约资源
2. **预测性提升 (Predictive Promotion)**：基于用户行为模式预测哪些记忆会被使用，提前提升
3. **混合粒度存储**：L3 也保留底层级的关键细节锚点（anchor points），防止过度抽象丢失关键细节
4. **分层检索**：检索时同时查询所有层级，按相关性排序，而非只查 L3

## 产生的结果

```
L1 → L2 合并过程:
  10 个原始对话片段
  → 摘要压缩为 3 个情节记忆
  → 存储到 L2 缓冲区

L2 → L3 合并过程:
  5 个关于"用户偏好静态类型语言"的相关情节
  → 语义抽象输出"用户偏好强类型系统"
  → 更新 L3 知识图谱节点
  → 更新相关链接 (类型安全、编译期错误检测等)
```

## 最大挑战

**层级间的信息一致性**：

不同层级的记忆可能存在矛盾。例如 L3 中存储的抽象是"用户偏好 Python"，但 L1 的最新对话显示用户正在询问 Rust。如果不处理这种矛盾，检索时会出现冲突。

解决方案：
- 新信息优先 (Recency Wins)：较新的具体事实可以暂时覆盖旧的抽象
- 矛盾标记：当检测到层级间矛盾时，标记相关记忆并触发回顾
- 逐步更新抽象：不立即推翻旧抽象，而是降低其置信度，准备更新

## 能力边界

- **层级过多导致管理开销**：超过 4-5 级后的边际收益急剧下降
- **提升策略依赖经验调参**：不同场景的最佳阈值差异很大
- **跨层检索延迟**：需要检索所有层级时，累积延迟可能影响用户体验
- **遗忘不可逆**：一旦降级到丢弃，信息永久丢失（除非外部备份）

## 区别对比

| 维度 | 三级分层 | 两级架构 | 单级存储 |
|------|---------|---------|---------|
| 信息粒度 | 多级可调 | 两级切换 | 单一粒度 |
| 整合质量 | 最高（渐进式） | 中等 | 最低 |
| 实现复杂度 | 高 | 中 | 低 |
| 检索策略 | 多级联查 | 两级分开查 | 直接查 |
| 信息损失 | 最小（渐进降级） | 中等（直接切级） | 最大（窗口滑动丢弃） |

## 核心优势

- **渐进式信息蒸馏**：每个层级都在前一层的输出上进一步提炼，信息质量逐步提升
- **时间维度平滑**：不会因为单次会话结束就丢失全部上下文，过渡自然
- **资源效率优化**：高成本操作（深度整合）只在高价值信息上执行
- **灵活性**：不同信息可以处于不同整合阶段，系统可以并行处理

## 工程优化

1. **预分配层级容量**：根据历史数据预测每个层级的最佳容量比例，避免容量瓶颈
2. **写时合并 (Copy-on-Merge)**：L1→L2 的合并使用写时复制策略，原始 L1 数据在合并后可以安全释放
3. **批次提升**：将多个满足 L2→L3 条件的记忆打包为 batch 处理，降低 LLM 调用次数
4. **索引分离**：L3 的检索索引和存储分离，索引可以频繁重建，存储只在合并时写入

## 场景判断

**适合使用分层合并的情况**：
- 需要处理长时间、多会话的持续交互
- Agent 需要同时支持短期精确回忆和长期模式识别
- 系统需要在高响应速度和深度理解之间取得平衡

**不适合使用分层合并的情况**：
- 单次会话、一次性交互（不需要长期记忆）
- 存储资源极度受限（层级管理需要额外开销）
- 对实现复杂度敏感的小型系统
