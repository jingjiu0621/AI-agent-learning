# 9 rag-advanced — 高级 RAG 与知识检索

检索增强生成（Retrieval-Augmented Generation, RAG）是将外部知识库与 LLM 生成能力相结合的核心范式。当 LLM 的知识截止、幻觉和领域专精三大短板无法通过模型自身解决时，RAG 通过实时检索外部知识来补充 LLM 的上下文，成为生产级 LLM 应用的标配架构。

## 本节概述

如果说 Prompt Engineering 是"教 LLM 如何回答"，那么 RAG 就是"给 LLM 提供它不知道的知识让它回答"。RAG 不是简单的"搜索 + LLM"拼凑，而是一个从索引构建、检索策略、查询理解到生成融合的完整系统工程。本节深入覆盖从基础索引到高级 Graph RAG 和 Code RAG 的全链路技术。

## 核心思想

1. **索引是 RAG 的地基**：分块策略、元数据标注、索引结构决定了检索的上限。垃圾进，垃圾出——索引质量直接决定检索质量。

2. **检索是精度与召回率的博弈**：稠密检索（语义相似）擅长理解意图但可能遗漏精确关键词，稀疏检索（BM25）精确匹配但无法理解语义。混合检索是工程最优解。

3. **RAG 需要"智能"而非"机械"**：Agentic RAG、Self-RAG、Corrective RAG 等高级模式让系统自己决定"什么时候检索、检索什么、检索结果是否可信"。

4. **知识图谱让 RAG 理解关系**：Graph RAG 不仅是"检索文档"，而是理解实体之间的关系，回答需要多跳推理的复杂问题。

5. **代码是特殊的"知识"**：代码 RAG 需要理解 AST、调用图、依赖关系——纯文本分块会破坏代码的结构语义。

6. **查询理解决定检索的上限**：用户不会用"检索语言"提问。查询分类、改写、分解是让模糊的用户意图匹配精确的索引结构的关键。

7. **评估驱动优化**：RAG 系统有检索和生成两个质量维度，缺少任何一个维度的评估都会导致系统优化方向偏斜。

## 关键工程启示

- 分块粒度决定了检索的上限：太大则噪声多，太小则上下文碎片化
- 纯向量检索不是万能的：在精确匹配场景（ID、代码、数字）中，BM25 仍然优于任何 embedding
- 元数据过滤能显著提升精度：时间范围、文档类型、来源——过滤比排序更便宜、更精准
- RAG 的延迟链路过长：索引 → 检索 → 重排序 → 生成，每个环节都引入延迟，需要缓存和预取策略
- 评估要分两端看：检索端看 Recall/Precision，生成端看 Faithfulness/Relevance
- Agentic RAG 引入决策开销：让 LLM 判断"是否需要检索"需要额外的 token 和时间
- Graph RAG 的构建成本高：实体抽取和关系构建需要大量 LLM 调用，适合高价值静态知识库
- 代码 RAG 不能按文本分块：函数定义、类继承、调用关系会被纯文本切分破坏
