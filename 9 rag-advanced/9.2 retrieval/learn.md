# 9.2 retrieval — 检索策略

检索是 RAG 系统的核心引擎。如果说索引决定了"什么可以被检索到"，检索则决定了"如何从索引中找到最相关的信息"。检索策略直接决定了 RAG 的上限——再好的生成模型也弥补不了检索的失败。

## 核心设计思想

### 1. 没有最好的检索方式，只有最合适的

```
查询类型 → 最佳检索方式

"Transformer 的论文是哪一年发表的？"
    → BM25 即可（关键词明确）

"解释一下注意力机制的工作原理"
    → Dense 检索更优（语义匹配）

"修复 PostgreSQL WAL 日志爆满的问题"
    → Hybrid 最佳（既有精确术语又有语义理解）

"分析用户行为数据，给出留存率优化建议"
    → Multi-stage + Re-ranking（需要从大量数据中精排）
```

### 2. 检索的核心困境

```
    ┌─────────────────────────────────────────────────────┐
    │                   检索不可能三角                       │
    │                                                     │
    │                   高精度 (Precision)                  │
    │                      /        \                      │
    │                     /          \                     │
    │            低延迟 ───────────────── 高召回 (Recall)    │
    │                                                     │
    │  任意两者可兼得，三者不可同时最优                       │
    │  - 高精度 + 高召回 = 多级检索 (高延迟)                 │
    │  - 低延迟 + 高召回 = 粗检索 (低精度)                   │
    │  - 高精度 + 低延迟 = 窄检索 (低召回)                   │
    └─────────────────────────────────────────────────────┘
```

### 3. 多级检索是生产标配

单次检索无法兼顾精度和召回。生产 RAG 系统的标准模式是：

```
Stage 1: 粗筛 (BM25 + Vector)          延迟 5-10ms
    └→ 从百万级候选 → Top-100
Stage 2: 精细排序 (RRF 融合 + 元数据过滤)  延迟 10-20ms
    └→ Top-100 → Top-30
Stage 3: 重排序 (Cross-Encoder)         延迟 50-200ms
    └→ Top-30 → Top-5/10
Stage 4: LLM 生成                       延迟 500ms-2s
    └→ Top-5 → 最终回答
```

每一级都在减少候选数量、增加计算精度、累积计算成本。

## 关键工程启示

1. **BM25 永不退场**：在精确匹配场景（产品名、ID、代码、法律条款），BM25 持续优于任何 embedding
2. **Hybrid 是生产标配**：纯稠密检索在实际场景中会漏掉大量精确匹配，RRF 融合几乎零成本带来 5-15% 的 Recall 提升
3. **重排序是性价比最高的质量投资**：用 Cross-Encoder 重排 Top-100 比换更大的 embedding 模型效果好得多
4. **检索延迟必须管理**：每增加一个检索阶段，延迟线性增加。需要为 P99 延迟设置预算
5. **元数据过滤 > 更好的检索**：在检索前用元数据（时间、类型、来源）剪枝候选集，成本远低于检索后重排
6. **分块大小与检索对齐**：chunk_size 决定了检索的粒度。小 chunk 精确但碎片化，大 chunk 完整但噪声多
7. **检索引擎选择是架构决策**：托管向量库减少运维但增加成本和 vendor lock-in，自建 FAISS 灵活但需要工程投入
