# 9.5 code-rag — Code RAG：代码库理解

## 概述

Code RAG（Code Retrieval-Augmented Generation）是将 RAG 技术应用于**代码理解、代码检索和代码生成**的专门分支。它的核心命题是：**代码不是自然语言——纯文本分块会破坏代码的结构语义，必须理解 AST、调用图、依赖关系等代码特有结构才能实现有效的检索增强**。

传统 RAG 将文档切分为连续文本块（Chunks），用向量相似度检索。但代码具有**层次结构**（文件→类→方法→语句）、**引用关系**（导入、继承、调用）和**非局部依赖**（跨文件类型定义、接口实现），简单文本分块会导致：

- 函数定义被截断，中间不完整
- 类与方法的包含关系丢失
- 跨文件引用无法追溯
- 语义相似的代码片段可能功能完全不同

Code RAG 正是为解决这些问题而生——它专门针对代码的**结构特性**设计索引和检索策略。

## 核心挑战

1. **代码的离散性与连续性矛盾**：代码在语法上是离散的（token 序列），在语义上是连续的（一个函数依赖另一个文件中的类型）。纯文本检索无法捕获这种双重性。

2. **结构语义的损失**：将代码视为普通文本进行分块会破坏 AST 子树完整性，导致检索到的片段无法直接使用。

3. **跨文件依赖追踪**：现代代码库中，一个功能点可能分布在 5-10 个文件中（类型定义、工具函数、主逻辑、测试）。如何将这些相关片段聚合到 LLM 上下文中是核心难题。

4. **查询的多样性**：用户可能问"这个函数的作用是什么？"、"帮我修这个 bug"、"添加一个用户认证功能"——不同类型的查询需要不同的检索策略和上下文构成。

5. **仓库规模**：中大型仓库可能有数万文件，如何在毫秒级延迟下从海量代码中精准定位相关信息。

## 与普通 RAG 的关键区别

| 维度 | 普通 RAG | Code RAG |
|------|---------|----------|
| 检索单元 | 自然语言段落/文本块 | 函数/类/方法/代码块 |
| 索引维度 | 语义向量 | 语义 + AST 结构 + 调用关系 |
| 分块策略 | 固定大小/语义分割 | AST 节点 + 语法单元 |
| 查询类型 | 事实性/总结性 | 功能性/结构性/修复性 |
| 上下文组装 | 拼接 Top-K 块 | 按依赖关系补充完整上下文 |
| 关键工具 | Embedding + Vector DB | Tree-sitter + AST + Graph DB |
