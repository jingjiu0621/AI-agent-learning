# 9.7 evaluation — RAG 评估体系

## 概述

RAG 评估（RAG Evaluation）是衡量 RAG 系统**检索质量和生成质量**的系统化方法。它的核心命题是：**RAG 系统有两个质量维度——检索到的文档是否相关（检索质量），以及基于检索结果生成的回答是否准确（生成质量）——缺少任何一个维度的评估都会导致系统优化方向偏斜。**

RAG 评估是出了名的困难，因为它不仅要评估 LLM 生成（本身就是一个开放问题），还要评估检索系统。当回答出错时，是检索到的文档不够好，还是 LLM 没有正确利用检索结果？这是整个评估体系要解决的核心问题。

## 评估的三层架构

```
Layer 1: 检索层评估 (Retrieval Quality)
  ┌─────────────────────────────────────────────┐
  │ Precision / Recall / MRR / NDCG             │
  │ 检索到的文档是否相关？是否排在前列？          │
  └─────────────────────────────────────────────┘
                       │
Layer 2: 生成层评估 (Generation Quality)        │
  ┌─────────────────────────────────────────────┐
  │ Faithfulness / Relevance / Completeness     │
  │ 生成的回答是否忠于检索结果？是否回答了问题？│
  └─────────────────────────────────────────────┘
                       │
Layer 3: 端到端评估 (End-to-End Quality)
  ┌─────────────────────────────────────────────┐
  │ Overall Score / User Satisfaction           │
  │ 用户是否满意？任务是否完成？                 │
  └─────────────────────────────────────────────┘
```

## RAG 评估的独特挑战

| 挑战 | 说明 | 与其他系统的区别 |
|------|------|----------------|
| **双维度评估** | 需要同时评估检索和生成 | 传统 QA 只需要评估回答 |
| **归因困难** | 回答错了是检索的问题还是 LLM 的问题？ | 需要拆解错误原因 |
| **参考标准缺失** | 很多场景没有标准答案 | 需要 LLM-as-Judge |
| **评估成本高** | 人工评估 RAG 需要同时看检索结果和生成回答 | 比纯 LLM 评估更耗时 |
| **质量维度多** | 有用性、忠实度、相关性、完整性、安全性 | 多维度综合评分 |
