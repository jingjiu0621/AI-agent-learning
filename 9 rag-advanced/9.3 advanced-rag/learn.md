# 9.3 advanced-rag — 高级 RAG 模式

## 概述

基础 RAG（Naive RAG）遵循"检索-读取"的固定流水线：每次用户查询都执行检索，将检索结果拼入 Prompt，再由 LLM 生成回答。这种模式在简单事实性问答上表现尚可，但在复杂推理、多步问答、噪声容错等场景中暴露了根本性缺陷。

高级 RAG 模式（Advanced RAG）的核心思想是：**让检索过程变得智能、有选择性、可反思、可纠错**。不再是"每次都检索"，而是"在需要的时候检索最合适的信息，并对检索结果进行批判性评估后再生成"。

从 2024 年初 Self-RAG 论文发表，到 2025 年 Agentic RAG 成为主流架构，高级 RAG 领域经历了爆发式发展，形成了六大经典的进阶模式：

| 模式 | 核心创新 | 一句话概括 |
|------|----------|-----------|
| **Agentic RAG** | 智能体自主决策 | Agent 决定"何时检索、检索什么、是否继续检索" |
| **Self-RAG** | 自我反思评估 | 模型通过 Reflection Tokens 自我判断检索必要性和结果质量 |
| **Corrective RAG** | 检索结果纠错 | 对检索结果进行评估和修正，必要时重新检索 |
| **Adaptive RAG** | 策略自适应切换 | 根据查询复杂度动态选择不同的检索策略 |
| **Speculative RAG** | 投机性草稿生成 | 先基于部分信息生成草稿，再用检索结果验证修正 |
| **Fusion RAG** | 多源融合生成 | 从多个检索策略/来源综合信息，消除矛盾，互补不足 |

## 核心演进脉络

### 从 Naive RAG 到 Agentic RAG 的演进

```
时间线演进

Naive RAG (2023)
   ┌─────────┐    ┌──────────┐    ┌──────────┐
   │ Query   │───▶│ Retrieve │───▶│ Generate │   固定流水线，每轮必检索
   └─────────┘    └──────────┘    └──────────┘

         │
         ▼ 加入反思能力

Self-RAG / Corrective RAG (2024)
   ┌─────────┐    ┌──────────┐    ┌──────────┐
   │ Query   │───▶│ Retrieve │───▶│ Reflect  │───▶ Generate
   └─────────┘    └──────────┘    └──────────┘      ▲
                                    │ (不满意)        │
                                    └────────────────┘ 反馈循环
         │
         ▼ 加入决策能力和工具

Agentic RAG (2024-2025)
   ┌─────────────────────────────────────────────────┐
   │  Agent 主循环                                    │
   │  ┌────────┐  ┌──────────┐  ┌────────┐          │
   │  │ Think  │─▶│ Observe  │─▶│ Act    │──▶ ...   │
   │  └────────┘  └──────────┘  └────────┘          │
   │       │            ▲                            │
   │       ▼            │                            │
   │  ┌──────────┐      │                            │
   │  │Retrieve  │──────┘                            │
   │  │(工具调用) │                                   │
   │  └──────────┘                                   │
   └─────────────────────────────────────────────────┘
```

### 三种范式的对比

| 维度 | Naive RAG | 分阶段 RAG (Self/Corrective/Fusion) | Agentic RAG |
|------|-----------|-------------------------------------|-------------|
| **检索策略** | 每次都检索 | 有条件检索 + 自我评估 | Agent 完全自主决策 |
| **决策主体** | 开发者固化逻辑 | 模型内置评估能力 | 外部 Agent 循环 |
| **多轮交互** | 不支持 | 有限反馈循环 | 完整的工具循环 |
| **灵活性** | 低 | 中 | 高 |
| **可控性** | 高 | 中 | 低（依赖 Agent 决策） |
| **Token 开销** | 高（每次检索） | 中（选择性检索） | 高（推理开销大） |
| **实现复杂度** | 低 | 中-高 | 高 |

### 模式之间的互补关系

- **Agentic RAG** 是元模式/编排层，其他模式可以作为其子策略被调用
- **Self-RAG** 和 **Corrective RAG** 聚焦于"评估与反思"，但 Self-RAG 是模型内生能力，Corrective RAG 是外部评估器
- **Adaptive RAG** 解决的是"策略选择"问题，与 Self-RAG 的"结果可信度判断"正交
- **Fusion RAG** 解决"多源信息冲突"问题，可以与任何检索策略组合
- **Speculative RAG** 专注于"延迟优化"，在延迟敏感场景中与其他模式配合

## 模式选择决策矩阵

不同的业务场景对延迟、精度、成本有不同要求，以下矩阵帮助选择最优模式组合：

| 查询类型 | 推荐模式 | 原因 |
|----------|---------|------|
| 简单事实问答 | Naive RAG / Adaptive RAG | 不需要复杂决策，Adaptive 可快速路由到简单检索 |
| 复杂多步推理 | Agentic RAG + Self-RAG | Agent 多步检索 + 自我反思确保质量 |
| 知识密集且需高可信度 | Corrective RAG + Fusion RAG | 纠错机制 + 多源验证 |
| 实时/低延迟场景 | Speculative RAG | 投机生成减少等待，草稿修正替代全量检索 |
| 成本敏感场景 | Self-RAG | 选择性检索减少不必要的 API 调用 |
| 开放域长文生成 | Fusion RAG | 多源信息互补，提升生成质量 |
| 高噪声/不可信知识库 | Corrective RAG | 多轮评估和纠错机制过滤噪声 |
| 混合负载（各种查询混合） | Adaptive RAG + Agentic RAG | 根据查询动态切换策略 |

## 关键设计权衡

### 延迟 vs 精度

```
延迟                    精度
◄─────────── Trade-off ───────────►

Naive RAG          Agentic RAG + Self-RAG
  低延迟                高延迟
  低精度（噪声多）        高精度（多轮验证）
  
   │                      │
   │  Speculative RAG     │
   │  在两者之间搭建桥梁   │
   │  接受中等延迟换高精度  │
   ▼                      ▼

优化策略：
- 缓存常见查询的检索结果
- 使用 Speculative RAG 预生成草稿
- 流式输出生成结果，减少首 token 延迟感知
```

### 成本 vs 效果

| 模式 | 每 Query 额外 Token | 效果提升 | 性价比 |
|------|-------------------|----------|--------|
| Naive RAG | 基础检索成本 | 基准 | ★★★ |
| Self-RAG | +10-30%（评估 Token） | 中-高 | ★★★★ |
| Corrective RAG | +30-60%（多轮检索） | 高 | ★★★ |
| Agentic RAG | +50-200%（推理+多轮工具调用） | 非常高 | ★★ |
| Fusion RAG | +40-80%（多路检索+融合） | 高 | ★★★ |
| Speculative RAG | +20-40%（草稿+验证） | 中 | ★★★★ |

### 复杂度 vs 可维护性

```
低                                  高
◄──────────────────────────────────────►

Naive RAG  Adaptive RAG  Fusion RAG  Agentic RAG
 3个组件      5-6个组件     6-8个组件    10-15个组件
 简单管道     条件分支      并行路由      Agent编排
 易调试       中等调试      中等调试      调试困难
 低维护       中等维护      中等维护       高维护
```

**工程建议**：从最简单的 Naive RAG 开始，通过评估识别瓶颈，然后有针对性地引入高级模式。不要为了"高级"而高级——每个模式都是一种 trade-off，而非纯收益。

## 前置知识

在深入学习本节之前，确保理解以下基础：

### 9.1 Indexing（索引策略）

- 分块策略（Chunking）决定了检索的最小单元粒度
- 元数据标注（Metadata Annotation）提供了过滤和上下文信息
- 多级索引（Multi-level Index）结构影响检索效率
- 高级 RAG 模式依赖高质量的结构化索引来做智能决策

### 9.2 Retrieval（检索策略）

- 稀疏检索（BM25）与稠密检索（Vector Search）的原理
- 混合检索（Hybrid Search）和重排序（Re-ranking）的工程实践
- 检索指标（Recall, Precision, MRR）的理解
- 高级 RAG 的"检索评估"环节直接建立在这些基础检索指标之上
