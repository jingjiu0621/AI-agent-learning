# 9.6 query-understanding — 查询理解与改写

## 概述

查询理解（Query Understanding）是 RAG 流水线中**最容易被低估但影响最大的环节**。它的核心命题是：**用户不会用"检索语言"提问——原始查询往往是模糊的、碎片化的、偏离索引分布的。在检索之前理解、改写、优化查询，是提升 RAG 效果性价比最高的手段。**

实践经验表明：同样的索引和检索系统，经过良好优化的查询理解层可以让 Recall@5 提升 20-40%，甚至比更换 Embedding 模型带来的收益更大。

## 为什么查询理解如此重要

```
原始用户查询                     改写后的检索查询
───────────────                 ───────────────────
"阿里财报"          →            "阿里巴巴集团 2024年Q4 财务报告"
"怎么修这个bug"     →            "List comprehension 中变量作用域问题的修复方法"
"苹果公司"          →            "Apple Inc. 公司介绍 业务范围" (vs "苹果水果")
"推荐系统怎么做"    →            "协同过滤 + 矩阵分解 推荐算法实现"
```

**不经过理解的查询直接检索，相当于让搜索引擎猜你的真实意图。**

## 核心挑战

1. **歧义消解**：同一查询在不同上下文中含义完全不同（"苹果" → 水果 / 公司）
2. **查询过短**：用户倾向于输入短查询，但短查询的信息量不足以支撑精确检索
3. **领域术语不匹配**：用户使用的术语和索引中的术语不一致（"支付" vs "结算" vs "清算"）
4. **隐式假设**：查询中包含用户未说出的前提假设（"优化一下" → 优化什么？什么标准？）
5. **查询与索引分布差异**：用户自然语言表达和文档的语言风格差异
