# 9.4 graph-rag — Graph RAG：知识图谱增强检索 总体概述

## 概述

Graph RAG（Graph-enhanced Retrieval-Augmented Generation）是一种将**知识图谱（Knowledge Graph）** 的结构化关系信息引入 RAG 流水线的技术范式。它的核心命题是：**当问题需要跨实体、多跳推理才能回答时，基于文档块（Chunk）的向量检索注定失效——必须理解实体之间的关系**。

传统向量 RAG 的检索单元是"文档片段"（Document Chunks），即使语义相似度匹配出色，它也无法回答这类问题：

- *"阿尔伯特·爱因斯坦提出的理论中，哪些后来被 LIGO 实验验证？"*
- *"辉瑞和 BioNTech 合作开发的疫苗，使用了哪种 mRNA 技术平台？"*
- *"图灵奖得主中，哪些同时也是诺贝尔奖得主？"*

这些问题需要的不是"找一段相似的文字"，而是**沿着实体关系链进行多跳推理**。这正是 Graph RAG 要解决的核心矛盾。

### 为什么需要 Graph RAG？

| 传统 RAG 的盲区 | Graph RAG 的解法 |
|----------------|-----------------|
| 文档块之间是孤立的，无法跨文档推理 | 实体链接跨越文档边界，形成知识网络 |
| 无法理解"A 发明了 B，B 被 C 改进"的链式关系 | 关系路径（Path）天然支持多跳推理 |
| 对"实体属性"查询（如"某人的出生日期"）依赖文本命中 | KG 直接将属性结构化存储，精确检索 |
| 无法区分同名实体（如"苹果"是水果还是公司） | 实体消歧（Entity Disambiguation）明确上下文 |
| 缺乏全局知识结构概览，无法做摘要性回答 | 社区检测（Community Detection）支持全局查询 |

## 核心思想

### 思想一：实体为中心的检索范式

传统 RAG 以"文档块"为检索的基本单元，文档块之间没有显式的关系。Graph RAG 将基本单元从文档块转变为**实体—关系—实体**的三元组结构：

```
文档中心检索                             实体中心检索
─────────────────                     ─────────────────

检索单元: 文本 Chunk                   检索单元: 三元组 (h,r,t)
存储方式: 向量嵌入                      存储方式: 图数据库 / 图结构
相似度: 余弦相似度                      相似度: 图结构匹配 + 语义相似
推理能力: 无法跨文档关联                 推理能力: 沿关系路径多跳推理
召回方式: Top-K 相似                    召回方式: 实体锚定 + 邻域扩展
上下文: Chunk 文本片段                   上下文: 子图（Subgraph）
```

### 思想二：结构化知识增强 LLM 生成

LLM 对精确事实、结构化关系和最新知识的掌握并不完美。知识图谱提供了：

1. **精确性**：关系是明确的、无歧义的（"爱因斯坦_出生地_乌尔姆"而不是"爱因斯坦出生于..."的文本近似）
2. **可追溯性**：每条事实都能追溯到来源文档，支持验证和反驳
3. **结构完整性**：实体的所有属性、关系被显式建模，不会"遗漏关键细节"
4. **可控性**：知识范围（Schema）明确，不会像向量检索那样引入无关噪声

### 思想三：局部检索 + 全局概览 双模式

Microsoft GraphRAG 论文（2024）的一个重要洞见是：**向量 RAG 擅长局部细节检索但不擅长全局主题总结，而 Graph RAG 能同时支持两种模式**。

```
局部检索（Local Query）                   全局检索（Global Query）
─────────────────────                    ─────────────────────

"爱因斯坦获得诺贝尔奖是因为什么？"           "20 世纪物理学有哪些重大突破？"

         │                                          │
         ▼                                          ▼
  定位实体: "爱因斯坦"                         社区检测: 物理学家社群
         │                                          │
         ▼                                          ▼
  展开关系: 获奖 → 光电效应                   社区摘要: 相对论、量子力学...
         │                                          │
         ▼                                          ▼
  子图作为上下文                             多社区摘要融合
  ─────────────────                          ─────────────────
  回答需关注单实体可达范围                     回答需覆盖全局主题分布
```

## 两种主要技术路线

### 路线一：显式知识图谱构建 + 检索（Explicit KG）

这是目前最主流的路线，核心步骤是：

1. **实体抽取**：从文档中识别命名实体（Named Entity Recognition）
2. **关系抽取**：识别实体之间的语义关系（Relation Extraction）
3. **图谱构建**：将实体和关系存入图数据库（Neo4j, ArangoDB, NetworkX）
4. **查询检索**：将用户问题映射为图查询（实体链接 + 子图匹配）
5. **上下文注入**：将子图序列化为文本作为 LLM 的上下文

```
文档 ──→ NER/RE ──→ 三元组 ──→ 图数据库 ──→ 子图检索 ──→ LLM 生成
         (离线构建)           (结构化存储)     (在线查询)
```

**优点**：知识精确可解释，支持复杂的多跳推理，知识可更新和管理。
**缺点**：构建成本高（LLM 调用量大），Schema 设计复杂，更新延迟。

### 路线二：隐式图谱感知检索（Graph-aware Retrieval）

这一路线不显式构建 KG，而是通过改进索引和检索策略来隐式利用"图结构信息"：

- **GraphToken**（2024）：在向量索引中加入实体边界的 Token 级标记，使检索能感知实体边界
- **KGP (Knowledge Graph Prompting)**：在检索阶段通过"伪三元组"结构增强查询
- **Graph-embedded Index**：将文档间的引用/共现关系编码为索引中的接近度信号

```
文档 ──→ 实体标记嵌入 ──→ 图感知索引 ──→ 结构感知检索 ──→ LLM 生成
         (检索时感知关系)      (索引含结构信号)
```

**优点**：无需完整 KG 构建流程，成本较低，适合快速验证。
**缺点**：关系表达能力受限，多跳推理不如显式 KG 精确。

### 路线对比

| 维度 | 显式 KG 路线 | 隐式图谱感知路线 |
|------|-------------|----------------|
| **知识精度** | 高（明确三元组） | 中（隐式信号） |
| **构建成本** | 高（全量 LLM 抽取） | 低（索引改进） |
| **多跳推理** | 强（图遍历） | 弱（信号衰减） |
| **可解释性** | 高（可追溯三元组） | 低（隐式编码） |
| **更新延迟** | 高（需要重新抽取） | 低（增量索引） |
| **成熟度** | 高（Neo4j, GraphRAG） | 低（学术探索阶段） |
| **适合场景** | 高价值静态知识库 | 动态快速变化的文档集 |

## Graph RAG vs Vector RAG 对比

| 对比维度 | Vector RAG | Graph RAG |
|---------|-----------|-----------|
| **检索单元** | 文本 Chunk（不定长文本块） | 三元组 / 子图（结构化知识单元） |
| **组织形式** | 向量空间中的稠密嵌入 | 图结构中的节点和边 |
| **匹配方式** | 语义相似度（余弦距离） | 实体匹配 + 结构匹配 + 路径推理 |
| **多跳能力** | ❌ 不支持（Chunk 间无显式关联） | ✅ 天然支持（沿着边遍历） |
| **精确事实查询** | ❌ 语义近似可能丢失精确匹配 | ✅ 三元组精确匹配 |
| **模糊语义查询** | ✅ 擅长语义相似匹配 | ❌ 需要先链接到实体 |
| **全局摘要** | ❌ 难以综合多文档 | ✅ 社区摘要支撑全局视图 |
| **构建成本** | 低（Chunking + Embedding） | 高（NER + RE + 图谱构建 + 社区检测） |
| **更新成本** | 低（重计算 Chunk 向量） | 高（需要重新抽取实体关系） |
| **查询延迟** | 低（向量索引近似搜索） | 中-高（实体链接 + 子图遍历） |
| **可解释性** | 中（返回文本片段） | 高（三元组路径可追溯） |
| **Schema 约束** | 无（自由文本） | 有（需定义 Ontology） |
| **领域适配** | 无需 Schema，直接嵌入 | 需要领域专家定义实体和关系类型 |
| **存储开销** | 向量维度 × Chunk 数量 | 节点数 + 边数 + 属性 |
| **扩展性** | 水平扩展简单（分片索引） | 图分片复杂（跨分区遍历代价高） |
| **代表性系统** | Pinecone, Milvus, Qdrant | Neo4j, Microsoft GraphRAG, Amazon Neptune |

```
  │
  ├── 查询涉及多步推理（"A 的 B 被 C 如何？"）？
  │     └── 是 → Graph RAG（需要关系理解）
  │
  ├── 知识库高度结构化（产品目录、知识图谱）？
  │     └── 是 → Graph RAG（精确匹配）
  │
  ├── 需要全局主题概括？
  │     └── 是 → Graph RAG（社区摘要）
  │
  └── 成本和构建复杂度敏感？
        └── 是 → Vector RAG + 混合检索（更实用）

并非二选一。生产系统通常是 Vector RAG + Graph RAG 的混合体：
  - Vector RAG 处理语义相似检索
  - Graph RAG 处理关系推理和精确匹配
  - 结果通过 Fusion 策略合并
```

## 工程启示

### 要做的

1. **Schema 先行**：在构建 KG 前，与领域专家一起定义清晰的 Ontology（哪些实体类型、哪些关系类型）。Schema 决定了 KG 的表达上限。
2. **质量 > 数量**：100 条高质量三元组好过 10000 条噪声三元组。在实体抽取阶段加入置信度过滤，用规则校验三元组的合法性。
3. **增量构建**：全量重建 KG 成本极高。设计增量更新策略——仅处理新增/变更的文档，用实体链接匹配已有实体。
4. **混合检索是标配**：纯 KG 检索无法处理模糊查询。Vector RAG 提供语义召回，Graph RAG 提供精确推理——两者互补而非替代。
5. **子图大小控制**：将整个 KG 灌入 LLM 上下文是不现实的。控制子图展开的深度（通常 1-3 跳）和宽度（Top-K 邻居），平衡上下文完整性和 Token 预算。
6. **缓存热点子图**：常见查询对应的子图可以预计算并缓存，避免每次查询都做图遍历。

### 不要做的

1. **不要对非结构化对话历史建 KG**：KG 适合稳定的、结构化的知识库。对话历史查询更适合向量检索。
2. **不要在 OLTP 场景用图遍历**：高频低延迟查询场景（如商品推荐），向量检索 + KV 缓存的性能远优于图查询。
3. **不要忽略实体消歧**："Python 是蛇还是编程语言？""Java 是岛还是编程语言？"——同名实体必须消歧，否则 KG 变成噪声。
4. **不要用 KG 做纯语义相似搜索**：KG 不擅长"找相似"。找"与产品 X 相似的产品"用向量嵌入，找"X 的供应商的竞争对手"用 KG。
5. **不要过度设计 Schema**：一开始用简单的二元关系（"实体-关系-实体"），随着需求逐步增加复合关系和属性。

## 局限性

1. **构建瓶颈**：知识图谱的构建是 Graph RAG 最明显的瓶颈。高质量 NER/RE 依赖强大的 LLM（GPT-4 级别），小型模型抽取质量不稳定。
2. **领域迁移成本高**：从一个领域切换到另一个领域，需要重新设计 Schema 和训练/调优 NER/RE 模型——这比替换 embedding 模型成本高得多。
3. **时效性挑战**：知识库中的事实会过时。KG 的更新需要重新抽取实体关系，不能像向量索引那样简单地重新 embedding。
4. **查询映射误差**：将用户自然语言问题映射到 KG 查询（实体链接 + 子图匹配）本身就会引入错误——链接错误的实体导致后续所有推理都建立在错误基础上。
5. **Token 使用效率**：子图的序列化（将图结构转为文本）通常不够紧凑。一个 10 节点 15 边的子图可能需要数百 token 才能完整描述。
6. **评估困难**：KG 的质量（完整性、准确性、时效性）和图查询的效果（召回率、精度）缺乏统一的评估标准，使得系统优化目标模糊。
7. **小规模数据不适用**：当知识库只有数百个文档时，构建 KG 的投入产出比很低——向量 RAG 可能就足够了。
