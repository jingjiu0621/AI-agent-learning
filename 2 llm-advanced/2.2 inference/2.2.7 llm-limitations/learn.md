# LLM 固有局限：幻觉、知识截止、推理谬误与缓解

## 简单介绍

LLM 虽然强大，但有其固有局限。理解这些局限对于构建可靠的 Agent 系统至关重要——很多 Agent 失败不是因为架构设计有问题，而是因为 LLM 的内在限制。核心局限包括：**幻觉**（编造事实）、**知识截止**（不知道最新信息）、**推理谬误**（逻辑错误或计算不准）。

## 三大局限

### 1. 幻觉（Hallucination）

LLM 生成的内容与事实不符——但看起来却很有信心。

| 类型 | 示例 | 原因 |
|------|------|------|
| 事实性幻觉 | "《红楼梦》的作者是巴金" | 知识混淆或编造 |
| 存在性幻觉 | "我见过你上次发的邮件" | 在训练数据中没有相关信息 |
| 一致性幻觉 | "今天是星期三，后天是星期六" | 逻辑推理错误 |

**根本原因**：LLM 是一个"下一个 Token 预测器"，它优化的目标是"生成看起来合理的 Token"，而不是"生成符合事实的 Token"。

### 2. 知识截止

```
模型: GPT-4 (2023-10 知识截止)
用户: "2024 年巴黎奥运会开幕式怎么样？"
模型: "我无法回答这个问题，因为我针对 2023 年 10 月之前的训练数据..."
```

知识截止的 Agent 系统影响：
- Agent 不知道最新的 API 版本变化
- Agent 不知道最近发生的事件
- Agent 可能会自信地使用过时的知识

### 3. 推理谬误

| 谬误类型 | 示例 | 解决方案 |
|---------|------|----------|
| 计算错误 | "285 × 437 = ?" 可能算错 | 用计算器工具 |
| 逻辑跳跃 | A→B, B→C, 错误推导 A→D | CoT、显式推理 |
| 确认偏误 | 只看到支持自己论点的证据 | 多个视角的 prompt |
| 遗忘 | 长对话中忘记早期提到的信息 | 外部记忆存储 |

## 缓解策略

### 针对幻觉

| 策略 | 方法 | 效果 |
|------|------|------|
| RAG（检索增强生成） | 用外部知识库提供事实支持 | 显著降低事实性幻觉 |
| 自我反思 | 让 LLM 检查和纠正自己的输出 | 部分有效 |
| 约束解码 | 限制输出只能来自已知事实 | 强依赖知识库 |
| 多模型交叉验证 | 多个模型对同一问题的回答做一致性检查 | 消耗大、效果好 |

### 针对知识截止

```python
# RAG 是解决知识截止的标准方案
def agent_with_rag(query: str):
    # 1. 检索最新信息
    docs = search_database(query)  # 搜索最新文档/网页
    
    # 2. 将检索结果注入 Prompt 上下文
    response = llm.chat(
        messages=[
            {"role": "system", "content": f"以下是相关信息，请基于它们回答：\n{docs}"},
            {"role": "user", "content": query}
        ]
    )
    return response
```

### 针对推理谬误

```python
# Chain-of-Thought + 外部工具验证
def robust_reasoning(problem: str):
    # 1. 让 LLM 展示推理步骤
    reasoning = llm.chat(f"逐步推理：{problem}")
    
    # 2. 用工具验证关键步骤
    if "计算" in problem:
        # 提取计算部分，用 Python 验证
        calc_result = python_executor.extract_and_compute(reasoning)
        if calc_result:
            reasoning += f"\n[验证] 计算结果确认：{calc_result}"
    
    return reasoning
```

## LLM 局限对 Agent 设计的影响

| Agent 组件 | LLM 局限影响 | 设计考虑 |
|-----------|-------------|---------|
| 工具选择 | 幻觉可能导致选错工具 | 后验校验 + 强制类型约束 |
| 参数生成 | 参数值可能不合法 | JSON Schema 校验 |
| 推理决策 | 推理谬误导致错误行动 | CoT + 外部验证 |
| 信息记忆 | 遗忘导致上下文丢失 | 外部队列存储 |
| 计划执行 | 计划可能不合理 | 验证 + 可回退 |

## 工程启示

- **永远不要信任 LLM 的输出**——在 Agent 系统中，所有 LLM 的输出都应有后验校验
- **用工具弥补 LLM 的弱点**——数学用计算器、事实查重用 RAG、实时信息用搜索
- **区分"LLM 的错误"和"系统的错误"**——很多 Agent 归因于"LLM 不够聪明"的问题，其实是系统设计没有做好容错
- **RAG 是幻觉缓解的第一道防线**——如果 Agent 有可靠的知识来源，幻觉率可以大幅降低
- **LLM 的知识截止是 Agent 需要工具的原因之一**——搜索工具、数据库查询工具本质上都是在弥补这个局限
