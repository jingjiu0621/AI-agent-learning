# 窗口扩展技术（YaRN, NTK-aware, 线性缩放）

## 简单介绍

上下文窗口扩展技术让在较短序列（如 4K）上训练的模型，能够在更长的序列（如 32K）上推理。这些技术的核心是**修改位置编码的参数**，让模型"认为"当前位置在训练时的可见范围内，从而避免降级。

## 为什么需要窗口扩展

LLM 训练时使用的序列长度是固定的（例如 4K Tokens），这意味着：
- 模型从来没有见过位置 > 4096 的编码
- RoPE/位置编码的参数只针对 0-4096 范围优化
- 直接外推 → 注意力质量下降 → 输出质量下降

## 三种主流扩展方法

### 1. 线性位置插值（Position Interpolation, PI）

最简单的方法：将超出训练范围的位置"压缩"回训练范围。

```python
def linear_interpolation(pos, original_max=4096, target_max=32768):
    """将位置线性映射到训练范围"""
    scale = original_max / target_max
    return int(pos * scale)

# 原来位置 8192 → 现在映射到 4096 范围内的 1024
# 所以模型"看到"的是训练时见过的位置
```

**优点**：最简单，效果稳定
**缺点**：所有位置都被压缩了，可能导致相邻 Token 的区分度下降

### 2. NTK-aware 插值（Neural Tangent Kernel）

核心洞察：**高频信息（相邻 Token 的区分）需要保留，低频信息（远距离关系）可以压缩**。

```
线性插值: 对所有频率使用相同的缩放因子 ❌
NTK-aware: 高频少压缩（保持局部区分），低频多压缩 ❗
```

NTK-aware 通过调整 RoPE 的 base 频率（从 10000 提高到更大值）来实现：

```python
def ntk_scaled_rope(base=10000, scale=8, d_model=4096):
    """NTK-aware 的 base 缩放"""
    # 将 base 按比例放大，使高频部分几乎不变
    new_base = base * scale ** (d_model / (d_model - 2))
    return new_base  # 例如从 10000 → 500000+
```

**优点**：对短距离关系保持好，长距离也有改善
**缺点**：参数选择需要经验

### 3. YaRN（Yet another RoPE extensioN method）

YaRN 是目前最先进的方法，结合了 PI 和 NTK 的优势，并加入了**注意力温度调节**：

```python

    """YaRN 的核心：对不同频率维度使用不同的缩放"""
    # 1. NTK-style：不同频率不同缩放
    # 2. PI-style：总体位置压缩
    # 3. 注意力温度：调 softmax 的温度防止注意力过度平滑
    # ...
```

YaRN 的关键改进：
1. **频率感知的缩放**：高频（靠近的 Token）用小缩放，低频（远离的 Token）用大缩放
2. **注意力温度**：长序列下注意力分布趋平，通过温度系数重新"锐化"

## 方法对比

| 方法 | 提出来源 | 复杂度 | 外推效果 | 推荐场景 |
|------|---------|--------|---------|---------|
| Linear PI | Meta (2023) | 低 | 4x 扩展可用 | 小幅度扩展 |
| NTK-aware | 开源社区 | 中 | 8x+ | 中等扩展 |
| YaRN | Peng et al. (2023) | 高 | 32x+ | 大规模扩展 |
| Dynamic NTK | 开源社区 | 中 | 动态适配 | 灵活场景 |

## 能力边界

| 因素 | 影响 |
|------|------|
| 训练长度 | 训练时见过的最大位置数是基础——从 0 扩展到 128K 几乎不可能 |
| 模型大小 | 更大的模型外推能力更强 |
| 数据类型 | 长序列的数值稳定性要求更高（位置值很大） |

即使最好的窗扩展，**从 4K 扩展到 128K 仍然是一个巨大的挑战**。

## 工程启示

- 当前主流模型（GPT-4-128K, Claude-200K）已经原生支持长上下文——不需要自己实现扩展
- 如果使用开源模型（Llama 2 4K → 32K），YaRN 是目前的最佳选择
- 窗口扩展是非微调方案——不需要 GPU 训练，可以在推理时直接使用
- 扩展后的模型在长序列上的**质量**通常不如原生在长序列上训练的模型
- Neculiar fact：扩展后的模型在某些短序列任务上的表现可能会**稍微下降**

## 与 Agent 系统的关系

- Agent 系统受益于窗口扩展——可以在一个上下文中处理更长的历史，减少对 RAG 的依赖
- 扩展后的窗口仍然不如原生长上下文模型的表现——关键任务建议使用原生支持长上下文的模型
- 扩展 + 压缩（如 LLMLingua）可以组合使用，进一步延长有效上下文
