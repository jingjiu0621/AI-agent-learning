# 长上下文评估（Needle In A Haystack）

## 简单介绍

长上下文评估用来衡量 LLM 在长序列中检索和处理信息的能力。最著名的评估方法是 **Needle In A Haystack（NIAH，大海捞针）**——在长文档中插入一个关键信息片段（"针"），测试模型能否在回答问题时正确提取它。

## Needle In A Haystack 方法论

### 基本流程

```
1. 准备一个长文本"草堆"（如 Paul Graham 的论文集）
2. 在草堆的某个位置插入一个"针"（如 "我最喜欢的编程语言是 Rust"）
3. 让 LLM 回答关于"针"的问题（如"最喜欢的编程语言是什么？"）
4. 测试不同的：
   - 草堆长度（4K, 8K, 16K, 32K, 64K...）
   - 针的位置（草堆的 5%, 25%, 50%, 75% 处）
   - 针的复杂度
5. 评估模型是否正确回答

┌────────────────────────────────────────────┐
│  草堆文本（很长）                            │
│  针: "我最喜欢的编程语言是 Rust" ← 插入     │
│  ... 更多草堆文本 ...                       │
└────────────────────────────────────────────┘

询问: "用户最推荐的编程语言是什么？"
期望: "Rust"
```

### 结果矩阵

```
             针的位置（由浅到深）
长度  ┌──────────────────────────────────┐
4K    │ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ │ 100%
8K    │ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ │ 100%
16K   │ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ ✅ │ 100%
32K   │ ✅ ✅ ✅ ❌ ✅ ✅ ❌ ✅ ✅ ✅ │ 80%
64K   │ ❌ ❌ ✅ ❌ ❌ ❌ ❌ ❌ ✅ ❌ │ 20%
      └──────────────────────────────────┘
```

## 发现的规律

### 1. "U 形"性能曲线

模型在文档开头和结尾的检索效果最好，在**中间**部分最差：

```
检索准确率
  100% ┤    ●●●●●       ●●●●●
  75%  ┤   ●     ●     ●
  50%  ┤  ●       ●   ●
  25%  ┤ ●         ● ●
  0%   ┤●           ●
       └──────────────────────────
         开头     中间     结尾
                针的位置
```

原因：模型对文档开头（Primary Effect，首因效应）和结尾（Recency Effect，近因效应）记忆更好。

### 2. 窗口长度依赖

当文档长度接近模型的训练长度时，性能急剧下降。

### 3. 模型间差异

| 模型 | 宣称窗口 | NIAH 有效窗口 | 可靠性 |
|------|---------|--------------|--------|
| GPT-4 | 128K | ~64K | 高 |
| GPT-4o | 128K | ~128K | 很高 |
| Claude 3 | 200K | ~100K+ | 高 |
| Claude 3.5 | 200K | ~200K | 很高 |
| Llama 2 | 4K | ~4K | 中 |
| Gemini 1.5 | 1M | ~500K+ | 实验性 |

## 多针变体（Multi-Needle）

更难的变体：在文档中插入**多个**针，测试模型是否能检索所有相关信息并综合它们：

```
草堆中的多个针:
  针 1: "Alice 最喜欢的编程语言是 Rust"
  针 2: "Bob 最喜欢的编程语言是 Python"
  针 3: "Charlie 最喜欢的编程语言是 Haskell"

问题: "Alice 和 Charlie 最喜欢什么语言？"
期望: "Alice 喜欢 Rust，Charlie 喜欢 Haskell"
```

## 评估的局限

1. **"针"的类型有限**：简单的"插入一个事实"不能覆盖所有长上下文场景
2. **草堆质量**：无意义的重复文本（如"The number is..."）与真实文档差异大
3. **检索 ≠ 推理**：找到信息与使用信息推理是不同的事
4. **位置偏差无法完全消除**：针的位置始终影响结果

## 工程启示

- 在 Agent 系统中，**重要的信息应该放在 Prompt 的开头或结尾**（避免中间）
- 不要相信模型的"宣称窗口大小"——要用实际任务的 NIAH 测试来确认有效窗口
- 如果文档长度超过模型的有效窗口，使用 RAG（检索增强生成）而非直接注入
- 在 Agent 的长对话中，关键信息应通过外部存储（Memory/State）而非依赖上下文窗口来保留
- NIAH 测试可以作为 Agent 系统选择模型的"准入测试"之一
