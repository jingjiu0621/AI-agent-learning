# 多语言 Tokenizer 的挑战与方案

## 简单介绍

多语言 Tokenizer 需要同时支持多种语言的编码。不同语言的字符集、语法结构、词汇量差异巨大——英语用 26 个字母就能表达一切，中文需要数千常用汉字，日文则混合了三种文字系统。如何在一个 Tokenizer 中公平地处理所有语言，是 LLM Tokenizer 设计中的核心难题。

## 核心矛盾：语言不公平性

### "英语偏见"问题

由于训练语料中英语占据主导（通常超过 80%），BPE Tokenizer 天然偏向英语：

```
同一概念在不同语言下的 Token 消耗：
  "hello" → 1 Token（英语）
  "你好" → 2-3 Tokens（中文）
  "こんにちは" → 4-5 Tokens（日文）
  "Привет" → 3-4 Tokens（俄文）
```

这意味着同样的语义内容，非英语用户需要支付更高的 Token 成本。

### 问题的根源

| 语言 | 字符数 | 特点 | Tokenizer 效果 |
|------|--------|------|---------------|
| 英语 | 26 字母 | 字母少、拼写规律 | 好 |
| 中文 | 5000+ 常用字 | 每个字独立语义 | 中（字频率不均） |
| 日文 | 汉字 + 假名 | 混合文字系统 | 差 |
| 阿拉伯文 | 28 字母 + 变体 | 右向左 + 连写 | 中 |
| 印地文 | 47 字母 + 连写 | 复杂连写 | 差 |
| 韩文 | 24 字母 + 组合 | 字母组合成方块字 | 中 |

## 优化方案

### 方案 1：Byte-level BPE（GPT-2）

以字节为初始 Token 而不是 Unicode 字符。优点是可以表示所有 Unicode 字符，缺点是序列更长（一个中文字符可能变成 3 个 Token）。

### 方案 2：扩展词表

给每种语言分配足够的词表空间：

```
单语言词表: 50,000 tokens（其中中文只有 2,000-5,000）
多语言词表: 100,000-250,000 tokens（每种语言 10,000+）
```

**缺点**：词表大了，模型的 embedding 层参数也大了。

### 方案 3：SentencePiece（Unigram）

不依赖空格分词，直接处理原始文本。Unigram 的概率模型能更公平地分配词表：

```
例子: "Hello你好世界"
  BPE Tokenizer: ["Hello", "你", "好", "世", "界"] → 5 Tokens
  Unigram (SentencePiece): ["Hello", "你好", "世界"] → 3 Tokens（依赖训练配置）
```

### 方案 4：语言特定 Tokenizer

不同语言使用不同的 Tokenizer 分支，模型内部做融合——但这种方法工程复杂，很少在通用 LLM 中使用。

## 不同模型的多语言表现

| 模型 | Tokenizer | 词表大小 | 多语言表现 |
|------|-----------|---------|-----------|
| GPT-4 | cl100k_base | ~100k | 主要语言好、小众语言差 |
| Llama 2 | SentencePiece BPE | 32k | 英语好、其他一般 |
| Llama 3 | tiktoken-based | 128k | 多语言显著改善 |
| Claude 3 | 自定义 | 未公开 | 多语言表现优秀 |
| Gemma | SentencePiece | 256k | 对多语言预留了充足词表 |

## 工程启示

- 如果目标用户是中文/日文/韩文用户，选择多语言 Tokenizer 表现好的模型（GPT-4o, Claude 3.5, Llama 3）
- 对于非英语场景，实际 Token 消耗通常比估算高 30-50%——预算要留足余量
- 中文字符在 BPE Tokenizer 中通常每字 1 Token，但生僻字或专有名词可能变成 2-3 Tokens
- 多语言系统设计中，Token 计数必须以实际模型为准，不要依赖通用估算公式
- Agent 开发中处理多语言输入时，注意 System Prompt 的语言也消耗 Token——中英文混合的 Prompt Token 消耗比纯英文高
