# BPE 算法详解：从字节到子词的合并策略

## 简单介绍

BPE（Byte Pair Encoding，字节对编码）是目前最主流的 Tokenizer 算法，被 GPT 系列、Claude、Llama 等几乎所有主流 LLM 采用。它的核心思想是**从单个字符开始，逐步合并最常共现的字符对**，最终生成一个子词词表。

## 基本原理

BPE 的训练过程是一个自底向上的合并过程：

```
训练语料: "low lower lowest"

第 0 步: 按字符拆分（每个字符 + 空格作为初始 Token）
  l o w _ l o w e r _ l o w e s t

第 1 步: 统计相邻字符对的频率
  "lo": 3次, "ow": 3次, "w_": 2次, ...
  合并最频繁的 "lo" → 词表新增 "lo"

  词表现在: [l, o, w, _, e, r, s, t, lo]

第 2 步: 再次统计（用合并后的序列）
  "low": 3次, ...
  合并 "low"

  词表现在: [l, o, w, _, e, r, s, t, lo, low]

...继续合并直到达到目标词表大小
```

### 编码过程（推理时）

```
文本: "lowerest"

1. 按初始字符切分: l, o, w, e, r, e, s, t
2. 从左到右贪心匹配: 
   - "l"+"o"+"w" = "low" ✅ 在词表中 → 合并
   - "e"+"r" = "er" ✅ 在词表中 → 合并
   - "e"+"s"+"t" = "est" ✅ 在词表中 → 合并
3. 结果: [low, er, est] → 3 个 Token
```

## 历史背景

BPE 最初是 1994 年提出的一种数据压缩算法。2016 年，Neural Machine Translation 领域首次将 BPE 用于 NLP 的 Subword Tokenization。2018 年 GPT 采用 BPE，之后成为 LLM 的标准方案。

以前的做法是：
- **Word-based Tokenizer**：词表 50 万+，遇到生僻词就是 OOV（Out of Vocabulary）
- **Character-based Tokenizer**：序列太长（每个字符一个 Token），丢失了单词级语义

BPE 解决了"词表大小 vs 序列长度"的矛盾。

## 核心矛盾

| 维度 | 大词表 | 小词表 |
|------|--------|--------|
| 序列长度 | 短 | 长 |
| 词表大小 | 大（更多参数） | 小 |
| OOV 问题 | 少 | 多 |
| 计算开销 | 大 | 小 |

BPE 通过子词方案取得了平衡：高频词是一个完整 Token，低频词被拆为多个子词 Token。

## 优化方向

1. **Byte-level BPE（GPT-2 方案）**：将字节作为初始 Token，而不是 Unicode 字符——这样不会遇到 OOV 问题（所有 Unicode 字符都能用字节表示）
2. **词表大小选择**：GPT-2: 50,257, GPT-4: ~100k, Llama 2: 32k——不同大小的权衡
3. **合并规则限制**：某些 BPE 实现会限制跨标点符号的合并

## 能力边界

- BPE 的词表是固定的——训练完成后无法动态扩展（除非重新训练）
- 对于极生僻的领域术语，仍然可能被切得很碎（例如医学术语）
- 不同语言的效果差异大——英语效果好，中文效果差（因为中文字符本身就是一个语义单元）

## 与其他方案对比

| 方案 | 词表灵活性 | 序列长度 | 语言公平性 |
|------|-----------|---------|-----------|
| BPE | 中 | 中 | 偏向英语 |
| WordPiece | 中 | 中 | 中等 |
| Unigram | 高 | 中 | 更好 |
| SentencePiece | 高 | 不固定 | 好（多语言原生支持） |

## 工程启示

- 使用 tiktoken 库精确计数 Token——不同模型用不同的 Tokenizer，计数方法不同
- Token 估算公式：英文约 0.75 个 Token/词，中文约 1.5-2 个 Token/字
- 长数字序列（如 CSV 中的数字）会消耗大量 Token——格式化输出可以减少
- Agent 开发中的 Token 预算管理必须基于实际 Tokenizer 计数，而非估算
